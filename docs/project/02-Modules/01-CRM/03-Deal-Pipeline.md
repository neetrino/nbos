# Deal Pipeline (Воронка сделок)

> NBOS Platform — полный цикл продажи от первого разговора до закрытия

## Назначение

Deal Pipeline — воронка для работы с квалифицированными клиентами (SQL). Здесь происходит основная работа продавца: выяснение потребностей, встречи, подготовка коммерческих предложений, договор и получение оплаты. В pipeline попадают только квалифицированные лиды — те, кто прошёл Lead Pipeline и получил статус SQL.

**Ответственный:** Seller (основная работа), Head of Sales (контроль и эскалация)

---

## Стадии воронки сделок

### Стадия 1: Start a Conversation (Начало разговора)

| Параметр | Описание |
|---|---|
| **Что происходит** | Первый полноценный диалог с квалифицированным клиентом. Seller представляет компанию, уточняет запрос |
| **Ответственный** | Seller |
| **Обязательные поля** | Контакт (имя, телефон/email), источник лида, ответственный Seller |
| **Stage Gate (условие перехода)** | Клиент подтвердил интерес, готов обсуждать проект |
| **Действия** | Звонок/сообщение клиенту, первичная презентация услуг |

---

### Стадия 2: Discuss What Is Needed (Обсуждение потребностей)

| Параметр | Описание |
|---|---|
| **Что происходит** | Глубокое выяснение потребностей: что нужно клиенту, какой тип продукта, какие функции, какие сроки и бюджет |
| **Ответственный** | Seller |
| **Обязательные поля** | Тип услуги, описание потребности, примерный бюджет клиента |
| **Stage Gate** | Потребности клиента зафиксированы, понятно что нужно обсудить на встрече |
| **Действия** | Подробный разговор, фиксация требований в карточке сделки, предварительная оценка возможности |

---

### Стадия 3: Meeting (Встреча)

| Параметр | Описание |
|---|---|
| **Что происходит** | Запланирована и проведена встреча (онлайн или офлайн). Демонстрация портфолио, обсуждение деталей, формирование доверия |
| **Ответственный** | Seller (при необходимости — с участием PM или CEO) |
| **Обязательные поля** | Дата встречи, результат встречи (заметка), уточнённые требования |
| **Stage Gate** | Встреча проведена, клиент готов получить коммерческое предложение |
| **Действия** | Назначение встречи (→ Calendar), проведение, фиксация результатов |
| **Автоматизация** | При назначении даты встречи — автоматическое событие в календаре Seller |

---

### Стадия 4: Can We Do It? (Можем ли мы это сделать?)

| Параметр | Описание |
|---|---|
| **Что происходит** | Внутренняя оценка: команда определяет техническую возможность, сроки и стоимость. Seller консультируется с PM/разработчиком |
| **Ответственный** | Seller + PM (или Lead Developer для технической оценки) |
| **Обязательные поля** | Тип проекта (WL / Mix / Custom Code), предварительная оценка сроков, предварительная оценка стоимости |
| **Stage Gate** | Внутренняя команда подтвердила возможность реализации. Определены тип, сроки, стоимость |
| **Действия** | Внутренняя консультация, техническая оценка, формирование scope |

---

### Стадия 5: Send Offer (Отправка предложения)

| Параметр | Описание |
|---|---|
| **Что происходит** | Коммерческое предложение (КП) подготовлено и отправлено клиенту. Включает scope, сроки, стоимость, условия оплаты |
| **Ответственный** | Seller |
| **Обязательные поля** | Документ КП (прикреплён или ссылка), сумма сделки, тип оплаты (Classic/Subscription), условия |
| **Stage Gate** | КП отправлено клиенту, зафиксирована дата отправки |
| **Действия** | Генерация КП, отправка клиенту (email/WhatsApp/мессенджер), фиксация в карточке |

> Подробнее о КП — см. [[04-Offers-and-Handoff]]

---

### Стадия 6: Get Answer (Ожидание ответа)

| Параметр | Описание |
|---|---|
| **Что происходит** | Клиент получил КП и принимает решение. Seller ожидает ответ, при необходимости — follow-up |
| **Ответственный** | Seller |
| **Обязательные поля** | Дата предполагаемого ответа |
| **Stage Gate** | Клиент дал ответ: согласен, отказ, или просит изменения |
| **Действия** | Follow-up звонки/сообщения, ответы на вопросы, корректировка КП при необходимости |
| **Автоматизация** | Напоминание Seller о follow-up через 3 дня, 7 дней, 14 дней после отправки КП |

**Возможные исходы:**
- Клиент согласен → переход на «Deposit & Contract»
- Клиент просит изменения → возврат на «Send Offer» с обновлённым КП
- Клиент отказывается → «Failed» с фиксацией причины

---

### Стадия 7: Deposit & Contract (Предоплата и договор)

| Параметр | Описание |
|---|---|
| **Что происходит** | Клиент согласился. Подписывается договор, Seller создаёт Invoice на предоплату, клиент оплачивает |
| **Ответственный** | Seller (создаёт Invoice) + Finance Director (контролирует оплату) |
| **Обязательные поля** | Сумма сделки (финальная), тип оплаты, план оплат, Invoice создан, договор подписан |
| **Stage Gate** | Первый Invoice оплачен (подтверждено Finance Director) |
| **Действия** | Создание Invoice из карточки сделки, подписание договора, контроль оплаты |

**Процесс создания Invoice:**
1. Seller нажимает «Создать счёт» в карточке сделки
2. Invoice попадает на доску счетов → стадия «Новый»
3. Finance Director выставляет счёт в госсистеме
4. Клиенту отправляется уведомление об оплате
5. После оплаты — Finance Director отмечает Invoice как «Оплачен»
6. Сделка автоматически переходит на следующую стадию

**Критический момент:** именно здесь Deal становится Order. При оплате первого Invoice:
- Создаётся **Order** (финансовая сущность)
- Создаётся или обновляется **Project** в Projects Hub
- Запускается **Handoff** процесс (Seller → PM)

---

### Стадия 8: Creating (Разработка)

| Параметр | Описание |
|---|---|
| **Что происходит** | Проект передан в разработку. Seller отслеживает прогресс на высоком уровне для информирования клиента |
| **Ответственный** | PM (разработка), Seller (клиентские вопросы по продажной части) |
| **Обязательные поля** | Project создан в Projects Hub, Handoff завершён |
| **Stage Gate** | Проект разработан и сдан клиенту (PM подтверждает) |
| **Действия** | Мониторинг статуса проекта, ответы на вопросы клиента о сроках |

> На этой стадии основная работа идёт в Projects Hub. Сделка в CRM является зеркалом статуса.

---

### Стадия 9: Get Final Pay (Получение финальной оплаты)

| Параметр | Описание |
|---|---|
| **Что происходит** | Проект сдан, ожидается финальная оплата. Актуально для Classic (50/50 или 30/30/40) — последний платёж |
| **Ответственный** | Finance Director (контроль оплаты), Seller (коммуникация с клиентом) |
| **Обязательные поля** | Финальный Invoice создан и отправлен |
| **Stage Gate** | Все Invoice по сделке оплачены |
| **Действия** | Создание финального Invoice, контроль оплаты |

**Для Subscription:** эта стадия может быть пропущена, так как оплата идёт ежемесячно и проект уже на Maintenance.

---

### Стадия 10: Maintenance Offer (Предложение обслуживания)

| Параметр | Описание |
|---|---|
| **Что происходит** | Все оплаты получены, проект сдан. Seller предлагает клиенту подписку на техническое обслуживание (Maintenance) |
| **Ответственный** | Seller |
| **Обязательные поля** | Предложение Maintenance отправлено (да/нет), результат |
| **Stage Gate** | Клиент принял или отклонил предложение Maintenance |
| **Действия** | Подготовка предложения Maintenance, переговоры о сумме и условиях |

**При согласии клиента:**
- Проект переводится в статус Maintenance в Projects Hub
- Создаётся запись в Subscriptions (ежемесячная подписка)
- Активируется автоматический биллинг

---

### Терминальные стадии

#### Failed (Проигранная сделка)

| Параметр | Описание |
|---|---|
| **Что происходит** | Сделка не состоялась на любой стадии pipeline |
| **Обязательные поля** | Причина проигрыша (выбор из списка + комментарий) |
| **Текущий объём** | ~58 записей (из Bitrix) |

**Причины проигрыша (типовые):**
- Клиент выбрал конкурента
- Не устроила цена
- Клиент передумал / отложил
- Нет бюджета
- Не удалось договориться по условиям
- Клиент не выходит на связь

#### Deal Won (Сделка выиграна)

| Параметр | Описание |
|---|---|
| **Что происходит** | Сделка полностью завершена: все оплаты получены, Maintenance предложен |
| **Текущий объём** | ~88 записей (из Bitrix) |
| **Автоматизация** | Расчёт бонуса Seller, обновление статистики, архивация сделки |

## Типы сделок и визуальное отличие

### New Client (Новый клиент)
- Стандартная карточка без дополнительных меток
- Полный цикл всех 10 стадий

### Extension Deal (Доработка)
- Визуально выделена: **цветная метка** (например, голубой тег «Extension»)
- **Привязана к существующему проекту** — поле «Linked Project» обязательно
- Может пропускать ранние стадии (клиент уже знаком)
- При Deal Won: не создаёт новый проект, а добавляет Work Package к существующему

### Upsell (Допродажа)
- Метка «Upsell» на карточке
- Может создать новый проект (если другой продукт) или расширить существующий

## Типы оплаты и их влияние на pipeline

| Тип оплаты | Стадия 7 (Deposit) | Стадия 9 (Final Pay) | Стадия 10 (Maintenance) | Бонус команде |
|---|---|---|---|---|
| **Classic 50/50** | Invoice на 50% | Invoice на 50% | Предложение Maintenance | После получения 100% |
| **Classic 30/30/40** | Invoice на 30% | Invoice на 30% (середина) + 40% (сдача) | Предложение Maintenance | После получения 100% |
| **Subscription** | Invoice на 1-й месяц | N/A (оплата ежемесячная) | Автоматический переход | После сдачи (~4 мес) |

## Карточка сделки (Deal Card)

### Основные поля

| Поле | Тип | Обязательное | Когда заполняется |
|---|---|---|---|
| Название сделки | Текст | Да | Стадия 1 |
| Контакт | Связь | Да | Стадия 1 (из лида) |
| Компания | Связь | Нет → Да | До стадии 7 |
| Тип сделки | Выбор: New / Extension / Upsell | Да | Стадия 1 |
| Linked Project (для Extension) | Связь с Project | Условно | Если тип = Extension |
| Источник лида | Авто из Lead | Да | Авто |
| Ответственный Seller | Сотрудник | Да | Стадия 1 |
| Тип услуги | Выбор из списка | Да | Стадия 2 |
| Описание потребности | Текст | Да | Стадия 2 |
| Дата встречи | Дата | Нет | Стадия 3 |
| Тип проекта | WL / Mix / Custom Code | Да | Стадия 4 |
| Оценка сроков | Число (дни/месяцы) | Да | Стадия 4 |
| Сумма сделки | Число | Да | Стадия 5 |
| Тип оплаты | Classic 50/50 / 30/30/40 / Subscription | Да | Стадия 5 |
| Документ КП | Файл/ссылка | Да | Стадия 5 |
| Налогооблагаемость | Tax / Free | Да | До стадии 7 |
| Дедлайн проекта | Дата | Да | Стадия 7 |
| Причина проигрыша | Текст + выбор | Условно | Если Failed |

## Автоматизация воронки сделок

| Триггер | Действие |
|---|---|
| Сделка создана из Lead (SQL) | Уведомление Seller, перенос данных контакта |
| Встреча назначена (стадия 3) | Событие в Calendar |
| КП отправлено (стадия 5 → 6) | Напоминание о follow-up через 3, 7, 14 дней |
| Invoice создан (стадия 7) | Invoice → доска счетов Finance Director |
| Invoice оплачен | Создание Order + Project + Handoff |
| Все оплаты получены | Расчёт бонуса Seller → Bonus Ledger |
| Deal Won | Архивация, обновление аналитики |
| Extension Deal Won | Добавление Work Package к существующему Project |
| Сделка в одной стадии > 14 дней | Уведомление Seller + Head of Sales |

## Метрики воронки сделок

| Метрика | Описание |
|---|---|
| Конверсия по стадиям | % перехода с каждой стадии на следующую |
| Win Rate | Deal Won / (Deal Won + Failed) |
| Средний цикл сделки | Среднее время от «Start a Conversation» до «Deal Won» |
| Средний чек | Средняя сумма закрытой сделки |
| Pipeline Value | Сумма всех сделок в активном pipeline |
| Forecast | Прогноз закрытия на основе стадий и вероятностей |
| Seller Performance | Сравнение показателей продавцов |
