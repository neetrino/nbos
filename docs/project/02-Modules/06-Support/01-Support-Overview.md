# Модуль Support — Обзор

> NBOS Platform — обработка клиентских обращений, контроль качества и поддержка проектов на maintenance

## Назначение

Модуль **Support** отвечает за обработку клиентских обращений по уже сданным или активным проектам. Когда у клиента возникает проблема, вопрос или запрос на изменение — он обращается к Neetrino, и запрос попадает в модуль Support.

**Ключевой принцип:** поддержка — это не просто «починить баг». Это классификация обращения, определение его стоимости (бесплатно/платно), выполнение работы и документирование результата.

---

## ITIL-lite модель: 4 категории обращений

NBOS использует упрощённую модель ITIL с четырьмя категориями тикетов:

### 1. Incident (Инцидент)

| Параметр | Значение |
|----------|----------|
| **Что это** | Что-то сломалось по вине Neetrino или непредвиденной причине |
| **Примеры** | Сайт упал, форма не работает, ошибка 500, данные потерялись, вёрстка поехала после обновления |
| **Оплата** | **Бесплатно** — всегда. Это наша ответственность |
| **Лимиты** | Без ограничений. Сколько бы инцидентов ни было — чиним бесплатно |
| **SLA** | Зависит от приоритета: P1 — немедленно, P2 — в течение дня, P3 — в течение недели |

### 2. Service Request (Сервисный запрос)

| Параметр | Значение |
|----------|----------|
| **Что это** | Стандартная просьба: настроить, обновить, помочь с чем-то |
| **Примеры** | Обновить контент на сайте, добавить email-аккаунт, настроить переадресацию, обновить SSL |
| **Оплата** | **Обычно бесплатно** в рамках maintenance/подписки |
| **Лимиты** | Разумные. Если клиент просит обновлять контент каждый день — это уже выходит за рамки |
| **Граница** | Если запрос требует разработки (> 1–2 часа работы) → переклассификация в Change Request |

### 3. Change Request (Запрос на изменение)

| Параметр | Значение |
|----------|----------|
| **Что это** | Новая функция, улучшение, доработка — то, чего не было в изначальном ТЗ |
| **Примеры** | Добавить личный кабинет, интегрировать новую платёжную систему, переделать дизайн раздела |
| **Оплата** | **Платно** — это новая работа, которая проходит через CRM как Extension Deal |
| **Процесс** | PM создаёт Extension Deal в CRM → стандартный цикл продажи → после оплаты → разработка |
| **Важно** | Клиенту объясняется, что это выходит за рамки maintenance и требует отдельного бюджета |

### 4. Problem (Проблема)

| Параметр | Значение |
|----------|----------|
| **Что это** | Повторяющаяся корневая причина инцидентов |
| **Примеры** | Сайт падает каждую неделю из-за нехватки памяти, форма периодически не отправляет данные |
| **Оплата** | **Бесплатно** — если причина в нашем коде/архитектуре. **Платно** — если нужна модернизация |
| **Процесс** | Анализ корневой причины → архитектурное решение → может стать Change Request |
| **Автоматизация** | Если один и тот же инцидент повторяется 3+ раз → система автоматически предлагает создать Problem-тикет |

## Визуальная схема классификации

```
Клиент пишет
      │
      ▼
  PM классифицирует
      │
      ├── Сломалось? ──────────────→ Incident (бесплатно)
      │
      ├── Настроить/обновить? ─────→ Service Request (бесплатно)
      │
      ├── Новая функция? ──────────→ Change Request (платно → Extension Deal)
      │
      └── Повторяющаяся проблема? ─→ Problem (анализ корневой причины)
```

## Поток обработки обращений

### Текущий процесс:

```
Клиент пишет (WhatsApp / чат)
      │
      ▼
PM получает сообщение
      │
      ▼
PM создаёт тикет в Support
      │
      ▼
PM классифицирует (категория + приоритет + billable)
      │
      ├── Incident / Service Request → PM назначает разработчика
      │     │
      │     ▼
      │   Разработчик исправляет / выполняет
      │     │
      │     ▼
      │   PM подтверждает и сообщает клиенту
      │
      └── Change Request → PM создаёт Extension Deal в CRM
            │
            ▼
          Стандартный цикл продажи (КП, договор, оплата)
            │
            ▼
          После оплаты → создаётся Extension → разработка
```

### Текущий триажёр: PM

На данный момент **PM** выполняет роль тriage-специалиста — принимает, классифицирует и распределяет обращения. В будущем планируется выделенная роль **Support Specialist**.

## Maintenance Policy (Политика обслуживания)

### Основные правила:

| Правило | Описание |
|---------|----------|
| **Баги — без лимита** | Исправление инцидентов всегда бесплатно, без ограничения количества |
| **Service Requests — разумный объём** | Стандартные настройки в рамках подписки. Нет жёсткого числового лимита |
| **Change Requests — всегда платно** | Новый функционал = новый бюджет |
| **Злоупотребление** | Если клиент пишет 30+ раз в месяц с нерелевантными запросами — PM сообщает CEO |

### Обработка злоупотреблений:

1. PM фиксирует паттерн: частота обращений, их характер, время затрат
2. PM отчитывается CEO с данными
3. CEO принимает решение:
   - Провести разговор с клиентом и установить границы
   - Изменить условия подписки (повысить стоимость)
   - В крайнем случае — прекратить сотрудничество

## SLA-модель (Service Level Agreement)

> На текущем этапе SLA — это **целевые показатели**, а не жёсткие контрактные обязательства. По мере роста компании SLA станет формальной частью договора.

### Приоритеты и SLA:

| Приоритет | Описание | Время реакции | Время решения |
|-----------|----------|---------------|---------------|
| **P1 — Critical** | Полная неработоспособность: сайт/приложение лежит, данные теряются, бизнес-процесс клиента парализован | 30–60 мин | 4–8 часов |
| **P2 — High** | Значительная проблема: ключевая функция не работает, но workaround есть | 2–4 часа | 1–2 дня |
| **P3 — Normal** | Некритичная проблема: мелкий баг, косметический дефект, не срочная настройка | 1 рабочий день | 3–7 рабочих дней |

### Определение приоритета:

| Фактор | P1 | P2 | P3 |
|--------|:---:|:---:|:---:|
| Бизнес клиента остановлен | ✅ | ❌ | ❌ |
| Ключевая функция не работает | ✅ | ✅ | ❌ |
| Есть workaround | ❌ | ✅ | ✅ |
| Косметический дефект | ❌ | ❌ | ✅ |
| Влияет на выручку клиента | ✅ | Возможно | ❌ |

## Ключевые поля тикета

| Поле | Тип | Описание |
|------|-----|----------|
| **Название** | Текст | Краткое описание проблемы |
| **Описание** | Rich Text | Детальное описание, скриншоты, шаги воспроизведения |
| **Категория** | Enum | Incident / Service Request / Change Request / Problem |
| **Приоритет** | Enum | P1 Critical / P2 High / P3 Normal |
| **Проект** | Project | К какому проекту относится обращение |
| **Product** | Product | К какому конкретному продукту (если применимо) |
| **Контакт** | Contact | Кто обратился (клиент) |
| **Статус** | Enum | New → Triaged → Assigned → In Progress → Resolved → Closed |
| **Исполнитель** | User | Кто выполняет работу по тикету |
| **Billable** | Boolean | Платно (Yes) / Бесплатно (No) |
| **Источник** | Enum | Subscription (подписка), Warranty (гарантия), Paid Extension (платная доработка) |
| **SLA дедлайны** | Dates | Дедлайн реакции, дедлайн решения — рассчитываются автоматически по приоритету |
| **Резолюция** | Text | Описание того, что было сделано для решения |
| **Вложения** | File[] | Скриншоты, логи, файлы |
| **Чат тикета** | Chat Thread | Обсуждение по тикету (как Task Chat) |

## Интеграция с другими модулями

### Support → CRM

Когда тикет классифицирован как **Change Request** и отмечен **Billable: Yes**:

```
Support Ticket (Change Request)
      │
      ▼
PM нажимает «Создать Extension Deal»
      │
      ▼
В CRM создаётся Extension Deal
  ├── Данные из тикета: описание, проект, контакт
  ├── Тип: Extension
  └── Привязка к существующему проекту
      │
      ▼
Стандартный pipeline CRM: КП → Договор → Оплата → Extension → Разработка
```

### Support → Projects Hub

- Тикеты отображаются в карточке проекта (вкладка **Support**)
- PM видит историю всех обращений по проекту
- Статистика: количество тикетов/месяц, среднее время решения

### Support → Tasks

- Из тикета создаётся задача для разработчика
- Задача привязана к тикету — при закрытии задачи тикет автоматически переходит в Resolved

### Support → Messenger

- Клиент пишет в WhatsApp → PM видит сообщение в мессенджере NBOS
- PM создаёт тикет прямо из сообщения (текст → описание тикета)

## Глобальные представления Support

### 1. Support Board (Доска поддержки)

Kanban-доска с колонками по статусам тикетов:

```
New → Triaged → Assigned → In Progress → Resolved → Closed
```

Фильтры: по проекту, приоритету, категории, исполнителю, дате.

### 2. Support в карточке проекта

Таблица тикетов конкретного проекта с возможностью фильтрации и сортировки.

### 3. Support Dashboard

Виджеты:
- Открытые тикеты по приоритету
- Среднее время решения
- Тикеты по категориям (pie chart)
- SLA compliance (% тикетов, решённых в срок)
- Топ проектов по количеству обращений
