# Интеграция с банком

## Общее описание

Банковская интеграция предназначена для **автоматизации верификации платежей**. В текущем процессе Финансовый директор ежедневно вручную проверяет банковский счёт и отмечает оплаченные счета в системе. В будущем этот процесс будет автоматизирован через банковское API или импорт выписок.

---

## Текущий процесс (MVP)

### Ежедневная работа Финансового директора

```
1. Утро: Финансовый директор открывает онлайн-банк
2. Проверяет входящие платежи за предыдущий день / текущий день
3. Для каждого платежа находит соответствующий Invoice в NBOS
4. Отмечает Invoice как "Paid" (нажатием кнопки)
5. Система запускает downstream-события:
   - Бонус продавцу
   - Оплата партнёру (если применимо)
   - WhatsApp-уведомление клиенту
   - Обновление статуса Order
6. Неопознанные платежи — помечает для разбора
```

### Ограничения текущего подхода

| Проблема | Описание |
|----------|----------|
| Ручная работа | ~30–60 минут ежедневно на проверку и сверку |
| Человеческий фактор | Возможны ошибки при сопоставлении платежей |
| Задержки | Платёж может быть обработан с задержкой в 1 день |
| Масштабируемость | При росте числа подписок ручной процесс станет узким местом |

### Что необходимо в MVP

Даже без банковского API система должна быть **подготовлена к будущей автоматизации**:

| Подготовка | Описание |
|------------|----------|
| Уникальный референс | Каждый Invoice получает уникальный номер / reference для идентификации платежа |
| Структурированные данные | Сумма, дата, плательщик — хранятся в стандартизированном формате |
| API-ready архитектура | Модуль оплаты спроектирован так, чтобы источник подтверждения (человек / API) был взаимозаменяем |
| Журнал оплат | Все подтверждения оплат логируются с указанием источника (manual / bank_api / import) |

## Будущий процесс (Фаза 3+)

### Автоматическое сопоставление платежей

```
Шаг 1: Invoice создаётся в системе
  ─────────────────────────────────
  Фиксируется: сумма, клиент, реквизиты компании, reference number
  
Шаг 2: Клиент оплачивает через банковский перевод
  ─────────────────────────────────
  Платёж поступает на счёт Neetrino
  
Шаг 3: Система проверяет банковское API
  ─────────────────────────────────
  Периодический запрос (каждые N часов) или webhook от банка
  Получение списка входящих транзакций
  
Шаг 4: Автоматическое сопоставление
  ─────────────────────────────────
  Для каждой входящей транзакции система ищет совпадение:
  → По reference number (если клиент указал в назначении платежа)
  → По сумме + отправителю
  → По сумме + дате (примерное совпадение)
  
Шаг 5a: Точное совпадение найдено
  ─────────────────────────────────
  → Invoice автоматически отмечается как Paid
  → Запускаются все downstream-события (бонус, партнёр, уведомление)
  → Финансовый директор получает уведомление: «Invoice #{X} auto-matched»
  
Шаг 5b: Неоднозначное / нет совпадения
  ─────────────────────────────────
  → Транзакция помещается в очередь «Требует проверки»
  → Финансовый директор вручную сопоставляет или создаёт запись
```

### Алгоритм сопоставления (Matching)

Система пытается сопоставить входящий платёж с открытым Invoice по следующей логике:

```
1. EXACT MATCH (точное совпадение):
   Сумма совпадает ±0
   AND Отправитель совпадает с Company / Contact
   AND Reference содержит номер Invoice
   → Confidence: 100% → Auto-match

2. HIGH CONFIDENCE (высокая уверенность):
   Сумма совпадает ±0
   AND Отправитель совпадает с Company / Contact
   (Reference не указан или не совпадает)
   → Confidence: 85% → Auto-match с уведомлением ФинДиректору

3. MEDIUM CONFIDENCE (средняя уверенность):
   Сумма совпадает ±0
   AND Отправитель НЕ совпадает (новый счёт / другое юрлицо)
   → Confidence: 50% → Предложение для ручной проверки

4. LOW CONFIDENCE (низкая уверенность):
   Сумма НЕ совпадает ни с одним открытым Invoice
   OR Отправитель неизвестен
   → Confidence: <50% → Ручная обработка обязательна
```

### Визуализация процесса

```
Банковская транзакция
        ↓
   [Matching Engine]
        ↓
  ┌─────┼─────────────┐
  ↓     ↓             ↓
Exact  High         Medium/Low
Match  Confidence   Confidence
  ↓     ↓             ↓
Auto   Auto +       Очередь
Paid   Notify       «Требует проверки»
  ↓     ↓             ↓
Events Events      ФинДиректор
fired  fired       решает вручную
```

## Способы интеграции с банком

### Вариант 1: Банковское API (предпочтительно)

| Аспект | Описание |
|--------|----------|
| **Принцип** | Прямое подключение к API банка для получения транзакций |
| **Частота** | Каждые 1–4 часа или по webhook (если банк поддерживает) |
| **Данные** | Список входящих транзакций: сумма, дата, отправитель, назначение |
| **Плюсы** | Полная автоматизация, минимальная задержка, высокая надёжность |
| **Минусы** | Зависимость от банка, сложность подключения, возможные ограничения API |

**Банки в Армении:**

| Банк | API-возможности | Статус |
|------|-----------------|--------|
| Ameriabank | Open Banking API — возможно | Требует исследования |
| Ardshinbank | Ограниченное API | Требует исследования |
| ACBA-Credit Agricole | API для юрлиц — возможно | Требует исследования |
| IDBank | Digital banking API | Требует исследования |

Конкретный банк и доступность API будут определены на этапе реализации. Многие армянские банки развивают цифровые сервисы, и Open Banking API становится доступнее.

### Вариант 2: Импорт банковской выписки (альтернатива)

| Аспект | Описание |
|--------|----------|
| **Принцип** | Финансовый директор скачивает выписку из онлайн-банка и загружает в NBOS |
| **Форматы** | CSV, Excel (XLS/XLSX), PDF (с OCR) |
| **Частота** | Ежедневно или по необходимости |
| **Данные** | Те же: сумма, дата, отправитель, назначение |
| **Плюсы** | Не зависит от банковского API, работает с любым банком |
| **Минусы** | Полуавтоматический процесс (нужен ручной шаг загрузки) |

**Процесс импорта:**

```
1. Финансовый директор скачивает выписку из онлайн-банка (CSV/Excel)
2. Загружает файл в NBOS → раздел «Финансы» → «Импорт выписки»
3. Система парсит файл и извлекает транзакции
4. Для каждой транзакции запускается алгоритм сопоставления (Matching)
5. Точные совпадения → auto-match
6. Неоднозначные → предложения для ручной проверки
7. Финансовый директор подтверждает / корректирует
```

**Маппинг полей выписки:**

| Поле в выписке | Поле в NBOS | Описание |
|----------------|-------------|----------|
| Дата операции | transaction_date | Когда поступил платёж |
| Сумма | amount | Сумма платежа |
| Отправитель | sender_name | Наименование отправителя |
| Назначение платежа | reference | Текст назначения (для поиска номера Invoice) |
| Номер документа | bank_doc_number | Внутренний номер банка |

### Вариант 3: Гибридный подход (рекомендуемый)

```
Фаза 1 (MVP):     Полностью ручной процесс
Фаза 2:           Импорт выписки (CSV/Excel) + auto-matching
Фаза 3:           Банковское API (если доступно) + auto-matching
```

## Журнал платежей

Все подтверждения оплат фиксируются в журнале:

| Поле | Описание |
|------|----------|
| **Invoice** | Номер счёта |
| **Сумма** | Сумма платежа |
| **Дата платежа** | Когда поступил платёж |
| **Источник подтверждения** | manual / bank_import / bank_api |
| **Confidence** | Уровень уверенности при auto-match (100% / 85% / manual) |
| **Подтвердил** | Кто подтвердил оплату (сотрудник или «system» для auto-match) |
| **Банковский reference** | Номер транзакции из банка |
| **Комментарий** | Дополнительные заметки |

## Обработка нестандартных ситуаций

| Ситуация | Обработка |
|----------|-----------|
| Частичная оплата | Финансовый директор отмечает partial payment, указывает сумму. Invoice остаётся в статусе Pending. |
| Переплата | Финансовый директор фиксирует переплату. Излишек учитывается при следующем Invoice. |
| Оплата за несколько Invoice одним платежом | Финансовый директор вручную распределяет сумму по нескольким Invoice. |
| Возврат | Создаётся запись Refund, привязанная к Invoice. Invoice.status → Refunded. |
| Ошибочный платёж (чужой) | Помечается как «Не наш», не привязывается к Invoice. |

## Безопасность

| Аспект | Меры |
|--------|------|
| **Доступ к банковскому API** | Только серверная часть NBOS, ключи хранятся в secrets |
| **Доступ к журналу платежей** | Только Финансовый директор и CEO |
| **Импорт выписки** | Только Финансовый директор может загружать файлы |
| **Auto-match** | Все auto-match решения логируются и могут быть отменены |
| **Аудит** | Полный аудит-лог всех действий с платежами |

---

## Метрики

| Метрика | Описание |
|---------|----------|
| Время от платежа до подтверждения | Среднее время обработки платежа |
| % auto-matched | Доля автоматически сопоставленных платежей |
| % ручной обработки | Доля платежей, требующих ручного вмешательства |
| Ошибки сопоставления | Количество неправильных auto-match (отменённых) |
