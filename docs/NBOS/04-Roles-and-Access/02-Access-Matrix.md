# Матрица доступов (RBAC)

> NBOS Platform — Role-Based Access Control

## Обзор

Доступ в NBOS построен на модели RBAC (Role-Based Access Control). Каждый модуль платформы имеет настройку уровня доступа для каждой роли. Доступ определяется комбинацией роли сотрудника и его привязки к конкретным проектам/отделам.

### Уровни доступа

| Уровень | Обозначение | Описание |
|---|---|---|
| **Full** | ✅ | Полный доступ: просмотр, создание, редактирование, удаление, экспорт |
| **Read** | 👁 | Только чтение: просмотр данных без возможности изменения |
| **Limited** | 🔶 | Ограниченный: доступ только к связанным/назначенным записям |
| **None** | ❌ | Нет доступа: модуль скрыт из навигации |

## Модель доступа к Credentials (паролям)

Пароли в NBOS классифицируются по уровням доступа:

### Уровни доступа к паролям

| Уровень | Описание | Кто видит | Примеры |
|---|---|---|---|
| **Secret** | Секретные пароли компании | Только CEO + явно указанные сотрудники | Корневые пароли серверов, банковские данные, ключи API платёжных систем |
| **Project Team** | Пароли привязанные к проекту | Участники проектной команды | Хостинг проекта, CMS admin, база данных проекта, env-ключи |
| **Department** | Пароли отдела | Все сотрудники отдела | Общие сервисы отдела, тестовые аккаунты |
| **All** | Общедоступные пароли | Все сотрудники компании | Wi-Fi офиса, общие рабочие инструменты |

### Персональный раздел «Мои пароли»

Каждый сотрудник имеет персональный раздел для хранения своих паролей:
- **Создание** — неограниченное количество записей
- **Просмотр** — только свои записи
- **Редактирование** — полное управление своими паролями
- **Архивирование** — перемещение в архив для старых паролей
- **Видимость** — никто другой не видит персональные пароли (кроме CEO при бэкапе)

## Ролевая модель в задачах

Внутри каждой задачи существуют 4 роли участника:

| Роль в задаче | Описание | Права |
|---|---|---|
| **Creator (Постановщик)** | Создал задачу | Полный контроль: редактирование, удаление, смена статуса, назначение исполнителей, закрытие задачи |
| **Assignee (Исполнитель)** | Отвечает за выполнение | Смена статуса (To Do → In Progress → Done), добавление комментариев, прикрепление файлов, запрос уточнений |
| **Co-Assignee (Соисполнитель)** | Помогает с выполнением | Ограниченные действия: добавление комментариев, прикрепление файлов, отметка подзадач. Не может менять основной статус |
| **Observer (Наблюдатель)** | Следит за прогрессом | Только чтение: просмотр задачи, комментариев, статуса. Получает уведомления об изменениях |

### Кто может назначать роли в задачах

| Действие | CEO | PM | Dev | Seller | Finance |
|---|---|---|---|---|---|
| Создать задачу | ✅ | ✅ | ✅ | ✅ | ❌ |
| Назначить исполнителя | ✅ | ✅ | 🔶 | 🔶 | ❌ |
| Добавить наблюдателя | ✅ | ✅ | ✅ | ✅ | ❌ |
| Закрыть задачу | ✅ | ✅ | 🔶 | 🔶 | ❌ |

> 🔶 — может назначать/закрывать только в рамках задач, где является Постановщиком

## Контекстные ограничения

### Привязка к проекту

Для ролей Dev, Jr Dev, Designer, QA доступ к данным ограничен проектами, в которых они состоят как участники команды. Назначение в команду проекта выполняется PM или Head of Delivery.

### Привязка к отделу

Для ролей Marketing, Head of Sales дашборды отдела показывают данные только своего подразделения.

### Временный доступ

CEO или PM могут предоставить временный доступ к конкретному паролю или модулю с указанием срока действия. После истечения срока доступ автоматически отзывается.

## Правила безопасности

1. **Принцип минимального доступа** — каждая роль видит только то, что необходимо для работы
2. **Аудит** — все действия с паролями, финансами и критичными данными записываются в Audit Log
3. **Шифрование** — все пароли хранятся в зашифрованном виде (E2E)
4. **Двухфакторная авторизация** — обязательна для ролей CEO, Finance Director, Tech Ops
5. **Бэкап паролей** — доступен только CEO, выполняется через ввод мастер-пароля, скачивается зашифрованный архив
6. **Разделение финансов и разработки** — Finance Director не видит задачи, разработчики не видят финансы
