# Интеграция с WhatsApp

## Общее описание

WhatsApp-интеграция в NBOS предназначена исключительно для **клиентской коммуникации**: уведомления о счетах, напоминания об оплате, подтверждения и информирование по проектам. WhatsApp **не используется** для внутренней командной коммуникации — для этого существует Telegram-интеграция.

Подключение осуществляется через **WhatsApp Business API** или сервис-агрегатор (аналогично тому, как работает Bitrix24 и другие CRM-системы). Агрегатор предоставляет единый API для отправки и получения сообщений, управления шаблонами и отслеживания доставки.

---

## Назначение и границы

| Аспект | Описание |
|--------|----------|
| **Основная цель** | Автоматические уведомления клиентам (счета, оплаты, напоминания) |
| **Вторичная цель** | Проектные группы Neetrino + Клиент для обсуждений по проекту |
| **НЕ используется для** | Внутренней командной коммуникации (для этого — Telegram) |
| **Тип подключения** | WhatsApp Business API через сервис-агрегатор |
| **Направление** | Исходящие сообщения (основное) + входящие (опционально) |

## Функциональные возможности

### 1. Автоматические сообщения клиентам

Система отправляет сообщения клиентам при наступлении определённых событий. Все сообщения строятся на основе предопределённых шаблонов.

| Событие | Описание сообщения | Получатель |
|---------|-------------------|------------|
| Invoice создан | Уведомление о выставленном счёте с суммой и дедлайном | Контактное лицо клиента |
| Оплата подтверждена | Подтверждение получения платежа | Контактное лицо клиента |
| Просрочка (1-я) | Напоминание о просроченном платеже | Контактное лицо клиента |
| Просрочка (2-я) | Повторное напоминание с предупреждением | Контактное лицо клиента |
| Просрочка (3-я) | Эскалация — просьба связаться | Контактное лицо клиента |
| Продление домена | Уведомление о необходимости продления и выставленном счёте | Контактное лицо клиента |
| Тикет решён | Сообщение о закрытии обращения в поддержку | Автор обращения |

### 2. Шаблоны сообщений

Каждый тип уведомления имеет предопределённый шаблон с переменными. Шаблоны проходят модерацию WhatsApp Business API (требование Meta).

| Шаблон | Текст | Переменные |
|--------|-------|------------|
| **invoice_created** | «Здравствуйте! Ваш счёт №{invoice_number} на сумму {amount} выставлен. Оплатите, пожалуйста, до {due_date}.» | invoice_number, amount, due_date |
| **payment_confirmed** | «Оплата по счёту №{invoice_number} на сумму {amount} получена. Спасибо!» | invoice_number, amount |
| **overdue_reminder_1** | «Напоминаем: оплата по счёту №{invoice_number} просрочена. Пожалуйста, произведите оплату.» | invoice_number |
| **overdue_reminder_2** | «Важно: счёт №{invoice_number} не оплачен. Просим связаться с нами для урегулирования.» | invoice_number |
| **overdue_reminder_3** | «Счёт №{invoice_number} не оплачен более {days} дней. Обслуживание может быть приостановлено. Свяжитесь с нами.» | invoice_number, days |
| **domain_renewal** | «Ваш домен {domain_name} требует продления. Счёт №{invoice_number} на сумму {amount} выставлен.» | domain_name, invoice_number, amount |
| **ticket_resolved** | «Ваше обращение №{ticket_number} решено. Если остались вопросы — напишите нам.» | ticket_number |

### 3. Триггеры отправки

Сообщения отправляются автоматически при наступлении системных событий:

```
Событие в системе → Проверка условий → Выбор шаблона → Подстановка данных → Отправка через API
```

| Триггер | Условие | Шаблон | Задержка |
|---------|---------|--------|----------|
| Invoice.stage → «Send Message» | Tax: после подтверждения бухгалтера; Tax-Free: сразу | invoice_created | ~1 час после создания |
| Invoice.paid = true | Финдиректор отметил как оплаченный | payment_confirmed | Немедленно |
| Invoice.stage → «Overdue» | Billing Day прошёл, не оплачен | overdue_reminder_1 | День B+1 |
| Overdue + 2 дня | Всё ещё не оплачен | overdue_reminder_2 | День B+3 |
| Overdue + 5 дней | Всё ещё не оплачен | overdue_reminder_3 | День B+6 |
| Domain.expiry — 2 месяца | Домен истекает через 2 месяца | domain_renewal | Автоматически |
| Ticket.status → Resolved | Тикет закрыт | ticket_resolved | Немедленно |

### 4. Клиентские WhatsApp-группы

Для каждого активного проекта может быть создана WhatsApp-группа:

| Параметр | Описание |
|----------|----------|
| **Участники** | PM от Neetrino + контактное лицо клиента (+ опционально CEO, Seller) |
| **Привязка** | Группа привязана к сущности Project в NBOS |
| **Назначение** | Обсуждения по проекту, обмен скриншотами, согласования |
| **Создание** | Вручную при переходе проекта в стадию Development (пункт Kickoff Checklist) |
| **Архивация** | При завершении проекта группа архивируется (не удаляется) |

### 5. Входящие сообщения (опционально)

В будущем возможна обработка входящих сообщений от клиентов:

```
Клиент пишет в WhatsApp → Агрегатор принимает → Webhook в NBOS
  → Если распознан клиент → привязка к проекту → создание тикета поддержки
  → Если не распознан → попадает в очередь для ручной обработки
```

Эта функция не является приоритетом для MVP и планируется на поздних фазах.

## Сценарии автоматизации

### Сценарий 1: Подписочный счёт

```
1. Billing Day приближается → система создаёт Invoice
2. Invoice.stage → "Create in Gov System" (через ~1 час)
   → Уведомление бухгалтеру в системе
3. Бухгалтер подтверждает создание в госсистеме
4. Invoice.stage → "Send Message"
   → WhatsApp клиенту: «Ваш счёт №X на сумму Y выставлен. Оплатите до Z.»
5. Ожидание оплаты...
6. Если оплачено → WhatsApp: «Оплата получена. Спасибо!»
7. Если не оплачено к Billing Day:
   → День B+1: «Напоминаем: оплата просрочена.»
   → День B+3: «Важно: счёт не оплачен. Просим связаться.»
   → День B+6: «Обслуживание может быть приостановлено.»
```

### Сценарий 2: Продление домена

```
1. Система обнаруживает: домен example.com истекает через 2 месяца
2. Auto → создаётся Invoice на продление
3. WhatsApp клиенту: «Ваш домен example.com требует продления. Счёт выставлен.»
4. После оплаты → создаётся задача "Продлить домен example.com"
5. Домен продлён → WhatsApp: «Домен example.com успешно продлён до [дата].»
```

### Сценарий 3: Тикет поддержки

```
1. Тикет поддержки решён (status = Resolved)
2. WhatsApp клиенту: «Ваше обращение №X решено. Если остались вопросы — напишите.»
```

## Техническая архитектура

### Подключение через агрегатор

```
NBOS Platform → [API Агрегатора] → WhatsApp Business API → Клиент
                                  ← Webhook (входящие)
```

| Компонент | Описание |
|-----------|----------|
| **Агрегатор** | Сервис-посредник, предоставляющий REST API для отправки/получения сообщений |
| **API Ключ** | Аутентификация NBOS в агрегаторе |
| **Webhook URL** | Эндпоинт NBOS для получения входящих сообщений и статусов доставки |
| **Шаблоны** | Регистрируются в WhatsApp Business API и управляются через агрегатор |

### Статус доставки

Агрегатор возвращает статус каждого отправленного сообщения:

| Статус | Описание |
|--------|----------|
| **Sent** | Сообщение отправлено в WhatsApp |
| **Delivered** | Сообщение доставлено на устройство клиента |
| **Read** | Клиент прочитал сообщение |
| **Failed** | Ошибка отправки (неверный номер, блокировка и т.д.) |

Статус доставки сохраняется в системе и доступен в карточке Invoice / тикета.

## Ограничения и правила

### Rate Limits (ограничения по частоте)

| Правило | Значение |
|---------|----------|
| Максимум сообщений одному клиенту в день | 5 |
| Минимальный интервал между сообщениями одному клиенту | 2 часа |
| Максимум напоминаний о просрочке | 3 (после 3-го — только ручная эскалация) |
| Время отправки | Только в рабочие часы (9:00–19:00) |

### Opt-Out (отказ от уведомлений)

- Клиент может запросить отключение автоматических WhatsApp-уведомлений
- Запрос фиксируется в карточке Contact → поле `whatsapp_opt_out = true`
- При opt-out уведомления отправляются по email или не отправляются (по согласованию)
- Обязательные уведомления (юридические, договорные) могут всё равно отправляться

### Номер телефона

- WhatsApp-номер берётся из поля `phone` в сущности Contact
- Если у контакта нет WhatsApp — сообщение не отправляется, создаётся задача для PM
- Формат номера: международный (+374XXXXXXXX)

## Журнал сообщений

Все отправленные и полученные WhatsApp-сообщения логируются в системе:

| Поле | Описание |
|------|----------|
| **Дата/время** | Когда сообщение было отправлено |
| **Направление** | Исходящее / Входящее |
| **Получатель** | Контакт + номер телефона |
| **Шаблон** | Какой шаблон использовался |
| **Текст** | Полный текст сообщения |
| **Статус** | Sent / Delivered / Read / Failed |
| **Связь** | Привязка к Invoice / Ticket / Project |

Журнал доступен:
- В карточке Invoice — история сообщений по этому счёту
- В карточке Contact — все сообщения этому клиенту
- В разделе «Коммуникации» — общий журнал всех сообщений

## Выбор агрегатора

При выборе агрегатора для WhatsApp Business API учитываются:

| Критерий | Важность |
|----------|----------|
| Наличие REST API для отправки/получения сообщений | Обязательно |
| Поддержка шаблонов сообщений | Обязательно |
| Webhook для статусов доставки и входящих | Обязательно |
| Работа в регионе (Армения) | Обязательно |
| Стоимость за сообщение | Важно |
| Поддержка русского/армянского языков | Желательно |
| SDK / готовые библиотеки для интеграции | Желательно |

Конкретный агрегатор будет выбран на этапе технической реализации.
