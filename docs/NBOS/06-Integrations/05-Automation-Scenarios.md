# Сценарии автоматизации (полный список)

## Общее описание

Этот документ содержит **полный реестр всех правил автоматизации** в платформе NBOS. Каждый сценарий описан в формате: **Триггер → Условия → Действия → Получатели**. Сценарии организованы по модулям.

---

## Формат описания

```
Триггер:      Событие, запускающее автоматизацию
Условия:      Дополнительные проверки перед выполнением
Действия:     Что система делает автоматически
Получатели:   Кто получает уведомления / задачи
```

## 1. CRM (Лиды и Сделки)

### CRM-01: Автоопределение источника лида

| Параметр | Значение |
|----------|----------|
| **Триггер** | Новый Lead создан через внешний канал |
| **Условия** | Канал поступления определён (Instagram, Facebook, сайт и т.д.) |
| **Действия** | Lead.source = название канала (Instagram → source = «Instagram») |
| **Получатели** | — |

### CRM-02: Обнаружение дублей лидов

| Параметр | Значение |
|----------|----------|
| **Триггер** | Новый Lead создан |
| **Условия** | Существует Lead с таким же phone или email |
| **Действия** | Предупреждение: «Возможный дубль — Lead #{existing_id} имеет тот же {field}». Новый лид не блокируется, но помечается |
| **Получатели** | Seller, ответственный за нового лида |

### CRM-03: SQL → автосоздание сделки

| Параметр | Значение |
|----------|----------|
| **Триггер** | Lead.status → SQL (Quality Lead) |
| **Условия** | Lead имеет привязанный Contact |
| **Действия** | Создаётся Deal в Deal Pipeline (стадия «Start a Conversation»). Данные копируются из Lead: контакт, источник, тип потребности |
| **Получатели** | Seller (назначенный на Lead) |

### CRM-04: Deal Won → автосоздание Order + Project

| Параметр | Значение |
|----------|----------|
| **Триггер** | Deal.status → Won AND первый Invoice оплачен |
| **Условия** | Договор подписан (contract_signed = true) |
| **Действия** | 1. Создаётся Order с данными из Deal. 2. Создаётся Project в Projects Hub. 3. Создаётся Product внутри Project (тип из Deal). 4. Генерируются задачи из шаблона |
| **Получатели** | PM (назначенный), Seller, CEO, Финансовый директор |

### CRM-05: Extension Deal Won → создание расширения

| Параметр | Значение |
|----------|----------|
| **Триггер** | Deal (Extension).status → Won AND оплата получена |
| **Условия** | Deal привязана к существующему Project |
| **Действия** | Создаётся новый Product (Extension) внутри существующего Project. Генерируются задачи из шаблона |
| **Получатели** | PM проекта, Seller |

## 2. Счета (Invoices)

### INV-01: Автосоздание подписочного счёта

| Параметр | Значение |
|----------|----------|
| **Триггер** | Billing Day приближается (за N дней) |
| **Условия** | Subscription.active = true AND текущая дата ≥ subscription.start_date |
| **Действия** | Создаётся Invoice: сумма из Subscription, tax_status из Subscription, привязка к Project, Contact, Company |
| **Получатели** | Финансовый директор (в dashboard появляется новый Invoice) |

### INV-02: Invoice → «Создать в госсистеме»

| Параметр | Значение |
|----------|----------|
| **Триггер** | Invoice создан (через ~1 час) |
| **Условия** | Invoice.tax_status = «Tax» (налогооблагаемый) |
| **Действия** | Invoice.stage → «Create in Gov System». Уведомление бухгалтеру |
| **Получатели** | Бухгалтер |

### INV-03: Tax-Free → пропуск госсистемы

| Параметр | Значение |
|----------|----------|
| **Триггер** | Invoice создан (через ~1 час) |
| **Условия** | Invoice.tax_status = «Tax-Free» |
| **Действия** | Invoice.stage → «Send Message» (пропускает стадию госсистемы) |
| **Получатели** | — |

### INV-04: Invoice → отправка сообщения клиенту

| Параметр | Значение |
|----------|----------|
| **Триггер** | Invoice.stage → «Send Message» (после подтверждения бухгалтера или автоматически для Tax-Free) |
| **Условия** | Контакт клиента имеет WhatsApp-номер AND whatsapp_opt_out ≠ true |
| **Действия** | WhatsApp-сообщение клиенту: шаблон invoice_created |
| **Получатели** | Контактное лицо клиента (WhatsApp) |

### INV-05: Просрочка — 1-е уведомление

| Параметр | Значение |
|----------|----------|
| **Триггер** | Billing Day прошёл AND Invoice.status ≠ Paid |
| **Условия** | Прошёл 1 день после Billing Day |
| **Действия** | Invoice.stage → «Overdue». WhatsApp клиенту: шаблон overdue_reminder_1 |
| **Получатели** | Контактное лицо клиента (WhatsApp), Финансовый директор (система) |

### INV-06: Просрочка — 2-е уведомление

| Параметр | Значение |
|----------|----------|
| **Триггер** | Invoice.stage = «Overdue» AND прошло 2 дня после просрочки |
| **Условия** | Invoice.status ≠ Paid |
| **Действия** | WhatsApp клиенту: шаблон overdue_reminder_2 |
| **Получатели** | Контактное лицо клиента (WhatsApp) |

### INV-07: Просрочка — 3-е уведомление (эскалация)

| Параметр | Значение |
|----------|----------|
| **Триггер** | Invoice.stage = «Overdue» AND прошло 5 дней после просрочки |
| **Условия** | Invoice.status ≠ Paid |
| **Действия** | WhatsApp клиенту: шаблон overdue_reminder_3. Эскалация: уведомление CEO |
| **Получатели** | Контактное лицо клиента (WhatsApp), Финансовый директор, CEO |

## 3. Затраты (Expenses)

### EXP-01: Автосоздание карточек затрат из плана

| Параметр | Значение |
|----------|----------|
| **Триггер** | 1-е число каждого месяца |
| **Условия** | Есть Planned Expenses с датой исполнения в текущем месяце |
| **Действия** | Для каждой запланированной затраты создаётся карточка Expense на доске «This Month». Данные копируются: сумма, получатель, категория, дедлайн |
| **Получатели** | Финансовый директор |

### EXP-02: Оплата партнёру (Classic Order)

| Параметр | Значение |
|----------|----------|
| **Триггер** | Order.payment_status → «Fully Paid» (все Invoice оплачены) |
| **Условия** | Order.type = Classic AND Order.partner существует |
| **Действия** | Создаётся Expense: сумма = 30% от суммы Order, получатель = Partner, дедлайн = +30 дней от текущей даты. Привязка к Order и Project |
| **Получатели** | Финансовый директор |

### EXP-03: Оплата партнёру (Subscription)

| Параметр | Значение |
|----------|----------|
| **Триггер** | Subscription Invoice → Paid |
| **Условия** | Subscription.partner существует |
| **Действия** | Создаётся Expense: сумма = партнёрский % от суммы Invoice, получатель = Partner, дедлайн = конец месяца |
| **Получатели** | Финансовый директор |

### EXP-04: Продление домена — Invoice + задача

| Параметр | Значение |
|----------|----------|
| **Триггер** | Domain.expiry_date — 2 месяца (за 2 месяца до истечения) |
| **Условия** | Domain.auto_renew = true OR Domain.notify = true |
| **Действия** | 1. Создаётся Invoice клиенту на сумму продления. 2. Создаётся Task «Продлить домен {domain_name}» для ответственного сотрудника |
| **Получатели** | Контактное лицо клиента (Invoice), Ops-сотрудник (Task) |

### EXP-05: Pass-through Invoice оплачен → задача на покупку

| Параметр | Значение |
|----------|----------|
| **Триггер** | Invoice (pass-through type) → Paid |
| **Условия** | Invoice связан с доменом / хостингом / лицензией |
| **Действия** | Создаётся Task: «Купить/продлить {service_name} для {project}». Назначается на ответственного сотрудника |
| **Получатели** | Ops-сотрудник (Task), PM проекта (уведомление) |

## 4. Бонусы (Bonuses)

### BON-01: Начисление бонуса продавцу

| Параметр | Значение |
|----------|----------|
| **Триггер** | Invoice → Paid |
| **Условия** | Invoice привязан к Deal AND Deal.seller существует AND условия бонуса выполнены (источник лида, тип оплаты) |
| **Действия** | Создаётся Bonus Entry: сумма = % от суммы Invoice (зависит от источника лида и типа), статус = Holdback, release_date = +30 дней |
| **Получатели** | Seller (уведомление: «Бонус начислен»), Финансовый директор |

### BON-02: Начисление бонуса исполнителю (Dev/PM/Designer)

| Параметр | Значение |
|----------|----------|
| **Триггер** | Work Package Done AND соответствующий Invoice Paid |
| **Условия** | Work Package привязан к Product AND Product привязан к Order с оплаченным Invoice |
| **Действия** | Создаётся Bonus Entry для каждого исполнителя: Dev, PM, Designer. Суммы и % определяются правилами бонусов. Статус = Holdback |
| **Получатели** | Каждый исполнитель (уведомление), Финансовый директор |

### BON-03: Holdback → Active (разблокировка)

| Параметр | Значение |
|----------|----------|
| **Триггер** | 30 дней прошло с момента delivery (сдачи работы клиенту) |
| **Условия** | Bonus.status = Holdback AND delivery_date + 30 дней ≤ текущая дата AND нет открытых жалоб/багов |
| **Действия** | Bonus.status → Active (доступен к выплате) |
| **Получатели** | Сотрудник (уведомление: «Бонус разблокирован»), Финансовый директор |

### BON-04: KPI-gate проверка

| Параметр | Значение |
|----------|----------|
| **Триггер** | Конец месяца (последний рабочий день) |
| **Условия** | Есть Active бонусы для проверки |
| **Действия** | Для каждого сотрудника проверяются KPI-условия. Если KPI выполнены → Bonus.status = Vested. Если KPI не выполнены → Bonus.status = Burned |
| **Получатели** | CEO (отчёт), каждый сотрудник (статус своих бонусов) |

### BON-05: Payroll run (формирование зарплатной ведомости)

| Параметр | Значение |
|----------|----------|
| **Триггер** | 1-е число месяца |
| **Условия** | Есть Vested бонусы за прошлый месяц |
| **Действия** | Active/Vested бонусы переводятся в Payable. Формируется ведомость: зарплата + бонусы для каждого сотрудника. Ведомость требует подтверждения CEO |
| **Получатели** | CEO (подтверждение), Финансовый директор (исполнение), HR |

## 5. Проекты (Projects)

### PRJ-01: Автогенерация задач из шаблона

| Параметр | Значение |
|----------|----------|
| **Триггер** | Новый Product создан внутри Project |
| **Условия** | Product.type определён (Website, Mobile App, CRM и т.д.) AND Project.type определён (White Label, Mix, Custom) |
| **Действия** | Генерируются задачи из шаблона для данной комбинации product_type + project_type. Задачи создаются со стандартными дедлайнами, приоритетами и назначениями |
| **Получатели** | PM проекта (уведомление: «Задачи созданы из шаблона»), исполнители задач |

### PRJ-02: Незаполненные Credentials

| Параметр | Значение |
|----------|----------|
| **Триггер** | Product создан AND required credential fields пусты |
| **Условия** | Поля Credentials, отмеченные как обязательные для данного типа Product, не заполнены |
| **Действия** | Создаётся Task: «Заполнить credentials для {product_name}». Назначается на ответственного (PM или клиент-менеджер) |
| **Получатели** | Ответственный за заполнение |

### PRJ-03: Все задачи Product выполнены

| Параметр | Значение |
|----------|----------|
| **Триггер** | Последняя задача в Product → Done |
| **Условия** | Все задачи Product имеют статус Done |
| **Действия** | Уведомление PM: «Все задачи продукта {product_name} завершены. Запустите чек-лист закрытия.» Product.status → Pending Closure |
| **Получатели** | PM проекта |

### PRJ-04: Изменение стадии Product → обновление статуса Project

| Параметр | Значение |
|----------|----------|
| **Триггер** | Product.stage изменилась |
| **Условия** | — |
| **Действия** | Project.computed_status пересчитывается на основе стадий всех Products. Если все Products на стадии Done/Closed → Project.status → Suggest Archival |
| **Получатели** | — (автоматическое обновление) |

### PRJ-05: Все Products закрыты → предложение архивации

| Параметр | Значение |
|----------|----------|
| **Триггер** | Последний Product в Project → Closed |
| **Условия** | Все Products закрыты AND нет активных Subscriptions |
| **Действия** | Уведомление PM и CEO: «Все продукты проекта {project} закрыты. Предлагается архивация.» |
| **Получатели** | PM, CEO |

## 6. Уведомления (Notifications)

### NOT-01: Маршрутизация уведомлений по каналам

Каждый тип уведомления направляется в определённый канал в зависимости от получателя и важности:

| Канал | Когда используется | Примеры |
|-------|-------------------|---------|
| **Система (in-app)** | Всегда — для всех уведомлений | Все события |
| **Telegram** | Для сотрудников (внутренние) | Задачи, дедлайны, SLA |
| **WhatsApp** | Для клиентов (внешние) | Счета, просрочки, тикеты |
| **Email** | Fallback + важные документы | Договоры, официальные письма |

### NOT-02: Шаблоны по типам событий

| Категория | Событие | Канал клиенту | Канал сотруднику |
|-----------|---------|---------------|------------------|
| Финансы | Invoice создан | WhatsApp | Система |
| Финансы | Invoice оплачен | WhatsApp (подтверждение) | Система + Telegram |
| Финансы | Invoice просрочен | WhatsApp (напоминание) | Система + Telegram |
| Задачи | Задача назначена | — | Система + Telegram |
| Задачи | Задача просрочена | — | Система + Telegram |
| Проекты | Дедлайн приближается | — | Система + Telegram |
| Поддержка | Тикет решён | WhatsApp | Система |
| Поддержка | SLA нарушен | — | Система + Telegram |
| CRM | Сделка выиграна | — | Система + Telegram |
| Бонусы | Бонус начислен | — | Система + Telegram |

## 7. Задачи (Tasks)

### TSK-01: Задача просрочена → эскалация

| Параметр | Значение |
|----------|----------|
| **Триггер** | Task.due_date < текущая дата AND Task.status ≠ Done |
| **Условия** | — |
| **Действия** | Уведомление исполнителю: «Задача {task_name} просрочена». Если просрочка > 2 дней → уведомление PM проекта |
| **Получатели** | Исполнитель, PM (при эскалации) |

### TSK-02: Конец спринта → сводка

| Параметр | Значение |
|----------|----------|
| **Триггер** | Sprint.end_date = текущая дата |
| **Условия** | Есть незавершённые задачи в спринте |
| **Действия** | Сводка для PM: «Спринт {sprint_name} завершился. Незавершённых задач: {count}. Список: ...» |
| **Получатели** | PM проекта (Telegram + система) |

### TSK-03: Изменение статуса → уведомление наблюдателям

| Параметр | Значение |
|----------|----------|
| **Триггер** | Task.status изменился |
| **Условия** | У задачи есть наблюдатели (watchers) |
| **Действия** | Уведомление каждому наблюдателю: «Задача {task_name} → {new_status}» |
| **Получатели** | Все наблюдатели задачи |

## 8. Поддержка (Support)

### SUP-01: Новый тикет → SLA-дедлайны

| Параметр | Значение |
|----------|----------|
| **Триггер** | Новый Ticket создан |
| **Условия** | Ticket.priority установлен |
| **Действия** | Автоматическое назначение SLA-дедлайнов: Critical → 4 часа, High → 8 часов, Medium → 24 часа, Low → 72 часа. Назначение ответственного (по правилам маршрутизации) |
| **Получатели** | Ответственный сотрудник, PM |

### SUP-02: SLA 75% — предупреждение

| Параметр | Значение |
|----------|----------|
| **Триггер** | Время SLA израсходовано на 75% |
| **Условия** | Ticket.status ≠ Resolved |
| **Действия** | Уведомление: «SLA тикета {ticket_title} — осталось 25% времени. Дедлайн: {sla_deadline}» |
| **Получатели** | Ответственный за тикет (Telegram + система) |

### SUP-03: SLA нарушен → эскалация

| Параметр | Значение |
|----------|----------|
| **Триггер** | SLA-дедлайн прошёл |
| **Условия** | Ticket.status ≠ Resolved |
| **Действия** | Ticket помечается как SLA Breached. Эскалация: уведомление PM и CEO |
| **Получатели** | Ответственный, PM, CEO (Telegram + система) |

### SUP-04: Повторяющийся инцидент → Problem ticket

| Параметр | Значение |
|----------|----------|
| **Триггер** | Ticket создан |
| **Условия** | За последние 30 дней ≥ 3 тикетов с тем же типом ошибки / категорией / для того же проекта |
| **Действия** | Предложение: «Обнаружен повторяющийся инцидент ({count} случаев за 30 дней). Создать Problem-тикет?» |
| **Получатели** | PM проекта, CEO |
