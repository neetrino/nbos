# Канон Работы С Модулями

> NBOS Platform - как прорабатывать каждый модуль, чтобы документация и реализация строились правильно

## Назначение

Этот файл фиксирует рабочий подход для всех следующих модулей NBOS.

Его цель:

- не переносить в канон сырые идеи без обсуждения;
- сначала понять бизнес-логику модуля от начала до конца;
- находить слабые места, пропуски и устаревшие решения;
- предлагать не только исходную идею, но и профессиональный вариант реализации;
- только после согласования переносить решение в документацию;
- потом уже на базе канона идти в реализацию.

---

## Основной Принцип

Для каждого модуля мы работаем в таком порядке:

1. сначала обсуждаем в чате;
2. потом согласовываем правильную бизнес-логику;
3. потом фиксируем её в `docs/NBOS`;
4. только после этого переходим к реализации или gap-анализу кода.

Если по какому-то моему пункту нет отдельного возражения от owner, это считается согласием и пункт принимается в рабочий канон.

---

## Как Разбирать Каждый Модуль

### 1. Сначала коротко пересказать модуль

Перед любыми правками нужно коротко и человеческим языком объяснить:

- зачем нужен модуль;
- где он начинается и где заканчивается;
- с какими сущностями он работает;
- какие соседние модули с ним связаны.

Цель: убедиться, что мы одинаково понимаем сам модуль.

### 2. Разобрать реальный бизнес-процесс от начала до конца

Нужно пройти модуль как рабочий процесс, а не как набор экранов.

Для каждого процесса смотреть:

- вход в процесс;
- основные этапы;
- обязательные поля и условия;
- кто принимает решение;
- где есть ручное действие, где автоматизация;
- чем процесс заканчивается;
- что передаётся в следующий модуль.

### 3. Сначала понять идею owner, потом улучшать её

Нужно не просто переписать исходную идею в docs.

Нужно:

- понять, что именно owner хочет получить;
- уловить бизнес-смысл, даже если описание неполное;
- найти, чего не хватает;
- найти противоречия и риски;
- предложить более профессиональный и устойчивый вариант, если он лучше для бизнеса.

### 4. Предлагать профессиональный вариант, а не только исходный

Для каждого важного решения нужно смотреть:

- как это обычно делают профессионально;
- где текущая идея может сломаться;
- где могут возникнуть потери денег, контроля или данных;
- как сделать проще для команды, но без потери дисциплины;
- как не смешивать зоны ответственности между модулями.

Если есть более сильный operational pattern, его нужно предлагать.

### 5. Находить узкие места и скрытые проблемы

По каждому модулю отдельно искать:

- устаревшие формулировки;
- пропущенные обязательные поля;
- отсутствующие этапы;
- неочевидные edge cases;
- смешение двух разных процессов в одном месте;
- логику, которая уже не совпадает с кодом;
- логику, которая есть в коде, но больше не нужна бизнесу.

### 6. Сравнивать три слоя

Каждый модуль нужно проверять через три вопроса:

- что написано в документации;
- что реально реализовано в коде;
- как это на самом деле должно работать для бизнеса.

По итогам различия помечать как:

- `OK`
- `PARTIAL`
- `MISSING`
- `STALE`
- `BUSINESS DECISION`

### 7. Только после согласования вносить в документацию

Пока решение обсуждается, оно остаётся в чате.

В `docs/NBOS` переносится только то, что уже согласовано:

- owner подтвердил явно;
- или не возразил на предложенный пункт.

### 8. Фиксировать не только целевую модель, но и cleanup

Если в старых docs или коде уже есть неправильная логика, её надо не забывать.

Для этого отдельно фиксировать:

- что уже совпадает с новым каноном;
- что устарело только в документации;
- что устарело в runtime и потом требует рефакторинга.

### 9. После канона переходить к реализации по gap-анализу

Когда канон модуля готов, следующий шаг:

- проверить код по этому канону;
- отметить `OK / PARTIAL / MISSING`;
- собрать implementation backlog;
- только потом писать или менять код.

---

## Что Должно Получиться По Каждому Модулю

Минимальный хороший результат по модулю:

- понятное module overview;
- описание ключевых процессов;
- обязательные правила и stage gates;
- границы модуля и handoff в соседние модули;
- список решений owner;
- cleanup register, если есть старые хвосты;
- потом gap-анализ между каноном и кодом.

---

## Формат Работы В Чате

При работе по модулю держать такой ритм:

1. коротко пересказать модуль или процесс;
2. предложить профессиональный подход;
3. выделить слабые места и чего не хватает;
4. дождаться комментариев owner;
5. считать молчание по отдельному пункту согласием;
6. после этого переносить решение в docs.

---

## Как Использовать Этот Файл Дальше

Этот файл является живым каноном рабочего подхода.

По мере прохода следующих модулей сюда можно добавлять новые правила:

- как лучше оформлять stakeholder decisions;
- как вести cleanup registers;
- как отделять business canon от implementation details;
- как фиксировать спорные решения между модулями.
