# Drive: навигация, Project hub и папки (зафиксировано)

## 1. Один физический файл — много бизнес-входов

- **`FileAsset`** — один объект (метаданные, версии, ключ в R2).
- **`FileLink`** — привязка к сущности (`PROJECT`, `DEAL`, `PRODUCT`, `TASK`, …).
- **Размещение в пользовательских папках** (`DriveFolder` / `DriveFolderItem` в Company/Personal) — отдельно от линков: «где лежит в NBOS-дереве», не копия файла.

Принцип: **не плодить копии** ради того, что файл виден и в Deal, и в Project, и в Library.

---

## 2. Company / Personal vs System Libraries

| Слой                                                | Смысл                                                                                                                                                                                           |
| --------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Company / Personal Drive**                        | Реальное дерево **`DriveFolder`**: создаётся пользователем, move/copy/remove между папками.                                                                                                     |
| **System Libraries** (Deals, Projects, Products, …) | Логические **входы** по правилам (`FileLink`, purpose, visibility…). Пункты **Projects**, **Deals**, **Clients** в сайдбаре Library — **разные входы**, не вложенность «Deals внутри Projects». |

Оба нужны: до появления проекта файлы живут в контексте **Deal / Client**; после — тот же `FileAsset` доступен из **Project** при наличии линка и прав.

---

## 3. Project как центральный хаб (целевое поведение продукта)

- В карточке **Project** пользователь ожидает **сводку** по связанным сущностям: Deal, Product, Client, … — как **секции / представления** (фильтр по `FileLink` и связям CRM/Project), а не обязательно отдельное «второе дерево» на каждую сущность в БД.
- **Реальное дерево папок** под проектом (создание подпапок пользователем, перетаскивание файлов) — целевой срез **«Папки проекта в Drive»** (см. `todo.md`), с размещением без дублирования `FileAsset`.

**До появления проекта:** файлы по **`FileLink` на DEAL / CLIENT` и т.д.** отображаются в соответствующих Library и карточках CRM/Clients без дерева проекта.

**После Won / создания Project:** политика наследования линков (`PROJECT` + существующие линки) задаётся отдельно в продуктовых правилах (авто-линк vs только виртуальный показ до явного действия) — зафиксировать в срезе реализации Project hub.

---

## 4. Deep links в Drive из карточек

- **Project:** `?projectId=` → Library → Projects, контекст загрузки на проект.
- **Product:** `?driveProductId=` → Library → Products, контекст на продукт.
- **Task:** `?driveTaskId=` → Library → Tasks & Work Spaces, контекст на задачу.

Реализация: см. `apps/web/src/features/drive/drive-deep-link.ts` и `DriveWorkspace`.

---

## 5. Что вынесено из backlog на отдельное решение

**Отдельные «физические» деревья `DriveFolder` под каждую сущность в Library** (Deal-only tree, Client-only tree, дублирующие Project) — **не** входит в автоматическое «да» без продуктового решения: высокая цена синхронизации. Предпочтительная модель: **одно дерево под Project** + виртуальные секции по сущностям + глобальные входы Library по Deal/Client.
