# SOP и шаблоны процессов

> NBOS Platform — модуль My Company: стандартные операционные процедуры и библиотека шаблонов

## Назначение

SOP (Standard Operating Procedure) — это **короткое описание процесса**, которое позволяет любому сотруднику выполнить задачу правильно с первого раза. SOP — не книга на 50 страниц, а практичный документ на 1–2 страницы с чёткими шагами и чеклистом.

**Ключевой принцип:** Если процесс повторяется больше двух раз — у него должен быть SOP. Если сотрудник уволился — новый должен выполнить тот же процесс с помощью SOP без потери качества.

---

## Формат SOP

Каждый SOP следует единому формату:

| Поле | Описание | Пример |
|------|----------|--------|
| **Название** | Краткое название процедуры | «Обработка нового лида» |
| **Цель** | Зачем эта процедура нужна | «Обеспечить обработку лида в течение 2 часов» |
| **Owner** | Кто отвечает за актуальность SOP | Head of Sales |
| **Участники** | Кто выполняет процедуру | Seller, Marketing Manager |
| **Вход (Input)** | Что запускает процедуру | Новый лид в CRM |
| **Выход (Output)** | Что является результатом | Лид классифицирован (SQL / SPAM / Frozen) |
| **Шаги** | Пронумерованный список действий | 1. Проверить источник... 2. Связаться... |
| **Чеклист** | Контрольные точки для проверки | ☐ Контакт установлен ☐ Источник зафиксирован |
| **Метрики** | Как измерить качество выполнения | Время обработки ≤ 2 часа, conversion rate |
| **Версия** | Номер версии и дата обновления | v1.2 — 2026-03-01 |

## Обязательные SOP на 2026 год

### 1. Lead Intake → CRM (Обработка нового лида)

| Параметр | Значение |
|----------|----------|
| **Owner** | Head of Sales |
| **Вход** | Новый лид из любого канала (Instagram, Facebook, WhatsApp, сайт, партнёр) |
| **Выход** | Лид классифицирован: SQL / SPAM / Frozen |
| **Время** | ≤ 2 часов с момента поступления |

**Шаги:**

1. Лид поступает в CRM на стадию «New»
2. Seller проверяет источник и контактные данные
3. Первый контакт — звонок или сообщение в течение 2 часов
4. Если контакт не установлен → 3 попытки с интервалом 1 день → «Didn't Get Through»
5. При установлении контакта → квалификация: есть ли бюджет, потребность, сроки?
6. Результат: SQL (создаётся Deal) / SPAM / Frozen (вернёмся позже)
7. Зафиксировать источник лида (для маркетинговой аналитики)

### 2. Presale → Proposal → Contract → Payment (Цикл продажи)

| Параметр | Значение |
|----------|----------|
| **Owner** | Head of Sales |
| **Вход** | SQL-лид конвертирован в Deal |
| **Выход** | Подписанный договор + первая оплата получена |

**Шаги:**

1. Провести Discovery-встречу (выяснить требования клиента)
2. Подготовить коммерческое предложение (Proposal/Offer)
3. Презентация КП клиенту, обсуждение, корректировки
4. Согласование финальных условий (тип оплаты, сроки, scope)
5. Формирование договора (из шаблона)
6. Подписание договора обеими сторонами
7. Выставление первого счёта (Invoice) в Finance
8. Получение предоплаты → статус Deal Won
9. Инициация Handoff к PM

### 3. Project Kickoff (Запуск проекта)

| Параметр | Значение |
|----------|----------|
| **Owner** | Head of Delivery |
| **Вход** | Deal Won + первая оплата получена |
| **Выход** | Проект создан, команда назначена, первый спринт запланирован |

**Шаги:**

1. Seller передаёт PM все материалы: КП, договор, контакты, требования
2. PM создаёт проект в Projects Hub (или обновляет для Extension)
3. PM создаёт Product (Work Package) в проекте
4. Назначается команда проекта (Developer, Designer, QA)
5. Kickoff-встреча с командой: обзор проекта, ожидания, дедлайны
6. Создание первого спринта, декомпозиция на задачи
7. Заполнение обязательных полей проекта: тип, стек, дедлайн
8. Создание аккаунтов и инфраструктуры (по чеклисту типа проекта)
9. Сохранение credentials в модуле Credentials

**Чеклист Kickoff:**
- ☐ Все требования переданы от Seller
- ☐ Проект создан в Projects Hub
- ☐ Команда назначена
- ☐ Первый спринт запланирован
- ☐ Аккаунты инфраструктуры созданы
- ☐ Credentials сохранены
- ☐ Клиент уведомлён о старте разработки

### 4. Sprint Planning (Планирование спринта)

| Параметр | Значение |
|----------|----------|
| **Owner** | PM |
| **Вход** | Завершение предыдущего спринта / старт проекта |
| **Выход** | Спринт создан, задачи распределены, дедлайны установлены |

**Шаги:**

1. Sprint Retrospective предыдущего спринта (что сделано, что нет, почему)
2. Обзор Backlog — приоритизация задач
3. Определение scope спринта (задачи на 1–2 недели)
4. Декомпозиция задач, оценка сложности (Story Points / часы)
5. Назначение задач на конкретных исполнителей
6. Установка дедлайнов с учётом рабочих графиков команды
7. Публикация спринта → уведомление команде

### 5. Release / Deployment Checklist (Чеклист деплоя)

| Параметр | Значение |
|----------|----------|
| **Owner** | Head of Delivery / Lead Developer |
| **Вход** | Спринт завершён, все задачи проверены |
| **Выход** | Код развёрнут на production без инцидентов |

**Чеклист:**
- ☐ Все задачи спринта в статусе Done
- ☐ Code Review пройден для всех PR
- ☐ QA-тестирование завершено, критические баги исправлены
- ☐ Staging-окружение проверено
- ☐ Backup базы данных создан
- ☐ Deployment plan подготовлен
- ☐ Rollback plan подготовлен
- ☐ Деплой выполнен
- ☐ Smoke-тесты на production пройдены
- ☐ Мониторинг: ошибки, производительность — 30 минут наблюдения
- ☐ Клиент уведомлён (если нужно)

### 6. Maintenance Triage (Сортировка обращений техподдержки)

| Параметр | Значение |
|----------|----------|
| **Owner** | Head of Support |
| **Вход** | Обращение клиента (баг, вопрос, запрос на доработку) |
| **Выход** | Обращение классифицировано и направлено ответственному |

**Шаги:**

1. Обращение поступает (через мессенджер, email, звонок)
2. Создаётся тикет в модуле Support
3. Первичная классификация:
   - **P1 (Critical):** Production down, данные потеряны → немедленная реакция
   - **P2 (High):** Критичная функция не работает → в течение 4 часов
   - **P3 (Medium):** Некритичный баг → в течение 1 рабочего дня
   - **P4 (Low):** Улучшение, вопрос → в порядке очереди
4. Назначение ответственного (Support Engineer или Developer)
5. Первый ответ клиенту в рамках SLA
6. Решение → подтверждение клиентом → закрытие тикета

### 7. Incident Response (Реагирование на критический инцидент)

| Параметр | Значение |
|----------|----------|
| **Owner** | Head of Delivery / DevOps |
| **Вход** | P1-инцидент: production down, утечка данных, критическая ошибка |
| **Выход** | Сервис восстановлен, корневая причина найдена, профилактика принята |

**Шаги:**

1. Зафиксировать инцидент: время обнаружения, симптомы, масштаб
2. Уведомить: PM проекта, Head of Delivery, CEO (если масштабный)
3. Уведомить клиента: «Мы знаем о проблеме, работаем над решением»
4. Диагностика и локализация проблемы
5. Устранение / откат к предыдущей версии
6. Проверка: сервис восстановлен, данные целы
7. Уведомить клиента: «Проблема решена»
8. Post-mortem: документировать причину, действия, время простоя
9. Превентивные меры: задача на устранение корневой причины

### 8. Employee Onboarding (Онбординг нового сотрудника)

| Параметр | Значение |
|----------|----------|
| **Owner** | HR Director |
| **Вход** | Решение о найме принято |
| **Выход** | Сотрудник продуктивен, все доступы выданы, обучение пройдено |

> Детальный чеклист описан в `02-Team-Employees.md`, секция «Онбординг»

**Ключевые этапы:**
1. Создание профиля в NBOS → все аккаунты и доступы (День 1)
2. Знакомство с командой и ментором (День 1)
3. Обучение: платформа, процессы, SOP (Неделя 1)
4. Первый проект / задачи (Дни 2–5)
5. Проверка по итогам первой недели (День 7)
6. Проверка по итогам испытательного срока (Месяц 1–3)

### 9. Employee Offboarding (Оффбординг сотрудника)

| Параметр | Значение |
|----------|----------|
| **Owner** | HR Director |
| **Вход** | Решение об увольнении / уходе сотрудника |
| **Выход** | Доступы отозваны, проекты переданы, расчёт выполнен |

> Детальный чеклист описан в `02-Team-Employees.md`, секция «Оффбординг»

**Ключевые этапы:**
1. Передача проектов и задач (Неделя до ухода)
2. Передача знаний (документация, записи) (Неделя до ухода)
3. Отзыв всех доступов (Последний день)
4. Ротация критичных паролей (Последний день)
5. Финальный расчёт: зарплата + бонусы (1–2 недели)
6. Exit-интервью (по желанию)

### 10. Monthly Financial Close (Ежемесячное финансовое закрытие)

| Параметр | Значение |
|----------|----------|
| **Owner** | Finance Director |
| **Вход** | 1-е число нового месяца |
| **Выход** | Все финансы предыдущего месяца закрыты, P&L сформирован |

**Шаги:**

1. Проверить все плановые расходы: все ли оплачены?
2. Проверить внеплановые расходы: все ли зафиксированы?
3. Проверить счета (Invoice): все ли оплаченные отмечены?
4. Проверить подписки: все ли платежи получены?
5. Сформировать Salary Board: базовые ЗП + Active бонусы
6. Произвести выплату зарплат (до 10-го числа)
7. Сформировать P&L отчёт за месяц
8. Сверить Cash Balance с банковскими выписками
9. Отчёт CEO: ключевые метрики, отклонения, прогноз

### 11. Quarterly Planning (Квартальное планирование)

| Параметр | Значение |
|----------|----------|
| **Owner** | CEO |
| **Вход** | Окончание текущего квартала |
| **Выход** | Rocks на следующий квартал определены, KPI обновлены |

**Шаги:**

1. Ревью завершённого квартала: Rocks — On Track / Off Track
2. Анализ Scorecard за квартал: тренды, проблемные метрики
3. Финансовый ревью: P&L квартала, прогноз на следующий
4. Определение 3–7 Rocks на следующий квартал
5. Назначение Owner для каждого Rock
6. Обновление Target-значений в Scorecard (если нужно)
7. Обновление персональных KPI (если роли/приоритеты изменились)
8. Коммуникация плана команде

### 12. Domain Renewal Process (Процесс продления домена)

| Параметр | Значение |
|----------|----------|
| **Owner** | Operations / Finance Director |
| **Вход** | Домен истекает через 2 месяца (авто-уведомление) |
| **Выход** | Домен продлён, клиент оплатил, затрата закрыта |

**Шаги:**

1. За 2 месяца: система уведомляет — домен X истекает DD.MM.YYYY
2. Автоматически создаётся Invoice для клиента (из шаблона Domain Renewal)
3. Invoice отправляется клиенту
4. Клиент оплачивает → Payment создан
5. Создаётся задача: «Продлить домен X на провайдере Y»
6. Специалист продлевает домен
7. Обновляет дату истечения в записи Domain
8. Expense (Pass-through) создаётся и закрывается как Paid

## Реализация SOP в платформе

### SOP — не просто документы

В NBOS каждый SOP — это не просто markdown-файл, а **живой процесс**, интегрированный в платформу:

```
SOP документ → Шаблон чеклиста → Автоматический запуск → Отслеживание выполнения
```

| Компонент | Описание |
|-----------|----------|
| **SOP Document** | Текстовое описание процесса (читается человеком) |
| **Checklist Template** | Набор шагов, привязанный к SOP (используется системой) |
| **Trigger** | Событие, запускающее чеклист (смена статуса, дата, вручную) |
| **Progress Tracking** | Визуальная полоса прогресса, уведомления при просрочке |

### Примеры автоматических триггеров

| SOP | Триггер |
|-----|---------|
| Onboarding | Новый сотрудник → статус Probation |
| Offboarding | Сотрудник → статус Terminated |
| Project Kickoff | Deal Won + Payment received |
| Monthly Financial Close | 1-е число каждого месяца |
| Domain Renewal | Домен истекает через 2 месяца |
| Sprint Planning | Sprint завершён / первый Sprint проекта |
| Deployment | Sprint → Ready for Deploy |

## Библиотека шаблонов (Templates Library)

### Типы шаблонов

| Тип | Описание | Примеры |
|-----|----------|---------|
| **Project Template** | Шаблон для создания нового проекта | Web-site (WL), Mobile App (Custom), E-commerce |
| **Task Template** | Шаблон для часто повторяющихся задач | «Настроить CI/CD», «Создать аккаунт провайдера» |
| **Checklist Template** | Шаблон чеклиста для процесса | Deployment checklist, Onboarding checklist |
| **Sprint Template** | Шаблон первого спринта (стартовые задачи) | «Setup Sprint» с базовыми задачами |
| **Document Template** | Шаблон документа (КП, договор, ТЗ) | Proposal template, Contract template |

### Project Templates

В зависимости от типа проекта (WL / Mix / Custom Code) система предлагает разные шаблоны:

| Тип проекта | Стандартные задачи | Чеклисты |
|-------------|--------------------|-----------| 
| **WL (White Label)** | Настройка шаблона, брендинг, контент, SEO-базовый | WL-деплой чеклист |
| **Mix (WL + Custom)** | WL-задачи + кастомная разработка | WL-деплой + Code Review |
| **Custom Code** | Полный цикл разработки: дизайн, фронт, бэк, интеграции | Полный Deployment checklist |

```
Тип проекта: Custom Code Web App
  └─ Стартовый Sprint:
     ├─ Задача: Создать репозиторий Git
     ├─ Задача: Настроить CI/CD
     ├─ Задача: Создать staging-окружение
     ├─ Задача: Создать аккаунт хостинга
     ├─ Задача: Настроить домен
     ├─ Задача: Создать БД
     ├─ Задача: Настроить мониторинг
     └─ Чеклист: Инфраструктура Ready
```

### Эволюция шаблонов

Шаблоны — это живые документы. Они обновляются по мере того, как команда учится:

```
v1.0 (начальный)
  └─ Проект завершён, выявлены пропущенные шаги
     └─ v1.1 (добавлены шаги)
        └─ Новый инцидент: забыли настроить мониторинг
           └─ v1.2 (добавлен пункт мониторинга)
              └─ Квартальный ревью: оптимизация порядка шагов
                 └─ v2.0 (переработанный шаблон)
```

**Правило обновления:** После каждого завершённого проекта PM проводит ретроспективу и предлагает обновления шаблонов. Head of Delivery утверждает изменения.

## Права доступа к SOP

| Роль | Чтение SOP | Редактирование SOP | Создание шаблонов |
|------|------------|--------------------|--------------------|
| **CEO** | Все | Все | Да |
| **Head of Department** | Все | SOP своего отдела | Да |
| **Operations Manager** | Все | Все | Да |
| **PM** | Все | Предложения (review) | Предложения (review) |
| **Сотрудник** | SOP своего отдела | Нет | Нет |

## Метрики качества SOP

| Метрика | Описание | Цель |
|---------|----------|------|
| **SOP Coverage** | % процессов, покрытых SOP | ≥ 80% |
| **Template Usage** | % проектов, использующих шаблоны | ≥ 90% |
| **SOP Freshness** | Средний возраст SOP (время с последнего обновления) | ≤ 6 месяцев |
| **Checklist Completion** | % чеклистов, выполненных полностью | ≥ 95% |
