# Editor, Content And Storage

> TipTap is the native document editor for NBOS Documents.

## Editor decision

Documents uses `TipTap` on top of ProseMirror for native document editing.

Decision:

```text
Native document editor = TipTap.
Native content source of truth = TipTap JSON in PostgreSQL.
Files and images = Drive File Asset in R2.
```

Why:

- it gives a structured document model;
- it supports rich editing without building a full word processor;
- it can render clean viewer mode;
- it is extendable for tables, checklists, callouts, mentions and embeds;
- it keeps the future path open for AI-assisted edits and collaboration.

## Editor scope

The first release should support:

- headings;
- paragraphs;
- bold / italic / underline;
- bullet lists;
- ordered lists;
- task/check lists;
- blockquote;
- code block / inline code;
- horizontal divider;
- links;
- images;
- tables;
- callouts / info boxes;
- attachments;
- undo / redo;
- clean paste from Google Docs / Word where practical.

The first release does not need:

- full Microsoft Word parity;
- complex page layout;
- headers/footers;
- track changes;
- multi-user live cursors;
- advanced comments workflow;
- equation editor;
- spreadsheet formulas.

## Content fields

For native documents, store:

```text
content_json  -> source of truth
content_html  -> rendered display cache
plain_text    -> search and preview
```

`content_json` must be valid TipTap JSON.

`content_html` is derived from JSON and may be regenerated.

`plain_text` is derived from JSON and used for search, snippets and previews.

## Autosave rule

Autosave must not create heavy history.

Correct behavior:

```text
User types in editor
  -> browser keeps local editor state
  -> autosave after debounce / pause
  -> update current draft/current content
  -> write simple updated timestamp when appropriate
```

Recommended autosave behavior:

- debounce after 5-15 seconds of inactivity;
- save on explicit `Save`;
- save on navigation warning if unsaved changes exist;
- show clear save state: Saving, Saved, Failed, Unsaved changes;
- retry failed autosave safely;
- never create one historical revision per keystroke.

## Version history decision

First release keeps document history light.

Canonical first version:

```text
current content
updated_at
updated_by
activity log
```

Not required in first release:

```text
full revision timeline
diff viewer
restore per version
approval revisions
```

Optional later extension:

```text
DocumentRevision
  id
  document_id
  version_number
  content_json
  content_html
  plain_text
  created_by
  created_at
  reason
```

If added later, revisions should be created only on meaningful actions:

- publish;
- manual checkpoint;
- restore;
- import;
- major edit confirmation.

## File and image handling

Files do not live inside `content_json`.

Correct:

```json
{
  "type": "image",
  "attrs": {
    "fileAssetId": "file_123",
    "src": "/api/files/file_123/view",
    "alt": "Process diagram"
  }
}
```

Incorrect:

```text
base64 image blob inside content_json
```

Upload flow:

```text
Editor upload
  -> Drive creates File Asset
  -> file stored in R2
  -> Documents inserts fileAssetId into TipTap node
  -> viewer loads file through backend-controlled URL
```

## Attachment handling

Documents may show attachments below or beside the document.

Attachment examples:

- PDF policy file;
- source DOCX;
- spreadsheet;
- screenshot;
- diagram;
- training material;
- exported PDF.

All attachments are Drive File Assets linked through `DocumentAttachment`.

## Database storage rationale

Native document text should live in PostgreSQL because:

- it is structured business content, not binary file storage;
- it must be searchable;
- it needs permissions and ownership;
- it needs reliable save/read behavior;
- it should be returned directly with document metadata;
- typical document sizes are safe for PostgreSQL at NBOS scale.

R2 is the right place for:

- images;
- PDFs;
- DOCX/XLSX;
- exports;
- large binary files;
- backup archives.

## Search content extraction

On save, the system should derive:

```text
TipTap JSON -> plain_text -> search index
```

Search should include:

- title;
- description;
- tags;
- section name;
- plain_text;
- owner;
- updated date filters.

First release can use PostgreSQL full text search. External search engines are not required until real volume or UX proves the need.

## Rendering rule

Viewer mode should render from trusted sanitized content.

Preferred:

```text
content_json -> server/client renderer -> sanitized HTML/view tree
```

If cached `content_html` is used, it must be generated by trusted renderer and sanitized.

## Editor UX rules

Editor should feel calm and focused:

- title always visible;
- save state always visible;
- toolbar compact and predictable;
- formatting controls use familiar icons;
- no overloaded corporate forms around the writing area;
- attachments easy to add;
- document metadata visible but not blocking writing;
- publish/archive actions separated from formatting actions;
- mobile reading must be excellent even if mobile editing is lighter.

## Failure states

Documents must handle:

- autosave failure;
- lost network;
- permission changed while editing;
- document archived while user has it open;
- file upload failure;
- attachment deleted or inaccessible;
- malformed content_json.

Correct behavior:

```text
Show clear state, preserve local draft when possible, do not silently lose text.
```
