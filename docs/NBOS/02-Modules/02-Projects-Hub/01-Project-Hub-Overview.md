# Projects Hub — Обзор центрального модуля

## 1. Назначение

**Projects Hub** — центральный модуль NBOS Platform, вокруг которого строится вся операционная деятельность компании Neetrino. Это единая точка правды для каждого проекта: все данные, связанные с проектом, доступны из одного места.

Принцип: **один проект = один бизнес / один бренд**. Проект — это НЕ один клиент. Один клиент (Contact) может иметь множество проектов для разных бизнесов.

> **Пример:** Контакт «Арам Петросян» — крупный предприниматель. У него бренд обуви, цветочный магазин и транспортная компания. В NBOS это три отдельных проекта, каждый со своими продуктами, заказами и финансами.

---

## 2. Роль Projects Hub в системе

Projects Hub — это **контейнер-агрегатор**, который объединяет:

| Что агрегирует | Описание |
|----------------|----------|
| Products | Все продукты (Website, Mobile App, CRM и т.д.), которые разрабатываются для бизнеса |
| Extensions | Все доработки к существующим продуктам |
| Orders | Все заказы (финансовая сторона каждого продукта/доработки) |
| Invoices & Payments | Все счета и платежи, связанные с проектом |
| Subscriptions | Подписки на техобслуживание и development |
| Tasks | Все задачи по всем продуктам и доработкам проекта |
| Credentials | Все пароли и доступы для этого проекта |
| Drive / Assets | Все файлы и документы |
| Domains | Все домены, привязанные к проекту |
| Support Tickets | Все тикеты клиентской поддержки |
| Chat | Проектный чат с топиками |
| Audit Log | Полная история изменений |

Каждый специалист заходит в проект и видит **только то, что относится к его роли**: PM видит задачи и спринты, Seller — заказы и счета, Developer — задачи и пароли, Finance Director — P&L и инвойсы.

## 3. Вычисляемый статус проекта

**Проект НЕ имеет вручную установленного статуса.** Его состояние вычисляется автоматически на основе статусов его продуктов (Products) и доработок (Extensions).

### Правила вычисления:

```
ЕСЛИ хотя бы один Product/Extension в статусе Development
  → Проект виден в табе «Development»

ЕСЛИ хотя бы один Product в статусе Maintenance (через активную подписку)
  → Проект виден в табе «Maintenance»

ЕСЛИ проект одновременно имеет Product в Maintenance И Extension в Development
  → Проект виден В ОБОИХ табах одновременно

ЕСЛИ все Products/Extensions в статусе Done/Closed И нет активной подписки
  → Проект виден в табе «Closed / Archived»
```

### Пример одновременного отображения:

| Проект | Product «Website» | Extension «Личный кабинет» | Табы |
|--------|-------------------|---------------------------|------|
| Brand X | Done → Maintenance | Development | **Maintenance** + **Development** |
| Brand Y | Development | — | **Development** |
| Brand Z | Done | Done | **Closed** |
| Brand W | Maintenance | QA | **Maintenance** + **Development** |

> Это не дубликаты — это один и тот же проект, видимый в нескольких filtered views одновременно.

## 4. Основные виды (табы) Projects Hub

Все табы — это **фильтрованные представления** одного и того же списка проектов, а не отдельные хранилища данных.

### 4.1. All Projects (Все проекты)

Полный список всех проектов компании. Отображаются все проекты независимо от статуса: активные, в разработке, на обслуживании, закрытые.

**Фильтры:**
- По типу (WL / Mix / Custom Code)
- По контакту / компании
- По ответственному (Seller, PM)
- По дате создания
- Поиск по названию

**Сортировка:** по дате создания, по дедлайну, по имени, по количеству активных задач.

### 4.2. Development (Разработка)

Проекты, у которых хотя бы один Product или Extension находится в стадиях: New, Creating, Development, QA, Transfer.

**Визуальное отличие:**
- Проекты с основным Product в разработке — обычная карточка
- Проекты, где разрабатывается только Extension (а основной продукт уже в Maintenance) — карточка с пометкой «Extension» и указанием какая именно доработка идёт

**Дополнительная информация на карточке:** прогресс-бар текущего этапа, количество открытых задач, ближайший дедлайн.

### 4.3. Maintenance (Техобслуживание)

Проекты с активной подпиской на техобслуживание (Subscription Contract в статусе Active).

**Информация на карточке:** сумма подписки, дата следующего платежа, статус последнего платежа (оплачен / просрочен), количество открытых тикетов.

### 4.4. Closed / Archived (Закрытые)

Проекты, у которых все продукты завершены (Done/Lost) и нет активной подписки. Все данные сохраняются и доступны для просмотра, но проект не отображается в рабочих табах.

## 5. Карточка проекта в списке

В списке Projects Hub каждый проект отображается как карточка (или строка таблицы, в зависимости от выбранного вида) со следующей информацией:

| Поле | Описание |
|------|----------|
| Название проекта | Имя бренда / бизнеса |
| Тип | WL (White Label), Mix, Custom Code |
| Контакт | Имя основного контакта |
| Компания | Юрлицо для биллинга |
| Seller | Ответственный продажник |
| PM | Ответственный проект-менеджер |
| Статус-индикаторы | Иконки/бейджи: Development, Maintenance, Extension |
| Продукты | Список продуктов с мини-статусами |
| Прогресс | Визуальный прогресс-бар |
| Дедлайн | Ближайший дедлайн по продуктам/доработкам |
| Финансы (для CEO/Finance) | Общая сумма заказов, статус оплат |

## 6. Виды отображения списка

| Вид | Описание |
|-----|----------|
| **Table** | Табличный вид с сортировкой по столбцам. Для быстрого обзора большого количества проектов |
| **Cards** | Карточный вид. Визуально более информативный, подходит для повседневной работы |
| **Board** | Канбан-подобное отображение по этапам (для Development-таба) |

## 7. Кто видит что в Projects Hub

| Роль | Видит в списке | Видит внутри проекта |
|------|---------------|---------------------|
| **CEO** | Все проекты, все данные | Полный доступ ко всем табам |
| **Finance Director** | Все проекты, финансовые показатели | Finance, Orders, Invoices, Subscriptions |
| **Seller** | Свои проекты + общий список (read-only) | Overview, Orders, Finance (частично), Chat (sales-топик) |
| **PM** | Свои проекты | Products, Extensions, Tasks, Credentials, Support, Chat, Drive |
| **Developer** | Назначенные проекты | Tasks, Credentials (назначенные), Chat (dev-топик), Drive |
| **Designer** | Назначенные проекты | Tasks, Drive, Chat (dev-топик) |
| **QA** | Назначенные проекты | Tasks, Support, Chat |
| **Tech Specialist** | Проекты по запросу | Credentials, Domains, Drive |
| **Marketing** | Только аналитика (через Dashboards) | Не заходит в Projects Hub напрямую |

## 8. Связь с другими модулями

```
CRM (Leads & Deals)
  │
  │  Deal Won + First Payment
  ▼
┌─────────────────┐
│  PROJECTS HUB   │ ◄──── единая точка входа
└────────┬────────┘
         │
    ┌────┼────┬────────┬──────────┬────────┬──────────┐
    ▼    ▼    ▼        ▼          ▼        ▼          ▼
 Tasks Finance Support Credentials Drive  Messenger Calendar
```

| Модуль | Связь с Projects Hub |
|--------|---------------------|
| **CRM** | Deal Won → создаётся Project в Hub |
| **Finance** | Orders, Invoices, Subscriptions, Expenses — всё привязано к Project |
| **Tasks** | Все задачи принадлежат Project (через Product/Extension) |
| **Support** | Тикеты привязаны к Project |
| **Credentials** | Пароли хранятся в контексте Project |
| **Drive** | Файлы организованы по Projects |
| **Messenger** | Каждый Project имеет свой чат с топиками |
| **Calendar** | Дедлайны проектов, встречи по проектам |
| **Dashboards** | Аналитика по проектам |

## 9. Ключевые принципы

1. **Project = Container.** Проект сам по себе не «делается» — делаются продукты и доработки внутри него.

2. **Статус = Computed.** Никто не ставит статус проекту вручную. Статус автоматически определяется из состояния содержимого.

3. **Views, Not Data Stores.** Табы Development, Maintenance, Closed — это фильтры, а не отдельные хранилища. Данные существуют в одном месте.

4. **Role-Based Context.** Каждый специалист видит свой срез проекта. PM не видит финансовые детали, Developer не видит суммы заказов, Finance Director не видит scrum-доски.

5. **One Project = One Business.** Не один клиент, а один бизнес. Клиент может иметь 10 проектов для 10 разных бизнесов.

6. **Everything Connected.** Каждая сущность внутри проекта (задача, инвойс, пароль, файл) имеет прямую привязку к проекту и может быть найдена через него.

## 10. Сценарии использования

### Сценарий 1: PM начинает день
PM открывает Projects Hub → таб Development → видит свои проекты с прогресс-барами → заходит в проект с ближайшим дедлайном → открывает Product → видит scrum-доску с задачами на текущий спринт.

### Сценарий 2: CEO проверяет состояние бизнеса
CEO открывает Projects Hub → таб All → видит общую картину: сколько проектов в разработке, сколько на обслуживании, сколько закрытых → заходит в конкретный проект → проверяет финансы, задачи, статус.

### Сценарий 3: Seller предлагает доработку
Seller открывает Projects Hub → находит проект клиента → видит, что основной продукт уже на Maintenance → предлагает клиенту доработку → создаёт Extension Deal в CRM → Deal проходит pipeline → создаётся Extension в проекте.

### Сценарий 4: Finance Director проверяет оплаты
Finance Director открывает Projects Hub → таб Maintenance → видит проекты с подписками → заходит в проект → таб Finance → видит историю платежей, просроченные инвойсы, P&L проекта.

## 11. Терминология

| Термин | Определение |
|--------|------------|
| **Project** | Контейнер для одного бизнеса/бренда клиента |
| **Product** | Конкретный результат работы (Website, App, Logo) |
| **Extension** | Доработка к существующему продукту |
| **Order** | Финансовая сущность: что продали, за сколько |
| **Subscription** | Подписка на регулярное обслуживание |
| **Contact** | Физическое лицо — человек-заказчик |
| **Company** | Юридическое лицо для выставления счетов |
| **Stage Gate** | Обязательное условие для перехода между стадиями |
