# Продукты и доработки (Products & Extensions)

## Обзор

Внутри каждого проекта ведётся конкретная работа — она выражается через **Products** (продукты) и **Extensions** (доработки). Product — это крупный результат (сайт, приложение), Extension — дополнительная работа к уже существующему продукту. Оба имеют собственный lifecycle, задачи и финансовую привязку через Order.

---

# ЧАСТЬ 1: Products (Продукты)

## 1. Что такое Product

**Product** — это конкретный результат работы, который Neetrino создаёт для клиента в рамках проекта. Один проект может содержать несколько продуктов одновременно.

> **Пример:** Проект «Brand X» содержит Product «Website» + Product «Mobile App» + Product «Logo». Каждый продукт имеет свой жизненный цикл, свои задачи и свой заказ.

### Типы продуктов:

| Тип | Описание |
|-----|----------|
| **Website** | Веб-сайт (landing, corporate, e-commerce) |
| **Mobile App** | Мобильное приложение (iOS, Android, кроссплатформенное) |
| **CRM** | CRM-система или внутренний инструмент |
| **Logo** | Логотип и фирменный стиль |
| **SMM** | Ведение социальных сетей |
| **SEO** | Поисковая оптимизация |
| **Other** | Другие типы продуктов |

### Связь типа проекта и продукта:

Тип проекта (WL / Mix / Custom Code) влияет на:
- **Процент бонуса delivery-команде:** WL — 7%, Mix — 10%, Custom Code — 15%
- **Шаблоны задач:** у каждого типа проекта свой набор стандартных задач
- **Чеклисты качества:** стандарты безопасности и качества зависят от типа
- **Stage Gate требования:** обязательные условия перехода между стадиями

---

## 2. Жизненный цикл продукта (Stage Gates)

Product проходит строго определённые стадии. Каждая стадия имеет **обязательные условия** (Stage Gates), без выполнения которых система блокирует переход на следующую стадию.

### Стадии:

```
New → Creating → Development → QA/Checking → Transfer → Done
                                                  ↓
                                              On Hold (пауза)
                                                  ↓
                                              Lost (отменён)
```

### Детальное описание стадий:

### 2.1. New (Новый)

Продукт только получен. Заполняется начальная информация.

| Параметр | Значение |
|----------|----------|
| Кто работает | PM, Seller |
| Обязательные поля | Название, тип продукта, описание, дедлайн, привязка к заказу |
| Stage Gate для перехода | Все обязательные поля заполнены, заказ создан и привязан |

### 2.2. Creating (Подготовка)

Подготовительный этап: создание аккаунтов, покупка домена, настройка окружения, сохранение всех доступов.

| Параметр | Значение |
|----------|----------|
| Кто работает | PM, Tech Specialist |
| Обязательные действия | Создать аккаунты (хостинг, DNS, почта, база данных и др.), купить домен, настроить среду разработки, сохранить все пароли в Credentials |
| Stage Gate для перехода | ВСЕ обязательные аккаунты созданы, ВСЕ пароли сохранены в системе, домен привязан к проекту |

**Критически важно:** система проверяет, что для данного типа продукта созданы все необходимые Credential-записи. Например, для Website обязательны: Domain Account, Hosting Account, DNS (Cloudflare), Database. Пока все эти записи не созданы и поля login/password не заполнены — переход заблокирован.

### 2.3. Development (Разработка)

Активная работа над продуктом. Основной этап, где происходит scrum/kanban-процесс.

| Параметр | Значение |
|----------|----------|
| Кто работает | PM, Developer, Designer, QA |
| Внутренняя организация | Scrum-доска или Kanban-доска (переключаемые) |
| Спринты | Если scrum — спринты с планированием (обычно 1–2 недели) |
| Шаблоны задач | Автоматически загружаются стандартные задачи для типа продукта |
| Еженедельно | Планирование спринта, демо клиенту |
| Stage Gate для перехода | Все задачи текущего этапа закрыты, .env-файл сохранён, deployment checklist выполнен |

#### Scrum-доска внутри Development:

```
Backlog → To Do → In Progress → Review → Done
```

Задачи перетаскиваются между столбцами. PM создаёт спринты и назначает задачи. Developer перемещает задачи по мере выполнения.

#### Стандартные шаблоны задач по типам продуктов:

**Website (White Label):**

| Задача | Этап | Обязательная |
|--------|------|-------------|
| Собрать требования и ТЗ | Sprint 1 | ✅ |
| Дизайн главной страницы | Sprint 1 | ✅ |
| Дизайн внутренних страниц | Sprint 1 | ✅ |
| Frontend-вёрстка | Sprint 2 | ✅ |
| Backend-интеграция | Sprint 2 | ✅ |
| Наполнение контентом | Sprint 3 | ✅ |
| SEO-базовая настройка | Sprint 3 | ✅ |
| Тестирование (кросс-браузерное) | Sprint 3 | ✅ |
| Deployment на production | Sprint 3 | ✅ |
| Подключение аналитики | Sprint 3 | ✅ |

**Mobile App (Custom Code):**

| Задача | Этап | Обязательная |
|--------|------|-------------|
| Архитектура и техзадание | Sprint 1 | ✅ |
| UI/UX дизайн | Sprint 1–2 | ✅ |
| Создать аккаунты App Store / Google Play | Sprint 1 | ✅ |
| Backend API разработка | Sprint 2–4 | ✅ |
| Frontend разработка | Sprint 2–4 | ✅ |
| Интеграция платёжной системы | Sprint 3–4 | По необходимости |
| Push-уведомления | Sprint 4 | По необходимости |
| Unit и интеграционные тесты | Sprint 4–5 | ✅ |
| Публикация в Store | Sprint 5 | ✅ |

Шаблоны настраиваются CEO/PM и могут обновляться со временем.

### 2.4. QA / Checking (Проверка качества)

Контроль качества, проверка по стандартам безопасности и готовности к production.

| Параметр | Значение |
|----------|----------|
| Кто работает | QA, PM, Developer |
| Чеклисты | Security Checklist, Quality Checklist, Deployment Readiness |
| Обязательные проверки | Зависят от типа продукта и типа проекта (WL/Mix/CC) |
| Stage Gate для перехода | ВСЕ пункты чеклистов пройдены, deployment на production выполнен |

#### Security Checklist (пример для Website):

- [ ] HTTPS настроен и работает
- [ ] Формы защищены от XSS и CSRF
- [ ] Пароли в .env, не в коде
- [ ] Бэкап базы данных настроен
- [ ] Rate limiting для API
- [ ] Административная панель защищена
- [ ] Файлы .env не в репозитории

#### Quality Checklist (пример для Website):

- [ ] Кросс-браузерное тестирование (Chrome, Firefox, Safari, Edge)
- [ ] Мобильная адаптация проверена
- [ ] PageSpeed ≥ 80 (mobile), ≥ 90 (desktop)
- [ ] Все ссылки рабочие (no broken links)
- [ ] Favicon и мета-теги настроены
- [ ] 404-страница настроена
- [ ] Контент проверен клиентом

### 2.5. Transfer (Передача)

Передача готового продукта клиенту. Клиент тестирует, даёт обратную связь, принимает работу.

| Параметр | Значение |
|----------|----------|
| Кто работает | PM (координация), Developer (финальные правки), клиент |
| Процесс | PM демонстрирует продукт клиенту → клиент тестирует → обратная связь → правки → повторная проверка → acceptance |
| Stage Gate для перехода | Клиент подтвердил приёмку (acceptance), все финальные правки сделаны, задача по доработкам закрыта |

### 2.6. On Hold (Приостановлен)

Продукт приостановлен по запросу клиента или по внутренней причине.

| Параметр | Значение |
|----------|----------|
| Причины | Клиент попросил паузу, внутренние ресурсные ограничения, ожидание информации от клиента |
| Правила | Не влияет на финансовые обязательства. При Subscription — платежи приостанавливаются по решению CEO |

### 2.7. Done (Завершён)

Продукт полностью завершён и принят клиентом.

| Параметр | Значение |
|----------|----------|
| Условия | Клиент подтвердил приёмку, все задачи закрыты, все пункты чеклистов пройдены |
| Триггеры | Создание Bonus Entry для Developer/PM/Designer; смена статуса Order → Fully Paid (для Classic); переход проекта на Maintenance (если есть подписка) |

### 2.8. Lost (Отменён)

Продукт отменён. Все данные сохраняются, но продукт не считается завершённым.

---

## 3. Прогресс-бар продукта

В верхней части карточки продукта отображается визуальный прогресс-бар, показывающий текущий этап:

```
[New]──[Creating]──[Development ████████░░░░]──[QA]──[Transfer]──[Done]
                          ▲ текущий этап (60%)
```

Прогресс внутри этапа вычисляется:
- **Development:** % закрытых задач от общего количества
- **QA:** % пройденных пунктов чеклиста
- **Transfer:** % принятых пунктов приёмки

# ЧАСТЬ 2: Extensions (Доработки)

## 4. Что такое Extension

**Extension** — это дополнительная работа к уже существующему продукту: новая функция, улучшение, дополнительный модуль. В отличие от Product, Extension имеет упрощённый жизненный цикл и обычно меньший объём.

> **Важно:** даже самая маленькая доработка (1 час работы) ДОЛЖНА иметь связанный Order, чтобы корректно работала бонусная система и P&L-учёт.

### Связи Extension:

- Extension всегда привязана к **Project**
- Extension опционально привязана к конкретному **Product** (к какому продукту доработка)
- Extension всегда имеет связанный **Order** (финансовая сторона)

---

## 5. Категории размера доработки

| Размер | Описание | Ориентировочный объём |
|--------|----------|-----------------------|
| **Micro** | Мелкая правка, быстрая задача | Менее 1 дня (1–4 часа) |
| **Small** | Небольшая доработка | 1–3 дня |
| **Medium** | Средняя доработка с несколькими задачами | 1–2 недели |
| **Large** | Крупная доработка, сопоставимая с мини-проектом | 2+ недели |

Размер влияет на:
- Структуру задач (Micro — 1-2 задачи, Large — полноценный scrum)
- Сложность чеклиста приёмки
- Ожидаемые сроки

---

## 6. Жизненный цикл Extension

Extension проходит упрощённый цикл по сравнению с Product:

```
New → Development → QA → Done
                          ↓
                        Lost
```

### Стадии:

### 6.1. New (Новая)

Доработка создана из Extension Deal в CRM. PM заполняет описание, назначает исполнителей.

| Stage Gate | Описание заполнено, Order привязан, исполнитель назначен |
|------------|--------------------------------------------------------|

### 6.2. Development (Разработка)

Внутри Extension создаётся мини scrum/kanban-доска с задачами.

| Организация | Kanban-доска: To Do → In Progress → Review → Done |
|-------------|--------------------------------------------------|

Для Micro и Small — обычно 1–3 задачи. Для Medium и Large — полноценный набор задач, возможно с разбивкой на спринты.

### 6.3. QA (Проверка)

Проверка доработки по чеклисту. Для Micro — минимальная проверка, для Large — полноценный QA.

### 6.4. Done (Завершена)

Все задачи закрыты → PM закрывает доработку через чеклист приёмки:

**Чеклист приёмки Extension:**
- [ ] Все задачи выполнены и закрыты
- [ ] Доработка протестирована
- [ ] Клиент проверил и подтвердил
- [ ] Deployment выполнен
- [ ] Документация обновлена (если нужно)

После закрытия чеклиста → статус Extension = Done → **триггер: создание Bonus Entry** для исполнителей на основе суммы связанного Order.

---

## 7. Extension и Bonus

```
Extension закрыта (Done)
  ↓
Order привязан (сумма известна)
  ↓
Bonus Entry создаётся автоматически
  ├── Developer Bonus: % от суммы (зависит от типа проекта)
  ├── PM Bonus: % от суммы
  └── Designer Bonus: % от суммы (если Designer участвовал)
```

Даже Micro-extension (1 час работы за 30,000 AMD) создаёт полноценный Bonus Entry, который проходит стандартный жизненный цикл бонуса: Incoming → Earned → Active → Paid.

## 8. Одновременная работа: Products + Extensions

Один проект может иметь множество продуктов и доработок одновременно в разных стадиях:

### Пример: Проект «Brand X»

| Сущность | Тип | Статус | Влияние на проект |
|----------|-----|--------|-------------------|
| Website | Product | Done → Maintenance | Проект в табе «Maintenance» |
| Добавить личный кабинет | Extension | Development | Проект в табе «Development» |
| Обновить логотип | Extension | Done | Не влияет (завершена) |
| Mobile App | Product | QA | Проект в табе «Development» |

**Результат:** Проект «Brand X» одновременно виден в табах **Maintenance** и **Development**.

Внутри карточки проекта (таб Products + Extensions) пользователь видит:

## 9. Шаблоны и стандарты

### 9.1. Шаблоны задач

При создании Product или Extension определённого типа система автоматически загружает шаблон стандартных задач:

| Тип продукта | Шаблон | Количество задач |
|--------------|--------|-----------------|
| Website (WL) | Website WL Template | ~15–20 |
| Website (Custom) | Website Custom Template | ~25–30 |
| Mobile App | Mobile App Template | ~30–40 |
| Logo | Logo Template | ~5–8 |
| CRM | CRM Template | ~25–35 |

Шаблоны управляются администратором и могут обновляться. PM может добавлять кастомные задачи поверх шаблона.

### 9.2. Стандарты качества

Для каждого типа продукта определены стандарты, которые проверяются на этапе QA:

- **Security Standards** — стандарты безопасности (зависят от типа)
- **Quality Standards** — стандарты качества (производительность, совместимость)
- **Deployment Standards** — стандарты развёртывания (CI/CD, мониторинг)

Документация по стандартам хранится в Drive проекта и привязывается к задачам QA-этапа.

### 9.3. Обязательные аккаунты по типу продукта

| Тип продукта | Обязательные Credentials |
|--------------|------------------------|
| Website | Domain Account, Hosting, DNS (Cloudflare), Database, Mail Service |
| Mobile App | App Store Account, Google Play Account, Backend Server, Database, Push Service |
| CRM | Server, Database, Mail Service, Storage |

Система проверяет наличие этих Credential-записей при переходе со стадии Creating на Development.

## 10. Сводка различий Product vs Extension

| Параметр | Product | Extension |
|----------|---------|-----------|
| Масштаб | Крупный результат (недели–месяцы) | Доработка (часы–недели) |
| Жизненный цикл | 8 стадий с полными Stage Gates | 4 стадии, упрощённые Stage Gates |
| Scrum внутри | Полноценный: спринты, планирование | Мини: простая kanban-доска |
| Шаблоны задач | Загружаются автоматически по типу | Задачи создаются вручную PM |
| Влияние на статус проекта | Определяет статус проекта | Влияет, если в Development |
| Обязательный Order | ✅ | ✅ (даже для Micro) |
| Бонус | ✅ | ✅ |
| Привязка к Product | — (сам является Product) | Опционально привязан к Product |
| Чеклисты | Security, Quality, Deployment | Чеклист приёмки PM |
