# Жизненный цикл проекта

## Обзор

Этот документ описывает полный жизненный цикл проекта в NBOS Platform — от первого обращения клиента (Lead) до архивации проекта. Каждый этап включает описание действий, ответственных ролей, обязательных условий перехода (Stage Gates) и автоматических событий.

---

## 2. Фаза 1: Lead (Лид)

### Описание
Потенциальный клиент обращается в Neetrino через один из каналов: Instagram, Facebook, сайт, холодный звонок, партнёрская рекомендация.

### Процесс

```
Обращение → New → Contact Established → Qualification (MQL)
                                              ↓
                                 ┌─────────────┴─────────────┐
                                 ↓                           ↓
                              SPAM / Frozen            Quality Lead (SQL)
                              (лид не подходит)              ↓
                                                     Переход в Deals
```

### Участники и роли

| Роль | Действие |
|------|----------|
| **Marketing** | Генерация лидов через рекламу и каналы |
| **Seller** | Первый контакт, квалификация, классификация лида |

### Этапы лида:

| Этап | Описание |
|------|----------|
| New | Лид только поступил, ещё не обработан |
| Didn't Get Through | Не удалось связаться |
| Contact Established | Связь установлена, идёт первичное общение |
| Qualification (MQL) | Маркетингово квалифицирован, идёт оценка потребности |
| SPAM | Нерелевантный лид, спам |
| Frozen | Лид заморожен (не готов сейчас, но потенциально интересен) |
| **Quality Lead (SQL)** | Лид квалифицирован: заинтересован в конкретном продукте/услуге |

### Stage Gate: Lead → Deal
- Лид получил статус SQL (Sales Qualified Lead)
- Создан Contact в системе (или найден существующий)
- Определён интерес клиента (какой продукт/услуга)

### Автоматика:
- При SQL → автоматически создаётся Contact (если новый клиент)
- Лид конвертируется в Deal в CRM

## 3. Фаза 2: Deal (Сделка / Продажа)

### Описание
Активная работа продавца с квалифицированным клиентом. Задача — довести до подписания договора и получения первого платежа.

### Этапы сделки:

| Этап | Описание | Ответственный |
|------|----------|--------------|
| Start a Conversation | Начало переговоров | Seller |
| Discuss Needs | Обсуждение потребностей, выяснение требований | Seller |
| Meeting | Встреча (онлайн или офлайн) | Seller |
| Can We Do It? | Внутренняя оценка: можем ли реализовать | Seller + PM/Tech Lead |
| Send Offer | Отправка коммерческого предложения клиенту | Seller |
| Get Answer | Ожидание ответа клиента | Seller |
| Deposit & Contract | Подписание договора, выставление первого счёта | Seller + Finance |
| Creating | Первый платёж получен, запуск подготовки проекта | PM |
| Get Final Pay | (для Classic) Получение финального платежа | Finance |
| Maintenance Offer | Предложение подписки на техобслуживание | Seller |

### Терминальные состояния:
- **Deal Won** — сделка успешно закрыта
- **Failed** — сделка проиграна

### Stage Gate: Deal → Order + Project

Обязательные условия для перехода сделки в заказ:

| Условие | Описание |
|---------|----------|
| Первый платёж получен | Invoice создан → отправлен → оплачен → Payment подтверждён Finance Director |
| Company заполнена | Юрлицо клиента с реквизитами для биллинга |
| Тип оплаты определён | Classic (50/50 или 30/30/40) или Subscription |
| Сумма согласована | Total amount в Deal заполнен |

### Автоматика:
- При статусе Deal Won → автоматически создаётся **Order** в системе
- Автоматически создаётся **Project** в Projects Hub (или привязывается к существующему, если это Extension Deal)
- Автоматически создаётся первый **Product** в проекте (на основе данных сделки)

## 4. Фаза 3: Order Creation (Создание заказа)

### Описание
Момент превращения сделки в реальный заказ. Это финансовая «точка правды».

### Что создаётся:

| Сущность | Описание |
|----------|----------|
| **Order** | Финансовая единица: сумма, тип оплаты, привязка к проекту |
| **Project** | Контейнер в Projects Hub (если новый клиент/бренд) |
| **Product** | Первый продукт в проекте (Website, App и т.д.) |
| **Invoice** | Первый счёт (если ещё не создан на этапе Deal) |

### Участники:
- **Finance Director** — подтверждает оплату, контролирует финансовую сторону
- **Seller** — передаёт информацию PM
- **PM** — принимает проект в работу

---

## 5. Фаза 4: Creating / Setup (Подготовка)

### Описание
PM заполняет данные проекта, создаёт аккаунты, покупает домен, настраивает окружение. Все доступы сохраняются в Credentials.

### Чеклист подготовки (пример для Website):

| Действие | Ответственный | Обязательно |
|----------|--------------|-------------|
| Заполнить описание проекта | PM | ✅ |
| Создать аккаунт хостинга | Tech / PM | ✅ |
| Купить домен | Tech / PM | ✅ |
| Настроить DNS (Cloudflare) | Tech / PM | ✅ |
| Создать базу данных | Tech / Dev | ✅ |
| Настроить почтовый сервис | Tech / PM | По необходимости |
| Настроить среду разработки | Dev | ✅ |
| Сохранить ВСЕ пароли в Credentials | PM / Tech | ✅ |
| Создать запись Domain в системе | PM | ✅ |
| Загрузить ТЗ и материалы в Drive | PM | ✅ |

### Stage Gate: Creating → Development
Система проверяет:
1. Все обязательные Credential-записи для данного типа продукта созданы
2. У каждой Credential-записи заполнены логин и пароль
3. Запись Domain создана и привязана к проекту
4. Описание проекта заполнено

**Блокировка:** если хотя бы одно условие не выполнено — переход на Development заблокирован. Система показывает список невыполненных условий.

### Участники:
- **PM** — координация, заполнение данных
- **Tech Specialist** — создание аккаунтов, покупка домена
- **Developer** — настройка среды разработки

---

## 6. Фаза 5: Development (Разработка)

### Описание
Основной рабочий этап. Задачи создаются (из шаблонов + кастомные), организуются в спринты, выполняются командой.

### Процесс

```
Спринт 1                 Спринт 2                 Спринт N
┌────────────────┐       ┌────────────────┐       ┌────────────────┐
│ Планирование   │       │ Планирование   │       │ Планирование   │
│ Выполнение     │──────►│ Выполнение     │──────►│ Выполнение     │
│ Review         │       │ Review         │       │ Review         │
│ Демо клиенту   │       │ Демо клиенту   │       │ Демо клиенту   │
└────────────────┘       └────────────────┘       └────────────────┘
```

### Оплата во время Development (Subscription модель):
При Subscription-модели клиент продолжает оплачивать ежемесячно во время разработки. Если разработка занимает 3 месяца, клиент оплатит 3 invoice за Development. Платежи приостанавливаются, если Neetrino не уложились в срок, и возобновляются после сдачи.

### Участники:
- **PM** — планирование спринтов, контроль, демо клиенту
- **Developer** — выполнение задач
- **Designer** — дизайн, UI/UX
- **QA** — тестирование (на поздних спринтах)

### Stage Gate: Development → QA
- Все задачи текущего этапа закрыты (статус Done)
- .env файл сохранён в Credentials
- Deployment checklist выполнен
- PM подтверждает готовность к QA

---

## 7. Фаза 6: QA (Контроль качества)

### Описание
Проверка продукта по чеклистам безопасности, качества и готовности к production.

### Процесс:
1. QA/PM проходит Security Checklist
2. QA/PM проходит Quality Checklist
3. PM проходит Deployment Readiness Checklist
4. При обнаружении проблем — создаются задачи на исправление → возврат в Development (частично)
5. После успешного прохождения всех чеклистов → переход на Transfer

### Участники:
- **QA** — тестирование, прохождение чеклистов
- **PM** — координация, проверка deployment
- **Developer** — исправление найденных проблем

### Stage Gate: QA → Transfer
- ВСЕ пункты Security Checklist пройдены
- ВСЕ пункты Quality Checklist пройдены
- Deployment на production выполнен и проверен
- Нет открытых критических задач

---

## 8. Фаза 7: Delivery / Transfer (Передача)

### Описание
Готовый продукт демонстрируется клиенту. Клиент тестирует, даёт обратную связь, после чего принимает работу.

### Процесс:

```
PM демонстрирует продукт клиенту
         ↓
Клиент тестирует (3–7 дней)
         ↓
    ┌────┴────┐
    ↓         ↓
 Замечания    Принято ✅
    ↓
 Финальные правки
    ↓
 Повторная проверка
    ↓
 Принято ✅
```

### Участники:
- **PM** — координация, коммуникация с клиентом
- **Developer** — финальные правки
- **Клиент** — тестирование и acceptance

### Stage Gate: Transfer → Done
- Клиент подтвердил приёмку (Acceptance)
- Все замечания исправлены
- Финальная версия на production

---

## 9. Фаза 8: Financial Close (Финансовое закрытие)

### Описание
Финансовое завершение заказа. Зависит от типа оплаты.

### Classic (50/50 или 30/30/40):

```
Первый платёж (50% или 30%) — получен до начала
         ↓
[Промежуточный платёж (30%) — для схемы 30/30/40]
         ↓
Финальный платёж (50% или 40%) — после сдачи проекта
         ↓
Order status → Fully Paid
```

**Важно:** финальный платёж получается ПОСЛЕ сдачи проекта клиенту. Без подтверждения Finance Director, что деньги получены, Order не переходит в Fully Paid.

### Subscription:

```
Ежемесячные платежи во время Development (3 месяца)
         ↓
Сдача проекта
         ↓
Development-платежи прекращаются
         ↓
Начинается Maintenance-подписка
         ↓
Клиент платит: Development amount + Maintenance amount
(например: 100,000 + 80,000 = 180,000/месяц)
         ↓
Со временем Development-часть завершается (12 месяцев)
         ↓
Остаётся только Maintenance
```

### Участники:
- **Finance Director** — подтверждение оплат, закрытие Order
- **Seller** — коммуникация с клиентом по оплате

## 10. Фаза 9: Bonus Trigger (Начисление бонусов)

### Описание
После финансового закрытия (или после сдачи проекта для Subscription) автоматически создаются бонусные записи.

### Момент триггера:

| Тип оплаты | Когда начисляется бонус |
|-------------|------------------------|
| Classic | После получения ПОЛНОЙ суммы (все Invoice оплачены) |
| Subscription | После сдачи проекта, когда Dev-платежи получены и клиент перешёл на Maintenance |

### Кому начисляется:

| Роль | Процент | Источник процента |
|------|---------|-------------------|
| **Developer** | 7% / 10% / 15% | Зависит от типа проекта (WL / Mix / Custom Code) |
| **PM** | Фиксированный % | Определяется CEO |
| **Designer** | Фиксированный % | Определяется CEO (если участвовал) |
| **Seller** | 5–10% | Зависит от источника лида |

### Holdback (удержание):

```
Бонус начислен (Earned)
         ↓
80% → Active (готов к выплате в следующем payroll)
20% → Holdback (удержание на 14–30 дней)
         ↓
Через 14–30 дней (если нет clawback)
         ↓
Holdback → Active
         ↓
Выплачивается в следующем payroll cycle
```

Holdback защищает компанию от ситуаций, когда клиент возвращает деньги или возникает спор.

### Участники:
- **Система** — автоматическое создание Bonus Entry
- **CEO** — контроль, утверждение (для нестандартных случаев)
- **Finance Director** — выплата бонусов в payroll

## 11. Фаза 10: Maintenance (Техобслуживание)

### Описание
После сдачи проекта Seller предлагает клиенту подписку на техобслуживание. Если клиент соглашается, создаётся Subscription Contract.

### Процесс:

```
Проект сдан
     ↓
Seller предлагает Maintenance подписку
     ↓
┌────┴────┐
↓         ↓
Отказ     Согласен
          ↓
     Subscription Contract создан
          ↓
     Ежемесячный биллинг:
     1-го числа → Invoice выставлен
     → Уведомление бухгалтеру (создать в госсистеме)
     → Уведомление клиенту (WhatsApp)
     → Если не оплачено → Overdue → повторные напоминания
     → Finance Director подтверждает оплату
```

### Что входит в Maintenance:
- Исправление ошибок и проблем (в рамках подписки, бесплатно)
- Техническая поддержка
- Мониторинг работоспособности
- Обновления безопасности

### Что НЕ входит (оформляется как Extension):
- Новые функции
- Изменение дизайна
- Добавление страниц/разделов
- Интеграция новых сервисов

### Эскалация проблем:
Если клиент злоупотребляет поддержкой (чрезмерно частые обращения без реальных проблем) → PM сообщает CEO → CEO принимает решение: продолжить обслуживание или отказать.

### Участники:
- **Seller** — продажа подписки, коммуникация
- **PM** — координация техобслуживания
- **Developer** — исправление проблем
- **Finance Director** — биллинг, контроль оплат
- **CEO** — эскалации

---

## 12. Фаза 11: Extension Cycle (Цикл доработок)

### Описание
Клиент заказывает дополнительную работу (новую функцию, улучшение). Этот цикл может повторяться многократно.

### Процесс:

```
Клиент запрашивает доработку
         ↓
Seller создаёт Extension Deal в CRM
         ↓
Deal проходит pipeline (упрощённый)
         ↓
Оплата получена → Order создан
         ↓
Extension создана в проекте
         ↓
Development → QA → Done
         ↓
Bonus начислен
         ↓
Клиент может заказать ещё одну доработку → цикл повторяется
```

### Влияние на проект:
- Если Extension в Development → проект появляется в табе Development (параллельно с Maintenance)
- После завершения Extension → если нет других активных разработок → проект остаётся только в Maintenance

### Участники:
- **Seller** — продажа доработки
- **PM** — организация работы
- **Developer** — выполнение
- **Finance Director** — биллинг

---

## 13. Фаза 12: Close / Archive (Закрытие / Архивация)

### Описание
Проект архивируется, когда клиент прекращает сотрудничество: отменяет подписку и не заказывает новую работу.

### Условия архивации:
- Все Products в статусе Done или Lost
- Нет активных Extensions
- Subscription Contract в статусе Cancelled или Expired
- Нет неоплаченных Invoice

### Что происходит при архивации:
- Проект перемещается в таб Closed/Archived
- Все данные **сохраняются** полностью (задачи, пароли, файлы, история)
- Audit Log фиксирует архивацию
- Проект можно «реактивировать» при необходимости (новый заказ от клиента)

### Участники:
- **PM** — инициирует архивацию
- **CEO** — подтверждает (для проектов с историей)

## 15. Граф переходов состояний

```
[Lead]
  │ SQL квалификация
  ▼
[Deal]
  │ First Payment received
  ▼
[Order + Project + Product]
  │ PM fills data, creates accounts
  ▼
[Product: Creating]
  │ All credentials saved, domain ready
  ▼
[Product: Development]
  │ All tasks done, deployment ready
  ▼
[Product: QA]
  │ All checklists passed
  ▼
[Product: Transfer]
  │ Client acceptance
  ▼
[Product: Done]
  │
  ├── Classic: Final payment → Order Fully Paid → Bonus
  │
  └── Subscription: Dev payments done → Maintenance starts → Bonus
        │
        ▼
  [Maintenance Phase]
        │
        ├── Client requests extension → [Extension Cycle] → back to Maintenance
        │
        └── Client cancels subscription → [Archive]
```

## 16. Ключевые правила жизненного цикла

1. **Проект создаётся только при получении первого платежа.** Нет оплаты — нет проекта.
2. **Stage Gates обязательны.** Система физически блокирует переход между стадиями до выполнения условий.
3. **Бонусы начисляются только после подтверждённой оплаты.** Для Classic — после полной оплаты. Для Subscription — после сдачи проекта.
4. **Holdback 20% на 14–30 дней** для защиты от возвратов и споров.
5. **Extension = полноценный заказ.** Даже микро-доработка проходит через Deal → Order → Development → Bonus.
6. **Проект никогда не удаляется.** Архивация сохраняет все данные.
7. **Один проект может быть в нескольких фазах одновременно:** основной Product в Maintenance + Extension в Development.
