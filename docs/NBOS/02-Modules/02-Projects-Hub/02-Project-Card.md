# Карточка проекта (Project Card)

## Обзор

При открытии проекта из списка Projects Hub пользователь попадает на **карточку проекта** — детальную страницу, разделённую на табы/секции. Каждый таб представляет определённый аспект проекта. Доступность табов определяется ролью пользователя (RBAC).

---

## 1. Overview (Обзор)

Главная страница проекта с ключевой информацией.

### Поля:

| Поле | Тип | Описание |
|------|-----|----------|
| Название проекта | String | Имя бренда / бизнеса |
| Статус | Computed | Вычисляемый из статусов Products/Extensions (бейджи: Development, Maintenance, Closed) |
| Тип проекта | Enum | White Label / Mix / Custom Code |
| Контакт | Link → Contact | Основной контакт (клиент) |
| Компания | Link → Company | Юрлицо для биллинга |
| Seller | Link → Employee | Ответственный продажник |
| PM | Link → Employee | Ответственный проект-менеджер |
| Дедлайн | Date | Общий дедлайн проекта |
| Описание | Rich Text | Описание проекта, особенности, технические требования |
| Дата создания | DateTime | Когда проект был создан в системе |

### Виджеты на Overview:

| Виджет | Описание |
|--------|----------|
| **Продукты** | Мини-карточки всех Products с текущим статусом и прогрессом |
| **Активные задачи** | Счётчик открытых задач / общее количество |
| **Финансовый сводка** | Общая сумма заказов, оплачено, остаток (видна CEO, Finance) |
| **Ближайшие дедлайны** | Список ближайших дедлайнов по продуктам/доработкам |
| **Последние действия** | 5 последних событий в проекте (из Audit Log) |

### Доступ:
Все роли, имеющие доступ к проекту, видят Overview. Финансовый виджет скрыт для Developer, Designer, QA.

---

## 2. Products (Продукты)

Список всех продуктов, которые разрабатываются или уже разработаны в рамках этого проекта.

### Отображение карточки продукта в списке:

| Поле | Описание |
|------|----------|
| Название | "Website", "Mobile App", "CRM System" и т.д. |
| Тип | Website, Mobile App, CRM, Logo, SMM, SEO, Other |
| Статус | New, Creating, Development, QA, Transfer, On Hold, Done, Lost |
| Прогресс-бар | Визуальный индикатор этапа (от 0% до 100%) |
| Дедлайн | Дата завершения |
| PM | Ответственный проект-менеджер |
| Кол-во задач | Открытые / Всего |

### Действие при клике:
Открывается **детальная карточка продукта** со своей scrum/kanban-доской задач, стадиями (Stage Gates), прогресс-баром, чеклистами. Подробно описано в `03-Products-and-Extensions.md`.

### Кто может создавать продукты:
CEO, PM, Seller (при создании проекта из Deal).

### Доступ:
PM, Developer, Designer, QA, CEO. Seller видит список, но не детальное содержимое scrum-досок.

---

## 3. Extensions (Доработки)

Список всех доработок (дополнительных работ) по данному проекту.

### Отображение карточки доработки в списке:

| Поле | Описание |
|------|----------|
| Название | "Добавить личный кабинет", "Обновить дизайн главной" |
| Размер | Micro / Small / Medium / Large |
| Статус | New, Development, QA, Done, Lost |
| Привязка к продукту | К какому продукту относится (опционально) |
| Сумма | Стоимость доработки (из связанного Order) |
| Дедлайн | Дата завершения |
| Исполнитель | Ответственный разработчик / PM |

### Действие при клике:
Открывается **детальная карточка доработки** с мини scrum/kanban-доской задач, чеклистом приёмки.

### Кто может создавать доработки:
CEO, PM, Seller (через Extension Deal в CRM).

### Доступ:
PM, Developer, Designer, QA, CEO. Seller видит список и суммы. Finance Director видит суммы и статус оплаты.

## 4. Orders (Заказы)

Финансовая сторона проекта: все коммерческие заказы (Product Orders, Extension Orders).

### Отображение:

| Поле | Описание |
|------|----------|
| Номер заказа | Уникальный идентификатор / ссылка на сделку |
| Тип | Product / Extension |
| Привязка | К какому продукту/доработке |
| Сумма | Общая сумма заказа |
| Тип оплаты | Classic 50/50, Classic 30/30/40, Subscription Monthly |
| Прогресс оплаты | Визуальный бар: оплачено / осталось |
| Статус | Active, Partially Paid, Fully Paid, Closed, Cancelled |
| Дедлайн | Дата полной оплаты |

### Вложенная информация (при раскрытии заказа):
- Список Invoices по этому заказу с их статусами
- Список Payments (факты оплат)
- Бонусные записи (Bonus Entries), привязанные к заказу
- Партнёр-реферал и его процент (если есть)

### Кто может создавать:
Заказы создаются автоматически при закрытии Deal в CRM. Ручное создание — CEO, Finance Director.

### Доступ:
CEO, Finance Director, Seller (свои заказы). PM видит только статус (без сумм). Developer и QA не видят этот таб.

---

## 5. Finance (Финансы)

Проектный P&L (Profit & Loss) — полная финансовая картина проекта.

### Секция «Доходы» (Revenue):

| Показатель | Описание |
|------------|----------|
| Доход от заказов | Суммы по всем Product & Extension Orders |
| Доход от подписки | Сумма полученных платежей по Subscription |
| Сквозные платежи | Деньги, полученные от клиента за домены, сервисы (pass-through) |
| **Итого доход** | Сумма всех поступлений |

### Секция «Расходы» (Expenses):

| Показатель | Описание |
|------------|----------|
| Затраты на домены | Покупка / продление доменов для этого проекта |
| Затраты на сервисы | Хостинг, облачные сервисы, лицензии |
| Партнёрские выплаты | Процент партнёру-рефералу |
| Бонусы команде | Бонусы Seller, PM, Developer, Designer |
| Прочие затраты | Другие расходы, привязанные к проекту |
| **Итого расходы** | Сумма всех затрат |

### Секция «Маржинальность»:

| Показатель | Формула |
|------------|---------|
| Валовая прибыль | Доход − Расходы |
| Маржинальность % | (Валовая прибыль / Доход) × 100% |

### Список инвойсов:
Все Invoice, привязанные к проекту. Фильтры по типу (Development, Extension, Subscription, Domain), по статусу (Paid, Overdue, Pending).

### Список платежей:
Все Payment, привязанные к инвойсам проекта. Хронологический порядок.

### Доступ:
CEO, Finance Director. Seller видит только суммы своих заказов. Остальные роли не видят этот таб.

## 6. Subscription (Подписка)

Детали подписки на техобслуживание, если она активна.

### Поля:

| Поле | Описание |
|------|----------|
| Статус подписки | Active, Paused, Cancelled, Expired |
| Тип | Maintenance Only, Development + Maintenance |
| Сумма в месяц | Сумма ежемесячного платежа |
| День биллинга | Число месяца, когда выставляется счёт (1–28) |
| Дата начала | Когда подписка была активирована |
| Дата окончания | Null = бессрочная |
| Налоговый статус | Tax / Tax-Free |

### Платёжная сетка (Payment Grid):

Горизонтальная таблица по месяцам текущего и следующего года:

```
         Янв    Фев    Мар    Апр    Май    Июн    ...
Сумма:   80,000 80,000 80,000 80,000 80,000 80,000
Статус:  ✅     ✅     ✅     ⏳     —      —
```

Цвета ячеек:
- 🟢 Зелёный — оплачено
- 🟡 Жёлтый — ожидание (счёт выставлен)
- 🔴 Красный — просрочено
- ⬜ Серый — будущий период

### История изменений суммы:
Если сумма подписки менялась — показывается история с датами изменений.

### Доступ:
CEO, Finance Director, Seller. PM видит статус подписки (Active/Paused) без сумм.

## 7. Tasks (Задачи)

Агрегированный вид ВСЕХ задач по всему проекту — из всех продуктов, доработок, а также самостоятельных задач проекта.

### Фильтры:

| Фильтр | Описание |
|--------|----------|
| По продукту | Задачи конкретного продукта / доработки |
| По статусу | Backlog, To Do, In Progress, Review, Done, Cancelled |
| По исполнителю | Задачи конкретного сотрудника |
| По приоритету | Critical, High, Normal, Low |
| По спринту | Задачи текущего / прошлого / следующего спринта |
| По дедлайну | Просроченные, на этой неделе, на следующей |

### Виды отображения:

| Вид | Описание |
|-----|----------|
| **Kanban** | Столбцы по статусам: Backlog → To Do → In Progress → Review → Done |
| **List** | Список задач с сортировкой и группировкой |
| **My Plan** | Только задачи текущего пользователя, отсортированные по приоритету и дедлайну |
| **Timeline / Gantt** | Задачи на временной шкале (для PM) |

### Создание задач:
Задачи создаются внутри конкретного Product или Extension (через их scrum/kanban-доски). Также можно создать «общую задачу проекта» не привязанную к конкретному продукту.

### Доступ:
PM (полный доступ), Developer (свои задачи + задачи команды), Designer, QA (свои задачи), CEO (read-only обзор). Seller видит только задачи, связанные с его действиями (например, «Подготовить KP»).

---

## 8. Support (Поддержка)

Все тикеты клиентской поддержки, связанные с этим проектом.

### Отображение:

| Поле | Описание |
|------|----------|
| Номер тикета | Уникальный ID |
| Категория | Incident, Service Request, Change Request, Problem |
| Приоритет | P1 Critical, P2 High, P3 Normal |
| Статус | New, Triaged, Assigned, In Progress, Resolved, Closed, Reopened |
| Описание | Краткое описание проблемы |
| Исполнитель | Кто работает над тикетом |
| Дата создания | Когда тикет создан |
| SLA | Дедлайн реакции / решения |

### Фильтры:
По статусу, по приоритету, по категории, по исполнителю, по дате.

### Действия:
- Создать тикет
- Назначить исполнителя
- Конвертировать Change Request в Extension Deal (если работа платная)
- Закрыть тикет с описанием решения

### Доступ:
PM, QA, Developer (назначенные тикеты), CEO. Support Specialist (если есть). Seller не видит тикеты.

---

## 9. Credentials (Учётные данные)

Все пароли, доступы, API-ключи и секреты, связанные с проектом.

### Категории:

| Категория | Примеры |
|-----------|---------|
| Admin | Административные панели CMS, дашборды |
| Domain | Аккаунты регистраторов доменов |
| Hosting | Серверы, хостинг-панели |
| Service | Cloudflare, CDN, мониторинг |
| App | App Store, Google Play аккаунты |
| Mail | Почтовые сервисы (Resend, SendGrid) |
| API Key | Ключи API внешних сервисов |
| Database | Neon, PlanetScale, Upstash (Redis) |
| Other | Прочие доступы |

### Отображение карточки:

| Поле | Описание |
|------|----------|
| Название | "Cloudflare Account", "Production DB" |
| Провайдер | Beget, Cloudflare, Neon, Upstash, etc. |
| URL | Ссылка для входа |
| Логин | (скрыт, показывается по клику, копирование в буфер) |
| Пароль | (скрыт, показывается по клику, копирование в буфер) |
| API Key | (скрыт, если есть) |
| Телефон | Привязанный номер телефона |
| Уровень доступа | Secret, Project Team, Department, All |
| Заметки | Дополнительная информация |

### Уровни доступа:

| Уровень | Кто видит |
|---------|-----------|
| **All** | Все сотрудники компании |
| **Department** | Все сотрудники определённого отдела |
| **Project Team** | Все участники данного проекта |
| **Secret** | Только конкретные сотрудники, указанные вручную CEO/PM |

### Аудит:
Каждый просмотр и редактирование пароля логируется в Audit Log: кто, когда, какое действие.

### Доступ:
CEO (все пароли), PM (пароли проекта, кроме Secret без разрешения), Developer (назначенные пароли), Tech Specialist (по запросу). Seller и Finance Director не видят пароли.

## 10. Drive / Assets (Файлы и документы)

Файловое хранилище проекта.

### Автоматическая структура папок:

```
📁 Project Name/
  ├── 📁 Design/
  │     ├── 📁 Mockups/
  │     ├── 📁 Assets/
  │     └── 📁 Brand/
  ├── 📁 Documents/
  │     ├── 📁 Contracts/
  │     ├── 📁 Offers/
  │     └── 📁 Technical/
  ├── 📁 Development/
  │     ├── 📁 Specs/
  │     └── 📁 Environments/
  └── 📁 Delivery/
        ├── 📁 Client Review/
        └── 📁 Final/
```

### Возможности:
- Загрузка файлов (drag-and-drop)
- Привязка файла к конкретной задаче, продукту, доработке
- Предпросмотр изображений и PDF
- Версионирование (при замене файла сохраняется предыдущая версия)
- Поиск по имени файла

### Доступ:
PM, Developer, Designer (полный доступ к файлам проекта), CEO, QA. Seller видит только папку Documents.

## 11. Domains (Домены)

Все доменные имена, привязанные к проекту.

### Отображение:

| Поле | Описание |
|------|----------|
| Доменное имя | example.com |
| Провайдер | Reg.ru, GoDaddy, Beget и т.д. |
| Дата покупки | Когда был приобретён |
| Дата истечения | Когда нужно продлить |
| Стоимость продления | Сумма продления |
| Сумма для клиента | Сколько клиент платит (может быть > стоимости) |
| Статус | Active, Expiring Soon (за 60 дней), Expired, Transferred |
| Автопродление | Включено / выключено |

### Автоматизация:
- За 60 дней до истечения → статус «Expiring Soon» + уведомление PM и Finance
- Автоматическое создание Invoice для клиента на продление
- Создание задачи для Tech Specialist на продление домена

### Доступ:
CEO, PM, Tech Specialist, Finance Director.

## 12. Chat (Чат проекта)

Проектный чат с системой топиков (аналог тем в Telegram).

### Структура топиков:

| Топик | Описание | Участники |
|-------|----------|-----------|
| **General** | Общие обсуждения по проекту | Все участники проекта |
| **Sales** | Обсуждения по сделке, коммерческие вопросы | Seller, CEO, PM |
| **Development** | Технические обсуждения, решения по разработке | PM, Developer, Designer, QA |
| **Task Chats** | Автоматические чаты по задачам (создаются при первом сообщении в задаче) | Участники задачи |

### Возможности:
- Отправка текстовых сообщений, файлов, изображений
- Упоминание участников (@mention)
- Создание задачи прямо из чата (выделить сообщение → «Создать задачу»)
- Поиск по сообщениям в рамках проекта
- Непрочитанные сообщения поднимают чат вверх в списке Messenger

### Доступ:
Определяется топиком. Developer не видит Sales-топик. Seller не видит Development-топик. General доступен всем участникам проекта.

Подробнее — в документации модуля **Messenger**.

## 13. Audit Log (Журнал изменений)

Полная история всех изменений в проекте.

### Фиксируемые события:

| Событие | Пример |
|---------|--------|
| Изменение статуса | «Product "Website" перешёл из Development в QA» |
| Изменение полей | «PM изменён с Анна → Давид» |
| Финансовые события | «Invoice #045 помечен как Paid» |
| Работа с паролями | «Developer Арсен просмотрел пароль "Production DB"» |
| Изменение доступов | «CEO добавил Developer Тигран к Secret-паролю "Master API Key"» |
| Создание/удаление | «PM создал Extension "Добавить чат-бот"» |
| Переход стадий | «Product "Website" не прошёл Stage Gate: поле "Domain Account" не заполнено» |

### Поля записи Audit Log:

| Поле | Описание |
|------|----------|
| Дата/время | Когда произошло событие |
| Пользователь | Кто совершил действие |
| Тип события | Create, Update, Delete, View, StatusChange |
| Сущность | Какой объект затронут (Product, Credential, Invoice и т.д.) |
| Старое значение | Что было до изменения |
| Новое значение | Что стало после изменения |
| IP / User Agent | Техническая информация (для security audit) |

### Фильтры:
По типу события, по пользователю, по сущности, по дате.

### Доступ:
CEO (полный доступ), Finance Director (финансовые события), PM (события по задачам и продуктам). Остальные роли не видят Audit Log.
