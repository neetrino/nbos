# Рабочий процесс Support (Workflow)

> NBOS Platform — ежедневные процедуры, эскалация и интеграция с CRM

## Назначение

Данный документ описывает **ежедневный рабочий процесс** поддержки: от утренней проверки тикетов до закрытия и коммуникации с клиентом. Также описаны процессы эскалации, связь с CRM через Change Request и будущие AI-автоматизации.

**Ключевой принцип:** поддержка работает как конвейер — каждое утро проверяем входящее, классифицируем, распределяем, контролируем SLA в течение дня, закрываем решённые вопросы вечером.

## Ежедневный рабочий процесс

### Утро: Проверка и триаж

| Шаг | Действие | Ответственный | Время |
|-----|----------|---------------|-------|
| 1 | Открыть Support Dashboard → проверить новые тикеты | PM / Support | 09:00 |
| 2 | Проверить WhatsApp-группы проектов за ночь | PM / Support | 09:00–09:15 |
| 3 | Создать тикеты из новых обращений клиентов | PM / Support | 09:15–09:30 |
| 4 | Триаж: назначить категорию и приоритет каждому новому тикету | PM / Support | 09:30–09:45 |
| 5 | Назначить исполнителей для каждого тикета | PM / Support | 09:45–10:00 |

### День: Мониторинг и коммуникация

| Шаг | Действие | Ответственный | Когда |
|-----|----------|---------------|-------|
| 6 | Мониторить SLA-дашборд — отслеживать жёлтые и красные зоны | PM / Support | Каждые 2–3 часа |
| 7 | Проверять статусы тикетов In Progress — запрашивать обновления у разработчиков | PM / Support | Середина дня |
| 8 | Коммуницировать с клиентами — информировать о прогрессе | PM / Support | По мере обновлений |
| 9 | Проверять SLA-предупреждения — реагировать на приближающиеся дедлайны | PM / Support | При получении уведомления |
| 10 | Обрабатывать P1 Critical тикеты — немедленная реакция | PM / Support + Dev | Немедленно при поступлении |

### Вечер: Закрытие и отчётность

| Шаг | Действие | Ответственный | Время |
|-----|----------|---------------|-------|
| 11 | Проверить тикеты в Resolved — подтвердить закрытие или переоткрыть | PM / Support | 17:00–17:15 |
| 12 | Связаться с клиентами по решённым тикетам — запросить подтверждение | PM / Support | 17:15–17:30 |
| 13 | Проверить SLA-отчёт за день — сколько решено, сколько просрочено | PM / Support | 17:30–17:45 |
| 14 | Подготовить заметки на утро (если есть незакрытые P1/P2) | PM / Support | 17:45–18:00 |

## Полный цикл обработки тикета

### Сценарий 1: Incident (типичный баг)

```
1. Клиент пишет в WhatsApp: «Сайт не загружается»
       │
2. PM получает уведомление в NBOS (интеграция с Messenger)
       │
3. PM создаёт тикет Support:
   - Категория: Incident
   - Приоритет: P1 Critical
   - Проект: Website клиента X
       │
4. PM назначает разработчика (кто знает проект)
   → Разработчик получает push-уведомление
       │
5. PM отвечает клиенту в WhatsApp:
   «Приняли. Разработчик уже подключился, разбираемся.»
   → SLA First Response: ✅ выполнен (15 мин)
       │
6. Разработчик диагностирует проблему → переводит в In Progress
       │
7. Разработчик исправляет проблему → деплоит фикс
       │
8. Разработчик переводит тикет в Resolved
   → PM получает уведомление
       │
9. PM пишет клиенту: «Проблема решена, проверьте, пожалуйста»
       │
10. Клиент подтверждает: «Всё работает, спасибо»
       │
11. PM закрывает тикет → Closed
    → SLA Resolution: ✅ выполнен (2 часа при лимите 4–8)
```

### Сценарий 2: Service Request (небольшое изменение)

```
1. Клиент пишет: «Можно обновить номер телефона в футере?»
       │
2. PM создаёт тикет:
   - Категория: Service Request
   - Приоритет: P3 Normal
       │
3. PM назначает разработчика
       │
4. Разработчик вносит изменение (5 мин работы)
   → Resolved
       │
5. PM уведомляет клиента → Closed
```

### Сценарий 3: Change Request (новый функционал)

```
1. Клиент пишет: «Хотим добавить личный кабинет на сайт»
       │
2. PM создаёт тикет:
   - Категория: Change Request
   - Billable: Да
       │
3. PM уведомляет Seller:
   «Клиент X запросил новый функционал — личный кабинет»
       │
4. Seller связывается с клиентом:
   - Уточняет требования
   - Оценивает стоимость
   - Создаёт Extension Deal в CRM
       │
5. Extension Deal проходит CRM pipeline:
   - Обсуждение → КП → Согласование → Оплата
       │
6. После оплаты:
   - Создаётся Extension в Projects Hub
   - Загружается шаблон задач
   - Разработка начинается
       │
7. После завершения разработки:
   - Extension закрывается как Done
   - Support-тикет закрывается автоматически
```

### Сценарий 4: Problem (повторяющиеся инциденты)

```
1. PM замечает: 3 инцидента за месяц по одному проекту — «ошибка 500 при оплате»
       │
2. PM создаёт Problem-тикет:
   - Привязывает 3 Incident-тикета
   - Назначает root cause analysis
       │
3. Разработчик проводит анализ:
   - Root Cause: утечка памяти при высокой нагрузке
   - Action Plan: оптимизация обработки платежей + настройка auto-scaling
       │
4. Разработчик реализует план (серия задач)
       │
5. Верификация: 14 дней наблюдения
   - Мониторинг: ноль инцидентов
       │
6. Problem-тикет закрывается
```

### Правила конвертации сообщений в тикеты

| Тип сообщения | Действие |
|---------------|----------|
| «Не работает...», «Ошибка...», «Сломалось...» | Создать тикет → Incident |
| «Можно изменить...», «Обновите...» | Создать тикет → Service Request или Change Request (зависит от объёма) |
| «Как сделать...?», «Где найти...?» | Ответить в чате, тикет не нужен |
| «Хотим добавить...», «Нужен новый...» | Создать тикет → Change Request |
| Благодарность, общение | Ответить в чате, тикет не нужен |

## Процесс эскалации

### Уровни эскалации

| Уровень | Условие | Кто эскалирует | Кому | Действие |
|---------|---------|----------------|------|----------|
| **L1** | P1 не получил ответ за 60 мин | Система (автоматически) | PM Lead | PM Lead берёт под контроль |
| **L2** | P1 не решён за 4 часа | Система (автоматически) | PM Lead + CEO | CEO принимает решение о привлечении ресурсов |
| **L3** | P2 не решён за 1 рабочий день | Система (автоматически) | PM Lead | PM Lead проверяет блокирующие факторы |
| **L4** | P2 не решён за 2 рабочих дня | Система (автоматически) | CEO | CEO принимает решение |
| **L5** | Любой тикет просрочен по SLA | Система (автоматически) | PM + CEO | Обязательный разбор причины |

### Ручная эскалация

PM может эскалировать тикет вручную в любой момент:

| Причина | Действие |
|---------|----------|
| Разработчик не может решить | PM привлекает другого разработчика или внешнего специалиста |
| Нужно решение бизнес-уровня | PM эскалирует CEO (скидка, компенсация, отказ в обслуживании) |
| Клиент угрожает уйти | PM эскалирует CEO для личного контакта |
| Техническая проблема за пределами компетенции | PM привлекает внешнего консультанта |

## Еженедельный Support Review

Каждую неделю PM / Support Lead проводит краткий обзор:

### Повестка (15–20 мин)

| # | Пункт | Описание |
|---|-------|----------|
| 1 | **Обзор метрик** | SLA Compliance %, среднее время решения, reopen rate |
| 2 | **Просроченные тикеты** | Разбор каждого просроченного тикета — причина, план |
| 3 | **Problem-тикеты** | Статус root cause analysis по активным Problem-тикетам |
| 4 | **Проблемные проекты** | Проекты с 5+ тикетов за неделю — нужен аудит? |
| 5 | **Ресурсы** | Достаточно ли ресурсов? Нужен ли дополнительный разработчик? |
| 6 | **Процесс** | Что можно улучшить в процессе поддержки? |

### Участники

- PM (обязательно)
- CEO (при наличии эскалаций или критических вопросов)
- Tech Lead (если есть системные проблемы)

## Связь Support → CRM (Change Control)

Полная цепочка от обращения клиента до реализации нового функционала:

```
┌──────────────────────────────────────────────────────────────────┐
│                    CHANGE CONTROL FLOW                            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  SUPPORT                CRM                  PROJECTS HUB        │
│  ─────────             ─────                 ────────────        │
│                                                                  │
│  Client Request   →   Seller notified   →   Extension created    │
│       │                    │                      │              │
│  Change Request       Extension Deal         Task Template       │
│  Ticket created       in Pipeline            loaded              │
│       │                    │                      │              │
│  Ticket: Waiting      Deal Pipeline:         Development:        │
│  for CRM              Discussion →           Sprint tasks        │
│       │               Offer →                     │              │
│       │               Payment                     │              │
│       │                    │                      │              │
│  Ticket: Closed  ←   Deal Won            →  Extension Done       │
│  (auto)                                                          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### Ключевые связи

| Точка связи | Описание |
|-------------|----------|
| **Support → CRM** | PM уведомляет Seller о Change Request. Тикет ссылается на Extension Deal. |
| **CRM → Projects Hub** | После оплаты Extension Deal создаётся Extension в проекте |
| **Projects Hub → Support** | После завершения Extension (Done) → Support-тикет закрывается |
| **Billable поле** | Связывает тикет с финансовой стороной: если Billable = Да, обязательна связь с Extension Deal |

## Уведомления в рабочем процессе

### Автоматические уведомления

| Событие | Кому | Канал |
|---------|------|-------|
| Новый тикет создан | PM / Support | Push + In-app |
| Тикет назначен разработчику | Developer | Push + In-app |
| Статус тикета изменён | PM, наблюдатели | In-app |
| SLA предупреждение (25% осталось) | PM | Push + In-app |
| SLA нарушен | PM + CEO | Push + Email + In-app |
| Тикет Resolved | PM | In-app |
| Тикет Reopened | PM + Developer | Push + In-app |
| Клиент добавил комментарий | PM + назначенный Developer | Push |
| Автозакрытие через 2 дня | Клиент (через WhatsApp) | WhatsApp |

### Уведомления клиенту (через WhatsApp)

| Событие | Сообщение клиенту |
|---------|-------------------|
| Тикет принят | «Ваше обращение принято. Номер: #TICKET-XXX. Работаем над решением.» |
| Назначен исполнитель | — (не уведомляем, внутренний процесс) |
| Обновление статуса | «Обновление по вашему обращению #TICKET-XXX: [описание прогресса]» |
| Тикет решён | «Проблема по обращению #TICKET-XXX решена. Пожалуйста, проверьте и подтвердите.» |
| Ожидание подтверждения | «Напоминание: пожалуйста, подтвердите решение по #TICKET-XXX. Тикет будет автоматически закрыт через 2 дня.» |
| Тикет закрыт | «Обращение #TICKET-XXX закрыто. Если проблема повторится, напишите нам.» |

## KPI Support-процесса

| KPI | Формула | Цель | Ответственный |
|-----|---------|------|---------------|
| **SLA Compliance** | Тикетов в рамках SLA / Всего × 100% | ≥ 90% | PM / Support Lead |
| **Avg Resolution Time** | Сумма Resolution Time / Количество тикетов | < 48 ч | PM / Support Lead |
| **Reopen Rate** | Reopened / Closed × 100% | < 10% | Development Team |
| **Tickets per Project/Month** | Инцидентов / Проектов на Maintenance | < 3 | Quality indicator |
| **Customer Response Time** | Среднее время ответа клиенту | < 2 ч | PM / Support |
| **Problem Resolution Rate** | Problem-тикетов закрытых / Созданных × 100% | ≥ 80% | Tech Lead |
