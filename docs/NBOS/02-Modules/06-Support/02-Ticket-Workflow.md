# Рабочий процесс тикетов (Ticket Workflow)

> NBOS Platform — детальный жизненный цикл тикета от создания до закрытия

## Обзор

Каждый тикет в модуле Support проходит строго определённый жизненный цикл. На каждом этапе известно: кто действует, какие поля обязательны, какие автоматизации срабатывают и какие переходы возможны.

**Ключевой принцип:** тикет не может быть закрыт без документированной резолюции. Каждое решение фиксируется для будущей аналитики и предотвращения повторных инцидентов.

---

## Жизненный цикл тикета

```
  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │   New → Triaged → Assigned → In Progress → Resolved → Closed    │
  │                                    │           │          │      │
  │                                    │           │          │      │
  │                                    └───────────┼──────────┘      │
  │                                    (Reopened)  │                  │
  │                                                │                  │
  │                                        Reopened ←─────────────   │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

### Возможные переходы:

| Из статуса | В статус | Кто выполняет | Условие |
|------------|----------|---------------|---------|
| New | Triaged | PM / Support | Категория и приоритет установлены |
| Triaged | Assigned | PM / Support | Исполнитель назначен |
| Assigned | In Progress | Исполнитель | Работа начата |
| In Progress | Resolved | Исполнитель | Работа выполнена, резолюция описана |
| Resolved | Closed | PM / Auto | Клиент подтвердил или прошло N дней |
| Resolved | Reopened | PM / Support | Клиент сообщил, что проблема не решена |
| Reopened | In Progress | Исполнитель | Возобновление работы |
| Любой | Cancelled | PM | Тикет отменён (дубликат, ошибка, неактуально) |

## Детальное описание стадий

### 1. New (Новый)

**Тикет только создан** — вручную PM или автоматически из сообщения клиента.

| Параметр | Значение |
|----------|----------|
| **Кто действует** | PM / Support Specialist |
| **Что происходит** | Тикет попадает в колонку New на Support Board. PM получает уведомление |
| **Обязательные поля** | Название, описание, проект, контакт |
| **Автоматизация** | Уведомление PM проекта о новом тикете |
| **SLA** | Таймер реакции начинает отсчёт с момента создания |

**Способы создания тикета:**

| Способ | Описание |
|--------|----------|
| Ручное создание | PM нажимает «+ Тикет» на Support Board или в карточке проекта |
| Из мессенджера | PM выделяет сообщение клиента → «Создать тикет» → текст → описание |
| Из задачи | Если задача порождает инцидент — создаётся связанный тикет |
| Автоматически (будущее) | WhatsApp-сообщение клиента → автоматический тикет |

---

### 2. Triaged (Классифицирован)

**PM анализирует обращение** и устанавливает ключевые параметры.

| Параметр | Значение |
|----------|----------|
| **Кто действует** | PM / Support Specialist |
| **Что происходит** | Определение категории, приоритета, оплачиваемости |
| **Обязательные поля** | Категория (Incident/Request/Change/Problem), Приоритет (P1/P2/P3), Billable (Yes/No) |
| **Автоматизация** | SLA дедлайны рассчитываются на основе приоритета |

**Матрица классификации:**

| Обращение клиента | Категория | Billable | Следующий шаг |
|-------------------|-----------|----------|---------------|
| «Сайт не открывается» | Incident | No | Назначить разработчика |
| «Обновите номер телефона на сайте» | Service Request | No | Назначить разработчика/PM |
| «Хочу добавить онлайн-оплату» | Change Request | Yes | Создать Extension Deal |
| «Форма третий раз за месяц ломается» | Problem | No | Назначить Lead Developer |

**При установке Billable: Yes (Change Request):**

```
PM отмечает Billable: Yes
      │
      ▼
Появляется кнопка «Создать Extension Deal»
      │
      ▼
PM заполняет: ориентировочный бюджет, описание работ
      │
      ▼
Extension Deal создаётся в CRM
      │
      ▼
Тикет получает ссылку на Deal
      │
      ▼
Работа начинается ТОЛЬКО после оплаты Deal
```

---

### 3. Assigned (Назначен)

**Исполнитель определён** — тикет ожидает начала работы.

| Параметр | Значение |
|----------|----------|
| **Кто действует** | PM назначает исполнителя |
| **Что происходит** | Исполнитель получает уведомление. Тикет появляется в его My Plan |
| **Обязательные поля** | Исполнитель (Assignee) |
| **Автоматизация** | Push-уведомление исполнителю; для P1 — дополнительно SMS/звонок |

**Правила назначения:**

| Приоритет | Правило назначения |
|-----------|-------------------|
| P1 Critical | PM назначает немедленно. Если основной разработчик недоступен — назначает другого |
| P2 High | PM назначает в течение рабочего дня |
| P3 Normal | PM назначает при следующем планировании |

---

### 4. In Progress (В работе)

**Исполнитель начал работу** над тикетом.

| Параметр | Значение |
|----------|----------|
| **Кто действует** | Исполнитель (Developer, Tech Specialist и т.д.) |
| **Что происходит** | Исполнитель анализирует проблему, вносит исправления, тестирует |
| **Действия исполнителя** | Писать в чат тикета (вопросы, статус), прикладывать файлы, создавать связанные задачи |
| **Автоматизация** | Мониторинг SLA — если приближается дедлайн решения → уведомление PM |

**Если исполнитель понимает, что задача сложнее, чем ожидалось:**

1. Пишет в чат тикета: описание проблемы, примерные сроки
2. PM может изменить приоритет или переназначить
3. Если нужна помощь — PM добавляет Co-Assignee

---

### 5. Resolved (Решён)

**Работа выполнена**, ожидается подтверждение клиента.

| Параметр | Значение |
|----------|----------|
| **Кто действует** | Исполнитель переводит в Resolved |
| **Что происходит** | Исполнитель описывает резолюцию. PM уведомляет клиента |
| **Обязательные поля** | **Резолюция** — текстовое описание того, что было сделано |
| **Автоматизация** | Уведомление PM; PM отправляет подтверждение клиенту |

**Требования к резолюции:**

Резолюция — обязательное текстовое поле, которое **невозможно оставить пустым** при переводе в Resolved. Она должна содержать:

- Что было причиной проблемы (для Incident/Problem)
- Что было сделано для решения
- Какие файлы/компоненты изменены (опционально, для технических тикетов)
- Рекомендации по предотвращению в будущем (для Problem)

**Примеры резолюций:**

```
Incident: «Ошибка 500 на странице оплаты»
Резолюция: «Причина: таймаут подключения к платёжному API из-за 
истекшего API-ключа. Обновлён API-ключ в .env. Добавлен мониторинг 
срока действия ключа — напоминание за 30 дней до истечения.»

Service Request: «Обновить контактный телефон»
Резолюция: «Обновлён номер телефона в шапке и футере сайта: 
+374 XX XXX-XX-XX → +374 YY YYY-YY-YY. Кэш очищен.»
```

---

### 6. Closed (Закрыт)

**Тикет завершён** — клиент подтвердил решение или прошёл период ожидания.

| Параметр | Значение |
|----------|----------|
| **Кто действует** | PM (вручную) или система (автоматически) |
| **Что происходит** | Тикет считается завершённым и попадает в архив |
| **Условия закрытия** | Клиент подтвердил ИЛИ прошло 7 дней без ответа после Resolved |
| **Автоматизация** | Автозакрытие через 7 дней без ответа с уведомлением: «Тикет автозакрыт» |

**Правила автозакрытия:**

1. Тикет переведён в Resolved
2. Клиенту отправлено сообщение: «Проблема решена? Если да — подтвердите, если нет — опишите, что не работает»
3. Если клиент не отвечает 7 дней → система отправляет напоминание: «Тикет будет автоматически закрыт через 2 дня»
4. Если ещё 2 дня нет ответа → тикет автоматически закрывается с пометкой «Auto-closed»

---

### 7. Reopened (Переоткрыт)

**Клиент сообщил, что проблема не решена** — тикет возвращается в работу.

| Параметр | Значение |
|----------|----------|
| **Кто действует** | PM (по обратной связи от клиента) |
| **Что происходит** | Тикет возвращается в In Progress с тем же или другим исполнителем |
| **Автоматизация** | Уведомление исполнителю; SLA таймер перезапускается |
| **Максимум** | Тикет может быть переоткрыт не более 3 раз. После этого → эскалация к Head of Delivery |

**При переоткрытии:**
- PM добавляет комментарий: что именно не работает (по обратной связи клиента)
- Предыдущая резолюция сохраняется в истории
- Если тот же исполнитель — он продолжает. Если нужен другой — PM переназначает

## Специальные потоки

### Поток 1: Change Request → Extension Deal

Когда тикет классифицирован как Change Request и отмечен Billable:

```
┌─────────────────────────────────────────────────────────────────┐
│  SUPPORT                        │  CRM                         │
│                                 │                               │
│  Тикет (Change Request)         │                               │
│  Billable: Yes                  │                               │
│       │                         │                               │
│       ▼                         │                               │
│  PM: «Создать Deal»  ──────────┼──→  Extension Deal создан     │
│       │                         │         │                     │
│       │                         │         ▼                     │
│  Тикет: статус «On Hold»       │    КП → Договор → Оплата     │
│  (ожидает оплаты)              │         │                     │
│       │                         │         ▼                     │
│       │  ◄──────────────────────┼──  Deal Won + Payment         │
│       ▼                         │                               │
│  Тикет: → Assigned              │                               │
│  Extension создана              │                               │
│  Разработка начинается          │                               │
└─────────────────────────────────────────────────────────────────┘
```

**Ключевой момент:** работа по Change Request **не начинается** до получения оплаты. Тикет переходит в статус On Hold и ожидает завершения сделки в CRM.

---

### Поток 2: Incident → Problem (автоматический)

Когда один и тот же инцидент повторяется:

```
Incident #1 (Resolved)
      │
      ▼
Incident #2 (Resolved)  ← Система: «Похожий инцидент зафиксирован ранее»
      │
      ▼
Incident #3 (Created)   ← Система: «3-й инцидент по той же теме!»
      │                     Автоматически предлагает создать Problem-тикет
      ▼
Problem Ticket создан
  ├── Ссылки на все 3 инцидента
  ├── Категория: Problem
  ├── Описание: «Повторяющийся инцидент — необходим анализ корневой причины»
  └── Назначение: Lead Developer / Architect
```

**Критерии определения «того же инцидента»:**
- Один и тот же проект + один и тот же продукт
- Совпадение ключевых слов в описании (с учётом нечёткого поиска)
- PM может вручную связать инциденты как «Related»

---

### Поток 3: Эскалация при нарушении SLA

```
Тикет создан
      │
      ▼
SLA таймер запущен (на основе приоритета)
      │
      ├── 80% времени реакции прошло ──→ Уведомление PM: «SLA Warning»
      │
      ├── 100% времени реакции прошло ─→ Уведомление PM + Head of Delivery
      │                                   «SLA Breach: реакция просрочена»
      │
      ├── 80% времени решения прошло ──→ Уведомление PM + Исполнитель
      │
      └── 100% времени решения прошло ─→ Уведомление PM + Head of Delivery + CEO
                                          «SLA Breach: решение просрочено»
```

**Таблица SLA-эскалации:**

| Событие | Получатели | Канал |
|---------|-----------|-------|
| 80% реакции | PM проекта | In-app уведомление |
| 100% реакции | PM + Head of Delivery | Push + In-app |
| 80% решения | PM + Исполнитель | In-app уведомление |
| 100% решения | PM + Head of Delivery + CEO | Push + In-app + Email |

## Чат тикета

Каждый тикет может иметь собственный чат — аналогично Task Chat:

| Параметр | Описание |
|----------|----------|
| **Создание** | При первом сообщении в тикете |
| **Участники** | PM (Creator), Assignee, Co-Assignees, Observers |
| **Уведомления** | Все участники получают уведомления о новых сообщениях |
| **Вложения** | Можно прикладывать файлы, скриншоты, логи |
| **Видимость** | В карточке тикета + в проектном разделе мессенджера (Support Chats) |

### Отличие от прямого общения с клиентом:

Чат тикета — это **внутренний** чат команды. Клиент его **не видит**. Для общения с клиентом PM использует WhatsApp / основной канал коммуникации.

## Метрики и аналитика

| Метрика | Описание | Формула |
|---------|----------|---------|
| **Время первой реакции** | От создания до Triaged | `triaged_at - created_at` |
| **Время назначения** | От создания до Assigned | `assigned_at - created_at` |
| **Время решения** | От создания до Resolved | `resolved_at - created_at` |
| **SLA Compliance** | % тикетов, решённых в рамках SLA | `(тикеты в SLA / всего) × 100%` |
| **Reopen Rate** | % переоткрытых тикетов | `(переоткрытые / всего) × 100%` |
| **Тикетов в месяц** | Объём обращений | Кол-во тикетов за период |
| **Среднее время решения** | По категории и приоритету | Средняя `resolved_at - created_at` |
| **Распределение по категориям** | Incident / Request / Change / Problem | Кол-во по каждой категории |

### Аналитика по проектам:

PM и CEO могут видеть рейтинг проектов по количеству обращений:

| Проект | Тикетов/мес | Avg решение | SLA % | Проблемные |
|--------|:-----------:|:-----------:|:-----:|:----------:|
| Brand X | 12 | 4ч | 92% | 1 |
| Brand Y | 3 | 2ч | 100% | 0 |
| Brand Z | 25 | 6ч | 78% | 3 |

Такая аналитика помогает выявить «проблемные» проекты, требующие архитектурного внимания или разговора с клиентом.

## Автоматизации

| Триггер | Действие |
|---------|----------|
| Тикет создан | Уведомление PM проекта |
| Тикет P1 создан | Push-уведомление PM + Head of Delivery |
| Категория = Change Request + Billable | Появляется кнопка «Создать Extension Deal» |
| Исполнитель назначен | Push-уведомление исполнителю |
| 80% SLA истекло | Warning-уведомление |
| 100% SLA истекло | Breach-уведомление с эскалацией |
| Тикет в Resolved 7 дней | Напоминание клиенту |
| Тикет в Resolved 9 дней | Автозакрытие |
| 3-й похожий инцидент | Предложение создать Problem-тикет |
| Тикет Reopened 3+ раз | Эскалация к Head of Delivery |
| Тикет Closed | Обновление SLA-статистики проекта |
