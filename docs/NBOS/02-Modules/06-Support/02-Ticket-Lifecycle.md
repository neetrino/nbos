# Жизненный цикл тикета

> NBOS Platform — от обращения клиента до подтверждения решения

## Назначение

Каждый тикет поддержки проходит чёткий жизненный цикл со определёнными стадиями, ответственными и правилами перехода. Это обеспечивает прозрачность для клиента, контроль SLA и аналитику качества поддержки.

**Ключевой принцип:** ни один тикет не должен «зависнуть» — система автоматически отслеживает SLA, эскалирует просроченные тикеты и напоминает ответственным о необходимости действий.

## Стадии тикета

```
New → Triaged → Assigned → In Progress → Resolved → Closed
                                            ↑          │
                                            └──────────┘
                                            (Reopened)
```

### Подробное описание стадий

| # | Стадия | Описание | Ответственный | Действия |
|---|--------|----------|---------------|----------|
| 1 | **New** | Тикет создан, ещё не рассмотрен | — | SLA-таймер запущен |
| 2 | **Triaged** | Категория и приоритет назначены | PM / Support | Классификация, оценка |
| 3 | **Assigned** | Назначен конкретному исполнителю | PM / Support | Исполнитель уведомлён |
| 4 | **In Progress** | Исполнитель работает над решением | Developer | Фикс, тестирование |
| 5 | **Resolved** | Решение применено, ожидаем подтверждения клиента | PM / Support | Клиент уведомлён |
| 6 | **Closed** | Клиент подтвердил решение или тикет закрыт автоматически | PM / Система | Тикет архивируется |
| 7 | **Reopened** | Клиент сообщил, что проблема не решена | Клиент / PM | Возвращается в In Progress |

### Правила переходов

| Переход | Условие | Кто инициирует |
|---------|---------|----------------|
| New → Triaged | PM назначил категорию и приоритет | PM / Support |
| Triaged → Assigned | Назначен исполнитель | PM / Support |
| New → Assigned | Можно пропустить Triage, если категория/приоритет очевидны | PM / Support |
| Assigned → In Progress | Исполнитель начал работу | Developer |
| In Progress → Resolved | Фикс применён и протестирован | Developer |
| Resolved → Closed | Клиент подтвердил или прошло N дней без ответа | PM / Система |
| Resolved → Reopened | Клиент сообщил, что проблема сохраняется | PM |
| Reopened → In Progress | Исполнитель возвращается к работе | Developer |
| Любой → Closed | PM принимает решение закрыть (дубликат, неактуально) | PM |

## Процесс триажа (Triage)

Триаж — ключевой этап, определяющий дальнейшую судьбу тикета. PM должен ответить на несколько вопросов:

### Дерево решений

```
Обращение клиента
       │
       ▼
Это баг / поломка?
       │
  ┌────┴────┐
  Да        Нет
  │         │
  ▼         ▼
Incident    Это запрос на настройку
            существующего функционала?
                  │
             ┌────┴────┐
             Да        Нет
             │         │
             ▼         ▼
      Service      Это запрос на
      Request      новый функционал?
                        │
                   ┌────┴────┐
                   Да        Нет
                   │         │
                   ▼         ▼
             Change       Повторяющаяся
             Request      проблема (3+ раз)?
                               │
                          ┌────┴────┐
                          Да        Нет
                          │         │
                          ▼         ▼
                       Problem    Incident
                                  (разовый)
```

### Действия по категориям

| Категория | Действие PM после триажа |
|-----------|--------------------------|
| **Incident** | Назначить приоритет → назначить разработчика → отслеживать SLA |
| **Service Request** | Назначить приоритет (обычно P3) → назначить разработчика |
| **Change Request** | Уведомить Seller → Seller создаёт Extension Deal → отдельный процесс CRM |
| **Problem** | Создать Problem-тикет → привязать связанные Incidents → назначить root cause analysis |

### Change Request — отдельный путь

Если тикет классифицирован как Change Request, он выходит из стандартного потока Support:

```
Change Request тикет
       │
       ▼
PM уведомляет Seller
       │
       ▼
Seller связывается с клиентом
       │
       ▼
Seller создаёт Extension Deal в CRM
       │
       ▼
Стандартный CRM-pipeline
       │
       ▼
При закрытии Extension Deal → Support-тикет закрывается автоматически
```

## SLA-трекинг

### Механизм отслеживания

SLA (Service Level Agreement) отслеживается автоматически с момента создания тикета.

| Метрика SLA | Описание | Расчёт |
|-------------|----------|--------|
| **First Response Time** | Время от создания тикета до первого ответа (Triage) | Создание → Triaged |
| **Resolution Time** | Время от создания тикета до решения | Создание → Resolved |
| **Time to Assign** | Время от создания до назначения исполнителя | Создание → Assigned |
| **Work Time** | Время фактической работы над тикетом | Assigned → Resolved |

### Визуальная индикация SLA

| Состояние | Цвет | Описание |
|-----------|------|----------|
| **В норме** | 🟢 Зелёный | Оставшееся время > 50% SLA |
| **Внимание** | 🟡 Жёлтый | Оставшееся время 25–50% SLA |
| **Критично** | 🟠 Оранжевый | Оставшееся время < 25% SLA |
| **Просрочено** | 🔴 Красный | SLA нарушен |

### Правила эскалации при нарушении SLA

| Условие | Действие |
|---------|----------|
| P1 не получил первый ответ за 60 мин | Уведомление PM Lead + push-нотификация |
| P1 не решён за 4 часа | Эскалация: уведомление PM Lead + CEO |
| P2 не решён за 1 рабочий день | Уведомление PM Lead |
| P2 не решён за 2 рабочих дня | Эскалация: уведомление CEO |
| Любой тикет просрочен по SLA | Автоматическое уведомление PM и CEO |

### Паузы SLA-таймера

В некоторых случаях SLA-таймер ставится на паузу:

| Причина паузы | Описание |
|---------------|----------|
| **Ожидание информации от клиента** | PM запросил дополнительные данные — таймер на паузе до ответа |
| **Resolved (ожидание подтверждения)** | Фикс применён — таймер на паузе, пока клиент не подтвердит или не reopenит |

## Автозакрытие тикетов

Тикеты в статусе **Resolved** автоматически закрываются, если клиент не ответил в течение определённого периода:

| Правило | Описание |
|---------|----------|
| **Период ожидания** | 5 рабочих дней после перехода в Resolved |
| **Уведомление клиенту** | Через 3 дня: «Если проблема решена, тикет будет закрыт через 2 дня» |
| **Автозакрытие** | Через 5 дней: тикет переходит в Closed с пометкой «Закрыт автоматически» |
| **Возможность reopenить** | Клиент может написать в WhatsApp → PM reopenит тикет |

## Support Dashboard

### Оперативная панель

Дашборд поддержки предоставляет мгновенный обзор текущего состояния всех тикетов.

#### Виджеты дашборда

| Виджет | Описание | Формат |
|--------|----------|--------|
| **Open Tickets** | Количество открытых тикетов по приоритету | Числовые карточки с цветом (P1: красная, P2: оранжевая, P3: синяя) |
| **SLA Status** | Тикеты в норме / внимание / просрочено | Прогресс-бары с цветовой индикацией |
| **By Status** | Распределение по стадиям (Kanban-мини) | Горизонтальная bar-chart |
| **Overdue Tickets** | Список просроченных тикетов с указанием проекта и ответственного | Таблица с сортировкой |

#### Метрики за период (неделя / месяц)

| Метрика | Описание | Целевое значение |
|---------|----------|------------------|
| **SLA Compliance %** | Процент тикетов, решённых в рамках SLA | ≥ 90% |
| **Average First Response** | Среднее время первого ответа | < 2 часов |
| **Average Resolution Time** | Среднее время решения | < 2 рабочих дней |
| **Tickets Closed / Week** | Количество закрытых тикетов за неделю | Для мониторинга тренда |
| **Reopen Rate** | Процент тикетов, которые были повторно открыты | < 10% |
| **Top Projects by Tickets** | Проекты с наибольшим количеством тикетов | Для выявления проблемных проектов |

#### Аналитические графики

| График | Описание | Применение |
|--------|----------|------------|
| **Trend: тикеты/неделя** | Количество новых и закрытых тикетов за каждую неделю | Растущий тренд новых → может потребоваться Support-специалист |
| **SLA Compliance trend** | Динамика % соблюдения SLA | Падение → проблема с ресурсами или процессами |
| **Category distribution** | Pie chart: Incident / Service Request / Change Request / Problem | Много Change Request → клиенты хотят новый функционал (потенциал для upsell) |
| **Resolution by developer** | Время решения в разрезе разработчиков | Выявление узких мест, распределение нагрузки |
| **Top 5 projects** | Проекты-лидеры по количеству тикетов | Высокое количество → аудит качества проекта |

## Жизненный цикл Problem-тикета

Problem-тикеты имеют расширенный жизненный цикл, направленный на устранение корневой причины.

```
3+ связанных Incident
       │
       ▼
PM создаёт Problem-тикет
       │
       ▼
Привязывает связанные Incidents
       │
       ▼
Root Cause Analysis (RCA)
       │
       ▼
План устранения корневой причины
       │
       ▼
Реализация плана (задачи)
       │
       ▼
Верификация: инциденты больше не повторяются
       │
       ▼
Problem Closed
```

| Поле Problem-тикета | Описание |
|---------------------|----------|
| **Related Incidents** | Список связанных инцидентов (3+) |
| **Root Cause** | Описание корневой причины (после анализа) |
| **Action Plan** | План действий по устранению |
| **Verification Period** | Период наблюдения после фикса (7–14 дней) |
| **Recurrence Count** | Количество повторных инцидентов после фикса (должно быть 0) |

## Метрики качества поддержки

| Метрика | Формула | Целевое значение | Интерпретация |
|---------|---------|------------------|---------------|
| **SLA Compliance** | Тикетов в SLA / Всего тикетов × 100% | ≥ 90% | Процесс работает штатно |
| **First Contact Resolution** | Решено с первого ответа / Всего × 100% | ≥ 40% | Эффективность триажа |
| **Reopen Rate** | Reopened / Closed × 100% | < 10% | Качество решений |
| **Mean Time to Resolve** | Среднее (Resolved - Created) | < 48 часов | Скорость команды |
| **Customer Satisfaction** | Оценка клиента (1–5) после закрытия | ≥ 4.0 | Удовлетворённость клиента |
| **Incident / Project / Month** | Инцидентов на проект в месяц | < 3 | Качество проектов |
