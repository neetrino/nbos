# Clients — Обзор модуля

## Назначение

`Clients` отвечает за клиентский контекст NBOS: кто этот человек, через какое юридическое лицо он работает с Neetrino, какие проекты, сделки, счета, подписки, тикеты и коммуникации связаны с ним.

Модуль не создаёт отдельную ручную сущность `Client Account`. Клиентский профиль появляется автоматически из связей `Contact` и `Company` с бизнес-сущностями платформы.

---

## 1. Главные сущности

### Contact

`Contact` — физическое лицо: человек, с которым Neetrino общается.

Примеры:

- Anna, которая обсуждает проект;
- Tigran, который отвечает за оплату;
- Samvel, который даёт технические уточнения.

Contact хранит персональную и коммуникационную информацию: ФИО, телефон, email, мессенджеры, язык, preferred channel, заметки.

### Company

`Company` — юридическая или биллинговая сущность, на которую выставляются счета.

Примеры:

- InvestOn LLC;
- ИП клиента;
- физлицо, если счёт выставляется на человека.

Company хранит реквизиты, tax status, legal address, bank details, primary contact и настройки для счетов.

### Client Portfolio

`Client Portfolio` — вычисляемый view, а не отдельная таблица и не отдельная editable-карточка.

Он собирает данные из Contact, Company, CRM, Projects Hub, Finance, Subscriptions, Support и Messenger, чтобы показать полную картину отношений с клиентом.

---

## 2. Что такое Client

`Client` — это не отдельный объект, который сотрудник создаёт руками.

Contact получает клиентский контекст автоматически, если он связан хотя бы с одной клиентской сущностью:

- Lead;
- Deal;
- Project;
- Company;
- Invoice;
- Subscription;
- Support Ticket;
- Client Service Record.

Если Contact связан с такими объектами, система показывает для него `Client Profile / Client Portfolio`.

---

## 3. Contact Type вместо процессных ролей

В старой формулировке поле `role` могло звучать как процессная роль человека. Новый канон:

`contact_type` — общий тип контакта для фильтрации и понимания, кем человек является для Neetrino.

Значения:

- `Client`;
- `Partner`;
- `Contractor`;
- `Other`.

Правила:

- если Contact создан из CRM / Deal / Project / Company, default = `Client`;
- если Contact создан из Partners, default = `Partner`;
- если Contact создан как внешний исполнитель, default = `Contractor`;
- значение можно поменять вручную в карточке Contact;
- `contact_type` не управляет стадиями CRM, Project, Invoice или Support.

Процессы должны опираться не на общий тип контакта, а на конкретные связи.

---

## 4. Процессные связи

Для реальной работы используются конкретные связи:

| Связь                                       | Значение                                      |
| ------------------------------------------- | --------------------------------------------- |
| `deal.contact`                              | Основной контакт сделки                       |
| `company.primary_contact`                   | Основной контакт компании                     |
| `company.billing_contact` / invoice contact | Контакт для финансовых вопросов и напоминаний |
| `project.contacts`                          | Люди клиента, участвующие в проекте           |
| `support_ticket.contact`                    | Человек, от которого пришёл тикет             |

Сотрудник не обязан выбирать сложную роль вручную. Он просто добавляет контакт в нужное место, а система понимает контекст по связи.

Если позже роль начнёт влиять на процесс, например без `Decision Maker` нельзя закрыть Transfer, такая роль добавляется в конкретном модуле, где она реально нужна.

---

## 5. Пример InvestOn

Компания InvestOn заказывает сайт.

1. В CRM появляется Lead от Anna.
2. Система создаёт или находит `Contact: Anna`.
3. Seller создаёт Deal и выбирает `Company: InvestOn`.
4. Anna становится основным контактом сделки через связь `deal.contact`.
5. Tigran добавляется как финансовый контакт, если он нужен для оплаты.
6. Samvel добавляется в проект как контакт для технических вопросов.
7. После Deal Won создаётся Project / Product, связанные с InvestOn и нужными контактами.
8. Invoice берёт юридические данные из Company, а коммуникацию по оплате — из billing contact / настроек компании.

Ни один из этих людей не требует ручного выбора сложной роли. Главное — правильно связать Contact с Deal, Company, Project или Invoice.

---

## 6. Границы интерфейса

### Contact Card

Карточка человека. Показывает персональные данные, компании, проекты, сделки, коммуникации и вход в Client Portfolio.

### Company Card

Карточка юридической / биллинговой сущности. Показывает реквизиты, tax status, invoice settings, контакты, проекты, счета и финансовую историю.

### Client Portfolio

Агрегированный экран 360-view. Открывается от Contact или от Company и показывает полную клиентскую картину.

## 7. Канонические решения

| Решение                                                             | Статус   |
| ------------------------------------------------------------------- | -------- |
| `Client Account` как отдельную ручную сущность не создаём           | Accepted |
| `Client Portfolio` является вычисляемым view                        | Accepted |
| `Contact` = человек, `Company` = биллинговая сущность               | Accepted |
| `role` переосмысливается как `contact_type`                         | Accepted |
| Процессные роли не создаются заранее, если они не влияют на процесс | Accepted |
| Клиентский профиль появляется автоматически через связи             | Accepted |
