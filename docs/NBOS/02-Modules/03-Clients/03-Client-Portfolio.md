# Портфель клиента (Client Portfolio)

## Обзор

**Client Portfolio** — это агрегированное представление всех взаимоотношений клиента с Neetrino. Когда CEO или Seller хотят увидеть «полную картину» по конкретному клиенту — все его проекты, финансы, подписки, тикеты — они открывают Client Portfolio.

Client Portfolio не является отдельной сущностью в базе данных. Это **вычисляемый view**, который собирает данные из Contact, Company, Projects, Orders, Subscriptions, Invoices, Support Tickets и других связанных сущностей.

Client Portfolio не равен `Client Account`. Сотрудники не создают его руками. Он появляется автоматически, когда Contact или Company имеют связи с клиентскими объектами платформы.

---

## 1. Точка входа

Client Portfolio доступен из нескольких мест:

| Источник         | Действие                                      |
| ---------------- | --------------------------------------------- |
| **Contacts**     | Кнопка «Портфель клиента» на карточке Contact |
| **Companies**    | Кнопка «Портфель» на карточке Company         |
| **Projects Hub** | Ссылка на контакт → переход к портфелю        |
| **CRM Deals**    | Ссылка на контакт сделки → переход к портфелю |

---

## 2. Структура портфеля

Client Portfolio организован в виде секций, каждая из которых агрегирует данные по всем проектам клиента.

### 2.0. Screen Layout

Client Portfolio открывается как полноценная страница или большой drawer / sheet, но не как маленький popup.

Рекомендуемая структура:

```
┌──────────────────────────────────────────────────────────────┐
│ Header: Client / Company name, health, quick actions          │
├──────────────────────────────────────────────────────────────┤
│ Summary Cards: LTV, MRR, Outstanding, Projects, Tickets       │
├──────────────────────────────────────────────────────────────┤
│ Tabs: Overview | Projects | Finance | Subscriptions |         │
│       Support | Communication | Files                         │
├──────────────────────────────────────────────────────────────┤
│ Active tab content                                            │
└──────────────────────────────────────────────────────────────┘
```

Если Portfolio открыт из Contact, header показывает человека и его компании. Если открыт из Company, header показывает Company и её primary / billing contacts.

### 2.1. Заголовок (Header)

| Поле                       | Описание                                    |
| -------------------------- | ------------------------------------------- |
| ФИО контакта               | Имя и фамилия                               |
| Тип контакта               | Client / Partner / Contractor / Other       |
| Компании                   | Список всех Company, привязанных к контакту |
| Телефон / Email            | Основные контактные данные                  |
| Мессенджеры                | WhatsApp, Telegram, Instagram               |
| Дата начала сотрудничества | Дата создания первого проекта               |
| Общий стаж                 | «Клиент с 2022 года (3 года)»               |

Дополнительно в header показываются:

- `Client Health` — Good / Warning / Risk;
- предупреждение по overdue invoices;
- быстрые кнопки: New Deal, New Ticket, Open Chat, Create Invoice;
- context badge: `Contact Portfolio` или `Company Portfolio`.

### 2.1.1. Tabs

| Tab           | Назначение                                                          |
| ------------- | ------------------------------------------------------------------- |
| Overview      | Краткая сводка: финансы, активные проекты, риски, последние события |
| Projects      | Все Projects / Products / Extensions в выбранном контексте          |
| Finance       | Invoices, payments, orders, outstanding balance, LTV                |
| Subscriptions | Активные, pending, on hold, cancelled, completed subscriptions      |
| Support       | Open tickets, recent tickets, SLA risks                             |
| Communication | История сообщений, звонков, заметок, reminders                      |
| Files         | Client Library / документы клиента, если Drive доступен             |

Default tab = Overview. Система запоминает последний открытый tab пользователя.

### 2.2. Финансовая сводка (Financial Summary)

Агрегированные финансовые показатели по всем проектам клиента.

#### Ключевые метрики:

| Метрика                             | Описание                             | Формула                                       |
| ----------------------------------- | ------------------------------------ | --------------------------------------------- |
| **Total Lifetime Value (LTV)**      | Общая сумма всех полученных платежей | Σ всех Payment по всем Orders и Subscriptions |
| **Monthly Recurring Revenue (MRR)** | Ежемесячный повторяющийся доход      | Σ сумм активных Subscription                  |
| **Outstanding Balance**             | Непогашенная задолженность           | Σ неоплаченных Invoice                        |
| **Total Orders Value**              | Общая стоимость всех заказов         | Σ total_amount всех Orders                    |
| **Paid Percentage**                 | Процент оплаты                       | (Оплачено / Общая сумма) × 100%               |

#### Визуализация:

```
┌──────────────────────────────────────────────────────────────┐
│  FINANCIAL SUMMARY                                            │
│                                                               │
│  💰 Lifetime Value:     4,850,000 AMD                         │
│  📊 Monthly Recurring:    180,000 AMD                         │
│  ⚠️  Outstanding:          120,000 AMD                         │
│  📈 Total Orders:       5,200,000 AMD                         │
│  ✅ Paid:                    93.3%                             │
│                                                               │
│  [▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░] 93.3% оплачено                      │
└──────────────────────────────────────────────────────────────┘
```

## 3. Быстрые действия (Quick Actions)

Из Client Portfolio можно выполнить ключевые действия без переключения между модулями:

| Действие                    | Описание                                                   | Кто может            |
| --------------------------- | ---------------------------------------------------------- | -------------------- |
| **Создать новую сделку**    | Открывает форму Deal с предзаполненным Contact и Company   | Seller, CEO          |
| **Создать Extension Deal**  | Открывает форму Extension Deal для выбранного проекта      | Seller, CEO          |
| **Просмотреть все Invoice** | Переходит к фильтрованному списку Invoice по этому клиенту | Finance, CEO, Seller |
| **Написать клиенту**        | Открывает окно отправки сообщения через WhatsApp/Email     | Seller, PM, CEO      |
| **Создать тикет**           | Создаёт Support Ticket с предвыбранным проектом            | PM, Support, CEO     |
| **Просмотреть чаты**        | Переходит к проектным чатам этого клиента                  | PM, Seller, CEO      |

Правила:

- Quick Action не создаёт дублирующий Contact или Company, а использует текущий context.
- Если действие требует Company, а Portfolio открыт от Contact с несколькими Company, система просит выбрать нужную Company.
- Если действие требует Project, а их несколько, система просит выбрать Project / Product.
- Если у пользователя нет доступа к действию, кнопка скрыта или disabled с коротким объяснением.

## 5. Доступ к Client Portfolio

| Роль                 | Уровень доступа                                                      |
| -------------------- | -------------------------------------------------------------------- |
| **CEO**              | Полный доступ: все секции, все клиенты                               |
| **Finance Director** | Финансовые секции: сводка, Invoice, Orders, Subscriptions            |
| **Seller**           | Полный доступ к своим клиентам: все секции, кроме детальных расходов |
| **PM**               | Ограниченный: проекты, задачи, тикеты (без финансовых деталей)       |
| **Developer**        | Нет доступа к Client Portfolio                                       |
| **Designer**         | Нет доступа к Client Portfolio                                       |
| **QA**               | Нет доступа к Client Portfolio                                       |

### 5.1. Masking финансовых данных

Финансовые данные не должны быть одинаково видны всем.

| Секция                  | CEO  | Finance         | Seller                  | PM                    | Support           |
| ----------------------- | ---- | --------------- | ----------------------- | --------------------- | ----------------- |
| LTV / MRR / Outstanding | full | full            | full for own clients    | masked                | hidden            |
| Invoices                | full | full            | summary for own clients | status only           | hidden            |
| Orders                  | full | full            | full for own clients    | project-level summary | hidden            |
| Subscriptions           | full | full            | full for own clients    | status / plan only    | status only       |
| Support Tickets         | full | read            | read                    | full for own projects | full for assigned |
| Communication           | full | finance-related | own clients             | project-related       | support-related   |

`masked` означает, что пользователь видит статус и риск, но не видит точные суммы.

## 6. Сценарии использования

### Сценарий 1: CEO оценивает ключевого клиента

CEO открывает Client Portfolio крупного клиента → видит LTV 15,000,000 AMD → 5 проектов → 3 активные подписки → 1 просроченный Invoice → принимает решение позвонить и мягко напомнить, а также предложить новый проект.

### Сценарий 2: Seller готовится к встрече

Seller открывает Client Portfolio перед встречей с клиентом → видит, что текущий Website в Maintenance, последняя доработка была 4 месяца назад → готовит предложение новой доработки или нового продукта (Mobile App).

### Сценарий 3: Finance Director проверяет дебиторку

Finance Director открывает Client Portfolio → видит 2 просроченных Invoice на общую сумму 250,000 AMD → связывается с Seller для эскалации → если не оплачено — инициирует приостановку обслуживания.

### Сценарий 4: PM оценивает загрузку по клиенту

PM открывает Client Portfolio → видит 2 активных проекта: один в Development, другой в Maintenance с Extension → оценивает, сколько ресурсов требуется на этого клиента → планирует спринты.

## 7. Связь с Contact и Company

Client Portfolio может быть открыт в двух контекстах:

### От Contact (человека):

Показывает все данные по ВСЕМ компаниям и проектам этого человека. Это полная картина взаимоотношений с физическим лицом.

```
Client Portfolio (Contact: Арам Петросян)
  ├── Company: ООО "АрамОбувь"
  │     └── Project "Обувной бренд" → данные...
  ├── Company: ИП Петросян
  │     └── Project "Цветочный магазин" → данные...
  └── Company: Aram Transport LLC
        └── Project "Транспорт" → данные...

  LTV = сумма по ВСЕМ компаниям и проектам
```

### От Company (юрлица):

Показывает данные только по проектам конкретной компании. Полезно для финансового анализа по юрлицу.

```
Client Portfolio (Company: ООО "АрамОбувь")
  └── Project "Обувной бренд" → данные...

  LTV = только по этой компании
```

### 7.1. Отличие от Contact Card

Contact Card — это карточка человека.

Она отвечает на вопросы:

- кто этот человек;
- как с ним связаться;
- какие у него контакты и предпочтения;
- какие компании и проекты связаны с ним кратко.

Client Portfolio отвечает на другой вопрос: какая полная бизнес-история у этого клиента с Neetrino.

Contact Card может показывать краткие counters и кнопки, но не должна становиться местом всей аналитики.

### 7.2. Отличие от Company Card

Company Card — это карточка юридической / биллинговой сущности.

Она отвечает на вопросы:

- на кого выставлять счёт;
- какой tax status;
- какие реквизиты;
- кто primary / billing contact;
- какие invoices и projects связаны с этой Company.

Client Portfolio от Company показывает агрегированную историю только этой Company, но не заменяет редактирование реквизитов.

## 8. Как появляется клиентский профиль

Client Portfolio доступен автоматически, если Contact или Company связаны хотя бы с одной клиентской сущностью:

- Lead;
- Deal;
- Project;
- Company;
- Invoice;
- Subscription;
- Support Ticket;
- Client Service Record.

Ручного создания `Client Account` не требуется.

Пример:

- Anna добавлена в Deal как основной контакт — для Anna доступен Client Portfolio;
- InvestOn создана как Company для выставления счетов — для InvestOn доступен Client Portfolio в контексте компании;
- Tigran добавлен как billing contact — он остаётся Contact, а его клиентский контекст появляется через связь с Company / Invoice.

### 8.1. Empty states

Если Portfolio открыт, но данных мало, экран должен помогать сотруднику продолжить работу.

| Ситуация                    | Что показываем                                                                              |
| --------------------------- | ------------------------------------------------------------------------------------------- |
| Contact без Company         | Подсказка “Добавьте Company для выставления счетов”                                         |
| Contact без Projects        | Кнопка “Create Deal” или “Link Project”                                                     |
| Company без billing contact | Подсказка “Primary contact будет использоваться для оплаты, пока billing contact не указан” |
| Нет Invoice                 | Секция Finance показывает zero-state, не ошибку                                             |
| Нет Support Tickets         | Секция Support показывает “No tickets yet”                                                  |

Empty state не должен выглядеть как ошибка системы.

## 10. Ключевые принципы

1. **Portfolio = View, не Data.** Client Portfolio не хранит собственных данных — он агрегирует данные из связанных сущностей в реальном времени.

2. **Два контекста:** от Contact (человек → все его бизнесы) и от Company (юрлицо → только его проекты).

3. **Нет отдельного Client Account.** Клиентский профиль создаётся связями, а не ручной карточкой.

4. **Финансовая прозрачность.** LTV, MRR, Outstanding Balance — ключевые метрики для принятия бизнес-решений.

5. **Quick Actions.** Портфель не только показывает данные, но и позволяет действовать: создать сделку, написать клиенту, создать тикет.

6. **Ролевой доступ.** CEO видит всё, Seller — своих клиентов, PM — только нефинансовую часть.

7. **Screen context first.** Один и тот же Portfolio может быть открыт от Contact или Company, и цифры должны соответствовать выбранному контексту.

8. **No duplicate data entry.** Portfolio не редактирует исходные данные напрямую, кроме быстрых действий, которые открывают правильную карточку Contact / Company / Deal / Invoice.

9. **Financial data is permissioned.** Финансовые суммы скрываются или маскируются для ролей, которым они не нужны.

## 11. Implementation requirements

### 11.1. Computed endpoint / view

Runtime должен иметь отдельный способ получить Client Portfolio:

- by Contact;
- by Company.

Endpoint / service не должен создавать новую сущность. Он агрегирует существующие связи.

### 11.2. Required data blocks

Минимальный response должен покрывать:

- context: contact или company;
- related contacts;
- related companies;
- projects;
- products / extensions;
- orders;
- invoices;
- subscriptions;
- support tickets;
- client services;
- communication summary;
- access mask для текущего пользователя.

### 11.3. Navigation

Каждый элемент в Portfolio должен открывать исходную сущность:

- Project → Project Sheet / Projects Hub;
- Product / Extension → Delivery card;
- Invoice → Invoice Card;
- Subscription → Subscription Card;
- Ticket → Support Ticket;
- Contact → Contact Card;
- Company → Company Card.

Portfolio не должен быть тупиком. Это навигационная карта клиента.
