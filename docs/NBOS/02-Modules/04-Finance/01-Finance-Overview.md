# Модуль Finance — Обзор

## Назначение модуля

Модуль **Finance** управляет всеми денежными потоками компании Neetrino. Каждое движение денег — входящее или исходящее — фиксируется, привязывается к проекту (или к компании, если это общая статья) и доступно для аналитики.

Главный принцип: **ни одна транзакция не должна остаться без привязки**. Любой доход или расход привязан как минимум к одному из трёх уровней учёта — компания, проект, заказ.

## Стороны финансового потока

### Доходная сторона (Revenue)

```
Клиент платит → Invoice → Payment → Revenue
```

Источники дохода:

- **Заказы (Orders)** — оплата за разработку (предоплата, вторая часть, полная оплата)
- **Подписки (Subscriptions)** — ежемесячные платежи за техобслуживание, разработку по подписке, партнёрские сервисы
- **Расширения (Extensions)** — доработки и дополнительные задачи по существующим проектам
- **Pass-through** — клиент платит за домены, хостинг, лицензии (Neetrino выступает посредником, иногда с наценкой)

### Расходная сторона (Costs)

```
Neetrino платит → Expense → привязка к проекту/компании
```

Категории расходов:

- **Плановые расходы** — периодические затраты с известной суммой и графиком (домены, хостинг, сервисы, инструменты)
- **Внеплановые расходы** — разовые или непредвиденные траты (маркетинг, закупки, новые инструменты)
- **Pass-through расходы** — покупки от имени клиента (домены, аккаунты App Store, Google Workspace)
- **Зарплаты** — ежемесячные выплаты сотрудникам
- **Бонусы** — процент от сделок (продажи, доставка, маркетинг)
- **Партнёрские выплаты** — процент партнёру за привлечённого клиента

## Три уровня финансового учёта

### 1. Уровень компании (Company Level)

Агрегированная финансовая картина Neetrino за период:

- Суммарный доход (все платежи)
- Суммарные расходы (зарплаты + бонусы + плановые + внеплановые + партнёрские)
- Валовая маржа, чистая маржа
- Остаток на счёте, дебиторская задолженность, предстоящие расходы

### 2. Уровень проекта (Project Level)

Финансовый профиль конкретного проекта (бренда/бизнеса клиента):

- Доход: оплаты за заказы + подписки + расширения
- Расходы: бонусы по заказам, домены, хостинг, сервисы, партнёрские выплаты
- Маржинальность проекта

### 3. Уровень заказа (Order Level)

Детализация одной конкретной сделки:

- Доход: сумма оплат по этому заказу
- Расходы: бонус продажнику, бонус доставке, партнёрская выплата
- Маржа заказа

## Налогообложение: Tax vs Tax-Free

Каждый счёт имеет признак **налогооблагаемости**, который наследуется от родительской сущности:

| Источник счёта          | Откуда берётся Tax/Tax-Free |
| ----------------------- | --------------------------- |
| Заказ (Order)           | Из настроек заказа/сделки   |
| Подписка (Subscription) | Из настроек подписки        |
| Домен/Сервис            | Из настроек домена/сервиса  |

**Если Tax (налогооблагаемый):**

- Счёт проходит стадию «Создать в гос. системе»
- Бухгалтеру отправляется уведомление с реквизитами
- Бухгалтер создаёт официальный счёт в государственной системе

**Если Tax-Free (не налогооблагаемый):**

- Стадия «Создать в гос. системе» пропускается
- Счёт сразу переходит к отправке клиенту
- Типичные случаи: зарубежные клиенты, pass-through платежи без наценки

Статус Tax/Tax-Free фиксируется при создании сущности и **не меняется** на протяжении всего жизненного цикла.

## Роли и доступы

| Роль                    | Доступ в модуле Finance                                                                                |
| ----------------------- | ------------------------------------------------------------------------------------------------------ |
| **CEO**                 | Полный доступ ко всем разделам, отчётам, суммам                                                        |
| **Финансовый директор** | Основной пользователь. Управляет счетами, расходами, зарплатами, бонусами. Видит все финансовые данные |
| **Бухгалтер**           | Получает уведомления о создании счетов в гос. системе. Работает с налоговыми документами               |
| **Продажник (Seller)**  | Видит: суммы счетов, статус оплаты, свои бонусы. Не видит: расходы, зарплаты других, P&L               |
| **Проект-менеджер**     | Видит финансовый статус своих проектов (оплачен/не оплачен). Не видит детальные суммы расходов         |
| **Разработчик**         | Видит только свои бонусы (если применимо)                                                              |

Финансовый директор — ключевой пользователь модуля. Его ежедневные задачи:

1. Проверка банка → отметка оплаченных счетов
2. Запуск автоматизации напоминаний по неоплаченным счетам
3. Обработка расходов текущего месяца
4. Контроль зарплатной ведомости и бонусов

## Связи с другими модулями

```
CRM (Deals/Orders) ──→ Invoices, Payments, Bonuses
Projects Hub ──→ Project P&L, Subscriptions, Domains
Partners ──→ Partner Accruals, Partner Balance, Payout Batches, Partner Service Subscriptions
My Company ──→ Salary Board, Employee Bonuses
Tasks ──→ Auto-tasks (купить домен, оплатить сервис)
Credentials ──→ Аккаунты провайдеров (домены, хостинг)
Drive ──→ Invoice/payment/expense proofs, payroll exports, partner payout docs
Notifications ──→ Напоминания об оплатах, просрочках
Dashboards ──→ MRR, Churn, Cash Flow, P&L виджеты
```

Finance не хранит файлы в собственном локальном attachment layer. Все proofs, scans, exports и supporting documents являются Drive File Assets с finance permissions. Finance остаётся source of truth для сумм и статусов, Drive - source of truth для файлов и версий.

## Ключевые автоматизации

| Триггер                       | Действие                                                                 |
| ----------------------------- | ------------------------------------------------------------------------ |
| Создание подписки             | Авто-генерация ежемесячных счетов по billing_day                         |
| 1-е число месяца              | Создание карточек всех плановых расходов этого месяца                    |
| Счёт создан                   | Через ~1 час → уведомление бухгалтеру (если Tax)                         |
| Счёт в гос. системе создан    | WhatsApp-сообщение клиенту                                               |
| Просрочка оплаты              | Серия напоминаний (1-е, через 2 дня — 2-е, далее эскалация)              |
| Заказ полностью оплачен       | Создание бонусов и, если применимо, Partner Accrual по правилам Partners |
| Домен оплачен клиентом        | Авто-задача «создать аккаунт + купить домен»                             |
| Домен истекает через 2 месяца | Напоминание → создание счёта для клиента                                 |
| Первая оплата по заказу       | Переход Deal → Order в CRM                                               |

## Принципы проектирования

1. **Прозрачность** — каждый рубль/драм привязан к проекту или статье расхода
2. **Автоматизация** — минимум ручной работы для повторяющихся операций
3. **Контроль** — система напоминает о просрочках, создаёт задачи при необходимости действий
4. **Аудит** — полная история изменений: кто, когда, что изменил
5. **Масштабируемость** — модуль рассчитан на рост количества проектов, клиентов и транзакций
