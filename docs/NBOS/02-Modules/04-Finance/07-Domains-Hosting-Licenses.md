# Домены и сервисы клиента (Client Services)

## Общая концепция

Этот раздел описывает единый контур:

`Client Services / Домены и сервисы клиента`

Сюда входят:

- домены;
- хостинг;
- Apple / Google developer accounts;
- Google Workspace;
- платные SaaS и API;
- лицензии и внешние сервисы вокруг проекта клиента.

Главная идея:

это не просто “расходы” и не просто “домены”.  
Это **живые карточки сервисов клиента**, у которых есть:

- проект;
- провайдер;
- дата продления;
- себестоимость;
- цена для клиента;
- credentials;
- правила биллинга;
- связь с `Invoice Card`, `Expense`, `Task`.

```text
Client Service Record
    ->
Invoice Card to client (if we-pay)
    ->
Payment
    ->
Task for purchase / renewal
    ->
Expense for our side
```

---

## Главная сущность

### `Client Service Record / Карточка сервиса клиента`

Это единый канонический объект для:

- `Domain`
- `Hosting`
- `Service`
- `Account`
- `License`

Он не отменяет специальные типы, а объединяет их одной моделью.

### Базовые поля карточки

| Поле                    | Описание                                                           |
| ----------------------- | ------------------------------------------------------------------ |
| `service_id`            | Уникальный идентификатор                                           |
| `project`               | Проект                                                             |
| `product`               | Продукт, если сервис относится к конкретному продукту              |
| `type`                  | Domain / Hosting / Service / Account / License                     |
| `name`                  | Название сервиса                                                   |
| `provider`              | Поставщик                                                          |
| `provider_account`      | Ссылка на Credentials                                              |
| `status`                | Active / Pending / Suspended / Expiring Soon / Expired / Cancelled |
| `billing_model`         | We Pay / Reminder Only                                             |
| `pricing_model`         | Fixed / Usage-based                                                |
| `frequency`             | Monthly / Yearly / One-time                                        |
| `our_cost`              | Наша фактическая себестоимость                                     |
| `client_charge`         | Сколько должен платить клиент, если сервис клиентский              |
| `tax_status`            | Tax / Free                                                         |
| `notifications_enabled` | Разрешены ли уведомления по связанным Invoice Cards                |
| `start_date`            | Дата начала                                                        |
| `renewal_date`          | Дата продления / следующей оплаты                                  |
| `notes`                 | Заметки                                                            |

---

## Почему это лучше, чем держать всё отдельно

Потому что домен, хостинг и Apple account по сути решают одну и ту же задачу:

- что именно обслуживаем;
- кому принадлежит;
- кто платит;
- когда продлевать;
- нужна ли карточка оплаты клиенту;
- нужно ли создавать расход;
- нужно ли создавать задачу после оплаты.

То есть различается только тип сервиса, а не вся бизнес-модель.

---

## Типы сервисов

### 1. Domain

Доменное имя проекта клиента.

Особенности:

- обычно ежегодное продление;
- почти всегда привязано к проекту;
- часто требует отдельного аккаунта у регистратора;
- может иметь наценку.

### 2. Hosting

Сервер или хостинг проекта.

Особенности:

- бывает ежемесячным или ежегодным;
- может быть we-pay (pass-through) или reminder-only;
- может иметь инфраструктурные параметры.

### 3. Service

Внешний SaaS / API / инструмент, который нужен проекту.

Примеры:

- Google Workspace
- Neon
- Upstash
- Resend
- Cloudflare Pro

### 4. Account

Отдельный клиентский аккаунт у провайдера.

Примеры:

- Apple Developer Account
- Google Play Account

### 5. License

Отдельная лицензия или право использования сервиса.

---

## Billing Model (Кто платит и что делает система)

Это одно из главных полей. Значения в БД: `WE_PAY`, `REMINDER_ONLY`.

**Важно:** `Client Service Record` — это контур **клиентских** сервисов. Наши собственные повторяющиеся расходы компании живут в **`Expense Plan`**, а не здесь.

### `We Pay / Оплачиваем мы` (`WE_PAY`)

Клиент платит нам, мы платим провайдеру (pass-through).

Логика:

1. создаётся `Invoice Card` клиенту;
2. клиент оплачивает;
3. после оплаты автоматически создаётся `Expense Card` и задача на покупку / продление;
4. Finance оплачивает провайдера и закрывает расход.

### `Reminder Only / Только напоминание` (`REMINDER_ONLY`)

Клиент платит провайдеру сам; мы помогаем не пропустить срок.

Логика:

1. `Invoice Card` и `Expense Card` **не создаются**;
2. система напоминает и ставит контрольную `Task` менеджеру;
3. менеджер сверяется с клиентом: оплатил ли сам, нужно ли обновить `renewal_date`.

---

## Pricing Model (Как считается сумма)

### `Fixed / Фиксированная`

Примеры:

- домен: `12 000 / год`
- Apple account: `$99 / год`
- hosting: `6 000 / месяц`

### `Usage-based / По факту использования`

Примеры:

- Neon
- Upstash
- Resend

Правило:

- карточка сервиса хранит последнюю известную или среднюю стоимость;
- перед созданием нового `Expense` или `Invoice Card` Finance может скорректировать фактическую сумму.

---

## Частота

### `Monthly / Ежемесячно`

Например:

- hosting
- Google Workspace
- часть SaaS

### `Yearly / Ежегодно`

Например:

- домены
- Apple Developer
- часть лицензий

### `One-time / Один раз`

Например:

- Google Play developer account

## Domain как частный случай

Для домена внутри общего объекта дополнительно важны:

- `domain_name`
- `expiry_date`
- `auto_renew`

### Жизненный цикл домена

```text
Domain record created
    ->
renewal date approaches
    ->
Invoice Card to client (if we-pay)
    ->
Payment
    ->
Task: renew domain
    ->
Expense
    ->
expiry_date updated
```

### Правило для Website Product

Если создаётся `Website Product`, система должна **предлагать** создать минимум одну доменную карточку.

Это лучше, чем жёсткое авто-создание без контроля, и лучше, чем надеяться, что сотрудники не забудут.

---

## Hosting как частный случай

Для hosting дополнительно могут храниться:

- hosting type: shared / VPS / cloud / managed;
- infrastructure notes;
- технические характеристики, если нужно.

Логика:

- если hosting клиентский и `WE_PAY`, он может создавать `Invoice Card`;
- если `REMINDER_ONLY`, создаётся только контрольная `Task`.

## Services / Accounts / Licenses

Для сервисов и аккаунтов логика такая же:

- есть карточка сервиса;
- есть renewal / due date;
- есть provider account;
- по правилам создаются `Invoice Card`, `Expense`, `Task`.

### Примеры

| Сервис                                  | Billing Model           | Частота          | Особенность           |
| --------------------------------------- | ----------------------- | ---------------- | --------------------- |
| Google Workspace                        | We Pay                  | Monthly          | часто pass-through    |
| Cloudflare Pro                          | We Pay or Reminder Only | Monthly / Yearly | зависит от проекта    |
| Apple Developer                         | We Pay                  | Yearly           | требует credentials   |
| Google Play                             | We Pay                  | One-time         | одноразовая покупка   |
| Domain (клиент сам платит регистратору) | Reminder Only           | Yearly           | только контроль срока |

---

## Credentials

Карточка сервиса должна ссылаться на `Credentials`, если есть внешний аккаунт.

Примеры:

- аккаунт регистратора домена;
- аккаунт Apple;
- аккаунт Google Workspace;
- API key сервиса;
- hosting panel login.

То есть:

`Client Service Record -> Credential`

а не хаотичное хранение “где-то в заметках”.

---

## Как всё связано с Finance

### 1. Invoice Card

Если сервис оплачивает клиент:

- из карточки сервиса создаётся `Invoice Card`;
- туда наследуются:
  - `tax_status`
  - `notifications_enabled`
  - проектный контекст
  - сумма для клиента

### 2. Payment

Когда клиент оплачивает карточку:

- создаётся `Payment`;
- сервисный кейс получает право перейти к следующему действию.

### 3. Task

После оплаты создаётся задача:

- купить домен;
- продлить домен;
- создать Apple account;
- оплатить hosting;
- подключить сервис.

### 4. Expense

После фактической покупки / продления создаётся `Expense`, который отражает наши реальные деньги.

## Канонические сценарии

### Сценарий 1. Новый домен

1. создаётся карточка домена;
2. указываются:
   - provider
   - our_cost
   - client_charge
   - tax_status
3. создаётся `Invoice Card`;
4. клиент платит;
5. создаётся задача на покупку;
6. после покупки создаётся `Expense`;
7. сохраняются credentials;
8. обновляется `expiry_date`.

### Сценарий 2. Продление домена

1. приближается `renewal_date`;
2. система создаёт `Invoice Card`, если сервис клиентский;
3. после оплаты создаётся задача `Renew Domain`;
4. после выполнения:
   - обновляется `renewal_date`
   - создаётся `Expense`

### Сценарий 3. Apple / Google account

1. создаётся карточка сервиса;
2. создаётся `Invoice Card` клиенту;
3. после оплаты создаётся задача на создание аккаунта;
4. credentials привязываются к проекту;
5. если сервис ежегодный — ставится новая `renewal_date`.

### Сценарий 4. Usage-based service

1. сервисная карточка хранит последнюю/ожидаемую стоимость;
2. Finance перед периодом уточняет сумму;
3. создаётся `Expense`;
4. если сервис pass-through — дополнительно создаётся `Invoice Card`.

---

## Напоминания и автоматизация

### Что должно автоматизироваться

- приближение `renewal_date`;
- создание `Invoice Card` для клиентских сервисов;
- создание `Task` после оплаты;
- напоминание, если credentials не заполнены;
- напоминание, если сервис скоро истекает.

### Что не должно смешиваться

Нельзя смешивать в одну сущность:

- сам сервис;
- `Invoice Card`;
- `Expense`;
- `Task`.

Правильная модель:

`Client Service Record` — источник,
а остальное уже производные действия и сущности.

## Что видеть в интерфейсе

### Главный экран

`Client Services / Домены и сервисы клиента`

Экран имеет **три view-режима** (переключатель в hero, выбор запоминается):

1. **List / Список** — полный реестр, серверная пагинация с подгрузкой по скроллу (infinite scroll), колонка платёжной стадии.
2. **Board (Status) / Доска по стадиям** — операционная воронка платежа.
3. **Board (12 месяцев) / Доска по месяцам** — селектор года + 12 колонок по месяцу `renewal_date`; в шапке колонки — счётчик и сумма, карточки подгружаются по скроллу внутри колонки.

Фильтры (общие для всех view): по проекту, типу, provider, billing model, renewal period, статусу сервиса.

#### Платёжная стадия (`payment_stage`) — вычисляемая, не поле БД

`Client Service Record` остаётся источником; стадия **вычисляется** из `renewal_date`, `billing_model` и связанных `Invoice Card` / `Expense Card` (канон запрещает смешивать их в одну сущность). Это проекция трёх слоёв статусов на одну ось.

| Стадия     | Условие                                                                                                                     |
| ---------- | --------------------------------------------------------------------------------------------------------------------------- |
| `Active`   | до `renewal_date` > 3 месяцев (или дата не задана) — действий нет                                                           |
| `Upcoming` | `renewal_date` через 2–3 месяца; для `WE_PAY` — `Invoice Card` ещё не создан; для `REMINDER_ONLY` — ждём контрольную задачу |
| `Invoice`  | только `WE_PAY`: до `renewal_date` < 2 месяцев → авто-создание `Invoice Card` (см. EXP-04); ждём оплату клиента             |
| `Pay now`  | только `WE_PAY`: клиент оплатил → создан `Expense Card`, платим провайдеру (`Due Now`)                                      |

Терминальной колонки `Paid/Done` **нет**: как только `Expense` оплачен и `renewal_date` уезжает на следующий период, карточка пересчитывается и возвращается в `Active`.

**Overdue** — не колонка, а **сквозной флаг** (красный лейбл + заливка карточки) поверх любой стадии: `Invoice Card` `Overdue`, либо `Expense Card` `Overdue`, либо `renewal_date` прошёл, а ни счёт, ни оплата не сделаны.

Порог «2 месяца» = граница стадии `Invoice` и совпадает с автоматизацией продления (`EXP-04`).

### На карточке сервиса (detail sheet)

Структура sheet сверху вниз:

1. **Header** — название, тип, статус, **Settings** (⚙) с действием удаления сервиса.
2. **Connections** — компактная панель действий под header:
   - **`WE_PAY`**: create invoice / expense / task → shared-формы с предзаполнением;
   - **`REMINDER_ONLY`**: только create task.
3. **Tabs** — `General` | `Invoices` | `Expenses` | `Tasks`:
   - **General** — поля карточки (basics, billing, dates, notes, proofs).
   - **Invoices** — список связанных `Invoice Card`; **Create invoice** только для `WE_PAY`.
   - **Expenses** — списки expense plans и expense cards; **Create** только для `WE_PAY`.
   - **Tasks** — список связанных задач; **Create task** → `QuickCreateTaskDialog` (оба billing model).

Должно быть видно:

- что это за сервис;
- кто провайдер;
- кто платит;
- сколько стоит;
- когда следующее продление;
- есть ли связанный `Invoice Card`;
- есть ли связанный `Expense`;
- есть ли credentials;
- есть ли задача на покупку / продление.

## Связь с другими модулями

```text
Projects Hub
    ->
Client Service Record
    ->
Invoice Card / Payment
    ->
Task / Work Space
    ->
Expense
    ->
P&L
```

Связи:

- `Projects Hub` — откуда приходит проектный контекст;
- `Finance / Invoices` — деньги от клиента;
- `Finance / Expenses` — наши деньги;
- `Tasks / Work Space` — исполнение;
- `Credentials` — аккаунты и доступы;
- `Notifications` — напоминания о сроках и недостающих действиях.
