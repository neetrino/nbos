# Расходы (Expenses)

## Общая концепция

Расходы — это все исходящие денежные потоки компании. Модуль расходов обеспечивает полный контроль над тем, куда уходят деньги Neetrino. Каждый расход привязан к категории и (по возможности) к проекту для корректного расчёта P&L.

Три типа расходов:

| Тип | Описание | Создание |
|-----|----------|----------|
| **Плановые (Planned)** | Известные периодические расходы | Авто-генерация на основе графика |
| **Внеплановые (Unplanned)** | Разовые, непредвиденные расходы | Вручную |
| **Pass-through** | Затраты от имени клиента | Автоматически при оплате клиентом |

---

## Плановые расходы (Planned Expenses)

### Определение

Плановые расходы — это затраты, которые компания знает заранее: они повторяются с определённой периодичностью и известной (или примерной) суммой.

### Примеры

- Домены (ежегодное продление)
- Хостинг / серверы (ежемесячно)
- SaaS-подписки (Figma, GitHub, Google Workspace)
- Инструменты разработки (лицензии)
- Аренда офиса (ежемесячно)
- Бухгалтерские услуги (ежемесячно)

### Поля плановой расходной записи

| Поле | Описание |
|------|----------|
| `planned_expense_id` | Уникальный идентификатор |
| `name` | Название расхода |
| `category` | Категория (Domain, Hosting, Service, Tools, Office и т.д.) |
| `amount` | Сумма |
| `currency` | Валюта |
| `frequency` | Периодичность: Monthly / Quarterly / Yearly / Multi-year |
| `billing_day` | День/месяц, когда нужно оплачивать |
| `project` | Привязка к проекту (опционально) |
| `description` | Описание |
| `provider` | Поставщик услуги |
| `is_pass_through` | Является ли pass-through расходом |
| `next_due_date` | Дата следующей оплаты |
| `auto_generate` | Авто-генерация карточек (да/нет) |

### Авто-генерация карточек расходов

```
1-е число каждого месяца:
  └─ Система проверяет все Planned Expenses
     └─ Для каждого, у которого в этом месяце есть дата оплаты:
        ├─ Создаётся карточка расхода (Expense Card)
        ├─ Стадия: "This Month"
        ├─ Сумма, категория, проект — из Planned Expense
        └─ Финансовый директор видит полный список на 1-е число
```

Финансовый директор 1-го числа каждого месяца чётко знает: «В этом месяце у меня N расходов на сумму X».

## Доска расходов (Expense Board)

### Стадии расхода (Kanban Board)

```
This Month → Pay Now → Delayed → On Hold → Old → Paid / Unpaid
```

| Стадия | Описание |
|--------|----------|
| **This Month** | Расходы текущего месяца. Авто-сгенерированные карточки попадают сюда |
| **Pay Now** | Требуют немедленной оплаты. Финдиректор вручную переводит приоритетные |
| **Delayed** | Оплата задерживается по какой-либо причине |
| **On Hold** | Отложено по решению руководства |
| **Old** | Старые неоплаченные расходы прошлых периодов |
| **Paid** | Оплачено. Финальная стадия |
| **Unpaid** | Закрыто без оплаты (отменено, списано) |

### Поля карточки расхода

| Поле | Описание |
|------|----------|
| `expense_id` | Уникальный идентификатор |
| `name` | Название |
| `type` | Planned / Unplanned / Pass-through |
| `category` | Категория расхода |
| `amount` | Сумма |
| `currency` | Валюта |
| `project` | Привязка к проекту (опционально) |
| `order` | Привязка к заказу (опционально) |
| `planned_expense_id` | Ссылка на плановый расход (если создан из Planned) |
| `due_date` | Срок оплаты |
| `stage` | Текущая стадия |
| `paid_date` | Дата фактической оплаты |
| `description` | Описание/комментарий |
| `created_by` | Кто создал |

### Ежедневная работа финансового директора

```
Утро:
  1. Открыть доску расходов
  2. Просмотреть "This Month" — что нужно оплатить
  3. Приоритетные → перевести в "Pay Now"
  4. Оплатить → отметить как "Paid"
  5. Проблемные → перевести в "Delayed" / "On Hold"
  6. Проверить "Delayed" — есть ли что оплатить сегодня
```

## Внеплановые расходы (Unplanned Expenses)

### Определение

Разовые, непредвиденные затраты, которые не были запланированы заранее.

### Примеры

- Срочная покупка нового инструмента
- Незапланированные маркетинговые расходы
- Ремонт оборудования
- Экстренная покупка лицензии
- Подарки клиентам/сотрудникам

### Процесс создания

1. Финансовый директор (или уполномоченный сотрудник) создаёт карточку вручную
2. Заполняет обязательные поля: название, сумма, категория, описание
3. Указывает проект (если расход проектный)
4. Карточка попадает на ту же доску расходов, в стадию "This Month" или "Pay Now"
5. Дальнейший процесс идентичен плановым расходам

### Обязательные поля при создании

| Поле | Обязательность |
|------|---------------|
| Название | ✅ Обязательно |
| Сумма | ✅ Обязательно |
| Категория | ✅ Обязательно |
| Описание | ✅ Обязательно |
| Проект | ⬜ Опционально |
| Заказ | ⬜ Опционально |

## Pass-through расходы

### Определение

Затраты, которые Neetrino совершает от имени клиента. Клиент платит Neetrino → Neetrino платит поставщику. Иногда с небольшой наценкой.

### Типичные pass-through расходы

| Расход | Описание | Наценка |
|--------|----------|---------|
| Доменные имена | Покупка/продление доменов для клиента | 0–20% |
| App Store аккаунт | Создание аккаунта разработчика Apple | Обычно 0% |
| Google Play аккаунт | Создание аккаунта разработчика Google | Обычно 0% |
| Google Workspace | Корпоративная почта и сервисы | 0–10% |
| SSL-сертификаты | Если платные | 0% |
| Специализированные сервисы | Cloudflare Pro, специфические API | Варьируется |

### Процесс pass-through расхода

```
1. Клиенту нужен домен/сервис
   │
2. Создаётся Invoice (тип: Domain/Service) для клиента
   ├─ В Invoice указывается: что именно покупаем, точная сумма (с наценкой если есть)
   │
3. Клиент оплачивает Invoice → Payment создан
   │
4. Авто-создание задачи: "Купить [домен/сервис]"
   ├─ Если аккаунт у провайдера уже есть → задача "Купить"
   └─ Если аккаунта нет → задача "Создать аккаунт + Купить"
   │
5. Специалист выполняет задачу:
   ├─ Создаёт аккаунт (если нужно)
   ├─ Покупает домен/сервис
   ├─ Сохраняет credentials в модуле Credentials
   └─ Создаёт запись Domain / Service с датой истечения
   │
6. Создаётся Expense (Pass-through) на фактическую сумму покупки
   │
7. Финдиректор отмечает расход как Paid
```

### Наценка при pass-through

Пример:
- Стоимость домена у провайдера: 10,000 драм
- Наценка Neetrino: 2,000 драм
- Сумма в Invoice для клиента: 12,000 драм
- Expense записывается на 10,000 драм (фактическая стоимость)
- Разница 2,000 = доход Neetrino

## Категории расходов

| Категория | Описание | Типичная привязка |
|-----------|----------|-------------------|
| **Domain** | Покупка/продление доменных имён | Проект |
| **Hosting** | Хостинг, серверы, VPS | Проект |
| **Service** | Внешние сервисы (API, SaaS) | Проект |
| **Marketing** | Маркетинговые расходы | Компания |
| **Salary** | Зарплаты сотрудников | Компания |
| **Bonus** | Бонусные выплаты | Проект/Заказ |
| **Partner Payout** | Выплаты партнёрам | Проект/Заказ |
| **Tools** | Инструменты и лицензии | Компания |
| **Office** | Офисные расходы | Компания |
| **Other** | Прочие расходы | Варьируется |

## Привязка к проектам и заказам

Каждый расход может быть привязан к:

| Уровень | Влияние на P&L |
|---------|----------------|
| **Компания** (без привязки к проекту) | Попадает только в Company P&L как операционный расход |
| **Проект** | Попадает в Project P&L и Company P&L |
| **Заказ** | Попадает в Order P&L, Project P&L и Company P&L |

Это обеспечивает точный расчёт маржинальности на каждом уровне.

## Уведомления и авто-задачи

| Событие | Действие |
|---------|----------|
| 1-е число месяца | Генерация карточек всех плановых расходов. Уведомление финдиректору |
| Расход просрочен | Напоминание финдиректору |
| Расход в "Delayed" более 7 дней | Повторное напоминание |
| Pass-through: клиент оплатил | Авто-задача на покупку домена/сервиса |
| Домен истекает через 2 месяца | Напоминание → авто-создание Invoice для клиента |
| Сервис с usage-based ценой | Напоминание проверить текущую сумму перед выставлением |

## Связи с другими модулями

```
Planned Expense ──→ Expense Card (авто, 1-е число)
Invoice (Domain) ──→ Payment ──→ Auto-task ──→ Expense (Pass-through)
Domain record ──→ Planned Expense (annual renewal)
Bonus Ledger ──→ Expense (Salary/Bonus category)
Partner Payout ──→ Expense (Partner Payout category)
All Expenses ──→ Company P&L (cost side)
Project Expenses ──→ Project P&L (cost side)
```
