# Finance Core Architecture (операционный журнал, вариант B)

## Назначение

Этот документ описывает **целевую архитектуру учёта денег в NBOS** на уровне принципов и понятий: как фиксировать операции так, чтобы отчётность была профессиональной (без полноценной двойной записи на старте), с возможностью **эволюции к double-entry** позже.

## Цели

1. **Прозрачный аудит** — любая денежная операция имеет источник, период и разрез (компания / проект / заказ / сотрудник).
2. **Две проекции учёта** — **accrual** (признание дохода/расхода по правилам бизнеса) и **cash** (движение денег по факту входа/выхода с банка или эквивалента).
3. **Закрытие периодов** — после закрытия месяца данные периода не «переписываются» задним числом без явной корректирующей записи.
4. **Мультивалютность** — операции в валюте клиента/банка с **зафиксированным курсом на дату операции** для консолидации отчётов.
5. **Идемпотентность** — повторная обработка одного и того же внешнего события (webhook банка, повторный клик) не создаёт дубликатов.
6. **Путь к double-entry** — текущая модель не должна блокировать появление полноценного плана счетов и проводок без «сжигания» истории.

Что **намеренно не входит в v1 журнала** (Phase 2+):

- Полный **chart of accounts** как в ERP.
- Готовые **бухгалтерские формы** уровня 1С (баланс, ОСВ) без дополнительной продуктовой спецификации.
- Замена NBOS Finance отдельным продуктом — не отдельная тема этого документа; важны **границы модуля** (см. ниже).

## Operational Journal (single-entry)

Основная идея варианта **B**: центральный **операционный журнал** — это **append-only** набор записей без двухсторонних проводок. Каждая строка — одна **экономическая величина** в одном направлении (доход или расход, или нейтральное техническое движение — по росту продукта), с явным признаком **какого базиса** она касается.

### Логические поля записи журнала

| Поле                | Назначение                                                                                                                                |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `id`                | Уникальный идентификатор записи                                                                                                           |
| `amount`            | Сумма (знак + направление по конвенции продукта: например + доход, − расход)                                                              |
| `currency`          | Валюта операции                                                                                                                           |
| `fx_rate_applied`   | Курс к **функциональной валюте компании** на дату операции (если валюта ≠ функциональной)                                                 |
| `functional_amount` | Сумма в функциональной валюте (derived или хранимое — решение реализации)                                                                 |
| `booked_at`         | Дата/время фиксации записи в системе                                                                                                      |
| `recognition_basis` | `accrual` \| `cash` — к какой проекции относится эта строка                                                                               |
| `posting_period_id` | Связь с **открытым** финансовым периодом (месяц)                                                                                          |
| `idempotency_key`   | Ключ идемпотентности: один ключ = максимум одна запись                                                                                    |
| `source_type`       | Тип источника домена (например `Payment`, `InvoiceCard`, `ExpenseCard`, `ExpensePayment`, `PayrollRun`, `SalaryLine`, `ManualAdjustment`) |
| `source_id`         | Идентификатор источника в том домене                                                                                                      |
| `description`       | Человекочитаемое пояснение                                                                                                                |
| `status`            | Активная запись / отменённая **сторнирующей** записью (не UPDATE суммы)                                                                   |

**Правила:**

- **Нет произвольных UPDATE** суммы: исправление — новая запись с корректирующим знаком или пара «reversal + correct» на уровне продукта.
- Любая интеграция с банком / платёжным шлюзом должна выставлять **стабильный** `idempotency_key` (идемпотентность по внешнему id платежа).

## Dimensional tagging (разрезы аналитики)

Каждая запись журнала несёт **теги измерений**, чтобы строить P&L и cash-flow по уровням из [01-Finance-Overview.md](./01-Finance-Overview.md):

| Измерение       | Обязательность       | Примечание                                                           |
| --------------- | -------------------- | -------------------------------------------------------------------- |
| `company_id`    | Да                   | Юридическое лицо / контур компании при мульти-юрлица (если появится) |
| `project_id`    | По правилам операции | Если операция относится к проекту                                    |
| `product_id`    | По правилам операции | Если операция относится к конкретному продукту                       |
| `order_id`      | По правилам операции | Детализация заказа                                                   |
| `employee_id`   | По правилам          | Зарплата, бонус сотрудника                                           |
| `department_id` | Опционально          | Для unit economics по отделам                                        |
| `partner_id`    | Опционально          | Партнёрские начисления / payout batches / partner service revenue    |

Пустые измерения допустимы только если операция по определению **компании целиком** (например часть оверхеда без разнесения).

## Accrual vs Cash

| Базис       | Вопрос                                        | Примеры признания                                                                               |
| ----------- | --------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| **Accrual** | Когда **экономически** признаём доход/расход? | Выручка по оказанию услуги/по правилам заказа; резерв под ожидаемый расход по политике компании |
| **Cash**    | Когда **деньги** вошли или вышли?             | Платёж на счёт, выплата зарплаты с банка                                                        |

NBOS строит **две проекции**:

- **P&L и маржинальность по проектам/заказам** — преимущественно на **accrual**-записях (согласовать с определениями в [06-PnL-Reports.md](./06-PnL-Reports.md)).
- **Cash flow и «сколько денег на счету»** — на **cash**-записях.

Одно и то же хозяйственное событие может порождать **две строки** с разным `recognition_basis`, либо одну строку с чёткой политикой — это уточняется при реализации, но **принцип двух базисов** в документации фиксируется.

## Period Close (закрытие периода)

1. **Финансовый период** — как минимум **календарный месяц** (расширение до недели — только если появится продуктовая необходимость).
2. Периоды бывают **open** / **closed**.
3. После **close**:
   - новые операции с датой внутри закрытого периода **запрещены** для пользователей (кроме роли с явным правом «reopen period» — редкий кейс);
   - исправления прошлых периодов — только **корректирующими** записями в **открытом** периоде с ссылкой на первичную запись.
4. Связь с **payroll cycle** — см. [05-Bonus-and-Payroll.md](./05-Bonus-and-Payroll.md): закрытие зарплатной ведомости за месяц N согласуется с политикой закрытия финансового периода.
5. Период нельзя закрывать, пока payroll run не закрыт или не перенесён осознанно, ключевые `Payment` / `Expense Payment` сверены, а ручные корректировки внесены.
6. `Expense Backlog` показывается отдельно в cash forecast и не должен автоматически попадать в текущий платёжный цикл.

## Multi-currency

1. Каждая операция хранит **валюту как есть** у источника (счёт клиента, валюта платежа).
2. На дату операции фиксируется **курс** к функциональной валюте компании (источник курса — политика: ручной ввод на день / интеграция / официальный курс ЦБ — **тип поля**, не конкретный провайдер).
3. Консолидированные отчёты строятся в **функциональной валюте**; пересчёт исторических курсов «задним числом» для старых операций — **запрещён** без отдельной корректирующей политики (иначе отчёты год к году несопоставимы).

## Эволюция к double-entry (без переписывания истории)

Текущая модель — **single-sided entries**. Переход к double-entry означает:

1. Введение **плана счетов** (иерархия счетов).
2. Замена или расширение строки журнала до **минимум двух ног** (debit/credit) с инвариантом **сумма debits = сумма credits** на один `journal_entry_group_id`.
3. Миграция: существующие single-entry строки маппятся на **стандартные шаблоны** проводок (например «поступление от клиента» → Дт Банк, Кт Выручка) — детали реализации позже.

Важно: **append-only** и **`idempotency_key` сохраняются**; меняется только форма группировки записей.

## Границы модуля Finance

NBOS **модульна** по структуре продуктовых областей (`02-Modules/*`, см. также слои в [02-Platform-Architecture-Layers.md](../../01-Platform-Overview/02-Platform-Architecture-Layers.md)). Модуль **Finance** отвечает за деньги компании: счета, платежи, расходы, подписки, отчёты, бонусный контур и зарплатная ведомость **как процессы учёта и выплат**.

**Внутри Finance:**

- Invoice Cards, Payments, статусы оплаты, привязка к заказу, проекту и продукту.
- Expense Plans, Expense Cards, Expense Payments, Expense Backlog.
- Bonus Board, Payroll Run, Salary Board, Employee Wallet.
- Операционный журнал и политика периодов/валют (этот документ).

**Вне Finance (но связано по ID и событиям):**

- **CRM:** сделки, стадии, лиды — источник сделки, не место хранения банковских остатков.
- **Projects Hub:** исполнение работ, статусы продуктов — триггеры для Delivery-бонусов, но не дублирование платёжного учёта.
- **My Company:** профиль сотрудника, базовая ставка — входные данные для payroll.

**Принцип:** бизнес-логика «что произошло в сделке/проекте» живёт в своих модулях; Finance **подписывается** на согласованные события и фиксирует денежные последствия и отчётные срезы. Так модуль Finance можно **наращивать** (новые отчёты, журнал, валюты), не смешивая код расчёта бонуса с кодом тасок — границы на уровне домена и интерфейсов между командами/пакетами при реализации.
