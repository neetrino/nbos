# Employee Wallet (кошелёк сотрудника)

## Назначение

**Employee Wallet** — это **read-only** интерфейс для сотрудника: единое место, где он видит **свои** финансы в компании — базовую зарплату, бонусы по проектам и заказам, ожидаемые суммы, удержания (holdback), историю выплат.

**Кошелёк не заменяет** рабочие доски финансового директора:

- **[Bonus Board](./05-Bonus-and-Payroll.md)** — контроль и операции по бонусам компании (в т.ч. всех сотрудников).
- **[Salary Board](./05-Bonus-and-Payroll.md)** — подготовка и закрытие зарплатной ведомости за период (раздел в том же документе).

Источник правды по суммам и статусам бонусов — **[03-Bonus-Payroll-Logic.md](../../03-Business-Logic/03-Bonus-Payroll-Logic.md)** и связанные процессы; Wallet **только отображает** эти данные в удобной для сотрудника форме (проекция поверх существующих сущностей, при необходимости с кэшем для скорости).

Операционный учёт компании и журнал: **[09-Finance-Core-Architecture.md](./09-Finance-Core-Architecture.md)** — кошелёк сотрудника **не** является отдельным «счётом» в журнале; для сотрудника это паспорт начислений и выплат из домена Bonus / Payroll.

## Роли и ответственность

| Роль                      | Интерфейс                         | Назначение                                                          |
| ------------------------- | --------------------------------- | ------------------------------------------------------------------- |
| Сотрудник                 | **Employee Wallet**               | Понятная картина своих доходов, мотивация, прозрачность по проектам |
| Финансовый директор / CEO | Bonus Board, Salary Board, отчёты | Управление, сверки, выплаты, контроль бюджета                       |

## Принципы UX

1. **Полнота для мотивации** — сотрудник видит не только то, что уже «жёстко» к выплате, но и **прогноз** (потенциальные бонусы по активным проектам), и **промежуточные** состояния («сделал работу — ждём оплату клиента», «в гарантийном удержании»), согласованно с бизнес-логикой.
2. **Разрез по проектам** — для каждого проекта видно вклад в доход (тип бонуса, сумма, статус). Дополнительно — **сводка по всем активным проектам**: «если завершу условия по всем текущим проектам — ориентировочный суммарный бонус» (на основе Incoming / формализованного прогноза).
3. **Согласованность терминов** — в UI кошелька используются **понятные группы** (см. маппинг ниже); технические имена состояний из логики не скрываются в подсказках / детализации карточки для доверия к цифрам.
4. **Нет редактирования** — сотрудник не меняет суммы и статусы; всё происходит через процессы CRM, Projects Hub и Finance (как сегодня).

## Карта состояний: логика ↔ Wallet UI

В [03-Bonus-Payroll-Logic.md](../../03-Business-Logic/03-Bonus-Payroll-Logic.md) описана **полная цепочка**:

`Incoming → Earned → Pending Eligibility → Vested → Holdback → Active → Paid` (+ `Clawback`).

На **[Bonus Board](./05-Bonus-and-Payroll.md)** в интерфейсе контроля часто используются **три колонки**: **Incoming / Active / Paid**.

В **Employee Wallet** группируем состояния так, чтобы сотрудник видел **смысл**, а не только три колонки доски:

| Состояние (канон логики)    | Группа в Wallet (рабочее название)  | Что видит сотрудник                                                                                                                                       |
| --------------------------- | ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Incoming                    | **Прогноз / потенциал**             | «Сколько могу получить, когда условия по проекту сработают» — сумма и проект                                                                              |
| Earned, Pending Eligibility | **Ожидание условий**                | «Работа зачтена / в пути — ждём оплату клиента или другое финансовое условие»                                                                             |
| Vested                      | **Подтверждено, дальше по графику** | «Финансовое условие выполнено; дальше — график выплаты / holdback по правилам delivery»                                                                   |
| Holdback                    | **Гарантийное удержание**           | Часть суммы удержана на период; отдельная строка и ссылка на политику holdback в логике                                                                   |
| Active                      | **К выплате в ближайшем цикле**     | Попадает в ведомость следующего payroll (как в [05-Bonus-and-Payroll.md](./05-Bonus-and-Payroll.md): бонусы месяца N выплачиваются в окне 1–10 числа N+1) |
| Paid                        | **Выплачено**                       | История; привязка к месяцу выплаты                                                                                                                        |
| Clawback                    | **Отзыв / корректировка**           | Отрицательное событие в истории с объяснением (возврат клиента и т.д.)                                                                                    |

**Holdback** и **Clawback** описаны в логике ([раздел Holdback / Clawback](../../03-Business-Logic/03-Bonus-Payroll-Logic.md)); в Wallet — краткие тексты-пояснения и суммы, без дублирования полных таблиц коэффициентов.

## Основные блоки экрана Wallet

### 1. Сводка периода (месяц)

- **Базовая зарплата (Fix)** за текущий период — из профиля сотрудника / кадрового контура (видимость суммы — по политике RBAC; минимум сотрудник видит «плановую» часть, если так решено в матрице доступов).
- **Бонусы к выплате** — сумма бонусов в состояниях, которые войдут в ближайший **payroll** (в первую очередь **Active**, согласованно с [05-Bonus-and-Payroll.md](./05-Bonus-and-Payroll.md)).
- **Ориентир «следующая выплата»** — дата/окно по политике компании (например 1–10 числа).

### 2. Разрез по проектам

Таблица или список карточек: **проект → заказ (если есть) → тип бонуса → сумма → группа из таблицы маппинга выше**.

Сотрудник может раскрыть проект и увидеть **все связанные бонусы** (включая неактивные и прогнозные).

### 3. Агрегат «Н активных проектов»

- Количество проектов, по которым есть ненулевой **Incoming / ожидание условий**.
- **Суммарный потенциал** (осторожная формулировка в UI: «если выполнятся условия по всем текущим проектам») — для мотивации и планирования.

### 4. История зарплат и бонусов

- Помесячная история: **итого к выплате** (Fix + бонусы за период), статус «выплачено».
- Детализация до уровня **бонусных строк** и ссылки на период ведомости (источник — Salary Board / payroll run).

### 5. Уведомления

Новые типы событий для Wallet **не обязательны**: существующие события из [13-Notifications](../13-Notifications/01-Notifications-System.md) по бонусам и payroll покрывают сценарий «узнать в канале уведомлений»; при появлении отдельного дайджеста Wallet — расширять отдельной задачей.

## Техническая оговорка (реализация)

- Wallet = **read model**: запросы к `Bonus`, `Employee`, данным payroll, при необходимости материализованное представление для быстрого списка.
- **Не** вводить вторую «сумму кошелька», не синхронизируемую с Bonus/Payroll.
- Ссылки на сотрудника: вход из **[Команда и сотрудники](../07-My-Company/02-Team-Employees.md)** — действие «Открыть кошелёк» / «Мои финансы».
