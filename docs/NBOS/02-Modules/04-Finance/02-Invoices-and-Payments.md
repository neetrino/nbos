# Счета и платежи (Invoices & Payments)

## Общая концепция

В NBOS `Invoice Card / Карточка инвойса` по смыслу — это не просто PDF-счёт, а **карточка денег, которые клиент должен нам заплатить**.

То есть для команды это главный объект контроля:

- сколько денег мы ждём;
- от кого;
- по какому источнику;
- до какого срока;
- нужен ли официальный счёт;
- отправлены ли напоминания;
- оплачено или нет.

```text
Источник (Order / Subscription / Domain / Service)
        ->
Invoice Card
        ->
Official Invoice request if needed
        ->
Awaiting Payment
        ->
Payment confirmed
```

---

## Главный принцип

Не нужно смешивать в одной логике четыре разные вещи:

1. саму карточку ожидаемой оплаты;
2. запрос на официальный счёт;
3. напоминания и уведомления;
4. факт поступления денег.

Правильнее:

- `Invoice Card` — главный объект;
- `Official Invoice` — небольшой подпроцесс внутри карточки;
- `Notifications` — отдельная автоматика;
- `Payment` — отдельная запись факта оплаты.

---

## Invoice Card (Карточка инвойса)

### Что это такое

`Invoice Card` — карточка суммы, которую клиент должен нам заплатить.

Она может быть создана из:

- `Order`
- `Subscription`
- `Domain`
- `Service / License`
- другого разрешённого финансового источника

### Основные поля карточки

| Поле                    | Описание                                                          |
| ----------------------- | ----------------------------------------------------------------- |
| `code`                  | Уникальный номер карточки                                         |
| `type`                  | Development / Extension / Subscription / Domain / Service / Other |
| `source_entity`         | Из чего создана карточка                                          |
| `project`               | Проект                                                            |
| `product`               | Продукт, если применимо                                           |
| `company`               | Компания-плательщик                                               |
| `contact`               | Контактное лицо                                                   |
| `amount`                | Сумма                                                             |
| `currency`              | Валюта                                                            |
| `tax_status`            | `Tax` или `Free`                                                  |
| `notifications_enabled` | Разрешены ли клиентские напоминания                               |
| `created_at`            | Дата создания                                                     |
| `due_date`              | Крайняя дата оплаты                                               |
| `money_status`          | Статус денег                                                      |
| `notes`                 | Заметки                                                           |

### Что наследуется из источника

При создании карточки должны автоматически подтягиваться:

- `tax_status`
- `notifications_enabled`
- проектный контекст
- source link
- при необходимости контакт / компания / продукт

Примеры:

- из `Subscription` приходит `tax_status`, `notifications_enabled`, `billing day`;
- из CRM / Order чаще всего `notifications_enabled = Off` по умолчанию;
- из `Domain / Service` подтягиваются правила сервиса и проекта.

## Money Status (Статус денег)

Это главный статус карточки.

### Канонические статусы

- `New / Новый`
- `Awaiting Payment / Ждём оплату`
- `Overdue / Просрочено`
- `On Hold / На паузе`
- `Paid / Оплачено`
- `Cancelled / Отменено`

### Когда какой статус используется

| Статус             | Смысл                                               |
| ------------------ | --------------------------------------------------- |
| `New`              | карточка только создана                             |
| `Awaiting Payment` | можно уже реально ждать деньги                      |
| `Overdue`          | срок оплаты прошёл, денег нет                       |
| `On Hold`          | вручную остановили активное взыскание / напоминания |
| `Paid`             | деньги подтверждены                                 |
| `Cancelled`        | эту оплату больше не ждём                           |

### Правила переходов

| Из                                                 | В                  | Как происходит                                                  |
| -------------------------------------------------- | ------------------ | --------------------------------------------------------------- |
| `New`                                              | `Awaiting Payment` | вручную или автоматикой, если карточка готова к ожиданию оплаты |
| `Awaiting Payment`                                 | `Overdue`          | автоматически по due date                                       |
| `Overdue`                                          | `On Hold`          | вручную                                                         |
| `On Hold`                                          | `Awaiting Payment` | вручную                                                         |
| `Awaiting Payment` / `Overdue` / `On Hold`         | `Paid`             | только после подтверждения денег                                |
| `New` / `Awaiting Payment` / `Overdue` / `On Hold` | `Cancelled`        | вручную                                                         |

### Важное правило для `Tax`

Если:

- `tax_status = Tax`
- и запрос на официальный счёт ещё **не отправлен**

то карточка **не должна** запускать клиентские напоминания.

Именно поэтому `Awaiting Payment` и автоматика напоминаний зависят не только от due date, но и от состояния блока `Official Invoice`.

---

## Official Invoice (Официальный счёт)

Это не отдельная большая сущность и не отдельная колонка доски.

Это **небольшой блок внутри карточки**.

### Минимальные поля блока

| Поле           | Описание                                            |
| -------------- | --------------------------------------------------- |
| `tax_status`   | `Tax` или `Free`                                    |
| `request_sent` | Отправлен ли запрос в бухгалтерскую WhatsApp-группу |
| `sent_at`      | Когда отправили                                     |
| `cancelled_at` | Когда отменили прошлый запрос                       |

### Логика

Если `tax_status = Free`:

- ничего в бухгалтерскую группу не отправляем;
- карточка может идти дальше без official invoice subprocess.

Если `tax_status = Tax`:

- в нужный момент отправляется сообщение в бухгалтерскую WhatsApp-группу;
- система отмечает `request_sent = Yes`;
- если запрос отменён, система фиксирует отмену;
- после отмены можно отправить заново.

### Кнопки

Минимально достаточно таких действий:

- `Send Official Invoice Request / Отправить запрос на счёт`
- `Cancel Previous Request / Отменить прошлый запрос`
- `Send Again / Отправить заново`

Можно реализовать это и одной последовательной кнопкой, если UI так удобнее, но логика должна оставаться именно такой.

## Notifications (Уведомления)

Уведомления не должны быть stage карточки.

Это отдельная автоматическая логика.

### Основное правило

Если:

- `notifications_enabled = Off`

то клиентских напоминаний нет.

Если:

- `notifications_enabled = On`
- карточка находится в рабочем статусе ожидания денег

то работают правила напоминаний.

### Каноническое правило напоминаний

Примеры допустимых автоматик:

- отправить запрос на официальный счёт за `N` дней до даты оплаты;
- через несколько дней отправить клиенту первое напоминание;
- потом второе напоминание;
- не отправлять напоминания, если карточка в `On Hold`;
- не отправлять напоминания, если для `Tax` ещё нет `request_sent = Yes`.

Для автоматических карточек (например, подписки) это должно запускаться без ручного участия.

Для ручных карточек пользователь должен иметь возможность отправить процесс сразу вручную.

---

## Action History (История действий)

Карточке нужна не столько “история оплат”, сколько **история действий**.

Туда попадает:

- карточка создана;
- сумма изменена;
- due date изменён;
- запрос на официальный счёт отправлен;
- запрос отменён;
- первое напоминание отправлено;
- второе напоминание отправлено;
- карточка поставлена на паузу;
- карточка отмечена как оплаченная;
- карточка отменена.

Это и есть главный журнал работы по карточке.

## Payment (Платёж)

### Что это такое

`Payment / Платёж` — это факт, что деньги реально пришли.

Он создаётся после подтверждения оплаты финансовым директором.

### Поля платежа

| Поле             | Описание                     |
| ---------------- | ---------------------------- |
| `payment_id`     | Уникальный идентификатор     |
| `invoice_id`     | Какая карточка была оплачена |
| `amount`         | Сумма                        |
| `payment_date`   | Дата поступления             |
| `payment_method` | Способ оплаты                |
| `confirmed_by`   | Кто подтвердил               |
| `notes`          | Заметки                      |

### Что происходит при подтверждении оплаты

1. создаётся `Payment`;
2. карточка получает статус `Paid`;
3. запускаются downstream-события в зависимости от источника:
   - order / deal logic;
   - subscription logic;
   - domain / service purchase tasks;
   - bonus logic, если применимо;
   - partner logic, если применимо: `Payment -> Partner Accrual -> Partner Balance`, но не Expense напрямую.

## Invoice Board (Доска инвойсов)

Пока модуль называется `Invoices`, в интерфейсе оставляем это название. Позже можно переименовать, если команда захочет более точный термин.

### Колонки доски

| Колонка            | Смысл          |
| ------------------ | -------------- |
| `New`              | новая карточка |
| `Awaiting Payment` | ждём деньги    |
| `Overdue`          | просрочка      |
| `On Hold`          | пауза          |
| `Paid`             | оплачено       |
| `Cancelled`        | отменено       |

### Для чего нужна доска

Это не бухгалтерская доска создания PDF.

Это **рабочая доска контроля денег к получению**.

Именно здесь Finance team видит:

- что должно быть оплачено;
- что просрочено;
- что на паузе;
- что уже пришло.

## Массовая автоматика для Subscription

Для подписок автоматика обязательна.

Пример:

1. система создаёт новую `Invoice Card`;
2. подтягивает `tax_status` и `notifications_enabled`;
3. если нужен официальный счёт — в нужный день автоматически шлёт запрос в бухгалтерскую WhatsApp-группу;
4. только после этого начинает клиентские напоминания;
5. после оплаты карточка закрывается как `Paid`.

Это особенно важно для сценариев, где в месяц создаются сотни карточек.

## Связь с другими модулями

### CRM / Orders

- manual development / extension invoices;
- логика перехода заказа и сделки после оплаты.

### Subscriptions

- массовое автосоздание карточек;
- наследование `tax_status` и `notifications_enabled`;
- связь с billing cycle.

### Domains / Services

- карточки на продление доменов, аккаунтов и сервисов;
- после оплаты создаются задачи на покупку / продление.

### Tasks / Work Space

- после оплаты некоторых карточек создаются задачи:
  - купить домен;
  - продлить сервис;
  - выполнить pass-through действие.

---

## Что важно не смешивать

В каноне нужно всегда помнить:

- `Invoice Card` — это деньги, которые нам должны;
- `Official Invoice request` — это запрос в внешнюю бухгалтерию;
- `Notifications` — это автоматика сообщений;
- `Payment` — это факт поступления денег.

Только при таком разделении модуль остаётся чистым и масштабируемым.
