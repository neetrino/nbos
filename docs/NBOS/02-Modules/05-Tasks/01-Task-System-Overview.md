# Система задач — Обзор

> NBOS Platform — атомарная единица работы, объединяющая людей, процессы и контроль

## Назначение

**Задача (Task)** — это атомарная единица работы в NBOS Platform. Всё, что нужно сделать — от разработки фичи до звонка клиенту — выражается через задачу. Задачи пронизывают всю платформу: они живут внутри Products, Extensions, Support, а также могут существовать отдельно.

**Ключевой принцип:** любая работа = задача. Если работа не оформлена как задача — она не существует для системы, не учитывается в прогрессе, не влияет на аналитику и не генерирует бонусы.

---

## Контексты существования задач

Задачи не существуют в вакууме — они всегда привязаны к определённому контексту:

| Контекст | Описание | Пример |
|----------|----------|--------|
| **Product** | Задачи разработки продукта | «Frontend-вёрстка главной страницы» |
| **Extension** | Задачи доработки | «Добавить форму обратной связи» |
| **Support** | Задачи по тикетам поддержки | «Исправить ошибку 500 на странице оплаты» |
| **Standalone** | Общие задачи без привязки к продукту | «Позвонить клиенту», «Проверить оплату», «Купить домен» |

### Иерархия привязки:

```
Project
  ├── Product → Tasks (scrum/kanban доска продукта)
  ├── Extension → Tasks (kanban доска доработки)
  ├── Support Ticket → Tasks (задачи из тикета)
  └── Standalone Tasks (общие задачи проекта)
```

Standalone-задачи привязаны к проекту, но не к конкретному продукту или доработке. Также возможны полностью свободные задачи вне проекта — для внутренних процессов компании.

## Роли участников задачи

Каждая задача имеет чёткое распределение ролей. Это **критическая функция** системы — роли определяют, кто что видит и кто что может делать.

| Роль | Название (RU) | Описание | Права |
|------|---------------|----------|-------|
| **Creator** | Постановщик | Создал задачу | Полный контроль: редактирование, удаление, переназначение, смена приоритета, закрытие |
| **Assignee** | Исполнитель | Ответственный за выполнение | Смена статуса, добавление комментариев, отметка времени, запрос на переназначение |
| **Co-Assignee** | Соисполнитель | Помогает с выполнением | Смена статуса (ограниченно), добавление комментариев, отметка времени |
| **Observer** | Наблюдатель | Следит за прогрессом | Только чтение. Получает уведомления при изменениях |

### Принцип «кто угодно — кому угодно»

В NBOS **любой сотрудник** может создать задачу для **любого другого** сотрудника, вне зависимости от иерархии:

- Seller создаёт задачу для PM: «Подготовить демо для клиента X»
- PM создаёт задачу для Developer: «Исправить баг на продакшене»
- Developer создаёт задачу для Finance Director: «Оплатить лицензию GitHub»
- CEO создаёт задачу для Marketing: «Подготовить пост для Instagram»
- Developer создаёт задачу для CEO: «Напоминание: согласовать архитектуру»

Это обеспечивает горизонтальную коммуникацию и снимает бюрократические барьеры. Задача — универсальный инструмент взаимодействия между любыми ролями.

## Статусы задачи

Каждая задача проходит через линейный жизненный цикл:

```
Backlog → To Do → In Progress → Review → Done
                                           ↓
                                       Cancelled
```

| Статус | Описание | Кто обычно переводит |
|--------|----------|---------------------|
| **Backlog** | Задача создана, но ещё не запланирована в работу | Creator / PM |
| **To Do** | Запланирована на спринт или на ближайшее выполнение | PM (при планировании) |
| **In Progress** | Исполнитель начал работу | Assignee |
| **Review** | Работа выполнена, ожидает проверки (code review, QA, приёмка PM) | Assignee |
| **Done** | Задача завершена и принята | Creator / PM / Reviewer |
| **Cancelled** | Задача отменена (потеряла актуальность, дубликат) | Creator / PM |

### Правила переходов:

- Backlog → To Do: PM включает задачу в спринт или отмечает как готовую к работе
- To Do → In Progress: исполнитель берёт задачу в работу
- In Progress → Review: исполнитель завершил работу, отправляет на проверку
- Review → Done: проверяющий (PM, Creator, другой разработчик) подтвердил выполнение
- Review → In Progress: проверяющий вернул задачу на доработку
- Любой статус → Cancelled: Creator или PM отменяет задачу

## Приоритеты

| Приоритет | Описание | Визуальный маркер |
|-----------|----------|-------------------|
| **Critical** | Блокирует работу, требует немедленного внимания | Красный |
| **High** | Важная задача, приоритет на текущий спринт | Оранжевый |
| **Normal** | Стандартная задача, выполняется в порядке очереди | Синий |
| **Low** | Можно отложить, выполняется при наличии свободного времени | Серый |

Приоритет влияет на:
- Сортировку задач на доске (Critical всегда наверху)
- Уведомления (Critical → мгновенное push-уведомление)
- SLA (для задач Support — время реакции зависит от приоритета)

## Поля карточки задачи

### Основные поля:

| Поле | Тип | Обязательное | Описание |
|------|-----|-------------|----------|
| **Название** | Текст | ✅ | Краткое описание задачи |
| **Описание** | Rich Text | ❌ | Детальное описание, ТЗ, ссылки |
| **Исполнитель (Assignee)** | User | ✅ | Ответственный за выполнение |
| **Постановщик (Creator)** | User (авто) | ✅ | Заполняется автоматически |
| **Соисполнители** | User[] | ❌ | Список помощников |
| **Наблюдатели** | User[] | ❌ | Кто следит за задачей |
| **Проект** | Project | ❌ | Привязка к проекту |
| **Product / Extension** | Entity | ❌ | Привязка к продукту или доработке |
| **Приоритет** | Enum | ✅ (default: Normal) | Critical / High / Normal / Low |
| **Дедлайн** | Date | ❌ | Крайний срок выполнения |
| **Спринт** | Sprint | ❌ | К какому спринту привязана |
| **Теги** | Tag[] | ❌ | Произвольные метки (bug, feature, design и т.д.) |
| **Вложения** | File[] | ❌ | Файлы, скриншоты, документы |
| **Оценка времени** | Duration | ❌ | Сколько часов/дней планируется на задачу |
| **Затраченное время** | Duration | ❌ | Сколько реально потрачено (заполняет исполнитель) |

## Чат задачи (Task Chat)

Каждая задача может иметь собственный чат — поток обсуждения, привязанный к задаче.

### Принцип работы:

1. Чат **создаётся при первом сообщении** — не раньше. Пустые задачи не создают лишних чатов
2. Чат виден **всем участникам задачи**: Creator, Assignee, Co-Assignees, Observers
3. Уведомления о новых сообщениях отправляются всем участникам
4. Чат задачи также виден в **проектном разделе «Task Chats»** — PM может просматривать все обсуждения по задачам проекта в одном месте

### Где отображаются Task Chats:

```
Messenger
  ├── Personal Chats (личные)
  ├── Group Chats (групповые)
  ├── Project Chats
  │     ├── General (основной чат проекта)
  │     ├── Client Chat (WhatsApp группа)
  │     └── Task Chats ← все чаты задач проекта
  └── ...
```

### Типичное использование:

- Developer пишет в чат задачи: «Не могу воспроизвести баг, нужен скриншот»
- PM отвечает: «Прилагаю видео от клиента»
- Observer (CEO) читает переписку, но не участвует

## Способы создания задачи

| Способ | Описание |
|--------|----------|
| **Scrum/Kanban доска** | Нажатие «+ Задача» в колонке доски (контекст продукта/доработки) |
| **Карточка проекта** | Создание задачи из таба Tasks внутри проекта |
| **Мессенджер** | Прямо из сообщения в чате: «Создать задачу» → текст сообщения становится описанием |
| **Глобальная кнопка** | Кнопка «+ Задача» из любого места платформы |
| **Шаблон** | Автоматическое создание набора задач при загрузке шаблона для нового продукта |
| **Тикет Support** | Из тикета поддержки автоматически создаётся задача для разработчика |

### Создание задачи из мессенджера:

Особенно удобно для ситуаций, когда клиент или коллега описывает проблему в чате. PM выделяет сообщение → «Создать задачу» → система подставляет текст сообщения в описание, PM выбирает исполнителя и проект.

## Связи между задачами (Linking)

Задачи могут быть связаны между собой для отражения зависимостей:

| Тип связи | Описание | Эффект |
|-----------|----------|--------|
| **Blocked by** | Задача B заблокирована задачей A | Задача B не может перейти в In Progress, пока A не в Done |
| **Blocks** | Задача A блокирует задачу B | Обратная сторона «Blocked by» |
| **Related to** | Задачи связаны по смыслу | Информационная связь без блокировки |
| **Duplicate of** | Задача является дубликатом другой | При закрытии оригинала дубликат автоматически отменяется |

Связи отображаются в карточке задачи и на Timeline/Gantt-представлении (стрелки зависимостей).

## Подзадачи (Subtasks)

Для сложных задач можно создавать подзадачи — вложенные задачи с упрощённой структурой:

- Подзадача наследует проект и контекст родительской задачи
- Подзадача может иметь своего исполнителя (отличного от родительской)
- Прогресс родительской задачи вычисляется автоматически: % закрытых подзадач
- Родительская задача не может быть закрыта, пока не закрыты все подзадачи

### Пример:

```
Задача: «Интеграция платёжной системы»
  ├── Подзадача: «Настроить аккаунт Stripe»        — Assignee: PM
  ├── Подзадача: «Реализовать API платежей»         — Assignee: Backend Dev
  ├── Подзадача: «Интегрировать на фронт»           — Assignee: Frontend Dev
  └── Подзадача: «Написать тесты»                   — Assignee: QA
```

## Повторяющиеся задачи (Recurring Tasks)

Для периодической работы можно настроить автоматическое создание задач по расписанию:

| Параметр | Описание |
|----------|----------|
| **Частота** | Ежедневно, еженедельно, ежемесячно, раз в квартал, произвольный cron |
| **Шаблон задачи** | Название, описание, исполнитель, приоритет, проект |
| **Автоназначение** | Задача автоматически назначается заданному исполнителю |
| **Дедлайн** | Автоматически рассчитывается от даты создания |

### Примеры recurring tasks:

- «Ежемесячная проверка доменов» — каждый 1-й числа, Assignee: Tech Specialist
- «Еженедельный отчёт по спринту» — каждый понедельник, Assignee: PM
- «Ежеквартальный аудит безопасности» — каждые 3 месяца, Assignee: QA Lead
- «Ежемесячный бэкап всех проектов» — каждый 15-й числа, Assignee: DevOps

## Уведомления по задачам

| Событие | Кому уведомление |
|---------|------------------|
| Задача создана | Assignee, Co-Assignees |
| Статус изменён | Creator, Observers |
| Комментарий добавлен | Все участники (кроме автора) |
| Приближается дедлайн (за 1 день) | Assignee |
| Дедлайн просрочен | Assignee, Creator, PM проекта |
| Задача переназначена | Новый Assignee, Creator |
| Задача заблокирована/разблокирована | Assignee заблокированной задачи |

| Модуль | Связь |
|--------|-------|

| **Extensions** | Задачи доработки на kanban-доске доработки |
| **Support** | Тикет → задача для исполнителя |

| **Messenger** | Создание задачи из сообщения, Task Chat |
