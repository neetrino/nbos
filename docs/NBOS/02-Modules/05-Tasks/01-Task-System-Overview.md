# Task System Overview

> NBOS Platform — универсальная система управления работой. Задача остаётся отдельной сущностью, а Work Space управляет её планированием и отображением.

## 1. Назначение модуля

Модуль `Tasks` — это общий task engine платформы. Он нужен не только для delivery-команды, но и для личных задач owner, внутренних задач компании, задач по финансам, support и любых операционных действий.

Канон:

- `Task` — отдельная рабочая единица
- `Work Space` — рабочее пространство, где задачи планируются и выполняются
- `Sprint`, `Backlog`, `Epics`, `Views`, `Rules` — подсущности и механики внутри `Work Space`

`Tasks` остаётся top-level модулем платформы. `Work Space` не выносится в отдельный top-level module, но является отдельной core-сущностью внутри него.

## 2. Граница модуля

`Tasks` не заменяет другие бизнес-сущности. Он обслуживает их.

Примеры:

- `Projects Hub` хранит `Project / Product / Extension`
- `Finance` хранит `Invoice / Subscription / Expense`
- `Support` хранит `Ticket`
- `Tasks` хранит работу, которую нужно выполнить по этим сущностям

При этом задача может существовать и без привязки к другой сущности.

Примеры standalone-задач:

- личная задача owner
- внутренняя задача marketing
- организационная задача finance
- напоминание CEO

## 3. Task и контекст

### 3.1. Основной принцип

Задача может существовать:

- без связей вообще
- с одним основным контекстом
- с несколькими дополнительными links

### 3.2. Основной контекст задачи

У задачи должен быть `primary context`, который определяет:

- где задача живёт по умолчанию
- в каком списке она открывается
- в какой аналитике она считается
- какое рабочее пространство её обычно управляет

Базовые типы primary context:

- `Standalone`
- `Project`
- `Product`
- `Extension`
- `Support Ticket`
- `Deal`
- `Invoice`
- `Subscription`
- `Company/Internal`

### 3.3. Дополнительные links

Помимо primary context, задача может иметь дополнительные связи.

Пример:

- основная задача живёт в `Finance Work Space`
- дополнительно привязана к `Invoice`
- и ещё связана с `Project`

Это позволяет строить сильную навигацию и аналитику, не ломая отдельность самой задачи.

## 4. Роли в задаче

| Роль          | Смысл               | Базовые права                                                    |
| ------------- | ------------------- | ---------------------------------------------------------------- |
| `Creator`     | Кто поставил задачу | редактирование, удаление, смена исполнителя, закрытие            |
| `Assignee`    | Главный исполнитель | работа по задаче, смена статуса, запрос review                   |
| `Co-Assignee` | Соисполнитель       | помощь в выполнении, комментарии, частичные действия             |
| `Observer`    | Наблюдатель         | чтение, уведомления, комментарии                                 |
| `Reviewer`    | Проверяющий         | подтверждение review / completion, если для задачи это требуется |

Принцип сохраняется прежний: любой сотрудник может поставить задачу любому сотруднику, если это не ограничено правами доступа.

## 5. Workflow Status задачи

### 5.1. Канонические статусы

`Workflow Status` — это собственный жизненный цикл задачи. Он не должен смешиваться с backlog и sprint planning.

Канон:

- `Open`
- `In Progress`
- `Review`
- `Completed`
- `Deferred`
- `Cancelled`

### 5.2. Смысл статусов

| Статус        | Описание                                  |
| ------------- | ----------------------------------------- |
| `Open`        | Задача существует и ожидает начала работы |
| `In Progress` | Исполнитель начал работу                  |
| `Review`      | Работа выполнена и ожидает проверки       |
| `Completed`   | Задача подтверждённо завершена            |
| `Deferred`    | Работа временно отложена                  |
| `Cancelled`   | Задача отменена и больше неактуальна      |

### 5.3. Что не является workflow status

Следующие вещи не должны быть task-status:

- `Backlog`
- `Future Sprint`
- `Active Sprint`
- `Blocked`
- `Overdue`
- `Waiting`

Это либо planning state, либо derived flag.

## 6. Planning отдельно от статуса

`Planning` и `Workflow Status` — это два разных измерения.

Пример:

- задача может быть `Open`, но лежать в `Backlog`
- задача может быть `Open`, но быть уже включённой в будущий спринт
- задача может быть `In Progress` и находиться в `Active Sprint`

Именно поэтому backlog нельзя использовать как основной статус задачи.

Planning-логика описана отдельно в `Work Space`.

## 7. Derived flags

Некоторые состояния не являются отдельными статусами, а вычисляются поверх задачи:

- `Overdue`
- `Blocked`
- `Waiting`
- `Near Deadline`
- `Needs Review`

Они влияют на:

- цвет карточки
- badge
- фильтры
- уведомления
- эскалацию

## 8. Completion Rules

`Completion Rules` — это правила, которые определяют, можно ли перевести задачу в `Completed`.

Это отдельный слой поверх статусов и он должен быть заложен в каноне сразу.

Примеры completion rules:

- нужен `Review`
- checklist должен быть завершён
- должен быть прикреплён хотя бы один файл
- должно быть заполнено конкретное поле
- creator должен подтвердить выполнение
- linked-entity condition должна быть выполнена

Если правило не выполнено, задача не может завершиться и система должна показать причину блокировки.

Это особенно важно для:

- delivery handoff tasks
- finance control tasks
- системных automation tasks
- задач, где нужна приёмка

## 9. Стандартные поля задачи

| Поле              | Смысл                                |
| ----------------- | ------------------------------------ |
| `title`           | краткое название                     |
| `description`     | подробное описание                   |
| `creator`         | постановщик                          |
| `assignee`        | основной исполнитель                 |
| `coAssignees`     | дополнительные исполнители           |
| `observers`       | наблюдатели                          |
| `reviewer`        | кто принимает review, если применимо |
| `workflowStatus`  | статус задачи                        |

| `links[]`         | дополнительные связи                 |

| `dueDate`         | дедлайн                              |
| `estimate`        | оценка времени/объёма                |

## 10. Связи задач

Базовые типы task linking:

- `Blocked by`
- `Blocks`
- `Related to`
- `Duplicate of`

Также поддерживаются:

- `Parent / Subtask`
- cross-module links to external entities

Родительская задача может иметь собственные completion rules, включая правило: не завершать до закрытия подзадач.

## 11. Task Checklist

Checklist внутри задачи — это обычный task-level инструмент выполнения.

Он используется для:

- списка шагов
- контроля мелких действий
- проверки полноты
- частичного прогресса

Важно:

- `Task Checklist` принадлежит модулю `Tasks`
- `Stage Gate Checklist` принадлежит delivery-логике `Projects Hub`

Эти два слоя не должны смешиваться.

## 12. Task Chat и Activity Feed

В будущем каждая задача должна поддерживать два связанных слоя:

- `Discussion`
- `Activity Feed`

### 12.1. Discussion

Человеческое общение по задаче:

- комментарии
- файлы
- уточнения
- mentions

### 12.2. Activity Feed

Системная история:

- создана задача
- изменён статус
- назначен исполнитель
- изменён дедлайн
- добавлен checklist item
- сработал automation rule
- задача почти просрочена

Task Chat будет развиваться вместе с модулем `Messenger`, но канонически уже считается частью модели задачи.

## 13. Способы создания задач

- из глобального `+ Task`
- из `Work Space`
- из карточки `Deal`
- из `Product / Extension`
- из `Support Ticket`
- из `Messenger`
- из recurring template
- из automation rule
- из launch blueprint

## 14. Интеграция с другими модулями

| Модуль         | Как связан                                              |
| -------------- | ------------------------------------------------------- |
| `Projects Hub` | Work Space продукта/доработки, delivery tasks           |
| `Support`      | тикет может создавать и связывать задачи                |
| `Finance`      | recurring и automation tasks, контроль оплат и сервисов |
| `CRM`          | подготовительные и handoff tasks                        |
| `Messenger`    | task chat и создание задач из сообщений                 |
| `Calendar`     | due dates, sprint dates, reminders                      |

## 15. Канонические правила

1. `Task` — отдельная сущность и может существовать без связей.
2. `Backlog` не является task-status.
3. `Workflow Status` и `Planning` хранятся раздельно.
4. `Work Space` — центральная planning-сущность модуля `Tasks`.
5. `Completion Rules` — отдельный обязательный архитектурный слой.
6. `Task Checklist` и delivery stage gates не смешиваются.
