# Шаблоны и чеклисты (Templates & Checklists)

> NBOS Platform — стандартизация процессов через шаблоны задач и контрольные списки качества

## Обзор

Шаблоны и чеклисты — два инструмента, которые обеспечивают **стандартизацию** и **повторяемость** процессов в Neetrino. Шаблоны автоматически создают набор типовых задач, а чеклисты гарантируют, что ни один важный шаг не будет пропущен.

**Ключевой принцип:** каждый новый продукт начинается не с нуля, а с проверенного набора задач. Каждый переход между стадиями контролируется обязательным чеклистом.

---

# ЧАСТЬ 1: Шаблоны задач (Task Templates)

## 1. Что такое шаблон задач

**Шаблон (Template)** — это предопределённый набор задач, который автоматически создаётся при запуске нового Product или Extension. Шаблон содержит типовые задачи с описаниями, оценками времени, зависимостями и вложенными чеклистами.

### Когда применяется шаблон:

1. PM создаёт новый Product (например, Website)
2. Система предлагает: «Загрузить шаблон задач для Website?»
3. PM подтверждает → автоматически создаётся 15–30 задач с настроенными зависимостями
4. PM адаптирует: удаляет ненужные задачи, добавляет специфические

### Что хранит шаблон:

| Поле шаблона | Описание |
|--------------|----------|
| **Название задачи** | Стандартное название (например, «Frontend-вёрстка») |
| **Описание** | Типовое описание задачи с инструкциями |
| **Роль исполнителя** | Не конкретный сотрудник, а роль: Developer, Designer, QA, PM |
| **Оценка времени** | Стандартная оценка (например, 16 часов) |
| **Приоритет** | Default-приоритет для задачи |
| **Зависимости** | Связи «blocked by» между задачами шаблона |
| **Чеклист задачи** | Вложенный чеклист для конкретной задачи |
| **Спринт (ориентировочный)** | К какому спринту обычно относится задача |
| **Теги** | Стандартные метки (design, backend, frontend, devops и т.д.) |

## 3. Управление шаблонами

### Кто управляет:

| Действие | CEO | PM | Developer |
|----------|-----|-----|-----------|
| Создание нового шаблона | ✅ | ✅ | ❌ |
| Редактирование шаблона | ✅ | ✅ | ❌ |
| Удаление шаблона | ✅ | ❌ | ❌ |
| Загрузка шаблона в продукт | ✅ | ✅ | ❌ |
| Кастомизация после загрузки | ✅ | ✅ | ❌ |

### Процесс кастомизации:

После загрузки шаблона PM может:
- **Удалить** задачи, которые не нужны для конкретного проекта
- **Добавить** специфические задачи клиента
- **Изменить** оценки времени (увеличить/уменьшить в зависимости от сложности)
- **Переназначить** роли на конкретных сотрудников
- **Перестроить** зависимости
- **Добавить/изменить** чеклисты внутри задач

Шаблон — **отправная точка**, не жёсткий каркас. PM адаптирует его под реальность конкретного проекта.

# ЧАСТЬ 2: Чеклисты (Checklists)

## 4. Два уровня чеклистов

В NBOS чеклисты работают на двух уровнях:

```
Уровень 1: Stage Gate Checklists
  → Контролируют переход ПРОДУКТА между стадиями
  → Обязательные, блокируют переход при невыполнении

Уровень 2: Task Checklists
  → Контролируют выполнение внутри ЗАДАЧИ
  → Список пунктов, которые нужно выполнить для завершения задачи
```

---

## 5. Stage Gate Checklists (Чеклисты стадий)

Stage Gate Checklists — это **обязательные контрольные списки**, которые нужно пройти для перехода продукта с одной стадии на другую. Система **блокирует переход**, если хотя бы один обязательный пункт не отмечен.

### 5.1. Creating → Development

> Цель: убедиться, что всё готово для начала разработки

| # | Пункт | Обязательный | Описание |
|---|-------|-------------|----------|
| 1 | Все аккаунты созданы | ✅ | Хостинг, DNS, база данных, почта — все Credential-записи заполнены |
| 2 | Все пароли сохранены в Credentials | ✅ | Ни один пароль не остался только у сотрудника — всё в системе |
| 3 | Среда разработки настроена | ✅ | Dev-сервер, репозиторий, CI/CD (если применимо) |
| 4 | Домен приобретён (если нужно) | ⚠️ По ситуации | Для Website обязательно, для внутренних систем — нет |
| 5 | ТЗ/требования зафиксированы | ✅ | Описание задачи или отдельный документ в Drive проекта |
| 6 | Дизайн согласован с клиентом | ⚠️ По ситуации | Для продуктов с UI-дизайном |

### 5.2. Development → QA/Checking

> Цель: убедиться, что разработка завершена и продукт готов к тестированию

| # | Пункт | Обязательный | Описание |
|---|-------|-------------|----------|
| 1 | Все задачи текущего спринта закрыты | ✅ | Нет задач в статусах To Do / In Progress / Review |
| 2 | Code review пройден | ✅ | Все PR-ы проверены и вмержены |
| 3 | Нет критических багов | ✅ | Все задачи с тегом «bug» и приоритетом Critical/High закрыты |
| 4 | .env файл сохранён в Credentials | ✅ | Все секреты проекта зафиксированы в системе |
| 5 | Deployment на staging выполнен | ✅ | Продукт доступен для тестирования на staging-сервере |
| 6 | Базовый smoke-test пройден | ✅ | Основные сценарии работают (авторизация, основные страницы, формы) |

### 5.3. QA/Checking → Transfer

> Цель: убедиться, что продукт соответствует стандартам качества и безопасности

| # | Пункт | Обязательный | Описание |
|---|-------|-------------|----------|
| 1 | Quality Checklist пройден | ✅ | Все пункты стандарта качества отмечены (см. раздел 6) |
| 2 | Security Checklist пройден | ✅ | Все пункты стандарта безопасности отмечены (см. раздел 6) |
| 3 | Deployment Checklist пройден | ✅ | Все пункты развёртывания выполнены (см. раздел 6) |
| 4 | Демо клиенту запланировано | ✅ | Дата и время демо зафиксированы в календаре |
| 5 | Все баги QA-этапа закрыты | ✅ | Найденные при тестировании баги исправлены |

### 5.4. Transfer → Done

> Цель: убедиться, что клиент принял работу и все обязательства выполнены

| # | Пункт | Обязательный | Описание |
|---|-------|-------------|----------|
| 1 | Клиент подтвердил приёмку | ✅ | Письменное подтверждение (в чате, email или подпись акта) |
| 2 | Нет открытых issue от клиента | ✅ | Все замечания клиента обработаны и закрыты |
| 3 | Все доступы задокументированы | ✅ | Клиент получил или имеет доступ ко всем аккаунтам |
| 4 | Handoff в Support завершён | ✅ | Если проект переходит на maintenance — передача техсвязи выполнена |
| 5 | Финальный бэкап выполнен | ✅ | Бэкап БД и кода на момент сдачи |
| 6 | Документация обновлена | ⚠️ По ситуации | Техническая документация в Drive проекта |

## 6. Стандарты качества (Technical Standards)

Чеклисты QA-этапа ссылаются на **документы стандартов** — подробные инструкции, описывающие **КАК** проверить каждый пункт.

### 6.1. Security Standards (Стандарты безопасности)

Для **Website**:

| # | Пункт проверки | Как проверить |
|---|----------------|---------------|
| 1 | HTTPS работает корректно | Проверить в браузере: замок, перенаправление HTTP → HTTPS |
| 2 | Формы защищены от XSS | Попробовать вставить `<script>` в поля ввода |
| 3 | CSRF-защита включена | Проверить наличие CSRF-токена в формах |
| 4 | SQL-инъекции невозможны | Проверить параметризацию запросов или ORM |
| 5 | Секреты не в коде | Проверить .env, убедиться что .env в .gitignore |
| 6 | Административная панель защищена | Проверить: сильный пароль, 2FA если возможно, IP-ограничение |
| 7 | Rate limiting настроен | Проверить ограничение запросов к API / формам |
| 8 | Backup настроен и протестирован | Проверить автоматический бэкап + восстановление |

Для **Mobile App** — дополнительные пункты:

| # | Пункт проверки | Как проверить |
|---|----------------|---------------|
| 9 | Данные шифруются при хранении | Проверить Keychain (iOS) / EncryptedSharedPreferences (Android) |
| 10 | API-ключи не хардкодятся | Проверить использование environment variables |
| 11 | Сертификат pinning настроен | Проверить защиту от MITM-атак |

### 6.2. Quality Standards (Стандарты качества)

Для **Website**:

| # | Пункт проверки | Критерий |
|---|----------------|----------|
| 1 | Кросс-браузерная совместимость | Chrome, Firefox, Safari, Edge — основные сценарии работают |
| 2 | Мобильная адаптация | iPhone SE (375px), iPhone 14 (390px), iPad (768px), Desktop (1440px) |
| 3 | PageSpeed Mobile | ≥ 80 баллов |
| 4 | PageSpeed Desktop | ≥ 90 баллов |
| 5 | Нет битых ссылок | Все ссылки ведут на рабочие страницы |
| 6 | Favicon установлен | Видно в табе браузера |
| 7 | Мета-теги заполнены | Title, description, og:image для каждой страницы |
| 8 | 404-страница настроена | Кастомная страница вместо дефолтной ошибки |
| 9 | Контент проверен | Нет опечаток, placeholder-текстов, lorem ipsum |
| 10 | Формы работают | Отправка, валидация, уведомления — протестированы |

### 6.3. Deployment Standards (Стандарты развёртывания)

| # | Пункт проверки | Описание |
|---|----------------|----------|
| 1 | Production-сервер настроен | Сервер работает, достаточно ресурсов |
| 2 | Домен привязан | DNS-записи настроены, домен резолвится |
| 3 | SSL-сертификат установлен | Let's Encrypt или другой, автоматическое обновление |
| 4 | .env на production актуален | Все переменные окружения соответствуют production-конфигурации |
| 5 | Бэкап базы данных настроен | Автоматический бэкап с ротацией (ежедневно) |
| 6 | Мониторинг включён | Uptime monitoring (UptimeRobot, Pingdom или аналог) |
| 7 | Логирование настроено | Ошибки логируются и доступны для анализа |

## 8. Task Checklists (Чеклисты внутри задач)

Task Checklist — это список пунктов **внутри конкретной задачи**, которые нужно выполнить.

### Отличие от подзадач:

| Параметр | Task Checklist | Подзадачи |
|----------|----------------|-----------|
| **Масштаб** | Мелкие пункты (5–15 минут) | Полноценные задачи (часы–дни) |
| **Назначение** | Один исполнитель задачи | Разные исполнители |
| **Статусы** | Только ✅ / ❌ | Полный цикл (Backlog → Done) |
| **Отчётность** | Не отображаются в аналитике | Отображаются как отдельные задачи |
| **Пример** | «Проверить на iPhone», «Проверить на iPad» | «Frontend-вёрстка», «Backend API» |

### Пример задачи с чеклистом:

```
Задача: «Deployment на production»
Исполнитель: Backend Developer

Чеклист:
  ☐ Обновить .env на production
  ☐ Запустить миграции БД
  ☐ Выполнить deployment (git pull / CI/CD)
  ☐ Проверить работу сайта после деплоя
  ☐ Проверить SSL-сертификат
  ☐ Очистить кэш
  ☐ Уведомить PM о завершении
```

### Свойства пунктов чеклиста:

| Свойство | Описание |
|----------|----------|
| **Текст** | Описание пункта |
| **Обязательный / Опциональный** | Обязательные пункты блокируют закрытие задачи |
| **Ссылка на документацию** | URL на стандарт или инструкцию — как именно выполнить пункт |

## 9. Хранение секретов (.env)

Отдельный обязательный пункт, который присутствует в нескольких Stage Gate чеклистах:

> **Все секреты проекта (.env файл, API-ключи, пароли сервисов) ДОЛЖНЫ быть сохранены в Credentials проекта до перехода на следующую стадию.**

### Почему это критически важно:

- Если разработчик уволится — доступы не потеряются
- При необходимости передачи проекта другому разработчику — всё в системе
- При инцидентах на production — быстрый доступ к конфигурации
- Аудит: кто и когда последний раз обновлял секреты

### Как система контролирует:

1. При попытке перевести Product из Development → QA система проверяет: «Есть ли Credential-запись типа .env для этого продукта?»
2. Если записи нет → переход блокируется с сообщением: «Сохраните .env файл в Credentials проекта»
3. PM или Developer создаёт запись → переход разблокируется

## 10. Связь стандартов и чеклистов

```
Стандарт (Standard Document)
  │
  ├── Описывает: ЧТО проверять и КАК проверять
  │     Пример: «Security Standard v2.0» → раздел XSS → «Вставить <script> в поле...»
  │
  └── Ссылается из → Stage Gate Checklist → пункт «Security Checklist пройден»
                        │
                        └── При раскрытии пункта PM видит ссылку на документ стандарта
```

### Хранение стандартов:

- Документы стандартов хранятся в **Drive** компании (раздел «Standards»)
- Каждый стандарт — отдельный документ с версионированием
- Стандарты обновляются CEO / Head of Delivery при необходимости
- При обновлении стандарта все ссылки из чеклистов автоматически ведут на актуальную версию
