# Коммерческие предложения и передача проекта (Handoff)

> NBOS Platform — от коммерческого предложения до передачи проекта в разработку

## Часть 1: Коммерческое предложение (Offer / КП)

### Назначение

Коммерческое предложение (КП) — документ, который Seller отправляет клиенту на стадии «Send Offer» в Deal Pipeline. КП фиксирует договорённости и формирует ожидания клиента: что будет сделано, в какие сроки, за какую стоимость.

### Когда создаётся КП

КП создаётся после:

1. Потребности клиента выяснены (стадия «Discuss What Is Needed»)
2. Встреча проведена (стадия «Meeting»)
3. Внутренняя оценка завершена (стадия «Can We Do It?»)

Seller переводит сделку на стадию «Send Offer» и формирует КП.

### Структура коммерческого предложения

| Раздел                        | Описание                                                          | Обязательность |
| ----------------------------- | ----------------------------------------------------------------- | -------------- |
| **Название проекта**          | Краткое название проекта/продукта                                 | Да             |
| **Описание scope**            | Что входит в работу — перечень функций, страниц, модулей          | Да             |
| **Deliverables (результаты)** | Что клиент получит: веб-сайт, приложение, дизайн, etc.            | Да             |
| **Тип проекта**               | White Label / Mix / Custom Code                                   | Да             |
| **Timeline (сроки)**          | Ориентировочные сроки разработки и сдачи                          | Да             |
| **Стоимость**                 | Общая сумма проекта                                               | Да             |
| **Условия оплаты**            | Тип оплаты: Classic 50/50, Classic 30/30/40, Subscription Monthly | Да             |
| **План оплат**                | Даты и суммы каждого платежа                                      | Да             |
| **Maintenance**               | Условия технического обслуживания после сдачи (опционально)       | Нет            |
| **Что не входит**             | Явное указание того, что выходит за scope                         | Рекомендовано  |

### Формат отправки

КП может быть отправлено клиенту в двух форматах:

- **Документ (PDF/DOCX)** — прикрепляется к карточке сделки и отправляется клиенту по email или мессенджеру
- **Ссылка** — если КП создаётся в онлайн-формате (Google Docs, Notion, или будущий генератор КП в NBOS)

Для MVP допустим и третий формат фиксации:

- **Screenshot / messenger proof** — скриншот переписки, в которой зафиксировано предложение и согласование цены / условий

### Offer Materials в системе

NBOS не требует сложного offer-builder на первом этапе. Вместо этого система хранит `Offer Materials`.

Каждый material может быть одного из типов:

- `FILE`
- `LINK`
- `MESSENGER_SCREENSHOT`

Для каждого material важно хранить:

- кто загрузил / зафиксировал
- когда отправлено клиенту
- является ли этот material текущим согласованным offer

Переход дальше по CRM разрешён только если в сделке существует хотя бы один valid offer material.

### Действия после отправки КП

1. Сделка переходит на стадию **«Get Answer»**
2. Фиксируется дата отправки КП
3. Запускаются автоматические напоминания Seller о follow-up (3, 7, 14 дней)
4. Seller ожидает реакцию клиента

### Возможные исходы

| Исход                    | Действие                                                             |
| ------------------------ | -------------------------------------------------------------------- |
| Клиент согласен          | → стадия «Deposit & Contract», подготовка договора и Invoice         |
| Клиент просит изменения  | Корректировка КП, повторная отправка (возврат на «Send Offer»)       |
| Клиент торгуется по цене | Переговоры, при необходимости — эскалация на Head of Sales или CEO   |
| Клиент молчит            | Follow-up по расписанию, через 30 дней — решение о переводе в Failed |
| Клиент отказывается      | → «Failed» с фиксацией причины                                       |

## Часть 2: Процесс передачи проекта (Handoff)

### Назначение

Handoff — это формализованный процесс передачи проекта от Seller к PM. Происходит в момент перехода **Deal → Order → Project**. Качество handoff напрямую влияет на успех доставки проекта.

**Ключевой принцип:** PM должен получить проект со **всем контекстом**, а не просто устное «новый проект, разберись». Все договорённости с клиентом, scope, условия оплаты и дедлайны должны быть зафиксированы и переданы.

### Когда происходит Handoff

```
Deal Pipeline                              Projects Hub
     │                                          │
     ├── Stадия 7: Deposit & Contract           │
     │       │                                   │
     │       ├── Invoice создан                  │
     │       ├── Invoice оплачен ──────────────► Order создан
     │       │                                   │
     │       └── HANDOFF STARTS ──────────────► Project создан/обновлён
     │                                           │
     ├── Стадия 8: Creating ◄────────────────── PM принял проект
     │                                           │
```

Handoff начинается сразу после подтверждения первой оплаты и создания Order.

---

### Обязательный Kickoff Checklist

При передаче проекта PM **обязан** пройти чек-лист. Проект не может перейти в статус «Development» без выполнения всех обязательных пунктов.

#### Блок 1: Scope и договорённости

| #   | Пункт                          | Описание                                                                      | Ответственный |
| --- | ------------------------------ | ----------------------------------------------------------------------------- | ------------- |
| 1   | **Scope подтверждён**          | PM прочитал и подтвердил понимание scope из КП. Все вопросы уточнены с Seller | PM            |
| 2   | **Milestones согласованы**     | Ключевые точки проекта определены: дизайн → разработка → тестирование → сдача | PM + Seller   |
| 3   | **Deliverables зафиксированы** | Перечень конечных результатов чётко записан в карточке проекта                | PM            |

#### Блок 2: Финансы и сроки

| #   | Пункт                      | Описание                                                                         | Ответственный |
| --- | -------------------------- | -------------------------------------------------------------------------------- | ------------- |
| 4   | **План оплат согласован**  | Тип оплаты (Classic/Subscription), суммы и даты всех платежей зафиксированы      | Seller → PM   |
| 5   | **Дедлайн установлен**     | Конечная дата сдачи проекта клиенту записана в Project и используется в спринтах | Seller → PM   |
| 6   | **Бюджет команды понятен** | PM понимает, сколько ресурсов можно выделить на проект (на основе суммы заказа)  | PM            |

#### Блок 3: Клиент и коммуникация

| #   | Пункт                         | Описание                                                                           | Ответственный |
| --- | ----------------------------- | ---------------------------------------------------------------------------------- | ------------- |
| 7   | **Контакты клиента собраны**  | Все контактные лица со стороны клиента зафиксированы: имена, телефоны, email, роли | Seller → PM   |
| 8   | **Канал коммуникации создан** | Создана группа WhatsApp (или другой канал) с клиентом, PM добавлен                 | Seller + PM   |
| 9   | **Клиент представлен PM**     | Клиент знает, что его проект ведёт конкретный PM, и имеет его контакты             | Seller        |

#### Блок 4: Технические детали

| #   | Пункт                            | Описание                                                                       | Ответственный |
| --- | -------------------------------- | ------------------------------------------------------------------------------ | ------------- |
| 10  | **Тип проекта определён**        | WL (White Label), Mix или Custom Code — зафиксирован в карточке проекта        | Seller → PM   |
| 11  | **Accesses/Credentials собраны** | Выявлены необходимые доступы от клиента (хостинг, домен, CMS, API-ключи, etc.) | PM            |
| 12  | **Требования к инфраструктуре**  | Определено, что нужно создать: домен, хостинг, база данных, сторонние сервисы  | PM + Tech Ops |

#### Блок 5: Риски и особенности

| #   | Пункт                       | Описание                                                                                       | Ответственный |
| --- | --------------------------- | ---------------------------------------------------------------------------------------------- | ------------- |
| 13  | **Риски задокументированы** | Все известные риски записаны: сложные интеграции, зависимость от клиента, нереалистичные сроки | PM            |
| 14  | **Особые условия клиента**  | Любые нестандартные договорённости: скидки, специальные условия, ограничения                   | Seller → PM   |
| 15  | **Партнёрская привязка**    | Если клиент от партнёра — зафиксирована связь и условия партнёрского вознаграждения            | Seller        |

### Процесс Handoff пошагово

#### Шаг 1: Автоматическая подготовка (система)

При оплате первого Invoice система автоматически:

1. Создаёт **Order** с данными из сделки (сумма, тип оплаты, план оплат)
2. Создаёт **Project** в Projects Hub (или привязывает к существующему для Extension)
3. Создаёт **Product** (или Work Package) внутри проекта
4. Переносит данные контакта и компании в проект
5. Назначает PM (вручную Head of Delivery или автоматически по правилам)
6. Генерирует пустой **Kickoff Checklist** для PM
7. Отправляет уведомление PM: «Новый проект назначен, требуется Handoff»

#### Шаг 2: Встреча Seller → PM (ручная)

Seller и PM проводят встречу (15–30 минут) по проекту:

- Seller рассказывает контекст: кто клиент, что нужно, какие нюансы
- PM задаёт уточняющие вопросы
- Вместе проходят Kickoff Checklist и заполняют пункты
- Определяют, какие доступы нужно запросить у клиента

#### Шаг 3: PM заполняет Kickoff Checklist (система)

PM проходит по каждому пункту чек-листа в системе:

- Заполняет обязательные поля в карточке проекта
- Прикрепляет документы (КП, договор)
- Фиксирует контакты клиента
- Создаёт группу коммуникации с клиентом

#### Шаг 4: Валидация и запуск (система)

Система проверяет, что все обязательные пункты Kickoff Checklist заполнены:

- Если всё заполнено → PM может перевести проект в статус **Development**
- Если есть незаполненные пункты → система блокирует переход и показывает что нужно заполнить

После перевода в Development:

- Проект появляется в списке «Development» в Projects Hub
- PM может создавать спринты и задачи
- Seller видит сделку на стадии «Creating»

### Handoff для Extension Deal

Для доработок процесс упрощён, так как проект уже существует:

| Этап               | Extension                             | New Project                  |
| ------------------ | ------------------------------------- | ---------------------------- |
| Создание Project   | Не нужно — привязка к существующему   | Создаётся новый              |
| Создание Product   | Создаётся Work Package внутри Project | Создаётся основной Product   |
| Назначение PM      | Обычно тот же PM                      | Назначается Head of Delivery |
| Kickoff Checklist  | Сокращённый (scope + сроки + оплата)  | Полный (15 пунктов)          |
| Канал коммуникации | Уже существует                        | Создаётся новый              |
| Контакты клиента   | Уже в системе                         | Собираются с нуля            |

---

### Сокращённый Kickoff Checklist для Extension

| #   | Пункт                                    | Обязательный |
| --- | ---------------------------------------- | ------------ |
| 1   | Scope доработки подтверждён              | Да           |
| 2   | Дедлайн установлен                       | Да           |
| 3   | План оплат зафиксирован                  | Да           |
| 4   | Новые технические требования (если есть) | Нет          |
| 5   | Задачи созданы в Scrum/Kanban            | Да           |

## Критерии качества Handoff

### Что определяет хороший Handoff

| Критерий                      | Описание                                                                        |
| ----------------------------- | ------------------------------------------------------------------------------- |
| **Полнота информации**        | PM имеет всё необходимое для начала работы без дополнительных вопросов к Seller |
| **Однозначность scope**       | Нет расхождений между тем, что продал Seller и что понял PM                     |
| **Реалистичность сроков**     | Дедлайн, заявленный клиенту, достижим при текущих ресурсах                      |
| **Финансовая ясность**        | План оплат понятен, все суммы зафиксированы, тип оплаты определён               |
| **Готовность инфраструктуры** | Понятно, какие доступы нужны, какие сервисы требуется создать                   |

### Что определяет плохой Handoff

| Проблема                         | Последствия                                                |
| -------------------------------- | ---------------------------------------------------------- |
| «Разберись сам, контекст в чате» | PM тратит дни на поиск информации, теряется контекст       |
| Scope не зафиксирован            | Scope creep — клиент ожидает больше, чем заложено в бюджет |
| Нереалистичный дедлайн           | Проект срывает сроки, конфликт с клиентом                  |
| Не собраны доступы               | Разработка блокируется, ожидание доступов от клиента       |
| Не создан канал коммуникации     | Хаотичная переписка, потеря информации                     |

## Автоматизация процесса

| Триггер                           | Автоматическое действие                                             |
| --------------------------------- | ------------------------------------------------------------------- |
| Первый Invoice оплачен            | Создание Order, Project, Product                                    |
| Project создан                    | Назначение PM, генерация Kickoff Checklist, уведомление PM          |
| Kickoff Checklist не заполнен 48ч | Напоминание PM + уведомление Head of Delivery                       |
| Все пункты Checklist заполнены    | Разблокировка перехода в Development                                |
| Extension Deal Won                | Создание Work Package в существующем Project, сокращённый Checklist |
| PM перевёл в Development          | Обновление стадии Deal → «Creating»                                 |

## Связь с другими модулями

| Модуль               | Связь через Handoff                                               |
| -------------------- | ----------------------------------------------------------------- |
| **CRM (Deals)**      | Источник данных: scope, сумма, тип оплаты, контакты               |
| **Finance (Orders)** | Создаётся Order — финансовая сущность проекта                     |
| **Projects Hub**     | Создаётся Project, Product/Work Package                           |
| **Tasks**            | После Handoff PM создаёт первые задачи и спринты                  |
| **Credentials**      | Фиксируются требования к доступам, создаются записи в Credentials |
| **Calendar**         | Дедлайн проекта попадает в календарь PM                           |
| **Messenger**        | Создаётся канал коммуникации с клиентом                           |
