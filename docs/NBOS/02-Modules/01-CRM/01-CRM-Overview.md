# Модуль CRM — Обзор

> NBOS Platform — управление продажами от лида до заказа

## Назначение

CRM-модуль в NBOS отвечает за **процесс продаж** — от момента первого контакта с потенциальным клиентом до подписания договора и получения первой оплаты. CRM **не** управляет доставкой проекта — только продажами.

**Ключевой принцип:** CRM заканчивается там, где начинается Projects Hub. После закрытия сделки и получения первой оплаты проект передаётся в зону ответственности PM.

For stale logic and removal tracking see `06-CRM-Cleanup-Register.md`.

## Основные сущности

### Лиды и Сделки — две раздельные сущности

В NBOS лиды (Leads) и сделки (Deals) — это **раздельные** сущности с разными pipeline, а не стадии одной воронки. Причина разделения:

| Характеристика           | Leads (Лиды)                                 | Deals (Сделки)                     |
| ------------------------ | -------------------------------------------- | ---------------------------------- |
| **Объём**                | Массовый поток: 100–200+ в месяц             | Квалифицированные: десятки в месяц |
| **Работа**               | Минимальная: зафиксировать, классифицировать | Полный цикл: встречи, КП, договор  |
| **Цель**                 | Отфильтровать качественных (SQL)             | Довести до сделки и первой оплаты  |
| **Ответственный**        | Seller / Marketing                           | Seller                             |
| **Терминальные статусы** | SPAM, Frozen, SQL (→ Deal)                   | Failed, Deal Won (→ Order)         |

Разделение предотвращает «замусоривание» воронки сделок сотнями необработанных лидов.

### Переход Lead → Deal

Лид переходит в сделку только когда получает статус **Quality Lead (SQL)** — Sales Qualified Lead. Это значит:

- Контакт установлен
- Клиент заинтересован в конкретной услуге
- Есть бюджет и сроки (хотя бы примерные)
- Лид классифицирован как качественный

При конвертации SQL → Deal автоматически:

1. Создаётся карточка Deal в воронке сделок на стадии «Start a Conversation»
2. Данные контакта (имя, телефон, email) переносятся из лида
3. Источник лида сохраняется для аналитики

## Основные сценарии сделок

### 1. Product Deal / Новый клиент

Стандартная сделка с новым клиентом, который ранее не работал с Neetrino. Полный цикл от знакомства до договора.

В каноническом `Deal Type` это обычно `PRODUCT`.

### 2. Extension Deal (Доработка)

Сделка на доработку для существующего проекта. Особенности:

- Проходит тот же pipeline, что и новая сделка
- Визуально выделена (тег/цвет) для быстрой идентификации
- **Привязана к существующему проекту** в Projects Hub
- Обычно меньше по объёму, быстрее проходит pipeline
- При закрытии не создаёт новый проект, а добавляет Work Package к существующему

### 3. Maintenance Deal (Обслуживание)

Сделка на техническое обслуживание уже существующего продукта Neetrino. Особенности:

- Всегда связана с существующим `Project` и конкретным `Product`
- Не является новым самостоятельным продуктом
- После `Deal Won` создаёт или активирует движение в `Subscription Board`, а не новый delivery-flow в CRM
- Может создаваться автоматически после `PRODUCT -> Deal Won` как подготовленная карточка для дальнейшего коммерческого согласования

## Типы оплаты в сделке

Тип оплаты определяется на стадии сделки и влияет на всю последующую финансовую логику:

### Classic 50/50

- 50% предоплата до начала работ
- 50% после сдачи проекта клиенту
- Бонус команде начисляется после получения полной суммы

### Classic 30/30/40

- 30% предоплата
- 30% в середине разработки (по milestone)
- 40% после сдачи проекта
- Бонус начисляется после получения всех платежей

### Subscription Monthly (Ежемесячная подписка)

- 1-я оплата до начала работ является одновременно подтверждением заказа и первой оплаченной subscription-итерацией
- Ежемесячные платежи во время разработки идут дальше по monthly cycle
- После сдачи проекта — переход на Maintenance с изменённой суммой
- Пример: 100,000/мес за dev → после сдачи 100,000 dev + 80,000 maintenance = 180,000/мес
- Бонус команде начисляется после сдачи проекта (после 3–4 месяца оплат)

## Жизненный цикл: от Сделки до Заказа

```
Lead (SQL) → Deal → [Pipeline стадии] → Deal Won + 1st Payment → Order → Project
```

### Момент перехода Deal → Order

Сделка становится заказом (Order) при выполнении **обоих** условий:

1. Сделка получает статус **Deal Won**
2. Первый Invoice оплачен (подтверждено Finance Director)

В этот момент:

- Создаётся **Order** — центральная финансовая сущность
- Создаётся или обновляется **Project** в Projects Hub
- Создаётся **Product** (или Work Package для Extension)
- Запускается процесс **Handoff** (передача от Seller к PM)

Для `payment_type = Subscription`:

- первый оплаченный invoice одновременно считается первой оплаченной subscription-итерацией;
- subscription создаётся сразу после `Deal Won`;
- в `Subscription Board` запись появляется как `Active`;
- месяц первого invoice отображается как уже оплаченный;
- следующий monthly invoice создаётся на ту же календарную дату следующего месяца.

## Ключевые представления (Views)

### 1. Lead Board (Доска лидов)

- Kanban-доска со стадиями: New → Didn't Get Through → Contact Established → Qualification (MQL) → Terminal
- Фильтры: по источнику, дате, ответственному Seller
- Массовые действия: перемещение в SPAM, Frozen

### 2. Deal Pipeline Board (Доска сделок)

- Kanban-доска с **7 рабочими стадиями** + 2 терминальных статуса (**Failed**, **Deal Won**). Последняя рабочая стадия — **Deposit & Contract**; Creating / остаток оплат / Maintenance billing — вне воронки (Projects, Finance)
- Визуальное отличие deal types по цвету карточек:
  - `PRODUCT` — белый
  - `EXTENSION` — оранжевый
  - `MAINTENANCE` — фиолетовый
- На каждой карточке: имя клиента, сумма, тип оплаты, дедлайн
- Фильтры: по `Deal Type` (`PRODUCT` / `EXTENSION` / `MAINTENANCE` / `OUTSOURCE`), Seller, сумме, периоду

### Marketing Attribution

Для Lead и Deal обязателен marketing block:

- `From` — верхний источник
- `Where` — канал внутри источника
- required linkage — partner/contact, когда это требуется источником

На создании карточки допускается временная неполнота, но перейти дальше первой meaningful stage без этого блока нельзя.

### 3. Sales Reports / module-owned analytics

- Конверсия по стадиям (Lead → SQL → Deal → Won)
- Средний цикл сделки
- Объём продаж по периодам
- Разбивка по источникам лидов
- Performance продавцов
- Прогноз выручки по текущему pipeline

Граница с `Reports / Analytics`:

```text
CRM owns sales facts and CRM-specific report definitions.
Reports / Analytics exposes these reports in the global catalog, scheduled reports and exports.
```

CRM не должен становиться отдельным глобальным BI-модулем. CRM показывает sales analytics там, где это помогает Sales работать с pipeline; глубокие сравнения периодов, exports и scheduled reports живут в `Reports / Analytics`.

## Интеграции с другими модулями

### CRM → Finance

- Seller создаёт Invoice из карточки сделки (на стадии «Deposit & Contract»)
- Invoice попадает на доску счетов Finance Director
- После оплаты первого Invoice при **Deposit & Contract** — переход в **Deal Won** и передача в Order / Projects (не промежуточные стадии CRM)

### CRM → Projects Hub

- При Deal Won создаётся проект в Projects Hub
- Extension Deal привязывается к существующему проекту
- Данные из сделки (scope, сумма, дедлайн, контакты) передаются в проект

### CRM → Contacts & Companies

- Каждая сделка привязана к Контакту (физическое лицо) и Компании (юрлицо)
- Один контакт может иметь множество сделок для разных проектов/компаний
- Компания хранит юридические данные для выставления счетов

### CRM → Partners

- Если Lead / Deal пришёл от партнёра — обязательна привязка к Partner.
- CRM фиксирует `Partner Referral Terms`: партнёр, Deal Type, Payment Type, зафиксированный процент.
- Deal Type определяет процент партнёра по Partner Commission Policy.
- Payment Type определяет ритм выплаты: Classic после сдачи и полной оплаты, Subscription после реально полученных subscription-платежей.
- CRM не создаёт Expense напрямую. После реальной оплаты Finance / Partners создают цепочку `Partner Accrual -> Partner Balance -> Payout Batch -> Expense Card`.

### CRM → Calendar

- Встречи из стадии «Meeting» автоматически попадают в календарь Seller
- Дедлайны по follow-up видны в календаре

### CRM → Messenger

- На стадии «Deposit & Contract» создаётся группа WhatsApp с клиентом
- Клиентские сообщения из Instagram/Facebook попадают в CRM-мессенджер

## Права доступа к CRM

| Роль             | Leads | Deals   | Действия                                   |
| ---------------- | ----- | ------- | ------------------------------------------ |
| CEO              | Full  | Full    | Всё                                        |
| Seller           | Full  | Full    | Полный цикл продажи                        |
| Head of Sales    | Full  | Full    | + аналитика команды                        |
| Marketing        | Read  | Limited | Аналитика источников, статусы для ROI      |
| PM               | None  | Limited | Видит только привязанные к проектам сделки |
| Finance Director | None  | None    | Не работает с CRM напрямую                 |
| Developer        | None  | None    | CRM скрыт                                  |
