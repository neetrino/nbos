## 1. Решённые уточнения (модель согласована с продуктом и админкой)

### 1.1. Сущность Product vs Deal Type `PRODUCT`

**Решение:** не смешивать уровни.

- **Deal Type** (справочник, 4 значения: `PRODUCT`, `EXTENSION`, `MAINTENANCE`, `OUTSOURCE`) — классификация **сделки и заказа** в CRM/Finance.
- **Product Type** (отдельный справочник в Settings → Lists) — **какая услуга** заказана, когда нужно детализировать работу (в первую очередь при Deal Type = `PRODUCT`).
- **Сущность Product** в модели данных — единица исполнения внутри **Project**; это не то же самое, что значение «PRODUCT» в Deal Type (совпадение слов только вербальное).

Зафиксировано в `01-Platform-Overview/03-Core-Entities-and-Data-Model.md` (§ 1.1).

### 1.2. Типы сделок / заказов

**Решение:** канон — **ровно четыре Deal Type** из System Lists, как в UI админки. Старые формулировки в части документов (**New / Upsell** как тип сделки) нужно постепенно привести к этой модели: «новый клиент» — сценарий, а не отдельный Deal Type; допродажа нового продукта существующему клиенту остаётся **Deal Type = PRODUCT** с привязкой к проекту.

Зафиксировано в `03-Core-Entities-and-Data-Model.md` (Deal, Order).

### 1.3. Переход Deal → Order

| Источник                     | Условие                                                                                   |
| ---------------------------- | ----------------------------------------------------------------------------------------- |
| `03-Deal-Pipeline.md`        | Order + Project при **оплате первого Invoice** (стадия 7)                                 |
| `01-Lead-to-Cash-Process.md` | Deal Won + первый платёж; также упоминается создание Product                              |
| `1.md`                       | Акцент на обязательных полях + оплата; сценарии новый клиент / второй продукт / доработка |

**Статус:** **Частично.** Логика близка, но в `New-Description` сильнее подчёркнуты сценарии привязки к проекту и типы оплаты; нужно свести в одну таблицу переходов.
