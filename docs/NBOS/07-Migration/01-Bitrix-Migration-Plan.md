# План миграции из Bitrix24

## Общее описание

Миграция из Bitrix24 в NBOS — критический процесс, от которого зависит непрерывность бизнес-операций Neetrino. Стратегия миграции — **поэтапная** (phased), с параллельной работой обеих систем в переходный период. Это минимизирует риски потери данных и простоя.

---

## Объём миграции

### Количественные показатели

| Сущность | Примерное количество | Критичность |
|----------|---------------------|-------------|
| Контакты (Contacts) | ~500+ | Критическая |
| Компании (Companies) | ~200+ | Критическая |
| Проекты (Projects) | ~100+ (активных ~30-50) | Критическая |
| Сделки (Deals) | ~300+ (активных ~20-40) | Высокая |
| Заказы (Orders) | ~200+ (активных ~30-50) | Высокая |
| Подписки (Subscriptions) | ~50+ (активных) | Критическая |
| Счета (Invoices) | ~1000+ (недавних ~200) | Средняя |
| Пароли/Credentials | ~300+ | Критическая |
| Домены | ~100+ | Высокая |
| Задачи (Tasks) | ~500+ (активных ~100) | Средняя |
| **Итого** | **~2000+ карточек/записей** | — |

### Что мигрировать, а что — начать с нуля

| Данные | Решение | Обоснование |
|--------|---------|-------------|
| Контакты + Компании | **Мигрировать полностью** | Основа для всех связей |
| Активные проекты | **Мигрировать** | Текущая работа не должна прерываться |
| Закрытые проекты (недавние) | **Мигрировать** | Нужны для истории и отчётности |
| Старые закрытые проекты (>2 лет) | **Не мигрировать** | Доступны в Bitrix как архив |
| Активные подписки | **Мигрировать** | Критично для ежемесячного биллинга |
| Активные заказы | **Мигрировать** | Текущие финансовые обязательства |
| Активные сделки (pipeline) | **Мигрировать** | Текущие продажи |
| Закрытые сделки | **Не мигрировать** | Доступны в Bitrix; статистика начнётся заново |
| Credentials/Пароли | **Мигрировать** | Критично для ежедневных операций |
| Домены | **Мигрировать** | Нужны для трекинга продлений |
| Счета (последние 6 мес) | **Мигрировать** | Текущие финансовые данные |
| Старые счета | **Не мигрировать** | Архив в Bitrix |
| Активные задачи | **Мигрировать** | Текущая работа |
| Завершённые задачи | **Не мигрировать** | Начать историю заново |
| Переписка/комментарии | **Не мигрировать** | Слишком сложно; Bitrix как архив |

## Стратегия миграции

### Принцип: поэтапная миграция с параллельной работой

```
        Bitrix24                          NBOS
        ────────                          ────
Фаза 1: [████████████]                   [подготовка]
Фаза 2: [████████████]                   [импорт данных]
Фаза 3: [██ read-only █]                 [████████████]
Фаза 4: [█ архив █████]                  [████████████]
Фаза 5:                                  [████████████]
        
        ←─── 2-3 месяца параллельной работы ───→
```

## Фаза 1: Подготовка данных (2 недели)

### Задачи

| # | Задача | Ответственный | Срок |
|---|--------|---------------|------|
| 1.1 | Экспорт контактов и компаний из Bitrix (CSV/API) | Разработчик | 2 дня |
| 1.2 | Экспорт проектов, сделок, заказов | Разработчик | 2 дня |
| 1.3 | Экспорт подписок с текущими суммами и billing dates | Разработчик + ФинДиректор | 2 дня |
| 1.4 | Экспорт паролей/credentials (зашифрованно) | Разработчик + Безопасность | 3 дня |
| 1.5 | Экспорт доменов с датами истечения | Разработчик | 1 день |
| 1.6 | Очистка данных: удаление дублей, исправление ошибок | PM + ФинДиректор | 3 дня |
| 1.7 | Маппинг полей: Bitrix → NBOS (таблица соответствий) | Разработчик + PM | 2 дня |
| 1.8 | Определение scope: что мигрируем, что оставляем | CEO + PM | 1 день |

### Маппинг сущностей Bitrix → NBOS

| Сущность Bitrix | Сущность NBOS | Примечание |
|-----------------|---------------|------------|
| Контакт (CRM) | Contact | Поля: имя, телефон, email, компания |
| Компания (CRM) | Company | Поля: название, ИНН, адрес, реквизиты |
| Сделка (CRM) | Deal | Стадии маппятся на новый пайплайн |
| Задача (Tasks) | Task | Статусы маппятся на новые |
| Проект (Groups/Projects) | Project | + Products внутри |
| Smart Process «Подписки» | Subscription | Сумма, billing day, тип |
| Smart Process «Заказы» | Order | Сумма, статус оплаты |
| Smart Process «Счета» | Invoice | Сумма, дата, статус |
| Хранилище паролей | Credentials Vault | Требуется перешифровка |
| Пользовательские поля «Домены» | Domain | Домен, дата истечения, клиент |

### Очистка данных (Data Cleaning)

| Проблема | Как решать |
|----------|-----------|
| Дубли контактов | Merge по телефону/email, оставить самую полную карточку |
| Неактуальные контакты | Пометить как Inactive, не мигрировать |
| Пустые поля | Заполнить обязательные; опциональные — оставить пустыми |
| Разный формат телефонов | Нормализация в формат +374XXXXXXXX |
| Неактивные подписки | Не мигрировать (только active) |
| Дубли компаний | Merge, проверить связи с контактами и проектами |

## Фаза 2: Импорт данных (2 недели)

### Порядок импорта

Порядок важен из-за зависимостей между сущностями:

```
1. Companies  ──→  (основа)
2. Contacts   ──→  (привязка к Companies)
3. Projects   ──→  (привязка к Companies + Contacts)
4. Subscriptions ──→ (привязка к Projects)
5. Orders     ──→  (привязка к Projects + Companies)
6. Credentials ──→ (привязка к Projects)
7. Domains    ──→  (привязка к Projects + Companies)
8. Deals (active) ──→ (привязка к Contacts + Companies)
9. Invoices (recent) ──→ (привязка к Orders / Subscriptions)
10. Tasks (active) ──→ (привязка к Projects)
```

### Задачи

| # | Задача | Ответственный | Срок |
|---|--------|---------------|------|
| 2.1 | Разработка скриптов импорта (по сущностям) | Разработчик | 3 дня |
| 2.2 | Тестовый импорт на staging-окружении | Разработчик | 2 дня |
| 2.3 | Валидация: проверка полноты и корректности данных | PM + ФинДиректор | 2 дня |
| 2.4 | Исправление ошибок импорта | Разработчик | 2 дня |
| 2.5 | Импорт Credentials (с перешифровкой) | Разработчик + Безопасность | 2 дня |
| 2.6 | Финальный импорт на production | Разработчик | 1 день |
| 2.7 | Верификация: spot-check 10% записей | PM + ФинДиректор | 2 дня |

### Валидация после импорта

| Проверка | Метод | Критерий успеха |
|----------|-------|-----------------|
| Количество записей | COUNT(*) в NBOS vs COUNT(*) в выгрузке | 100% совпадение |
| Связи между сущностями | Проверка FK (Contact → Company, Project → Company) | Все связи сохранены |
| Суммы подписок | Сумма всех активных подписок в NBOS vs Bitrix | Совпадение ±0 |
| Credentials | Расшифровка тестового набора паролей | Все расшифровываются корректно |
| Домены | Проверка дат истечения | Все даты корректны |
| Активные сделки | Стадии и суммы | Совпадение с Bitrix |

## Фаза 3: Параллельная работа (1–2 месяца)

### Правила параллельной работы

| Процесс | Где выполняется | Примечание |
|---------|-----------------|------------|
| Новые лиды и сделки | **NBOS** | CRM полностью в NBOS |
| Подписочный биллинг | **NBOS** | Все счета создаются в NBOS |
| Управление задачами | **NBOS** | Новые задачи только в NBOS |
| Финансовый учёт | **NBOS** | Затраты, бонусы, payroll |
| Просмотр истории | **Bitrix** (read-only) | Старые переписки, закрытые сделки |
| Credentials | **NBOS** | Все пароли в NBOS Vault |
| Домены | **NBOS** | Трекинг продлений в NBOS |

### Обучение команды

| # | Активность | Аудитория | Формат | Длительность |
|---|-----------|-----------|--------|-------------|
| 1 | Общий обзор NBOS | Вся команда | Презентация + демо | 2 часа |
| 2 | CRM: лиды и сделки | Sales | Практический воркшоп | 2 часа |
| 3 | Projects: управление проектами | PM, Dev, Design | Практический воркшоп | 3 часа |
| 4 | Finance: счета, подписки, затраты | ФинДиректор, бухгалтер | Практический воркшоп | 3 часа |
| 5 | Support: тикеты и SLA | Support, PM | Практический воркшоп | 1 час |
| 6 | Credentials Vault | PM, Dev | Практический воркшоп | 1 час |
| 7 | Telegram-бот и уведомления | Вся команда | Инструкция + настройка | 30 мин |

### Мониторинг и отчётность

| Метрика | Как отслеживать | Целевое значение |
|---------|-----------------|------------------|
| % процессов в NBOS | Ежедневный чек-лист | 100% к концу Фазы 3 |
| Обращения к Bitrix | Логи доступа / самоотчёт | Снижение до 0 |
| Ошибки данных | Тикеты от команды | < 5 за неделю |
| Удовлетворённость команды | Еженедельный опрос | > 7/10 |

## Риски и митигация

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| **Потеря данных при миграции** | Низкая | Критическое | Bitrix продолжает работать как backup. Все экспорты проверяются до удаления. Валидация после импорта. |
| **Сопротивление команды** | Средняя | Высокое | Поэтапный переход. Обучение. Параллельная работа — команда привыкает постепенно. Сбор обратной связи. |
| **Автоматизации не работают** | Средняя | Высокое | Параллельная работа 1-2 месяца. Ручные процессы как fallback. Постепенное включение автоматизаций. |
| **Безопасность паролей** | Низкая | Критическое | Зашифрованный экспорт. Перешифровка при импорте. Никогда не хранить в открытом виде. Аудит доступа. |
| **Неполный маппинг данных** | Средняя | Среднее | Тщательный маппинг на Фазе 1. Тестовый импорт. Spot-check после импорта. |
| **Bitrix API ограничения** | Низкая | Среднее | Альтернативный экспорт через CSV. Использование нескольких API-ключей. Экспорт по частям. |
| **Время миграции превышает план** | Средняя | Среднее | Буфер в плане (+1-2 недели). Приоритизация — критичное мигрируется первым. |
