# Бонусная и зарплатная система

## Общее описание

Каждый сотрудник Neetrino имеет **фиксированную зарплату (Fix)** и **переменную часть (бонусы)**. Бонусы — это основной мотивационный инструмент: они составляют значительную часть дохода и напрямую привязаны к результатам работы. Система построена так, чтобы сотрудник видел связь «сделал работу → получил деньги».

---

## Структура дохода сотрудника

```
Общий доход = Fix (базовая зарплата) + Variable (бонусы)

Где:
  Fix = фиксированная ежемесячная зарплата
  Variable = сумма всех бонусов за месяц (Cap: max 100-200% от Fix)
```

## Бонус продажника (Sales Bonus)

### Триггер начисления (sales bonus)

**Когда создаётся начисление:** только когда **впервые получены деньги от клиента** по сделке/заказу — не при выставлении счёта и не при черновике инвойса.

**Как это стыкуется с Finance (уже в каноне счетов):**

1. По сделке/заказу создаётся **Invoice Card** — сумма, которую клиент должен оплатить (`02-Modules/04-Finance/02-Invoices-and-Payments.md`).
2. Клиент платит → финансы **подтверждают поступление** → создаётся запись **`Payment`** (факт денег), карточка инвойса переходит в **`Paid`**.
3. Именно это событие (**подтверждённый платёж / оплаченная карточка**) должно запускать расчёт **sales bonus** по правилам ниже. Для схемы «первые деньги» — **первый** такой платёж по заказу, который квалифицируется под политику (идемпотентно: повторно тот же расчёт на ту же сделку не выполнять, если политика не задаёт дробление по траншам).

Термин **`Payment`** в коде — это не «любая строка», а **факт оплаты** после подтверждения; в UI пользователь видит оплаченный инвойс и дальше уже отрабатывают связанные процессы (в т.ч. бонус).

### Две независимые ставки Seller / Assistant (без общего пула 80/20)

Начисление sales bonus **не** строится как «один пул → деление 80/20». Задаются **два отдельных контролируемых процента**:

- `seller_rate` — доля **Seller** от базы;
- `assistant_rate` — доля **Seller Assistant** от той же базы.

Они **не обязаны** сохранять фиксированное соотношение: админ может поднять или опустить ставку продавца или ассистента **независимо** (в рамках политики компании). Сумма `seller_rate + assistant_rate` — это **общий процент на команду продаж по сделке**, но он **не выводится** автоматически из 80/20 — только из строки политики в БД.

**База расчёта (тип оплаты сделки)**

- **Classic:** база = **общая сумма заказа / сделки** (как в Finance).
- **Subscription:** база = **сумма первого оплаченного месяца** (первый период подписки после подтверждённого `Paid`).

**Формулы**

```text
Начисление Seller      = база × seller_rate
Начисление Assistant   = база × assistant_rate
```

Один сотрудник в обеих ролях получает **(seller_rate + assistant_rate) × база** на свой внутренний учёт.

**Привязка к полю `From` (маркетинговый блок сделки)**

Какую строку политики применить, определяет **верхний уровень источника** — поле **`From`** (в каноне CRM: `Marketing From` на Deal; те же значения, что и у лида при конвертации): **Sales / Marketing / Partner / Client**. См. `02-Modules/01-CRM/02-Lead-Pipeline.md`, `03-Deal-Pipeline.md`.

- При расчёте сохранять **snapshot** выбранной категории `From` и применённых ставок на момент первого квалифицирующего платежа, чтобы последующая смена справочника не переписывала историю.

**Стартовая таблица ставок** (хранится в `Bonus Policy` / БД, **редактируется в UI** без смены кода):

| `From`        | Classic (база = **общая сумма** заказа) | Subscription (база = **сумма 1-го месяца**) |
| ------------- | --------------------------------------- | ------------------------------------------- |
| **Sales**     | Seller **8%**, Assistant **2%**         | Seller **80%**, Assistant **20%**           |
| **Marketing** | Seller **6%**, Assistant **1%**         | Seller **60%**, Assistant **10%**           |
| **Partner**   | Seller **6%**, Assistant **1%**         | Seller **60%**, Assistant **10%**           |
| **Client**    | Seller **4%**, Assistant **1%**         | Seller **40%**, Assistant **10%**           |

Столбец Subscription — **стартовые** значения; их можно изменить так же, как Classic. Начисления за **2-й и далее месяцы** подписки — **отдельное** правило в `Bonus Policy` (по умолчанию без автоповтора этой строки).

**Пример Classic:** заказ **1 000 000**, `From` = Sales → Seller **80 000** (8%), Assistant **20 000** (2%).

**Пример Subscription:** первый месяц **100 000**, `From` = Client → Seller **40 000** (40%), Assistant **10 000** (10%).

**Когда деньги попадают в «банк» начислений**

- По умолчанию: **полные** начисления по формулам выше — с **первого** подтверждённого платежа по сделке / первого оплаченного периода подписки; **идемпотентность** по deal/order (не дублировать тот же расчёт).
- Для классики в **несколько траншей** без изменения общей суммы заказа не начислять второй раз ту же пару сумм, если политика не задаёт иное (например половина с первого платежа, половина со второго — только явная **Billing / Bonus Policy**).

**Данные и UI (требования к продукту)**

- **БД:** модель ставок с ключами минимум `from_category` (Sales | Marketing | Partner | Client), `payment_model` (CLASSIC | SUBSCRIPTION_FIRST_MONTH), `seller_percent`, `assistant_percent`, `effective_from`, `is_active`; опционально audit и комментарий.
- **UI:** экран настройки (например Settings → Bonus Policy / Compensation) — таблица как выше, **редактирование каждого процента отдельно**, предпросмотр по тестовой сумме.
- Без заполненного **`From`** на сделке выбор строки политики невозможен → **блокировать** начисление sales bonus до заполнения (как и прочие marketing gates в CRM), если нет явно утверждённого fallback.

**KPI** по-прежнему применяется к **выплате** из накопленного учёта за месяц, а не к пересчёту ставок на сделке.

Реализация в коде: поля Seller / Assistant на Deal, идемпотентность начислений, Employee Wallet / отчёты.

### KPI Gate (порог выполнения плана)

Бонус корректируется в зависимости от выполнения месячного плана продаж:

| Выполнение плана | Коэффициент бонуса | Результат               |
| ---------------- | ------------------ | ----------------------- |
| ≥ 70%            | 100%               | Полный бонус            |
| 50% – 69%        | 50%                | Половина бонуса         |
| < 50%            | 0%                 | Бонус сгорает полностью |

**Пример (KPI к выплате, не к пересчёту ставок по сделке):**

- В «банке» начислений продавца за месяц накопилась сумма X после всех сделок.
- План: 5 000 000 ₽/мес; факт: 3 800 000 ₽ (76%) → KPI ≥ 70% → коэффициент **100%** к выплате X.
- Если бы факт был < 50%, выплата за месяц по KPI могла бы обнулиться — **начисления в учёте** остаются для аудита, логика списания по политике Finance.

## Универсальная Bonus Policy

`Bonus Policy` применяется не только к продавцам. Это универсальный слой правил мотивации для всех отделов.

| Роль      | От чего может зависеть бонус                                                                                                                                                    |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Seller    | категория **`From`** (Sales/Marketing/Partner/Client), тип оплаты (Classic / Subscription first month), две ставки `seller_rate` / `assistant_rate` из `Bonus Policy`, KPI gate |
| Developer | product category, product type, сложность, роль в проекте, уровень                                                                                                              |
| PM        | project delivery, deadline, scope variance, acceptance                                                                                                                          |
| Designer  | product type, дизайн-объём, правки, acceptance                                                                                                                                  |
| Marketing | MQL, SQL, CPL, revenue from marketing leads                                                                                                                                     |
| Support   | SLA, tickets closed, reopen rate, maintenance quality                                                                                                                           |

Код должен содержать безопасный `Bonus Policy Engine`, а интерфейс должен позволять настраивать параметры правил: проценты, thresholds, effective dates, employee overrides и статус active/inactive.

## Subscription Product Bonus Release / Выпуск бонусов по подписочным продуктам

> **Граница с sales bonus:** здесь речь о **delivery / product bonus** (PM, Developer, Designer) и **Product Bonus Pool** после сдачи конкретного продукта / доработки внутри проекта. **Не путать** с **sales bonus** Seller/Assistant по сделке (ставки от `From`, база Classic или первый месяц подписки) — тот блок выше в § «Две независимые ставки».

Для продуктов, проданных по подписке, нельзя автоматически выплачивать весь delivery bonus сразу, если клиент платит частями. Но после сдачи продукта компания должна закрывать бонусы максимально быстро из тех денег, которые уже получила именно по этому продукту / extension.

### Product Bonus Pool

Для каждого **Product / Extension** формируется отдельный `Product Bonus Pool / Бонусный фонд продукта`.

`Project` — это оболочка клиента/работы. В одном проекте может быть несколько продуктов или доработок, которые продаются, оплачиваются и закрываются отдельно. Поэтому pool нельзя считать одним общим фондом проекта: бонусы специалистам считаются и выпускаются **по каждому Product / Extension отдельно**, по его заказу, оплатам и delivery status.

Пример:

```text
Seller:     100,000
PM:          30,000
Developer: 120,000
Designer:   50,000
Total:     300,000
```

### До сдачи продукта

До `Product Done / Acceptance` delivery bonus не активируется автоматически.

Возможен только ручной `Early Bonus Release`:

- Finance/CEO выбирает сотрудника и сумму;
- система показывает, сколько денег уже получено по этому Product / Extension;
- если release выше доступного фонда, появляется `Over Funding`;
- если release выше planned bonus, появляется `Extra Bonus`;
- reason обязателен.

### После сдачи продукта

После сдачи продукта система автоматически выпускает бонусы на максимально возможную сумму из доступного продуктового фонда.

```text
Product Done
-> Product Received Amount
-> minus already released bonuses
-> Available Product Bonus Funding
-> auto-release remaining planned bonuses up to available funding
```

Если денег хватает на весь остаток, система активирует весь остаток бонусов сразу. Если денег не хватает, система активирует частично.

По умолчанию частичный release распределяется пропорционально между оставшимися плановыми бонусами.

```text
Remaining:
PM:         30,000
Developer:120,000
Designer:  50,000
Total:    200,000

Available funding: 100,000

Auto-release:
PM:         15,000
Developer: 60,000
Designer:  25,000
```

Finance/CEO может изменить предложенные суммы до approval payroll:

- уменьшить сумму и перенести остаток на следующий месяц;
- увеличить сумму в рамках фонда;
- выплатить сверх planned bonus как `Extra Bonus`;
- выплатить сверх полученных денег как `Over Funding`;
- указать reason и approval для исключений.

### Новая оплата после частичного release

Если после сдачи продукта бонусы закрыты не полностью, каждая новая оплата клиента по этому Product / Extension снова запускает проверку:

```text
New client payment
-> update available funding
-> auto-release remaining bonuses if Product / Extension is Done
```

Это отражает принцип компании: платить бонусы максимально быстро, как только бизнес может это сделать по конкретному продукту, но не скрывать кассовый риск.

## Бонус разработки (Delivery Bonus)

### Кто получает

- **Developer** (разработчик)
- **PM** (проектный менеджер)
- **Designer** (дизайнер)

### Триггер начисления

```
Product.status = "Done" (клиент принял)
  AND Payment условие выполнено (зависит от модели оплаты)
  → Бонус разработки начисляется
```

### Процент бонуса по типу проекта

| Тип проекта     | % бонуса от суммы заказа | Обоснование                           |
| --------------- | ------------------------ | ------------------------------------- |
| **White Label** | ~7%                      | Минимум кастомной работы              |
| **Mix**         | ~10%                     | Средняя сложность                     |
| **Custom Code** | ~15%                     | Максимальная сложность и трудозатраты |

### Зависимость от модели оплаты

#### Classic (50/50 или 30/30/40)

```
Условие: 100% оплаты получено + Проект сдан клиенту
  → Бонус начисляется полностью
```

- Бонус выплачивается **после** получения последнего платежа и подтверждения приёмки
- Если проект сдан, но оплата не полная — бонус в статусе ожидания

#### Subscription (ежемесячные платежи)

```
Условие: Проект сдан клиенту + Клиент перешёл на Maintenance (≈ месяц 4)
  → Бонус начисляется полностью
```

- Бонус выплачивается когда компания уже получила достаточно платежей (обычно месяц 4)
- К этому моменту клиент перешёл на Maintenance и продолжает платить стабильно
- Связь для специалиста очевидна: сдал работу → получил бонус

### Milestone-бонусы для длинных проектов

Для проектов длительностью **> 2 месяцев** возможна схема milestone-бонусов:

| Milestone              | % от общего бонуса | Условие                                    |
| ---------------------- | ------------------ | ------------------------------------------ |
| Mid-point (50% работы) | 50%                | Согласованный промежуточный результат сдан |
| Delivery (100% работы) | 50%                | Финальная сдача и приёмка                  |

Решение о milestone-бонусах принимает CEO на этапе планирования проекта.

### Cap (ограничение переменной части)

- Максимальный бонус за месяц: **100–200% от базовой зарплаты**
- Если бонусы превышают Cap → остаток переносится на следующий месяц
- Cap применяется для предотвращения резких скачков в фонде оплаты труда

## Бонус маркетинга (Marketing Bonus)

### Структура

| Компонент        | Описание                                            |
| ---------------- | --------------------------------------------------- |
| Fix-бонус за KPI | Фиксированная сумма за выполнение KPI маркетинга    |
| % от продаж      | Процент от общей суммы продаж с маркетинговых лидов |

### Пример расчёта

```
KPI маркетинга выполнен ≥ 100%:
  Fix-бонус: 50 000 ₽

Общая сумма продаж с маркетинговых лидов: 8 000 000 ₽
  % бонус: 1% × 8 000 000 = 80 000 ₽

Итого бонус маркетинга: 130 000 ₽
```

### Условия

- Fix-бонус выплачивается только при выполнении KPI ≥ заданного порога
- % от продаж привязан к лидам, которые были помечены как Marketing-source
- Отслеживается через Lead → Deal → Order → Payment цепочку

## Состояния бонуса (Bonus States)

Обобщённая модель для разных типов бонусов. Выплаты управляются через доступный product bonus pool, ручные release-решения Finance/CEO, KPI, cap и clawback.

Каждый бонус проходит через определённые состояния в своём жизненном цикле:

```
Incoming → Earned → Pending Eligibility → Vested → Active → Paid
                                                        ↓
                                                    Clawback
```

| Состояние               | Описание                                                                      | Когда                              |
| ----------------------- | ----------------------------------------------------------------------------- | ---------------------------------- |
| **Incoming**            | Будущий бонус — Product / Extension ещё не сдан, но видна потенциальная сумма | При создании Order                 |
| **Earned**              | Работа выполнена — бонус заработан, но финансовое условие не выполнено        | Product.status = Done              |
| **Pending Eligibility** | Ожидание финансового условия (оплата клиентом)                                | Работа сдана, ждём оплату          |
| **Vested**              | Финансовое условие выполнено, бонус подтверждён                               | Payment получен                    |
| **Active**              | Бонус готов к выплате в следующем месяце                                      | Разрешён к payroll по pool/KPI/cap |
| **Paid**                | Бонус выплачен                                                                | Включён в зарплату                 |
| **Clawback**            | Бонус отозван (возврат / chargeback от клиента)                               | Refund от клиента                  |

### Отображение в Employee Wallet

Полная цепочка состояний выше используется в учёте и расчётах. В интерфейсе **[Employee Wallet](../02-Modules/04-Finance/08-Employee-Wallet.md)** для сотрудника состояния **сворачиваются в понятные группы** (прогноз, ожидание условий, к выплате, выплачено, clawback) без изменения определений таблицы — см. маппинг в Wallet.

## Bonus Board (Доска бонусов)

Каждый сотрудник видит свою **персональную доску бонусов** с тремя разделами:

### Структура доски

| Раздел       | Что отображается   | Описание                                            |
| ------------ | ------------------ | --------------------------------------------------- |
| **Incoming** | Будущие бонусы     | Проекты в работе — видна потенциальная сумма бонуса |
| **Active**   | Активные бонусы    | Бонусы, готовые к выплате в следующем месяце        |
| **Paid**     | Выплаченные бонусы | История всех полученных бонусов                     |

### Карточка бонуса содержит

| Поле                | Описание                                            |
| ------------------- | --------------------------------------------------- |
| Product / Extension | Название продукта / доработки и родительский проект |
| Сумма заказа        | Общая сумма Order                                   |
| Сумма бонуса        | Расчётная сумма бонуса                              |
| Тип                 | Sales / Delivery / Marketing                        |
| Состояние           | Текущий статус (Incoming → Active → Paid)           |
| Дата начисления     | Когда бонус стал Active                             |
| Дата выплаты        | Когда был / будет выплачен                          |

### Мотивационный эффект

Сотрудник видит:

- Сколько он **заработает** в будущем (Incoming) — мотивация завершать проекты
- Сколько он **получит** в следующем месяце (Active) — конкретная цифра
- Сколько он **получил** за всю историю (Paid) — чувство прогресса

## Clawback (Возврат бонуса)

### Триггер

```
Клиент запросил возврат (Refund) или произошёл Chargeback
  → Бонус, связанный с данным Payment, отзывается
```

### Механизм

- Clawback вычитается из **будущих бонусов** сотрудника (не из зарплаты)
- Если у сотрудника нет будущих бонусов — вычет переносится на следующие месяцы
- **Исключительные случаи** (не по вине сотрудника) — CEO принимает решение

### Пример

```
Продажник получил бонус 50 000 ₽ за сделку
Клиент через месяц запросил refund
  → 50 000 ₽ вычитается из бонусов следующего месяца
  → Если в следующем месяце бонусы = 80 000 ₽
  → Продажник получит 80 000 - 50 000 = 30 000 ₽
```

## Payroll Cycle (Цикл выплат)

### Зарплатный цикл

```
Месяц N:
  - Зарплата (Fix) = выплачивается по графику компании
  - Бонусы заработанные в месяце N
      → Состояние: Active (готовы к выплате)
      → Выплата: между 1 и 10 числом месяца N+1
```

### Salary Board (Зарплатная доска)

Отдельная доска, где видна ежемесячная зарплата каждого сотрудника:

| Поле                   | Описание                      |
| ---------------------- | ----------------------------- |
| Сотрудник              | Имя и должность               |
| Fix                    | Базовая зарплата              |
| Бонусы (текущий месяц) | Сумма Active-бонусов          |
| Итого к выплате        | Fix + Active бонусы           |
| Бонусы (Incoming)      | Будущие бонусы (для прогноза) |

Зарплатная доска доступна:

- **CEO** — видит всех сотрудников
- **Финансовый директор** — видит всех (управляет выплатами)
- **Сотрудник** — видит только свои данные

## Сводная таблица бонусных правил

| Параметр     | Sales                                                                                           | Delivery (Dev/PM/Design)   | Marketing                  |
| ------------ | ----------------------------------------------------------------------------------------------- | -------------------------- | -------------------------- |
| Триггер      | Первый квалифицирующий Payment по сделке                                                        | Done + Payment             | KPI + Payment              |
| %            | Две ставки × база по **`From`** и Classic/Subscription (см. § выше; стартовая таблица в каноне) | 7–15% (от типа проекта)    | Fix + 1% от объёма         |
| KPI Gate     | Да (план продаж)                                                                                | Нет                        | Да (KPI маркетинга)        |
| Clawback     | Да                                                                                              | Да                         | Да                         |
| Cap          | 100–200% от Fix                                                                                 | 100–200% от Fix            | 100–200% от Fix            |
| Milestone    | Нет                                                                                             | Да (для проектов > 2 мес.) | Нет                        |
| Цикл выплаты | Месяц N → выплата 1–10 N+1                                                                      | Месяц N → выплата 1–10 N+1 | Месяц N → выплата 1–10 N+1 |

## Исключительные ситуации

| Ситуация                                  | Решение                                          |
| ----------------------------------------- | ------------------------------------------------ |
| Проект отменён клиентом в процессе        | CEO решает — частичный бонус или нет             |
| Refund по вине компании                   | Clawback не применяется, CEO решает              |
| Сотрудник уволился до выплаты бонуса      | Active-бонусы выплачиваются, Incoming — сгорают  |
| Проект затянулся по вине клиента          | Бонус начисляется по стандартной схеме при сдаче |
| Два продажника работали над одной сделкой | Бонус делится по согласованию CEO                |
