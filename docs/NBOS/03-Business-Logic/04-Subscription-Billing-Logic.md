# Логика биллинга подписок

## Общее описание

Подписка (Subscription) — это фиксированный ежемесячный платёж от клиента за Maintenance (техобслуживание) и/или продолжающуюся разработку (Development). Система подписок обеспечивает стабильный ежемесячный доход компании и требует автоматизированного процесса выставления счетов, уведомлений и контроля оплат.

---

## Типы подписок

| Тип | Описание | Пример |
|-----|----------|--------|
| **Maintenance Only** | Только техобслуживание (поддержка, мелкие правки, мониторинг) | Сайт сдан, клиент платит 80 000/мес за поддержку |
| **Development + Maintenance** | Разработка + техобслуживание параллельно | Месяц 4+: 100 000 dev + 80 000 mnt = 180 000/мес |
| **Development Only** | Только разработка (подписочная модель оплаты) | Первые 3 месяца: 100 000/мес за разработку |
| **Partner Service** | Оплата за аутсорсную услугу (маркетинг, SEO и т.д.) | Партнёр предоставляет услугу через Neetrino |

---

## Атрибуты подписки

Каждая подписка содержит следующие данные:

| Поле | Описание | Пример |
|------|----------|--------|
| **Проект** | Привязка к проекту в Projects Hub | «BrandX — Website» |
| **Сумма** | Ежемесячная сумма платежа | 180 000 ₽ |
| **Billing Day** | День месяца для выставления счёта | 5-е число |
| **Tax Status** | Налогооблагаемый / Не налогооблагаемый | Tax / Tax-Free |
| **Дата начала** | Когда подписка начала действовать | 01.03.2025 |
| **Дата окончания** | Когда подписка завершится (если известно) | null (бессрочная) |
| **Тип** | Maintenance / Development / Dev+Mnt / Partner | Maintenance Only |
| **Контакт** | Контактное лицо клиента | Иван Петров |
| **Компания** | Юридическое лицо для выставления счёта | ООО «BrandX» |

### Tax Status (статус налогообложения)

| Статус | Поведение |
|--------|-----------|
| **Tax** | Счёт попадает на стадию «Создать в госсистеме» → бухгалтер выставляет официальный счёт |
| **Tax-Free** | Счёт пропускает стадию госсистемы → прямой перевод без официального счёта |

Tax Status берётся из подписки и наследуется каждым Invoice, созданным из этой подписки. Статус устанавливается при создании подписки и не меняется в течение её жизни.

## Billing Cycle (Цикл выставления счетов)

### Автоматический процесс

```
Шаг 1: Создание Invoice
  ────────────────────────────────────
  За N дней до Billing Day → Invoice создаётся автоматически
  Статус: "New"
  
Шаг 2: Создание в госсистеме (через ~1 час)
  ────────────────────────────────────
  Invoice переходит на стадию "Create in Gov System"
  → Уведомление бухгалтеру
  → Бухгалтер создаёт официальный счёт в государственной программе
  (Пропускается, если Tax-Free)
  
Шаг 3: Отправка сообщения клиенту (через ~1 час / ~1 день)
  ────────────────────────────────────
  Invoice переходит на стадию "Send Message"
  → WhatsApp-сообщение клиенту: «Ваш счёт выставлен, оплатите до [дата]»
  
Шаг 4: Ожидание оплаты
  ────────────────────────────────────
  Клиент оплачивает до Billing Day
  
Шаг 5: Проверка оплаты
  ────────────────────────────────────
  Финансовый директор проверяет банк ежедневно
  Если оплачено → Invoice.status = "Paid" → ячейка в Grid зелёная
  
Шаг 6: Просрочка (если не оплачено к Billing Day)
  ────────────────────────────────────
  Invoice переходит на стадию "Overdue"
  → 1-е уведомление о просрочке отправляется клиенту
  
Шаг 7: Повторное уведомление (+2 дня после просрочки)
  ────────────────────────────────────
  → 2-е уведомление: «Оплата просрочена, возможно отключение»
  
Шаг 8: Эскалация (при длительной неоплате)
  ────────────────────────────────────
  Финансовый директор решает: напомнить ещё раз / приостановить / CEO
```

### Визуализация стадий Invoice подписки

```
New → Create in Gov System → Send Message → [Ожидание] → Paid
                                                ↓
                                            Overdue → Overdue Notification 2 → ...
                                                              ↓
                                                          Эскалация
```

## Ежедневная работа Финансового директора

### Утренний процесс

1. **Открыть банк** — проверить поступления за вчера/сегодня
2. **Отметить оплаченные** — перевести оплаченные Invoice в статус «Paid»
3. **Запустить автоматизацию** — нажать кнопку «Запустить процесс»:
   - Неоплаченным клиентам отправляются уведомления
   - Invoice переходят на следующие стадии автоматически
   - Просроченные получают соответствующие уведомления

### Важно: порядок действий

```
1. Проверить оплаты в банке
2. Отметить оплаченные Invoice как "Paid" (чтобы они НЕ получили напоминание)
3. Запустить автоматизацию (напоминания уходят только неоплаченным)
```

## Разные Billing Days

Клиенты платят в разные дни месяца. Система учитывает индивидуальный Billing Day для каждой подписки.

| Billing Day | Создание Invoice | Уведомление клиенту | Дедлайн оплаты | Просрочка |
|-------------|-----------------|---------------------|-----------------|-----------|
| 5-е число | 1-го числа | 1-го числа | 5-е число | 6-е число |
| 15-е число | 10-го числа | 10-го числа | 15-е число | 16-е число |
| 20-е число | 15-го числа | 15-го числа | 20-е число | 21-е число |

### Процесс для 1-го числа (массовый)

Большинство подписок (≈90%) имеют Billing Day = 5-е число:
- 1-го числа: массовое создание Invoice → уведомления бухгалтеру → уведомления клиентам
- 5-го числа: массовая проверка оплат → отметка Paid → автоматизация напоминаний
- 6–7 числа: повторные проверки и напоминания о просрочке

### Процесс для других дат

Invoice создаются автоматически за N дней до каждого индивидуального Billing Day. Финансовый директор видит их на доске и обрабатывает по мере поступления.

## Subscription Grid View (Сетка подписок)

### Описание

Subscription Grid — это основной инструмент для визуального контроля подписок. Формат: **матрица «проекты × месяцы»**.

### Структура

```
                 │  Янв  │  Фев  │  Мар  │  Апр  │  Май  │ ... │  Дек  │
─────────────────┼───────┼───────┼───────┼───────┼───────┼─────┼───────┤
Project Alpha    │ 180k  │ 180k  │ 180k  │ 180k  │ 180k  │ ... │ 180k  │
                 │  ✅   │  ✅   │  ✅   │  🟡   │       │     │       │
─────────────────┼───────┼───────┼───────┼───────┼───────┼─────┼───────┤
Project Beta     │       │       │ 100k  │ 100k  │ 180k  │ ... │ 180k  │
                 │       │       │  ✅   │  ✅   │  🟡   │     │       │
─────────────────┼───────┼───────┼───────┼───────┼───────┼─────┼───────┤
Project Gamma    │ 80k   │ 80k   │ 80k   │ 80k   │ 80k   │ ... │ 80k   │
                 │  ✅   │  ✅   │  🔴   │       │       │     │       │
─────────────────┼───────┼───────┼───────┼───────┼───────┼─────┼───────┤
  ИТОГО          │ 260k  │ 260k  │ 360k  │ 360k  │ 440k  │ ... │ 440k  │
```

### Цветовая кодировка ячеек

| Цвет | Статус | Описание |
|------|--------|----------|
| 🟢 Зелёный | Paid | Оплачено |
| 🔴 Красный | Overdue | Просрочено |
| 🟡 Жёлтый | Pending | Счёт выставлен, ожидание оплаты |
| ⬜ Пустая | — | Подписка ещё не началась / не существует |
| 🔵 Синий | Forecast | Прогноз на будущие месяцы |

### Функциональность Grid

| Функция | Описание |
|---------|----------|
| **Просмотр суммы** | В каждой ячейке видна сумма и статус |
| **Итоги по месяцам** | Нижняя строка — сумма всех подписок за месяц |
| **Прогноз** | Будущие месяцы показывают ожидаемые поступления |
| **Пустые ячейки** | Видно, когда подписка началась (до этого — пусто) |
| **Изменение суммы** | Возможность изменить сумму подписки (с какого месяца) |
| **Фильтры** | По типу подписки, по статусу оплаты, по сумме |
| **Клик по ячейке** | Открывает Invoice подписки за этот месяц |

### Изменение суммы подписки

Сумма подписки может меняться со временем:
- Согласование с клиентом → изменение суммы начиная с определённого месяца
- В Grid видна актуальная сумма для каждого месяца
- История изменений сохраняется

**Пример:**
```
Project Beta:
  Мар-Мар: 100 000 (Development)
  Апр+:    180 000 (Development + Maintenance)
```

## Переход Development → Maintenance

### Типичный сценарий

```
Месяц 1-3: Development Only (100 000/мес)
  → Проект сдан
Месяц 4+:  Development + Maintenance (100 000 + 80 000 = 180 000/мес)
  → Через N месяцев: Maintenance Only (80 000/мес)
```

### В системе

1. Подписка создаётся с типом «Development Only» и суммой 100 000
2. После сдачи проекта PM / Финансовый директор меняет:
   - Тип → «Development + Maintenance»
   - Сумма → 180 000
   - Дата изменения фиксируется
3. Когда Development-период заканчивается:
   - Тип → «Maintenance Only»
   - Сумма → 80 000

## Pass-through подписки (домены и сервисы)

### Описание

Некоторые подписки — это «транзитные» платежи: клиент платит Neetrino, а Neetrino оплачивает сторонний сервис для клиента.

### Примеры

| Сервис | Описание | Периодичность |
|--------|----------|---------------|
| Доменное имя | Покупка/продление домена | Ежегодно |
| Google Workspace | Рабочая почта | Ежемесячно (сумма зависит от объёма) |
| App Store аккаунт | Аккаунт разработчика Apple | Ежегодно |
| Хостинг | Серверы, CDN | Ежемесячно |

### Особенности

- Tax Status может быть **Tax-Free** (не нужен счёт в госсистеме)
- Сумма может меняться от месяца к месяцу (зависит от потребления)
- В Subscription Grid отображается как обычная подписка
- При создании Invoice для домена → автоматически создаётся задача на покупку/продление

### Процесс для доменов

```
1. За 2 месяца до истечения домена → система создаёт Invoice клиенту
2. Клиент оплачивает
3. После оплаты → создаётся задача "Продлить домен [name]"
4. Ответственный сотрудник продлевает домен
5. Подтверждение → задача закрыта
```

## Автоматизация уведомлений

### Типы уведомлений

| Событие | Канал | Получатель | Текст (шаблон) |
|---------|-------|-----------|----------------|
| Invoice создан | WhatsApp | Клиент | «Здравствуйте! Ваш счёт на [сумма] выставлен. Оплатите до [дата].» |
| Invoice создан | Система | Бухгалтер | «Создать счёт в госсистеме: [проект], [сумма], [реквизиты]» |
| Просрочка (1-я) | WhatsApp | Клиент | «Напоминаем: оплата [сумма] за [проект] просрочена.» |
| Просрочка (2-я) | WhatsApp | Клиент | «Оплата за [проект] не получена. Возможна приостановка обслуживания.» |
| Оплата получена | Система | Финансовый директор | «Оплата [сумма] от [клиент] получена.» |

### Расписание уведомлений

```
День B-5:  Invoice создан → уведомление бухгалтеру
День B-4:  Уведомление клиенту (WhatsApp)
День B:    Billing Day (дедлайн оплаты)
День B+1:  1-е уведомление о просрочке
День B+3:  2-е уведомление о просрочке
День B+7:  Эскалация → Финансовый директор решает

(B = Billing Day)
```

## Правила и ограничения

1. **Каждая подписка привязана к проекту** — не может существовать без проекта
2. **Tax Status наследуется** — Invoice берёт статус из Subscription
3. **Сумма может меняться** — но история изменений сохраняется
4. **Invoice создаётся автоматически** — вручную создавать не нужно
5. **Оплата подтверждается только Финансовым директором** — после проверки банка
6. **Уведомления не отправляются оплаченным** — сначала отметить Paid, потом запускать автоматизацию
7. **Предоплата возможна** — клиент может оплатить несколько месяцев вперёд
8. **Subscription Grid — источник правды** для прогноза ежемесячного дохода
