# Процесс управления изменениями (Change Control)

## Общее описание

Клиенты на техобслуживании (Maintenance) регулярно обращаются с запросами. Не каждый запрос — это платная работа. Процесс Change Control позволяет корректно классифицировать запросы, определить, что бесплатно (входит в Maintenance), а что оплачивается отдельно, и обеспечить правильный учёт всех работ для начисления бонусов и P&L.

---

## Классификация запросов

### Типы запросов (ITIL-based)

| Тип                 | Описание                                                   | Оплата                                               | Результат в системе         |
| ------------------- | ---------------------------------------------------------- | ---------------------------------------------------- | --------------------------- |
| **Incident**        | Что-то сломалось / баг по нашей вине                       | Бесплатно                                            | Support Ticket (задача)     |
| **Service Request** | Настройка / конфигурация в рамках обслуживания             | Бесплатно (обычно)                                   | Задача в проекте            |
| **Change Request**  | Новая функциональность / доработка                         | **Платно**                                           | Extension Deal в CRM        |
| **Problem**         | Повторяющаяся проблема, требующая анализа корневой причины | Бесплатно (анализ) / Платно (если нужен рефакторинг) | Задача + возможно Extension |

### Таблица принятия решений

```
Клиент написал запрос
  ↓
PM анализирует:
  ├── Это баг / наша ошибка? → Incident → Бесплатно → Task
  ├── Это настройка, входящая в Maintenance? → Service Request → Бесплатно → Task
  ├── Это новая функция / значительное изменение? → Change Request → Платно → Extension Deal
  └── Это повторяющаяся проблема? → Problem → Анализ → Может стать Change Request
```

## Incident (Инцидент)

### Определение

Инцидент — это нарушение нормальной работы продукта, возникшее по вине Neetrino или из-за технических причин, которые должны были быть предотвращены.

### Процесс

1. Клиент сообщает о проблеме (WhatsApp / чат)
2. PM создаёт задачу (Support Ticket) в проекте
3. Приоритет определяется по влиянию на бизнес клиента:
   - **Critical** — сайт/приложение не работает → реагирование < 1 часа
   - **High** — ключевая функция не работает → реагирование < 4 часов
   - **Medium** — некритичная проблема → реагирование < 1 рабочего дня
   - **Low** — косметический дефект → реагирование < 3 рабочих дней
4. Разработчик исправляет проблему
5. PM подтверждает клиенту, что исправлено
6. Задача закрывается

### Правила

- **Не создаёт Order** — это гарантийная работа
- **Не начисляет бонус** — входит в Maintenance
- Если инциденты по одному проекту приходят слишком часто → Problem Management

## Service Request (Сервисный запрос)

### Определение

Запрос на настройку, изменение конфигурации или мелкую корректировку, которая входит в рамки Maintenance-контракта.

### Примеры

| Запрос                           | Тип             | Оплата    |
| -------------------------------- | --------------- | --------- |
| Заменить фото на главной         | Service Request | Бесплатно |
| Обновить текст в разделе «О нас» | Service Request | Бесплатно |
| Добавить новый email для формы   | Service Request | Бесплатно |
| Обновить SSL-сертификат          | Service Request | Бесплатно |
| Изменить цвет кнопки             | Service Request | Бесплатно |

### Процесс

1. Клиент пишет запрос
2. PM создаёт задачу в проекте (простая задача, не Extension)
3. Разработчик / PM выполняет
4. Задача закрывается

## Change Request (Запрос на изменение) — ПЛАТНАЯ РАБОТА

### Определение

Change Request — это запрос на новую функциональность, значительное изменение существующей или работу, выходящую за рамки Maintenance.

### Примеры

| Запрос                          | Тип            | Оплата     |
| ------------------------------- | -------------- | ---------- |
| Добавить онлайн-оплату на сайт  | Change Request | **Платно** |
| Создать новый раздел / страницу | Change Request | **Платно** |
| Интеграция с CRM / ERP          | Change Request | **Платно** |
| Редизайн существующего раздела  | Change Request | **Платно** |
| Мобильная версия приложения     | Change Request | **Платно** |
| Добавить мультиязычность        | Change Request | **Платно** |

### Полный процесс (Flow)

```
Шаг 1: Запрос клиента
  ──────────────────────
  Клиент пишет в WhatsApp / чат проекта

Шаг 2: Классификация PM
  ──────────────────────
  PM определяет: это Maintenance (бесплатно) или новая работа (платно)?
  Решение принимается через mini-IDS (не в чате!)

Шаг 3: Передача продажнику
  ──────────────────────
  Если платно → PM информирует Seller
  PM передаёт: описание запроса, предварительную оценку, контекст

Шаг 4: Extension Deal в CRM
  ──────────────────────
  Seller создаёт Extension Deal:
    - Тег: "Extension"
    - Привязка к существующему проекту
    - Привязка к контакту клиента
    - Визуально отличается (цвет/тег) от новых сделок

Шаг 5: Стандартный Deal Pipeline
  ──────────────────────
  Extension Deal проходит стандартные стадии:
    Start a Conversation → Discuss Needs → Meeting* →
    Can We Do It → Send Offer → Get Answer →
    Deposit & Contract → Deal Won
  (*Meeting может быть пропущена для мелких доработок)
  После Deal Won — Order, Delivery Board, финальные оплаты и Maintenance вне воронки CRM

Шаг 6: Order + Extension в проекте
  ──────────────────────
  Deal Won + Payment received:
    → Order создаётся
    → Extension создаётся внутри существующего Project
    → Задачи создаются в Extension (Scrum/Kanban)

Шаг 7: Выполнение работ
  ──────────────────────
  Разработка → QA → Transfer → Done

Шаг 8: Закрытие Extension
  ──────────────────────
  Extension.status = Done
    → Чек-лист приёмки пройден
    → Клиент подтвердил
    → Бонус начисляется (по правилам Delivery Bonus)
```

### Визуализация в системе

```
Project "BrandX"
  ├── Product: Website (status: Maintenance)
  ├── Extension 1: "Онлайн-оплата" (status: Development) → активно
  ├── Extension 2: "Мультиязычность" (status: Done) → завершено
  └── Extension 3: "SEO-модуль" (status: New) → только что создано
```

## Mini-IDS: принятие решений

### Что такое Mini-IDS

Mini-IDS (Issue → Discussion → Solution) — это формализованный процесс принятия решений, который используется вместо спонтанных решений в чатах.

### Формат

| Компонент                 | Описание                      | Пример                                                         |
| ------------------------- | ----------------------------- | -------------------------------------------------------------- |
| **Проблема (Issue)**      | Что именно запрашивает клиент | «Клиент просит добавить фильтрацию товаров»                    |
| **Варианты (Options)**    | Возможные решения             | 1) Бесплатно в рамках Maintenance 2) Платно как Extension      |
| **Решение (Decision)**    | Что выбрали и почему          | «Платно — это новая функциональность, не входит в Maintenance» |
| **Ответственный (Owner)** | Кто исполняет                 | «Seller создаёт Extension Deal»                                |
| **Дедлайн (Deadline)**    | До какого срока               | «КП клиенту до 15.03»                                          |

### Правило

> Любое решение о том, является ли работа бесплатной (Maintenance) или платной (Change Request), принимается **НЕ в чате**, а через формальный mini-IDS. Это защищает компанию от необоснованных бесплатных работ и даёт чёткий audit trail.

## Problem (Проблема)

### Определение

Problem — это повторяющийся инцидент или системная проблема, требующая анализа корневой причины (Root Cause Analysis).

### Процесс

1. PM замечает, что клиент обращается с похожими проблемами регулярно
2. Создаётся Problem-задача для анализа
3. Определяется корневая причина:
   - Если архитектурная проблема → может потребоваться рефакторинг → Change Request
   - Если проблема в настройке → Service Request
   - Если проблема в нашем коде → Incident (исправление)
4. На основе анализа выбирается путь решения

### Эскалация

Если клиент пишет слишком часто (> 2 раз в месяц с проблемами):

- PM информирует CEO
- CEO принимает решение:
  - Кардинально решить техническую проблему
  - Продолжить обслуживание в штатном режиме
  - Предложить клиенту расширенный Maintenance (за доп. плату)
  - В крайнем случае: отказать в обслуживании (решение CEO)

## Micro Extension (для задач < 1 часа)

### Принцип

Даже очень маленькие платные работы (< 1 часа) должны быть оформлены как Extension + Order. Это критично для:

- **Бонусов** — разработчик получает бонус за любую оплаченную работу
- **P&L** — точный учёт доходов по каждому проекту
- **Истории** — полная трассировка: запрос → Deal → Order → Payment → Bonus

### Процесс для Micro Extension

```
1. PM определяет: работа платная, но очень маленькая (< 1 часа)
2. PM информирует Seller
3. Seller создаёт Extension Deal (упрощённый путь):
   - Может пропустить стадии Meeting и Can We Do It
   - Сумма обычно минимальная (фиксированный тариф)
4. Клиент соглашается → Payment → Order → Extension
5. Задача выполнена → Extension.status = Done → Бонус начислен
```

### Не путать с бесплатной работой

| Критерий                                  | Бесплатно (Maintenance) | Платно (Micro Extension) |
| ----------------------------------------- | ----------------------- | ------------------------ |
| Занимает < 15 мин, простая настройка      | ✅ Service Request      | —                        |
| Занимает < 1 час, но это новая функция    | —                       | ✅ Micro Extension       |
| Клиент просит изменить то, что мы сделали | ✅ Incident/SR          | —                        |
| Клиент просит сделать что-то новое        | —                       | ✅ Micro Extension       |

## Сводная матрица классификации

| Вопрос                                      | Да →                            | Нет →              |
| ------------------------------------------- | ------------------------------- | ------------------ |
| Это наш баг / ошибка?                       | **Incident** (бесплатно)        | Следующий вопрос ↓ |
| Входит в рамки Maintenance-контракта?       | **Service Request** (бесплатно) | Следующий вопрос ↓ |
| Это повторяющаяся проблема?                 | **Problem** (анализ → решение)  | Следующий вопрос ↓ |
| Это новая функция / значительное изменение? | **Change Request** (платно)     | Уточнить у PM/CEO  |

## Метрики и контроль

| Метрика                      | Описание                                 | Цель                        |
| ---------------------------- | ---------------------------------------- | --------------------------- |
| Incidents per Project/Month  | Количество инцидентов по проекту в месяц | < 2                         |
| Change Request Conversion    | % запросов, ставших Extension Deals      | Отслеживание                |
| Extension Revenue            | Доход от доработок                       | Рост                        |
| Average Extension Size       | Средняя сумма доработки                  | Для прогнозирования         |
| Time to Classification       | Время от запроса до классификации        | < 4 часов                   |
| Maintenance Cost per Project | Затраты на бесплатные работы по проекту  | Контроль убыточных проектов |

## Правила и ограничения

1. **Решение о бесплатно/платно** — только через mini-IDS, не в чате
2. **Любая платная работа** — обязательно Extension Deal + Order (даже < 1 часа)
3. **PM не может самостоятельно** давать скидки или делать работу бесплатно без согласования
4. **Extension Deal** проходит тот же пайплайн, что и новая сделка
5. **Бонус за Extension** начисляется по тем же правилам, что и за основной проект
6. **Частые инциденты** (> 2/мес) → эскалация к CEO
7. **CEO принимает решение** об отказе в обслуживании в крайних случаях
