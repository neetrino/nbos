# Анализ: сущность Product в истории git и пути восстановления

**Дата:** 2026-03-31  
**Запрос:** подтвердить, что Product раньше был реализован, затем убран по ошибке; оценить откат/мердж/повторную реализацию.

---

## 1. Ключевой коммит

| Поле      | Значение                                                   |
| --------- | ---------------------------------------------------------- |
| Хеш       | `02e2575`                                                  |
| Дата      | 2026-03-14                                                 |
| Сообщение | `feat: tasks system refactor + remove products/extensions` |

**Явно в сообщении:** удалены модели **Product** и **Extension**, enum’ы, API-модули `products` / `extensions`, фронтенд-вкладки; задачи переведены на **TaskLink** (полиморфные связи с PROJECT, DEAL, LEAD, ORDER, …). На **Deal** перенесены `pmId` и `deadline` (раньше часть логики жила у Product).

**Масштаб:** ~80 файлов, существенные изменения Prisma seed, API, web.

**Родитель:** `02e2575~1` — последний коммит **с** моделями `Product` и `Extension` в `schema.prisma`.

---

## 2. Что было в схеме до удаления (сводка)

Фрагмент из `02e2575~1:packages/database/prisma/schema.prisma`:

- **`Product`:** `projectId`, `name`, `productType` (enum `ProductTypeEnum`: WEBSITE, MOBILE_APP, CRM, …), `status` (`ProductStatusEnum`: NEW, CREATING, DEVELOPMENT, …), `pmId`, `deadline`, `checklistTemplateId`; связи: `order` 1:1, `extensions[]`, `tasks[]`, `tickets[]`.
- **`Extension`:** `projectId`, `productId?`, `name`, `size`, `status`, `assignedTo`, `order` 1:1, `tasks[]`.
- **`Order`:** `productId` **или** `extensionId` (уникальные FK), плюс `dealId`, `projectId`, тип заказа.
- **`Deal`:** без полей `productType` / `pmId` / `deadline` на карточке (их не было в показанном фрагменте старой модели — они появились после рефакторинга).

То есть **полноценная сущность Product в БД и API действительно существовала** до `02e2575`.

---

## 3. Текущее состояние репозитория (после `02e2575` и последующих коммитов)

- В **`schema.prisma` нет** `model Product` и `model Extension`.
- **`Order`** привязан к проекту и сделке; логика «один заказ — один продукт» в данных **не воспроизведена** так, как в старой схеме.
- **`Deal`** содержит `productType` (строка), `pmId`, `deadline` — компенсация за отсутствие Product.
- Задачи: **Task** + **TaskLink**, гибкие привязки к сущностям.
- После `02e2575` добавлено много функционала: RBAC, Clerk→NextAuth, сделки/инвойсы, **изменение воронки Deal** (`f36508a`), и т.д.

Документация в `docs/NBOS` (Core Entities, продукто-центричный Hub) **описывает Product как целевую модель** — она **ближе к старой схеме**, чем к текущему коду после удаления.

---

## 4. Варианты действий

### A. Полный `git revert 02e2575`

**Плюсы:** вернёт Product/Extension и старые модули одним коммитом.  
**Минусы:** почти наверняка **конфликты** со всеми последующими изменениями (сотни правок в тех же файлах). Ручное разрешение сопоставимо с полной переписью. **Не рекомендуется** как первый шаг.

### B. Частичный откат только `schema.prisma` + миграции из старого коммита

**Минусы:** текущий код API/web **не соответствует** старой схеме; приложение сломается, пока не восстановите сервисы и UI. Нужна **согласованная** цепочка: схема → миграция → API → web.

### C. Восстановление «вперёд» (рекомендуется)

1. **Спроектировать новую миграцию** под целевую модель из `docs` (Deal Type из справочников, Product Type, связь Order–Product, Extension при необходимости) — не копировать слепо старый enum, а согласовать с § 1.1 Core Entities.
2. **Постепенно:** вернуть модули `products` / `extensions` в API или объединить с существующими `projects`/`orders`.
3. **Задачи:** оставить TaskLink, добавить привязку задач к `productId` где нужно (как в старой модели `Product.tasks`).
4. **Снять с Deal** дублирующие поля (`pmId`/`deadline`/`productType`), когда логика снова на Product.

Так вы **сохраняете** новые правила (воронка Deal, auth, RBAC, финансы) и **добавляете** Product, а не откатываете весь репозиторий.

### D. «Лучшее из двух версий» (документы + код)

- **Документы:** уже ведут к Product + продукто-центричному Hub; доработать `02-Modules/02-Projects-Hub/*` при появлении реализации.
- **Код:** старый `ProductsTab` / `products.service.ts` из `02e2575~1` — **ориентир по полям и маршрутам**, не единственный источник истины (дизайн мог устареть).
