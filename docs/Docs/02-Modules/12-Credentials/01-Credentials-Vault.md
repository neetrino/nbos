# Хранилище паролей (Credentials Vault)

> NBOS Platform — встроенный менеджер паролей с корпоративным контролем доступа

## Назначение

Модуль Credentials — это **встроенный менеджер паролей**, заменяющий внешние решения (1Password, LastPass) и таблицы Excel. Все учётные данные компании хранятся в зашифрованном виде с гранулярным контролем доступа и полным аудит-логом.

**Ключевой принцип:** ни один пароль не должен передаваться через мессенджеры или почту. Все учётные данные хранятся в Credentials Vault с контролем: кто видит, кто копировал, кто изменял.

---

## Категории учётных данных

| Категория | Описание | Примеры |
|-----------|---------|---------|
| **Admin** | Административные аккаунты компании | Панели управления, root-доступы |
| **Domain** | Доменные регистраторы | Beget, Reg.ru, Namecheap, GoDaddy |
| **Hosting** | Серверы и хостинг | Beget, DigitalOcean, Hetzner, AWS, Vercel |
| **Service** | Внешние сервисы | Cloudflare, Neon, Upstash, Resend, Google Workspace, Sentry |
| **App** | Аккаунты магазинов приложений | App Store (Apple Developer), Google Play Console |
| **Mail** | Почтовые аккаунты | Корпоративная почта, SMTP-настройки |
| **API Key** | API-ключи и токены | Stripe API, OpenAI API, Telegram Bot Token |
| **Database** | Подключения к базам данных | Connection strings, DB credentials |
| **Other** | Прочее | WiFi-пароли, коды доступа в офис |

## Поля карточки Credential

### Основные поля

| Поле | Тип | Обязательное | Шифрование | Описание |
|------|-----|-------------|-----------|----------|
| **Название** | Текст | ✅ | ❌ | Понятное название (напр. «Beget — проект ClientBrand») |
| **Категория** | Enum | ✅ | ❌ | Admin / Domain / Hosting / Service / App / Mail / API Key / Database / Other |
| **Провайдер** | Выбор из списка | ❌ | ❌ | Beget, Reg.ru, Cloudflare, Neon, Upstash, Resend и др. |
| **URL** | URL | ❌ | ❌ | Ссылка на страницу входа |
| **Login** | Текст | ❌ | ✅ AES-256 | Логин / email для входа |
| **Password** | Текст | ❌ | ✅ AES-256 | Пароль |
| **API Key** | Текст | ❌ | ✅ AES-256 | API-ключ или токен |
| **env data** | Multiline Text | ❌ | ✅ AES-256 | Содержимое .env файла или группа переменных окружения |
| **Phone** | Телефон | ❌ | ❌ | Привязанный номер телефона (для 2FA) |
| **Notes** | Rich Text | ❌ | ❌ | Дополнительные заметки, инструкции |

### Поля доступа и привязки

| Поле | Тип | Описание |
|------|-----|----------|
| **Access Level** | Enum | Secret / Project Team / Department / All |
| **Allowed Employees** | Пользователи | Для уровня Secret: список сотрудников с доступом |
| **Project** | Связь | К какому проекту привязан credential |
| **Owner** | Пользователь | Кто создал / управляет этим credential |
| **Created** | DateTime | Дата создания |
| **Modified** | DateTime | Дата последнего изменения |
| **Modified By** | Пользователь | Кто последний изменял |

## Уровни доступа

### 1. Secret (Секретный)

| Параметр | Значение |
|----------|---------|
| **Кто видит** | ТОЛЬКО явно указанные сотрудники в поле «Allowed Employees» |
| **Применение** | Административные аккаунты компании, доступ к финансовым системам, root-пароли критической инфраструктуры |
| **Пример** | Root-доступ к основному серверу, аккаунт Apple Developer, главный аккаунт Google Workspace |

### 2. Project Team (Команда проекта)

| Параметр | Значение |
|----------|---------|
| **Кто видит** | Все участники привязанного проекта |
| **Применение** | Проектные учётные данные: хостинг проекта, домен проекта, API-ключи проекта |
| **Пример** | Пароль от Beget-аккаунта для проекта «ClientBrand», Neon DB connection string |

### 3. Department (Отдел)

| Параметр | Значение |
|----------|---------|
| **Кто видит** | Все сотрудники указанного отдела |
| **Применение** | Общие инструменты отдела, сервисы для командной работы |
| **Пример** | Аккаунт Figma для отдела дизайна, GitHub Organization для отдела разработки |

### 4. All (Все)

| Параметр | Значение |
|----------|---------|
| **Кто видит** | Все сотрудники компании |
| **Применение** | Общедоступная информация, не представляющая угрозы безопасности |
| **Пример** | Пароль от Wi-Fi в офисе, код домофона |

### 5. Personal — «Мои пароли»

| Параметр | Значение |
|----------|---------|
| **Кто видит** | Только сам сотрудник |
| **Применение** | Личные рабочие пароли сотрудника |
| **Пример** | Личный пароль от корпоративной почты, персональный API-токен |

## Безопасность

### Шифрование

| Аспект | Реализация |
|--------|-----------|
| **Алгоритм** | AES-256 на уровне отдельных полей |
| **Что шифруется** | Login, Password, API Key, env data |
| **Что НЕ шифруется** | Название, категория, провайдер, URL, заметки |
| **Хранение ключей** | Ключи шифрования хранятся отдельно от данных (key management) |
| **Передача** | Все данные передаются по HTTPS/TLS |

### Механика просмотра

1. Зашифрованные поля отображаются как `••••••••` по умолчанию
2. Для просмотра пользователь нажимает кнопку **«Показать»** (👁)
3. Факт просмотра **логируется** в аудит (кто, когда, какой credential)
4. Поле автоматически скрывается через 30 секунд
5. Кнопка **«Копировать»** — копирует значение в буфер обмена
6. Буфер обмена **автоматически очищается через 30 секунд**

### Защита от прямого доступа к БД

- Credential **не может быть прочитан** напрямую из базы данных (зашифрован)
- Доступ возможен только через API с проверкой прав
- Каждый запрос к credential проверяет: авторизация → уровень доступа → аудит-запись

## Резервное копирование (Backup)

### Критическая важность

Потеря паролей и учётных данных может парализовать работу компании. Резервное копирование — **обязательная** функция.

### Ручной экспорт

| Параметр | Описание |
|----------|---------|
| **Кто может** | Admin / Owner (CEO) |
| **Аутентификация** | Требуется повторный ввод пароля администратора |
| **Шифрование экспорта** | Пароль-защита архива (задаётся при экспорте) |
| **Формат** | Зашифрованный ZIP с JSON-файлами внутри |
| **Содержимое** | Все credentials с полями, организованные по проектам и категориям |
| **Рекомендация** | Ежемесячный экспорт, хранение на локальном жёстком диске |

### Автоматический бэкап

| Параметр | Описание |
|----------|---------|
| **Частота** | Еженедельно (настраиваемо) |
| **Хранение** | Зашифрованный архив отправляется на защищённое хранилище администратора |
| **Ротация** | Хранение последних 12 бэкапов (3 месяца) |
| **Уведомление** | Администратор получает уведомление о каждом бэкапе |

### Процедура экспорта

```
Admin нажимает «Экспорт всех credentials»
       │
       ▼
Система запрашивает пароль администратора
       │
       ▼
Admin вводит пароль → верификация
       │
       ▼
Система запрашивает пароль для архива
       │
       ▼
Admin задаёт пароль шифрования архива
       │
       ▼
Генерация зашифрованного ZIP/JSON
       │
       ▼
Скачивание файла + запись в аудит-лог
```

## Представления (Views)

### 1. Глобальное представление (Global View)

- Все credentials, к которым у пользователя есть доступ
- Фильтры: по категории, проекту, провайдеру, уровню доступа
- Сортировка: по имени, дате создания, дате изменения
- Поиск по названию и провайдеру

### 2. Проектное представление (Project View)

- Доступно из карточки проекта → вкладка **«Credentials»**
- Показывает только credentials, привязанные к данному проекту
- PM и Dev-команда используют при работе над проектом

### 3. Личное представление (My Passwords)

- Раздел «Мои пароли» — персональные credentials сотрудника
- Не видны никому, кроме владельца
- Каждый сотрудник управляет своими паролями самостоятельно

### 4. Административное представление (Admin View)

- Доступно только CEO
- Полный доступ ко всем credentials компании
- Управление уровнями доступа
- Просмотр аудит-лога
- Экспорт / бэкап

## Аудит-лог

### Отслеживаемые события

| Событие | Что фиксируется |
|---------|-----------------|
| **Просмотр** | Кто нажал «Показать» для зашифрованного поля, какой credential, когда |
| **Копирование** | Кто нажал «Копировать» для Login/Password/API Key, когда |
| **Создание** | Кто создал credential, все значения полей, когда |
| **Изменение** | Кто изменил, какие поля изменены (old → new для нешифрованных), когда |
| **Удаление** | Кто удалил credential, когда (credential перемещается в корзину, не удаляется окончательно) |
| **Экспорт** | Кто выполнил экспорт/бэкап, когда, сколько credentials экспортировано |
| **Изменение доступа** | Кто изменил уровень доступа или список Allowed Employees, когда |

### Хранение логов

- Логи хранятся **бессрочно** (не удаляются)
- Доступны для просмотра: CEO, Admin
- Фильтрация: по сотруднику, credential, типу события, периоду
- Экспорт аудит-лога в CSV/PDF

## Связь с другими модулями

### Credentials → Projects Hub
- Credentials привязаны к проектам
- В карточке проекта → вкладка «Credentials» показывает все пароли проекта
- При создании проекта — список credentials пуст, заполняется по мере работы

### Credentials → Finance (Domains, Hosting, Licenses)
- Аккаунты доменных регистраторов и хостинг-провайдеров хранятся в Credentials
- Из карточки домена/хостинга можно перейти к связанному credential
- При создании записи Domain/Hosting — предложение создать credential

### Credentials → My Company (Team)
- Каждый сотрудник имеет раздел «Мои пароли»
- При увольнении сотрудника: admin получает уведомление о credentials, где сотрудник был в Allowed Employees

### Credentials → Drive
- Credentials **не хранятся** в Drive (другой механизм шифрования)
- env-файлы лучше хранить в Credentials (зашифрованы), а не в Drive

## Рекомендации по использованию

| Рекомендация | Описание |
|-------------|---------|
| **Один credential — одна цель** | Не объединять несколько аккаунтов в одну запись |
| **Привязка к проекту** | Все проектные credentials должны быть привязаны к проекту |
| **Регулярная смена паролей** | Критичные пароли (Admin, Hosting) — раз в квартал |
| **Минимальный доступ** | Использовать Secret для всего, что не нужно видеть всем |
| **Ежемесячный бэкап** | CEO выполняет экспорт и сохраняет на защищённый носитель |
| **env data вместо файлов** | Хранить .env содержимое в поле env data, а не как файл в Drive |
