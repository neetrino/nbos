# Система уведомлений (Notifications)

> NBOS Platform — кросс-модульная система уведомлений, гарантирующая, что ни одно важное событие не останется без внимания

## Назначение

Модуль **Notifications** — это **кросс-модульная** система, которая связывает все модули NBOS в единый информационный поток. Каждое значимое событие в платформе (создание счёта, просрочка задачи, новый лид, нарушение SLA) автоматически генерирует уведомление нужным людям через нужный канал.

**Ключевой принцип:** уведомления — это не просто «звонок колокольчика». Это механизм **эскалации**, **напоминания** и **побуждения к действию**. Каждое уведомление содержит контекст: что произошло, кто должен действовать и ссылку на объект в системе.

## Каналы доставки

NBOS поддерживает четыре канала уведомлений, каждый — для своей аудитории и контекста:

| Канал | Аудитория | Назначение | Приоритет |
|-------|-----------|------------|-----------|
| **In-App** | Сотрудники Neetrino | Основной канал для всех внутренних уведомлений | Обязательный |
| **Telegram** | Сотрудники Neetrino | Оперативные уведомления за пределами платформы | Высокий |
| **WhatsApp** | Клиенты | Счета, напоминания об оплате, статусы проектов | Высокий |
| **Email** | Клиенты / Партнёры | Формальные коммуникации (акты, отчёты) | Опциональный |

### In-App (внутри платформы)

- **Иконка колокольчика** в шапке NBOS с бейджем непрочитанных уведомлений
- **Notification Center** — выдвижная панель со списком всех уведомлений
- Фильтрация по категориям: Finance, Projects, Tasks, CRM, Support, System
- Статусы: непрочитано → прочитано → архив
- Клик по уведомлению → переход к связанному объекту (счёт, задача, сделка)

### Telegram (внутренняя команда)

- Каждый сотрудник привязывает свой Telegram-аккаунт при регистрации в NBOS
- Уведомления приходят от бота Neetrino
- Содержат краткую информацию + deep link в NBOS
- Используется для: назначения задач, дедлайны, эскалации, SLA-нарушения

### WhatsApp (клиенты)

- Интеграция через WhatsApp Business API
- Шаблонные сообщения (pre-approved templates)
- Используется для: отправки счетов, напоминаний об оплате, уведомлений о доменах, статусов по тикетам
- Ограничение: не более 3 напоминаний об одном и том же событии (anti-spam)

### Email (формальный канал)

- Опциональный канал для формальных коммуникаций
- Используется для: отчётов, актов, формальных уведомлений партнёрам
- Не дублирует In-App и Telegram без необходимости

## Категории уведомлений

### A) Финансовые уведомления (Finance)

| Событие | Получатели | Каналы | Эскалация |
|---------|-----------|--------|-----------|
| Счёт создан | Finance Director | In-App | — |
| Оплата получена | Seller + Finance Director | In-App + Telegram | — |
| Счёт просрочен (Overdue) | Finance Director + CEO | In-App + Telegram | +CEO при просрочке > 3 дней |
| Подписка — оплата не получена | Finance Director | In-App + Telegram | +CEO при просрочке > 7 дней |
| Домен/хостинг истекает через 2 месяца | Ops + Finance Director | In-App | Начало процесса продления |
| Домен/хостинг истекает через 1 месяц | Ops + Finance Director + CEO | In-App + Telegram | Эскалация: подключается CEO |
| Плановый расход наступил (due today) | Finance Director | In-App | — |
| Плановый расход просрочен | Finance Director + CEO | In-App + Telegram | +CEO автоматически |
| Бонус начислен | Сотрудник | In-App + Telegram | Мотивационное уведомление |
| Статус бонуса изменён (Active → Paid) | Сотрудник | In-App + Telegram | — |

**Логика эскалации финансовых уведомлений:**

```
Счёт Overdue:
  День 0 → Finance Director (In-App)
  День 1 → Finance Director (In-App + Telegram) + Клиент (WhatsApp — напоминание)
  День 3 → Finance Director + CEO (In-App + Telegram) + Клиент (WhatsApp — повторное)
  День 7 → CEO (In-App + Telegram) + Клиент (WhatsApp — финальное предупреждение)
```

### B) Проектные уведомления (Projects)

| Событие | Получатели | Каналы | Примечание |
|---------|-----------|--------|------------|
| Новый проект создан | PM + Команда | In-App + Telegram | Старт работ |
| Дедлайн проекта через 7 дней | PM + CEO | In-App | Предупреждение |
| Дедлайн проекта через 1 день | PM + CEO + Команда | In-App + Telegram | Эскалация на всю команду |
| Дедлайн проекта пропущен | PM + CEO | In-App + Telegram | Критическое |
| Стадия продукта изменена | PM | In-App | Информационное |
| Все задачи в продукте выполнены | PM | In-App + Telegram | Триггер для запуска closing checklist |
| Stage Gate Validation не пройден | PM | In-App + Telegram | Содержит список невыполненных условий |

**Механизм Stage Gate Notification:**

При попытке перевести продукт/проект на следующую стадию система проверяет обязательные условия (Stage Gate). Если условия не выполнены, PM получает уведомление с детализацией:

```
⚠️ Stage Gate Validation Failed
Проект: ClientName — Website
Переход: Development → QA
Невыполненные условия:
  ✗ 3 задачи в статусе "In Progress" (нужно: 0)
  ✗ Нет записи QA Engineer в команде проекта
  ✓ Все критические баги закрыты
```

### C) Уведомления по задачам (Tasks)

| Событие | Получатели | Каналы | Примечание |
|---------|-----------|--------|------------|
| Задача назначена на вас | Исполнитель | In-App + Telegram | Призыв к действию |
| Дедлайн задачи приближается (1 день) | Исполнитель | In-App + Telegram | Предупреждение |
| Задача просрочена | Исполнитель + PM | In-App + Telegram | PM подключается |
| Статус задачи изменён | Автор + Наблюдатели | In-App | Информационное |
| Комментарий в чате задачи | Участники задачи | In-App | Только для подписанных |
| Задача создана из чата | Автор + Исполнитель | In-App + Telegram | С контекстом из чата |
| Задача переоткрыта (Reopened) | Исполнитель + PM | In-App + Telegram | Требуется повторная работа |

### D) CRM-уведомления (Sales)

| Событие | Получатели | Каналы | Примечание |
|---------|-----------|--------|------------|
| Новый лид получен | Назначенный Seller | In-App + Telegram | Быстрая реакция важна |
| Стадия сделки изменена | Seller | In-App | Информационное |
| Сделка выиграна (Deal Won) | Seller + PM + CEO | In-App + Telegram | Празднование + начало действий |
| Сделка проиграна (Deal Failed) | Seller + Head of Sales | In-App + Telegram | Анализ причин |
| Лид не обработан > 24 часов | Seller + Head of Sales | In-App + Telegram | Эскалация за бездействие |
| Follow-up deadline пропущен | Seller | In-App + Telegram | Напоминание о действии |

**Уведомление Deal Won — особый случай:**

Это одно из немногих «праздничных» уведомлений. Содержит:
- Название клиента и сумму сделки
- Тип сделки (New / Extension / Upsell)
- Имя Seller, который закрыл сделку
- Призыв к действию для PM: начать процесс Handoff

### E) Уведомления Support (Поддержка)

| Событие | Получатели | Каналы | Примечание |
|---------|-----------|--------|------------|
| Новый тикет создан | PM / Support owner | In-App + Telegram | Привязка к проекту |
| Дедлайн SLA приближается | Назначенный исполнитель | In-App + Telegram | До истечения < 2 часов |
| SLA нарушен (Breach) | PM + CEO | In-App + Telegram | Критическое, всегда доставляется |
| Тикет переоткрыт | Исполнитель + PM | In-App + Telegram | Требует повторного внимания |
| Тикет разрешён | PM | In-App | Информационное |
| Комментарий клиента в тикете | Исполнитель | In-App + Telegram | Ответ клиента |

**SLA Breach — уведомление с наивысшим приоритетом:**

```
🚨 SLA BREACH
Тикет: #1234 — "Сайт не открывается"
Проект: ClientName — E-commerce
Приоритет: P1 (Critical)
SLA: 2 часа | Прошло: 2 ч 15 мин
Исполнитель: Иван Петров
→ Требуется немедленная реакция
```

Это уведомление **игнорирует** режим «Не беспокоить» и всегда доставляется во все каналы.

### F) Системные уведомления (System)

| Событие | Получатели | Каналы | Примечание |
|---------|-----------|--------|------------|
| Учётные данные просмотрены | Владелец записи | In-App | Аудит доступа к секретам |
| Учётные данные изменены | Владелец + Admin | In-App + Telegram | Контроль безопасности |
| Права доступа изменены | Пользователь + Admin | In-App | Информирование об изменениях |
| Сотрудник добавлен в проект | Сотрудник + PM | In-App + Telegram | Новое назначение |
| Сотрудник удалён из проекта | Сотрудник + PM | In-App | Информационное |
| Новый сотрудник добавлен в систему | Admin + CEO | In-App | Информационное |

## Клиентские уведомления (WhatsApp)

Отдельная категория — уведомления для клиентов Neetrino, доставляемые через WhatsApp:

| Событие | Шаблон сообщения | Ограничения |
|---------|-----------------|-------------|
| Счёт создан | "Здравствуйте, {name}! Выставлен счёт №{invoice_id} на сумму {amount}. Ссылка для оплаты: {link}" | 1 сообщение |
| Счёт — до оплаты 1 день | "Напоминаем: срок оплаты счёта №{invoice_id} — завтра, {date}." | 1 сообщение |
| Счёт просрочен (1-е напоминание) | "Счёт №{invoice_id} просрочен. Просьба оплатить в ближайшее время." | День 1 после due date |
| Счёт просрочен (2-е напоминание) | "Повторное напоминание: счёт №{invoice_id} ожидает оплаты." | День 3 после due date |
| Счёт просрочен (финальное) | "Финальное напоминание по счёту №{invoice_id}. При отсутствии оплаты работы будут приостановлены." | День 7, максимум 3 сообщения |
| Домен требует продления | "{name}, срок действия домена {domain} истекает {date}. Для продления необходимо оплатить счёт." | За 2 месяца до истечения |
| Milestone проекта достигнут | "{name}, этап '{milestone}' вашего проекта завершён. Подробности: {link}" | Опционально, по настройке PM |
| Тикет поддержки решён | "Ваш запрос #{ticket_id} решён. Если вопрос остался, ответьте на это сообщение." | После закрытия тикета |

**Правила клиентских уведомлений:**
- Максимум **3 напоминания** по одному событию (anti-spam)
- Интервал между напоминаниями: 1, 3 и 7 дней
- Клиент может ответить → ответ попадает в Messenger модуль NBOS
- Время отправки: только в рабочие часы (9:00–19:00 по часовому поясу клиента)
- Язык сообщения: определяется настройками клиента (русский / английский)

## Настройки уведомлений

### Персональные настройки (для каждого сотрудника)

Каждый сотрудник может настроить свои предпочтения в разделе **Settings → Notifications**:

| Настройка | Описание | По умолчанию |
|-----------|----------|-------------|
| **Каналы по категориям** | Выбор каналов для каждой категории уведомлений | In-App: всё, Telegram: критичное |
| **Do Not Disturb** | Отключение некритичных уведомлений вне рабочих часов | Выкл (9:00–18:00) |
| **Звук уведомлений** | Включить/выключить звуковые оповещения In-App | Вкл |
| **Email-дайджест** | Ежедневная или еженедельная сводка по email | Выкл |
| **Подписка на проекты** | Получать все уведомления по выбранным проектам | Только назначенные |

### Приоритеты уведомлений

Система использует три уровня приоритета:

| Приоритет | Описание | Поведение |
|-----------|----------|-----------|
| **Critical** | SLA breach, P1 инцидент, критическая просрочка оплаты | Всегда доставляется, игнорирует DND |
| **High** | Дедлайн задачи, новый лид, оплата получена | Доставляется по настроенным каналам |
| **Normal** | Информационные: смена статуса, комментарии | Подчиняется DND и настройкам |

### Priority Override (приоритетный перехват)

Некоторые уведомления **невозможно отключить**, так как они критичны для бизнеса:

- SLA Breach (нарушение SLA поддержки)
- P1 инциденты
- Просрочка оплаты > 7 дней (для Finance Director)
- Проект — дедлайн пропущен (для PM и CEO)
- Учётные данные скомпрометированы (для Admin)

## Механизм работы: Event → Notification

Вся система уведомлений построена на **событийной архитектуре** (event-driven):

### Цепочка: Событие → Правило → Уведомление

```
[Системное событие]
        ↓
[Notification Engine: оценка правил]
        ↓
[Определение получателей + каналов]
        ↓
[Рендеринг шаблона сообщения]
        ↓
[Отправка по каналам: In-App / Telegram / WhatsApp / Email]
        ↓
[Логирование: кому, когда, что отправлено]
```

### Структура правила уведомления (Notification Rule)

| Поле | Описание | Пример |
|------|----------|--------|
| **Event** | Системное событие-триггер | `Invoice.status = Overdue` |
| **Condition** | Дополнительное условие (опционально) | `Invoice.amount > 1000` |
| **Recipients** | Роли или конкретные пользователи | Finance Director + CEO |
| **Channels** | Каналы доставки | In-App + Telegram |
| **Template** | Шаблон сообщения | "Счёт #{id} для {client} просрочен на {days} дн." |
| **Priority** | Приоритет уведомления | High |
| **Escalation** | Правило эскалации (опционально) | Через 3 дня → добавить CEO |

### Примеры правил

**Правило 1: Счёт просрочен**
```
Event:      Invoice.status changed to "Overdue"
Condition:  —
Recipients: Finance Director (role) + CEO (role)
Channels:   In-App + Telegram
Template:   "⚠️ Счёт #{invoice.id} для {client.name} просрочен. 
             Сумма: {invoice.amount}. Просрочка: {overdue_days} дн."
Priority:   High
Escalation: Day 3 → CEO (Telegram), Day 7 → Client (WhatsApp)
```

**Правило 2: Задача назначена**
```
Event:      Task.assignee changed
Condition:  Task.assignee != null
Recipients: Task.assignee (dynamic)
Channels:   In-App + Telegram
Template:   "📋 Вам назначена задача: {task.title}
             Проект: {project.name}
             Дедлайн: {task.due_date}
             Автор: {task.creator.name}"
Priority:   High
```

**Правило 3: SLA Breach**
```
Event:      Ticket.sla_status changed to "Breached"
Condition:  —
Recipients: Ticket.assignee + PM (project role) + CEO
Channels:   In-App + Telegram (всем)
Template:   "🚨 SLA BREACH: Тикет #{ticket.id} — {ticket.title}
             Проект: {project.name}
             SLA: {sla_limit} | Прошло: {elapsed_time}
             Исполнитель: {assignee.name}"
Priority:   Critical (игнорирует DND)
```

## Эскалация уведомлений

Система поддерживает автоматическую эскалацию — если на уведомление не отреагировали, круг получателей расширяется:

| Уровень | Время | Действие |
|---------|-------|----------|
| **L1** | Сразу | Уведомление прямому ответственному |
| **L2** | +1–3 дня | Подключение руководителя (PM, Head of Sales, Finance Director) |
| **L3** | +5–7 дней | Подключение CEO |

### Примеры эскалации

**Просроченный счёт:**
- L1 (Day 0): Finance Director — In-App
- L1 (Day 1): Finance Director — Telegram + Client — WhatsApp
- L2 (Day 3): Finance Director + CEO — Telegram + Client — WhatsApp (2-е напоминание)
- L3 (Day 7): CEO — Telegram + Client — WhatsApp (финальное)

**Тикет без реакции:**
- L1 (0 ч): Исполнитель — In-App + Telegram
- L2 (SLA - 1 ч): Исполнитель + PM — In-App + Telegram
- L3 (SLA breach): PM + CEO — In-App + Telegram

**Задача просрочена:**
- L1 (Day 0): Исполнитель — In-App + Telegram
- L2 (Day 2): Исполнитель + PM — In-App + Telegram
- L3 (Day 5): PM + CEO — In-App + Telegram

## Аудит и аналитика уведомлений

### Лог уведомлений

Все отправленные уведомления сохраняются в логе:

| Поле | Описание |
|------|----------|
| Дата/время | Когда отправлено |
| Событие | Какое событие вызвало уведомление |
| Получатель | Кому отправлено |
| Канал | Через какой канал (In-App / Telegram / WhatsApp / Email) |
| Статус доставки | Доставлено / Прочитано / Ошибка |
| Связанный объект | Ссылка на счёт, задачу, тикет и т.д. |

### Метрики для аналитики

- Количество уведомлений по категориям (тренд)
- Процент прочтения (In-App) по категориям
- Среднее время реакции на уведомление (от отправки до действия)
- Количество эскалаций (сигнал проблем в процессах)
- Ошибки доставки по каналам

## Принципы проектирования

1. **Event-driven** — все уведомления генерируются автоматически в ответ на системные события
2. **Контекстность** — каждое уведомление содержит достаточно информации для принятия решения
3. **Эскалация** — система автоматически расширяет круг получателей при отсутствии реакции
4. **Настраиваемость** — сотрудники контролируют, что и как они получают (в рамках разрешённого)
5. **Anti-spam** — ограничение частоты повторных уведомлений, группировка однотипных
6. **Аудитируемость** — полный лог: кому, когда, что отправлено и прочитано ли
7. **Надёжность** — критические уведомления доставляются всегда, независимо от настроек DND

## Роли и доступы

| Роль | Настройка уведомлений | Notification Center | Управление правилами |
|------|----------------------|--------------------|--------------------|
| **CEO** | Свои настройки | Все уведомления | Полный доступ |
| **Admin** | Свои настройки | Все уведомления | Полный доступ к правилам |
| **Finance Director** | Свои настройки | Финансовые + системные | Нет |
| **PM** | Свои настройки | Проектные + задачи + поддержка | Нет |
| **Seller** | Свои настройки | CRM + финансовые (свои) | Нет |
| **Developer** | Свои настройки | Задачи + личные | Нет |
