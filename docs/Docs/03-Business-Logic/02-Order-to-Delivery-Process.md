# Процесс Order-to-Delivery (от заказа до сдачи)

## Общее описание

Order-to-Delivery охватывает весь жизненный цикл проекта от момента создания заказа (Deal Won + первый платёж) до полной сдачи клиенту и перехода на техобслуживание. Этот процесс включает создание проекта, продуктов внутри проекта, управление разработкой через Scrum/Kanban и контроль качества.

---

## Создание заказа и проекта

### Триггер создания Order

```
Deal.status = "Deal Won"
  AND Invoice.status = "Paid" (первый платёж подтверждён Финансовым директором)
  → Order создаётся автоматически
  → Project создаётся в Projects Hub
```

### Что создаётся при новом заказе

| Сущность | Описание | Источник данных |
|----------|----------|-----------------|
| **Order** | Финансовая карточка заказа | Из Deal: сумма, тип оплаты, реквизиты |
| **Project** | Проект в Projects Hub | Из Deal: название, контакт, компания, дедлайн |
| **Product** | Продукт внутри проекта | Из Deal: тип продукта (Website, App, CRM и т.д.) |

### Что создаётся при Extension (доработке)

- Order — новый (отдельная финансовая единица)
- Project — **не создаётся** (используется существующий)
- Product/Extension — создаётся **внутри существующего проекта** как отдельный блок работ

---

## Иерархия сущностей в разработке

```
Project (один бренд / один бизнес)
  ├── Product 1 (Website)
  │     ├── Sprint 1
  │     │     ├── Task 1.1
  │     │     ├── Task 1.2
  │     │     └── ...
  │     ├── Sprint 2
  │     └── ...
  ├── Product 2 (Mobile App)
  │     └── ...
  ├── Extension 1 (новая функция для Website)
  │     └── Tasks...
  ├── Credentials (логины, пароли, доступы)
  ├── Domains (доменные имена)
  ├── Assets (файлы, дизайны)
  └── Chats (привязанные чаты)
```

## Стадии продукта (Product Stages)

Каждый Product внутри проекта проходит следующие стадии. Переход между стадиями контролируется **Stage Gates** — набором обязательных полей и чек-листов.

### Обзор стадий

| # | Стадия | Описание | Кто отвечает |
|---|--------|----------|--------------|
| 1 | **New** | Продукт только создан, данные из Deal перенесены | PM |
| 2 | **Creating** | Подготовка инфраструктуры: аккаунты, домены, среды | PM + DevOps |
| 3 | **Development** | Активная разработка: спринты, задачи, код | Dev Team |
| 4 | **QA / Checking** | Проверка качества, тестирование, аудит | QA + PM |
| 5 | **Transfer** | Передача клиенту, приёмка | PM + Client |
| 6 | **Done** | Проект завершён, клиент принял | — |
| 6a | **Lost** | Проект потерян / отменён | CEO решает |
| 6b | **On Hold** | Проект приостановлен | CEO / PM |

---

### Стадия 1: New

**Цель:** Зафиксировать все начальные данные о продукте.

**Обязательные поля для перехода на Creating:**

| Поле | Описание |
|------|----------|
| Тип продукта | Website / Mobile App / CRM / Other |
| Тип проекта | White Label / Mix / Custom Code |
| Модель оплаты | Classic (50/50 или 30/30/40) / Subscription |
| Дедлайн | Дата сдачи проекта клиенту |
| PM назначен | Проектный менеджер привязан к продукту |
| Бриф | Документ с требованиями от клиента загружен |
| Kickoff Checklist | Все пункты handoff от продажника закрыты |

---

### Стадия 2: Creating (Подготовка инфраструктуры)

**Цель:** Создать все необходимые аккаунты, среды и доступы до начала разработки.

**Обязательный чек-лист (зависит от типа продукта):**

| # | Пункт | Для Website | Для App | Для CRM |
|---|-------|:-----------:|:-------:|:-------:|
| 1 | Домен куплен/привязан | ✅ | — | — |
| 2 | Хостинг/сервер создан | ✅ | ✅ | ✅ |
| 3 | Аккаунт Cloudflare / CDN | ✅ | — | — |
| 4 | База данных (Neon/Supabase) | ✅ | ✅ | ✅ |
| 5 | Redis/Upstash | По потребности | По потребности | По потребности |
| 6 | Email-сервис (Resend) | ✅ | ✅ | ✅ |
| 7 | App Store / Google Play аккаунт | — | ✅ | — |
| 8 | Git-репозиторий создан | ✅ | ✅ | ✅ |
| 9 | CI/CD pipeline настроен | ✅ | ✅ | ✅ |
| 10 | Файл .env заполнен и сохранён | ✅ | ✅ | ✅ |
| 11 | Все пароли сохранены в Credentials | ✅ | ✅ | ✅ |
| 12 | Дизайн-макеты готовы/получены | ✅ | ✅ | ✅ |

**Stage Gate:** Пока все обязательные пункты (в зависимости от типа продукта) не выполнены, переход на Development заблокирован.

**Автоматизация:** При создании продукта определённого типа автоматически создаются задачи из шаблона (Template Tasks).

---

### Стадия 3: Development (Разработка)

**Цель:** Реализация продукта через спринты с задачами.

**Управление разработкой:**

- Внутри каждого Product — **Scrum-доска** (или Kanban, переключаемый режим)
- Спринты создаются PM, длительность обычно 1–2 недели
- Задачи создаются из шаблонов и вручную

**Стандартные шаблоны задач (зависят от типа проекта):**

| Тип проекта | Шаблон задач |
|-------------|-------------|
| **White Label** | Настройка темы, брендирование, контент, SEO-базовый, тестирование |
| **Mix** | Настройка основы, кастомные компоненты, интеграции, контент, тестирование |
| **Custom Code** | Архитектура, фронтенд, бэкенд, интеграции, тестирование, документация |

**Задачи имеют:**

| Атрибут | Описание |
|---------|----------|
| Постановщик | Кто создал задачу |
| Исполнитель | Кто выполняет |
| Соисполнитель | Помогает в выполнении |
| Наблюдатель | Видит задачу, получает уведомления |
| Приоритет | Critical / High / Medium / Low |
| Дедлайн | Срок выполнения |
| Спринт | К какому спринту привязана |
| Чат задачи | Обсуждение внутри задачи |

**Stage Gate для перехода на QA:**

- Все задачи текущего спринта закрыты
- Код задеплоен на staging-среду
- Базовое self-тестирование проведено разработчиком

---

### Стадия 4: QA / Checking (Проверка качества)

**Цель:** Убедиться, что продукт соответствует стандартам качества и безопасности.

**Чек-лист качества (Quality Checklist):**

| # | Категория | Пункты проверки |
|---|-----------|----------------|
| 1 | **Функциональность** | Все заявленные функции работают; формы отправляются; навигация корректна |
| 2 | **Адаптивность** | Корректное отображение на Desktop, Tablet, Mobile |
| 3 | **Производительность** | PageSpeed ≥ 80; время загрузки < 3 сек |
| 4 | **SEO** | Meta-теги, sitemap, robots.txt, alt-тексты |
| 5 | **Безопасность** | HTTPS, XSS-защита, CSRF-токены, Content Security Policy |
| 6 | **Кросс-браузерность** | Chrome, Safari, Firefox, Edge |
| 7 | **Контент** | Все тексты на месте, изображения оптимизированы, ссылки рабочие |

**Чек-лист безопасности (Security Checklist):**

| # | Пункт |
|---|-------|
| 1 | SSL-сертификат установлен и работает |
| 2 | Все API-ключи в переменных окружения (не в коде) |
| 3 | Rate limiting настроен |
| 4 | Логирование ошибок подключено |
| 5 | Бэкапы базы данных настроены |
| 6 | Пароли хранятся в хэшированном виде |

**Чек-лист деплоя (Deployment Checklist):**

| # | Пункт |
|---|-------|
| 1 | Production-среда настроена |
| 2 | DNS-записи корректны |
| 3 | .env production заполнен и сохранён в Credentials |
| 4 | CI/CD деплой на production протестирован |
| 5 | Мониторинг (uptime) подключён |
| 6 | Файл .env и все секреты сохранены в проекте |

**Stage Gate для перехода на Transfer:**
- Все чек-листы пройдены на 100%
- Критических багов нет
- Продукт задеплоен на production

---

### Стадия 5: Transfer (Передача клиенту)

**Цель:** Клиент принимает готовый продукт.

**Процесс:**

1. PM уведомляет клиента о готовности проекта
2. Клиенту предоставляется доступ к продукту
3. Клиент тестирует и проверяет
4. Если есть замечания → фиксируются как задачи, исправляются
5. Клиент подтверждает приёмку (формально или в чате)

**Stage Gate для перехода на Done:**

- Клиент подтвердил приёмку продукта
- Все задачи на доске Development закрыты
- Финансовая сторона закрыта (зависит от модели оплаты, см. ниже)

---

### Стадия 6: Done (Завершён)

**Условия закрытия проекта как Done:**

- Клиент подтвердил приёмку
- Все задачи закрыты
- Финансовая сторона закрыта (все оплаты получены для Classic; достаточный объём платежей для Subscription)
- Все credentials, домены и доступы задокументированы
- **Триггер:** начисление бонуса разработчику / PM / дизайнеру

## Влияние модели оплаты на Delivery

### Classic Model (50/50 или 30/30/40)

```
Timeline:
  ──────────────────────────────────────────────────────
  Предоплата     │  Разработка  │  Финальная    │ Done
  50% (или 30%)  │              │  оплата       │
                 │              │  50% (или 40%)│
  ──────────────────────────────────────────────────────

  Бонус Dev = после 100% оплаты + продукт сдан клиенту
```

**Правила:**
- Разработка начинается **после** получения предоплаты
- Проект сдаётся клиенту **после** получения полной оплаты
- Бонус начисляется только при выполнении **обоих** условий: полная оплата + приёмка клиентом

### Subscription Model (ежемесячные платежи)

```
Timeline:
  ──────────────────────────────────────────────────────────────────
  Мес.1     │ Мес.2    │ Мес.3     │ Мес.4 (Maintenance) │ Мес.5...
  100k dev  │ 100k dev │ 100k dev  │ 100k dev + 80k mnt  │ 180k
  ──────────────────────────────────────────────────────────────────
  1-я оплата  Разработка  Сдача         ↑
              (Development)           Бонус Dev начисляется здесь

  Бонус Dev = после сдачи проекта (≈ месяц 4, переход на Maintenance)
```

**Правила:**
- Клиент платит ежемесячно в течение всего цикла разработки
- Если разработка затянулась — клиент **не платит** за дополнительные месяцы до сдачи
- После сдачи: к сумме Development добавляется сумма Maintenance
- Бонус Dev начисляется на 4-м месяце (когда компания уже получила достаточно денег)

## Стандартные шаблоны по типу проекта

### White Label (WL)

| Этап | Типовые задачи |
|------|---------------|
| Creating | Покупка домена, настройка хостинга, установка шаблона |
| Development | Брендирование, наполнение контентом, базовая SEO-оптимизация |
| QA | Проверка по чек-листу WL (сокращённый) |
| Transfer | Демонстрация клиенту, правки по замечаниям |

### Mix (шаблон + кастом)

| Этап | Типовые задачи |
|------|---------------|
| Creating | Домен, хостинг, шаблон, настройка среды разработки |
| Development | Кастомные компоненты, интеграции, контент, адаптация дизайна |
| QA | Полный чек-лист качества |
| Transfer | Демонстрация, тестирование клиентом |

### Custom Code (с нуля)

| Этап | Типовые задачи |
|------|---------------|
| Creating | Полная инфраструктура: сервер, БД, CDN, CI/CD, среды |
| Development | Архитектура, фронтенд, бэкенд, API, интеграции, тестирование |
| QA | Расширенный чек-лист + security audit + нагрузочное тестирование |
| Transfer | Staging-демо → production → приёмка |

## Параллельные состояния проекта

Проект может находиться одновременно в нескольких состояниях:

| Комбинация | Описание |
|-----------|----------|
| Development only | Новый проект в разработке |
| Maintenance only | Проект сдан, на техобслуживании |
| Development + Maintenance | Проект на Maintenance, но идёт доработка (Extension) |

В Projects Hub проект отображается во всех соответствующих табах (фильтрах):
- **All** — все проекты
- **Development** — проекты с активной разработкой (основной продукт или Extension)
- **Maintenance** — проекты на техобслуживании
- **Done/Closed** — завершённые проекты без активных подписок

---

## Правила и ограничения

1. Переход между стадиями Product блокируется, если Stage Gate не пройден
2. Обязательные поля проверяются системой автоматически
3. При незаполненных полях автоматически создаётся задача ответственному специалисту
4. Все пароли и доступы должны быть сохранены в Credentials до закрытия проекта
5. Файл .env и секреты должны быть продублированы в системе
6. Проект не может быть закрыт как Done, если есть открытые задачи
7. Бонус не начисляется до выполнения финансовых условий
