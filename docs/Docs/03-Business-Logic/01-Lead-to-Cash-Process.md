# Процесс Lead-to-Cash (от лида до денег)

## Общее описание

Lead-to-Cash — это полный цикл от первого контакта с потенциальным клиентом до получения оплаты и запуска проекта. В NBOS этот процесс разделён на две отдельные сущности: **Лиды (Leads)** и **Сделки (Deals)**, каждая со своим пайплайном и правилами перехода между стадиями.

Ключевой принцип: лиды и сделки — это **раздельные сущности**. Ежемесячно в систему поступают сотни лидов, но только квалифицированные попадают в сделки. Это предотвращает засорение CRM-пайплайна и позволяет продажникам фокусироваться на реальных возможностях.

## Пайплайн лидов (Lead Pipeline)

### Стадии лидов

| Стадия | Описание | Действия |
|--------|----------|----------|
| **New** | Новый лид поступил в систему | Фиксация источника, контактных данных, первичная информация |
| **Didn't Get Through** | Не удалось дозвониться / связаться | Повторные попытки связи (до 3-х), фиксация попыток |
| **Contact Established** | Контакт установлен | Первичный разговор состоялся, выявлен базовый интерес |
| **Qualification (MQL)** | Маркетинговая квалификация | Оценка по критериям: бюджет, потребность, сроки, ЛПР |
| **SPAM** | Спам / нерелевантный | Закрытая стадия — лид не является целевым |
| **Frozen** | Заморожен | Клиент заинтересован, но не сейчас. Возврат через N месяцев |
| **Quality Lead (SQL)** | Квалифицированный лид | Лид готов к работе → **создаётся Сделка в Deal Pipeline** |

### Правила квалификации (MQL → SQL)

Лид получает статус SQL только при выполнении всех условий:

- Подтверждённая потребность в услуге, которую Neetrino предоставляет
- Наличие бюджета (хотя бы примерного понимания)
- Определён ЛПР (лицо, принимающее решение)
- Клиент выразил готовность обсуждать сотрудничество

### Источники лидов

Источник лида фиксируется при создании и влияет на бонус продажника:

| Источник | Описание | Влияние на бонус |
|----------|----------|------------------|
| Cold Call | Холодный звонок продажника | Максимальный % бонуса |
| Marketing | Входящий лид из маркетинга | Стандартный % бонуса |
| Existing Client | Существующий клиент (допродажа) | Сниженный % бонуса |
| Partner Referral | Лид от партнёра | Сниженный % (часть идёт партнёру) |
| Inbound (сайт/соцсети) | Органический входящий запрос | Стандартный % бонуса |

## Пайплайн сделок (Deal Pipeline)

### Стадии сделок

| # | Стадия | Описание | Stage Gate (обязательные поля) |
|---|--------|----------|-------------------------------|
| 1 | **Start a Conversation** | Первичный контакт по сделке | Контакт привязан, источник лида указан, тип услуги выбран |
| 2 | **Discuss What Is Needed** | Обсуждение потребностей | Описание потребности заполнено, предварительный тип продукта |
| 3 | **Meeting** | Встреча (онлайн/офлайн) | Дата встречи указана, участники отмечены, протокол встречи |
| 4 | **Can We Do It?** | Оценка реализуемости | Тип проекта (WL/Mix/Custom), предварительные сроки, техническая оценка |
| 5 | **Send Offer** | Отправка коммерческого предложения | Сумма сделки, тип оплаты (Classic/Subscription), файл КП прикреплён |
| 6 | **Get Answer** | Ожидание ответа клиента | Дата отправки КП, дедлайн для ответа |
| 7 | **Deposit & Contract** | Предоплата и договор | Сумма предоплаты, реквизиты компании/физлица, договор подписан |
| 8 | **Creating** | Проект в работе (ожидание оплаты) | Счёт на предоплату создан |
| 9 | **Get Final Pay** | Получение финальной оплаты | Все счета выставлены и оплачены |
| 10 | **Maintenance Offer** | Предложение техобслуживания | Условия Maintenance согласованы, сумма подписки |
| — | **Failed** | Сделка провалена | Причина отказа обязательна |
| — | **Deal Won** | Сделка выиграна | Все оплаты подтверждены, проект готов к передаче |

### Детальное описание ключевых переходов

#### Transition: Discuss → Meeting
- Продажник обязан заполнить поле «Потребность клиента» (текстовое описание)
- Выбран хотя бы один тип продукта (Website, App, CRM и т.д.)
- Указана примерная сумма сделки

#### Transition: Can We Do It → Send Offer
- Техническая команда подтвердила реализуемость (через задачу или комментарий)
- Выбран тип проекта: White Label / Mix / Custom Code
- Указана модель оплаты: Classic (50/50 или 30/30/40) или Subscription
- Рассчитаны предварительные сроки
- Определён дедлайн проекта

#### Transition: Send Offer → Get Answer
- Файл коммерческого предложения прикреплён к сделке
- Дата отправки зафиксирована
- Установлен дедлайн ответа от клиента

#### Transition: Deposit & Contract → Creating
- Счёт на предоплату создан в системе
- Договор загружен и подписан
- Реквизиты компании или физлица привязаны (сущность Company)
- Статус налогообложения указан (Tax / Tax-Free)

## Extension Deals (Сделки на доработку)

Extension Deal — это сделка на дополнительную функциональность для **существующего** проекта. Проходит тот же пайплайн, что и новая сделка, но визуально отличается:

- **Тег «Extension»** — автоматически присваивается при создании из проекта
- **Цветовая маркировка** — отличный цвет карточки на доске (например, оранжевый vs синий для новых сделок)
- Привязка к существующему проекту обязательна
- При Deal Won → создаётся Extension в рамках существующего проекта (не новый проект)

### Правила создания Extension Deal

1. PM определяет, что запрос клиента — это платная доработка (не Maintenance)
2. PM информирует продажника (Seller)
3. Seller создаёт Extension Deal в CRM с тегом Extension
4. Привязывает к существующему проекту и контакту
5. Далее сделка проходит стандартный пайплайн

## Момент перехода: Deal Won → Order → Project

### Условия перехода Deal → Order

```
Deal Won = TRUE
  AND Первый платёж получен (Invoice.status = Paid)
  AND Договор подписан
  → Сделка становится Заказом (Order)
```

### Что происходит при создании Order

1. **Order создаётся** со всеми данными из Deal (сумма, тип оплаты, сроки, реквизиты)
2. **Project создаётся** в Projects Hub (или обновляется, если Extension)
3. **Product создаётся** внутри Project (Website, Mobile App, CRM и т.д.)
4. Order переходит под контроль **Финансового директора**
5. Продажник получает уведомление о необходимости kickoff

## Handoff: от продажника к проектному менеджеру

### Обязательный Kickoff Checklist

При передаче проекта от Seller к PM заполняется чек-лист:

| # | Пункт | Ответственный |
|---|-------|---------------|
| 1 | Бриф проекта заполнен и передан | Seller |
| 2 | Контактное лицо клиента указано | Seller |
| 3 | Группа в WhatsApp создана (или обновлена) | Seller |
| 4 | Тип проекта и модель оплаты зафиксированы | Seller |
| 5 | Дедлайн проекта установлен | Seller |
| 6 | Дизайн-референсы / ТЗ переданы | Seller |
| 7 | PM назначен на проект | CEO / Seller |
| 8 | Kickoff-встреча проведена с PM | Seller + PM |
| 9 | PM подтвердил приём проекта | PM |

Пока все пункты не закрыты — проект не может перейти на стадию Development.

## После сдачи проекта: Maintenance как подписка

### Процесс предложения Maintenance

1. **Проект сдан** — клиент подтвердил приёмку
2. **Seller связывается** с клиентом и предлагает Maintenance-подписку
3. На стадии **Maintenance Offer** в Deal Pipeline фиксируется:
   - Сумма ежемесячной подписки
   - Что входит в Maintenance (техподдержка, мелкие правки, мониторинг)
   - Дата начала подписки
4. При согласии клиента → создаётся **Subscription** в системе, привязанная к проекту
5. Проект получает статус **Maintenance** в Projects Hub

### Типичный сценарий для Subscription-модели оплаты

Для проектов с подписочной оплатой переход в Maintenance происходит автоматически:
- Во время разработки (≈3 месяца) клиент платит за Development
- После сдачи (с 4-го месяца) к сумме Development добавляется Maintenance
- Пример: 100 000 (dev) → 100 000 (dev) + 80 000 (maintenance) = 180 000/мес

## Сводная схема процесса

```
[Источник лида]
      ↓
   LEAD PIPELINE
   New → Didn't Get Through → Contact Established → Qualification (MQL)
                                                          ↓
                                            ┌─────────────┼─────────────┐
                                          SPAM         Frozen        SQL ──→ DEAL PIPELINE
                                                                              ↓
                                                     Start a Conversation
                                                              ↓
                                                    Discuss What Is Needed
                                                              ↓
                                                          Meeting
                                                              ↓
                                                      Can We Do It?
                                                              ↓
                                                        Send Offer
                                                              ↓
                                                        Get Answer
                                                              ↓
                                                   Deposit & Contract
                                                              ↓
                                                          Creating
                                                              ↓
                                                      Get Final Pay
                                                              ↓
                                                    Maintenance Offer
                                                         ↓        ↓
                                                      Failed    Deal Won
                                                                   ↓
                                                        ORDER + PROJECT
                                                                   ↓
                                                         Handoff → PM
                                                                   ↓
                                                         DELIVERY PROCESS
```

## Метрики и аналитика по процессу

| Метрика | Описание | Где отслеживается |
|---------|----------|-------------------|
| Lead Conversion Rate | % лидов, ставших SQL | Dashboard маркетинга |
| SQL → Deal Won Rate | % квалифицированных лидов → выигранные сделки | Dashboard продаж |
| Average Deal Cycle | Среднее время от SQL до Deal Won | Dashboard продаж |
| Revenue by Source | Выручка по источнику лида | Финансовая аналитика |
| Win/Loss Ratio | Соотношение выигранных/проигранных сделок | Dashboard продаж |
| Time per Stage | Среднее время на каждой стадии | CRM аналитика |

## Правила и ограничения

1. **Лид не может стать сделкой напрямую** — только через статус SQL
2. **Сделка не может быть закрыта как Won** без подтверждённого первого платежа
3. **Extension Deal** обязательно привязана к существующему проекту
4. **Каждый переход стадии** проверяет заполнение обязательных полей (Stage Gate)
5. **Причина Failed** — обязательное поле при закрытии сделки как неуспешной
6. **Один контакт может иметь несколько сделок** — для разных проектов/брендов
7. **Одна компания может иметь несколько проектов** — один бренд = один проект
