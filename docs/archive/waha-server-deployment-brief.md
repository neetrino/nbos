# WAHA — логика в NBOS и бриф для сервера

Цель документа: объяснить **зачем** WAHA в платформе, **как** он вписан в модули, и **что именно** должен подготовить человек на **своём сервере**, чтобы дальше команда NBOS могла подключить приложение. Канон продуктовой модели: `docs/NBOS/06-Integrations/01-WhatsApp-Integration.md`.

---

## 1. Логика в платформе: что это и для чего

**WAHA** — это не «модуль NBOS», а **внешний сервис** (self-hosted), который даёт HTTP API к обычному WhatsApp через схему как у **WhatsApp Web** (подключение по **QR**). NBOS не хранит протокол WhatsApp внутри себя: он говорит с WAHA по REST, а WAHA держит сессию с телефоном.

Цепочка в продукте:

```text
Пользователь / автоматика в NBOS
  → Notification Engine или Messenger (исходящее во внешний канал)
    → WhatsAppWebAdapter (в приложении)
      → WAHA (на вашем сервере)
        → QR-подключённый WhatsApp-аккаунт
          → группы и редко личные чаты
```

**Зачем бизнесу:**

- Главный сценарий — **проектные WhatsApp-группы** (клиент + команда), напоминания по **счетам**, сообщения в **бухгалтерскую** группу, обновления **support**.
- **Уведомления (Notifications):** системные события (invoice, просрочка, сервисные напоминания и т.д.) могут уходить в WhatsApp **через адаптер → WAHA**, чаще в **группу**, а не массовая рассылка 1:1.
- **Мессенджер (Messenger):** переписка с клиентом во **внешней зоне** (External): история, вложения через Drive, граница с CRM/Support/Finance. Внутренний чат сотрудников в WhatsApp **не** смешивается с этим путём.

**Имена в интерфейсе и настройках NBOS:**

| Имя                       | Смысл                                                                                                                                 |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **Messenger**             | Модуль чатов; внешний WhatsApp — часть **внешних** разговоров.                                                                        |
| **WhatsApp** (интеграция) | Провайдер внешнего канала; в реестре настроек — ключ **`WHATSAPP__WAHA_INSTANCE`** = **базовый URL** вашего WAHA (в проде **HTTPS**). |
| **WhatsAppWebAdapter**    | Кодовое имя слоя в NBOS между бизнес-логикой и WAHA (без прямых вызовов WAHA из модулей CRM/Finance и т.д.).                          |

Официальный **Meta WhatsApp Cloud API** в каноне **не** целевой путь MVP; выбран путь Web/QR через WAHA (или позже другой совместимый адаптер).

## 2. Что нужно «создать или установить» в целом (картина целиком)

Чтобы **после подключения** всё работало по задумке, нужны **два контура**:

1. **Сервер WAHA** (зона специалиста по инфраструктуре) — Docker, URL, TLS, секреты, стабильная сессия, доступность API.
2. **Приложение NBOS** (зона команды продукта/backend) — реализация/докрутка `WhatsAppWebAdapter`, очереди, webhook от WAHA к API, переменные окружения, привязка групп/контекстов в данных.

Пока второй контур доводится по roadmap (Phase 7), **первый контур уже можно подготовить**: рабочий инстанс WAHA с QR и тестовой группой — это входные данные для команды приложения.

## 3. Разделение зон ответственности

| Зона                    | Кто                                          | Что делает                                                                                                                                     |
| ----------------------- | -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| **A. Сервер WAHA**      | Специалист на своём сервере (DevOps / админ) | Установка, сеть, TLS, volumes, безопасность API, мониторинг, резервная копия конфигурации, QR-сессия, тест «отправил/получил» на стороне WAHA. |
| **B. Учётка WhatsApp**  | Бизнес / владелец процесса                   | Отдельный **номер** под сервис (не личный), участие в **тестовой группе**, кто сканирует QR и следит за баном/лимитами WhatsApp.               |
| **C. NBOS приложение**  | Разработчики NBOS                            | Адаптер, URL + секреты в конфиге, webhook endpoint, связь с Messenger/Notifications, хранение id групп/разговоров, Drive для вложений.         |
| **D. Сеть между A и C** | Совместно                                    | Разрешить исходящие вызовы из NBOS API к WAHA; разрешить **входящие** webhook от WAHA к публичному URL API NBOS (когда path будет согласован). |

Специалист на сервере **не** обязан писать код NBOS; он обязан отдать **стабильный защищённый endpoint** и параметры доступа. Команда NBOS **не** обязана администрировать ваш VPS, но должна получить **точный base URL** и правила авторизации.

## 4. Только поле работы специалиста на своём сервере (чеклист)

Ниже — то, что нужно **сделать на стороне сервера**, чтобы «ваша часть» была закрыта до подключения NBOS.

1. Установить среду (**Docker** — типичный вариант) по документации **[WAHA (devlikeapro)](https://github.com/devlikeapro/waha)**.
2. Запустить контейнер(ы) с **persistent volume** для сессий (иначе частый повторный QR).
3. Настроить **reverse proxy + TLS**: публичный **HTTPS** URL (без самоподписанного сертификата для прод-связки с NBOS, если приложение не доверяет CA).
4. Включить и сохранить в secret store **API key / авторизацию** WAHA (если используется в вашей сборке).
5. Ограничить доступ к API (firewall, VPN, allowlist IP прод-серверов NBOS — по политике безопасности).
6. Выполнить **QR-login** служебного номера; проверить в Swagger/UI WAHA: **health**, отправку сообщения в **тестовую группу**, приём входящих, **рестарт** контейнера без потери сессии (если volume настроен верно).
7. **Передать команде NBOS письменно:**
   - базовый **URL** инстанса (то, что ляжет в `WHATSAPP__WAHA_INSTANCE`),
   - как передавать **авторизацию** (заголовок, query — как в вашей конфигурации WAHA),
   - контакт на случай падения WAHA или истечения сессии.

Пункт **webhook** (URL на NBOS API): специалист **настраивает в WAHA** тот адрес, который **даст** команда NBOS после реализации endpoint; до согласования path можно оставить заглушку или отложить, но **исходящий** канал (NBOS → WAHA) уже проверяется по URL и ключу.

---

## 5. Что сделает команда NBOS после получения вашего инстанса

- Прописать в конфигурации приложения **base URL** и секреты (в т.ч. через системные списки / env — см. UI интеграций `WHATSAPP__WAHA_INSTANCE`).
- Реализовать или дорисовать **WhatsAppWebAdapter**: send, health, обработка webhook, статусы сессии (`connected` / `qr_required` / … по канону).
- Связать доставку с **Notifications** и копией исходящего в **Messenger**, вложения — с **Drive** (по канону).

Детали roadmap: `docs/PHASE_7_INTEGRATIONS_MIGRATION_WORK_PLAN.md` (блок WhatsApp / WAHA).
