Шаг 1 — Диагностика (как я понял ваш бизнес)
1) Краткий пересказ модели

Вы digital/IT-компания (агентство/продуктовая студия), где главная единица управления это Project.

Текущий жизненный цикл выглядит так:

Лид → Deal (CRM)

Инвойс → Оплата → Order/Contract

Проект стартует: домен/сервер/аккаунты/доступы/пароли/Drive/чат

Delivery: Scrum/boards/задачи/разработка

Финансы проекта: доходы, расходы, бонусы (dev/sales), домены/серверы, подписки

Переход в maintenance/subscription: ежемесячные инвойсы, доп. работы

Клиент просит доп. функционал → нужен новый заказ, но он должен быть привязан к тому же проекту, чтобы вся история была в одном месте

Проблема Bitrix: все части процесса есть, но размазаны по разным бордам/сущностям, нет “единого окна проекта”, где любой сотрудник видит все, что ему нужно по проекту, согласно своей роли.

Модель монетизации

Проектные работы (фикс/этапы/милстоуны): аванс + финальная часть.

Абонентка / maintenance / subscription: регулярные счета.

Доп. заказы (extensions/changes): upsell в рамках существующего проекта.

Вероятно есть партнерские продажи/рефералы (вы упомянули Partner).

Основные источники дохода

Разработка / внедрение (основные проекты)

Поддержка / сопровождение (повторяющийся доход)

Доп. услуги и расширения (change requests)

(Возможное) партнерство/перепродажа/интеграции (если есть)

Ключевые процессы (как есть)

Commercial → Delivery → Finance → Support цепочка:

Lead/Deal management (CRM)

Offer/Proposal/Scope фиксация

Invoice/Payment tracking

Project initiation (доступы, домены, файлы, коммуникация)

Task/Scrum execution

Cost tracking (сервер/домен/сервисы/бонусы)

Change request / additional order creation

Subscription invoicing & renewal

Knowledge + passwords + assets management

Коммуникация по проекту (мессенджер)

Роли и зоны ответственности (по вашему описанию)

Минимально читаются такие роли (точное распределение уточним):

Sales / Seller: лиды, сделки, предложения, заказы, допродажи, “видит финансы проекта на своем уровне”

PM / Developer (у вас Dev Team = PM+Dev в одном лице): ведение проекта, задачи, статусы, доступы, документы, коммуникации

Finance / Accountant: контроль поступлений/долгов, инвойсы, расходы, подписки

(Возможное) Ops/Admin: домены, сервера, учетные записи, доступы, структура Drive

Потенциальные риски и узкие горлышки (сразу по делу)
1) Главный риск: отсутствует “Project as a System of Record”

Сейчас проект не является “истиной” по всем связанным данным. Это приводит к:

потере контекста,

ручному поиску информации,

ошибкам в доступах/паролях/обязательствах,

разрыву между продажами, производством и финансами.

2) Операционные риски

Неполная связность заказов: доп.работы живут отдельно, историю проекта теряете.

Финансовая дисциплина: расходы/бонусы/подписки могут учитываться непоследовательно, если нет жесткой модели.

Риск безопасности: пароли и доступы “везде и нигде”. Это почти гарантированная утечка в будущем, если масштабироваться.

Срыв SLA по support/maintenance: если задачи и коммуникации распылены, клиентские запросы тонут.

Нечеткие права доступа: seller видит лишнее или не видит нужное, PM не видит оплаты, finance не видит статусы.

3) Узкие горлышки роста

Вы (или 1–2 ключевых человека) становитесь “живым маршрутизатором информации”.

Нет единого стандарта проекта (шаблонов, чеклистов, Definition of Done по стадиям).

Невозможно нормально измерять KPI (маржа на проект, загрузка, скорость поставки, churn подписок).

4) Стратегическая ошибка (если не исправить сейчас)

Если вы построите “свой Bitrix”, но без строгой объектной модели Project + Order + Subscription + Cost + Access, вы просто перенесете хаос в собственный интерфейс. Дорого, больно, и потом переделывать в 3 раза хуже.
