# Project Hub: продукто-центричная навигация (утверждено)

**Статус:** целевая модель UI/IA для переработки Projects Hub.  
**Связь с Q1:** сущность **Product** обязательна; основной рабочий контекст — **страница продукта**, не «всё сразу в проекте».

---

## 1. Иерархия экранов

| Уровень                              | Путь (логический)                          | Что видит пользователь                                                                                                                                                                                                                         |
| ------------------------------------ | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Список проектов**                  | `/projects`                                | Как сейчас: список / карточки, фильтры Development / Maintenance / Closed и т.д.                                                                                                                                                               |
| **Оболочка проекта**                 | `/projects/:projectId`                     | **Не** полный набор вкладок «как раньше на всём проекте». Это **оболочка**: название, клиент, ключевые поля проекта + **список всех Product** в этом проекте (Neetrino Web, Neetrino App, Netfor CRM …). Отсюда пользователь выбирает продукт. |
| **Продукт (основной рабочий экран)** | `/projects/:projectId/products/:productId` | Весь функционал, который ранее ожидался «на проекте целиком» для работы по поставке: вкладки и данные **в разрезе этого Product**.                                                                                                             |

---

## 2. Вкладки внутри Product (целевой набор)

На странице **Product** размещаются вкладки, привязанные к **этому** продукту, например:

- Overview
- Tasks & Support
- Credentials
- Finance
- (остальные по модулю — по тем же правилам привязки к Product)

**Правило:** задачи, тикеты, финансы по продукту, пароли, относящиеся к этой линии — в контексте **Product**, если иное явно не задано (например, общие пароли компании — по правилам Vault).

---

## 3. Переключение между продуктами без выхода со страницы

В **шапке страницы Product**:

1. **Видны все Product этого проекта** (например три: Neetrino Web, Neetrino App, Netfor CRM).
2. Пользователь может **переключиться на другой продукт** того же проекта — открывается та же структура вкладок, но данные **другого** Product (задачи, финансы и т.д. этого продукта).
3. Кнопка **«Посмотреть всё»** (или аналогичное название) — переход к **агрегированному** представлению по проекту (см. § 5).

---

## 4. Зачем отдельный Product

Каждый Product — **отдельный элемент** с собственным Order, Creating, задачами и т.д. Это согласовано с моделью данных и сценариями Deal / Extension / Maintenance.

---

## 5. Агрегированный просмотр «по всем продуктам проекта» (редкий сценарий)

**Потребность:** те же типы вкладок (задачи, финансы …), но **сразу по всем Product** проекта (все задачи в одном месте, финансы сводно по вкладкам — **не** по одному продукту).

**Принцип:** сценарий **редкий**; реализовать **только если не ломает и не перегружает** первую версию.

**Варианты реализации (от простого к сложному):**

| Вариант                                     | Описание                                                                                                                            | Когда                                       |
| ------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------- |
| **A. Отложить**                             | Только продукто-центричные экраны; агрегат вручную через несколько вкладок                                                          | MVP                                         |
| **B. Отдельный режим «Все продукты»**       | Маршрут вида `/projects/:id/overview-all` или переключатель на странице проекта: списки задач/сводных таблиц с колонкой **Product** | Фаза 2, если нужно без дублирования логики  |
| **C. Те же вкладки, фильтр «все продукты»** | На существующих списках (задачи, счета) фильтр по проекту + «все продукты»                                                          | Если API уже отдаёт агрегаты по `projectId` |

Рекомендация: **MVP = A**; заложить в UI **кнопку «Посмотреть всё»** как вход в **B** или **C**, когда будет готова бэкенд-агрегация и дизайн сводных таблиц.
