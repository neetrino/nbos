## Решение — п.3 Category / Type

**Accepted:** C-hybrid.

DB сохраняет оба поля:

- `category` = grouping, Category Board columns, quick filters, Delivery slots, Finance/product handoff grouping.
- `credentialType` = secret format, dynamic fields, ENV editor, reveal/copy/export behavior and validation.

**UI rule:** не показывать два параллельных dropdown `Category` + `Credential Type` как sibling controls. Scope выбран через tab; category задается/preset-ится context/board/slot/compact control; credentialType выбирается как `What is stored?`.

**Enum rule:** не добавлять `ENV`, `SSH`, `RECOVERY_CODES` в category; это credentialType formats.

**Затронутые места для реализации:**

- Prisma enums `CredentialCategoryEnum`, `CredentialTypeEnum`
- `product-access-slots.ts`, Delivery Board bind/create
- `product-done-readiness.ts` handoff categories
- List filters, seed, migration mapping

**Можно начинать** form/sheet по rules из `06-Credentials-UX-Decisions.md`.

## Решение — access model / уровни доступа

**Статус:** direction accepted; implementation requires Platform Access Foundation above.

**Accepted concepts:**

- Platform access foundation: role levels + personal employee overrides + manual per-credential override.
- `ProjectTeamMember` and `ProductTeamMember` are separate concepts.
- Product team does not automatically get project-level access to all products.
- Project team/admin level is higher and can manage project/participants/settings by policy.
- Manual override in credential Sheet lists only manually added employees and supports `View` / `Edit`.
- Delete remains high-level RBAC permission, not a manual credential level.
